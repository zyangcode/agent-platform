本文是 `单向多Agent优化` 目录下 01-10 文档的最终收口版，用作后续实现前的主入口。

本文只做设计总结，不包含代码实现。

## 2. 最终结论

最终方案一句话：

```text
保留 DefaultTeamRuntimeService 作为 TeamRuntimeService 默认入口，内部用 LangGraph4j 替换手写编排。
```

更完整地说：

```text
LangGraph4j 只作为 Team Runtime 的编排内核。
Planner / Executor / Reviewer 继续复用现有实现。
MCP / Skill / Context / Memory / RAG / Trace / Token / SSE 继续走项目已有链路。
第一版不做 checkpoint、human-in-the-loop、任务并发、子图和框架级替换。
```

## 3. 方案演进说明

早期 `01` 文档曾倾向：

```text
新增 LangGraphTeamRuntimeService
下线 DefaultTeamRuntimeService 作为 Bean
```

后续阅读本项目源码后，`06` 文档修正为更稳方案：

```text
保留 DefaultTeamRuntimeService 名字
保留 TeamRuntimeService 接口
保留 DefaultAgentRuntimeService -> TeamRuntimeService 调用关系
DefaultTeamRuntimeService 内部改为调用 LangGraph4j graph
```

最终以 `06` 之后的判断为准。

原因：

```text
DefaultTeamRuntimeService 现在已经是默认 Team 入口。
外部 Bean、接口、Gateway、Web SSE 不需要变化。
当前目标是升级 Team 内核，不是新增一条运行模式。
名字看不出 LangGraph4j 的问题，可以通过 core.team.graph 包结构解决。
```

最终不建议新增并行入口：

```text
LangGraphTeamRuntimeService
```

除非后续确实需要 A/B 双运行时或灰度切换。

## 4. 改造目标

### 4.1 要达成

```text
把 DefaultTeamRuntimeService.run() 里的手写流程拆成 LangGraph4j StateGraph。
保留 Team 模式对外行为。
让 retry / replan / final 变成显式条件边。
让 Planner / Executor / Reviewer 的职责更清楚。
为后续可视化、并发、checkpoint 留扩展点。
```

### 4.2 不达成

第一版不做：

```text
不重写单 Agent Runtime。
不替换 ModelInvokeService。
不替换 AgentToolDispatcher。
不替换 McpToolExecutor / MCP Client。
不让 LangChain4j AgentExecutor 接管工具。
不引入 Spring AI Alibaba Agent Framework。
不新增 Team DB 表。
不启用 LangGraph4j checkpoint saver。
不做 graph 级 human approval。
不做任务并发。
不做拖拽编排。
```

## 5. 总体架构

最终架构：

```text
DefaultAgentRuntimeService
  -> TeamRuntimeService
     -> DefaultTeamRuntimeService
        -> TeamGraphFactory / CompiledGraph
           -> BuildContextNode
           -> PlanNode
           -> ValidatePlanNode
           -> ScheduleNode
           -> ExecuteBatchNode
           -> ReviewNode
           -> RouteAfterReview
           -> FinalAnswerNode
```

现有能力复用：

```text
BuildContextNode
  -> AgentContextBuilder
  -> AgentToolResolver

PlanNode
  -> TeamPlanner
  -> DefaultTeamPlanner

ExecuteBatchNode
  -> TeamExecutor
  -> DefaultTeamExecutor
  -> AgentToolDispatcher
  -> Skill / MCP

ReviewNode
  -> TeamReviewer
  -> DefaultTeamReviewer

FinalAnswerNode
  -> TeamFinalAnswerBuilder
  -> ModelInvokeService fallback
```

横切能力：

```text
TeamGraphSupport 或等价 helper
  -> TeamEventSink
  -> TraceService
  -> TokenUsageService
  -> ObjectMapper
  -> TeamRunLimiter
```

## 6. Graph 形态

第一版 graph：

```text
START
  -> build_context
  -> plan
  -> validate_plan
  -> schedule
  -> execute_batch
  -> review
  -> route_after_review

route_after_review:
  retry  -> execute_batch -> review -> route_after_review
  replan -> plan -> validate_plan -> schedule -> execute_batch -> review -> route_after_review
  final  -> final_answer -> END
```

关键判断：

```text
route_after_review 是确定性代码节点，不是 LLM Supervisor。
Planner 只生成计划。
Executor 只执行任务。
Reviewer 只审查，不执行、不规划、不调用工具。
FinalAnswerNode 统一生成最终答案。
```

## 7. Node 职责

### 7.1 build_context

职责：

```text
校验 conversationId
调用 AgentContextBuilder.build
调用 AgentToolResolver.resolve
写 context.build Trace span
```

不发 Team SSE。

### 7.2 plan

职责：

```text
调用 TeamPlanner.plan
记录 planner model usage
emit team_plan
```

首次规划：

```text
PlanTeamCommand(userInput, context, availableTools)
```

replan：

```text
PlanTeamCommand(userInput, context, availableTools, previousPlan, executionResults, review)
```

### 7.3 validate_plan

职责：

```text
limiter.checkTaskCount
TaskPlanValidator.validate
```

校验：

```text
goal 非空
tasks 非空
taskId 唯一
dependsOn 无环
taskType 合法
TOOL_TASK suggestedTool 可用
MODEL_TASK 不带 suggestedTool
```

### 7.4 schedule

职责：

```text
TaskDependencySorter.sort
首次执行：输出全部排序任务
replan：只输出新增且未完成 task id
retry：不走 schedule，由 route_after_review 指定 scheduledTasks
```

第一版串行执行，不做并发分层。

### 7.5 execute_batch

职责：

```text
按 scheduledTasks 串行执行
每个任务前 limiter.checkTimeout
emit team_task_start
TOOL_TASK 前 emit team_tool_call
调用 TeamExecutor.execute
记录 model usage
统计 tool call
emit team_tool_result
emit team_task_result
```

不直接调用：

```text
AgentToolDispatcher
McpToolExecutor
McpClient
```

这些仍封装在 `DefaultTeamExecutor` 后面。

### 7.6 review

职责：

```text
TeamAnswerDraftBuilder.build
TeamReviewer.review
记录 reviewer model usage
emit team_review
```

输出：

```text
answerDraft
review
reviewResults
```

### 7.7 route_after_review

职责：

```text
if review failed and retryTasks not empty:
  route = RETRY
  scheduledTasks = [first retry task]
  emit team_retry

else if review failed and replanRequired:
  route = REPLAN
  previousPlan = plan
  emit team_retry

else:
  route = FINAL
```

### 7.8 final_answer

职责：

```text
TeamFinalAnswerBuilder.build
必要时 fallbackModelAnswer
聚合 totalUsage
emit team_final
返回 AgentRunResult 所需字段
```

## 8. TeamGraphState 设计

`TeamGraphState` 是 graph 的业务状态契约，建议继承：

```text
org.bsc.langgraph4j.state.AgentState
```

第一版不启用 checkpoint，因此可以保存 DTO 引用。

字段分类：

### 8.1 核心输入

```text
command: AgentRunCommand
conversationId: Long
runSpanId: Long
step: Integer
```

### 8.2 上下文与工具

```text
context: AgentContextDTO
availableTools: List<AgentToolDTO>
```

### 8.3 计划与调度

```text
plan: TaskPlanDTO
previousPlan: TaskPlanDTO
scheduledTasks: List<TeamTaskDTO>
```

### 8.4 执行结果

```text
taskExecutionResults: List<TeamTaskExecutionResultDTO>
executionResults: List<ExecutionResultDTO>
```

### 8.5 审查与最终回答

```text
answerDraft: String
review: ReviewResultDTO
finalAnswer: String
usage: ModelUsageDTO
```

### 8.6 历史调用

```text
planResults: List<TeamPlanResultDTO>
reviewResults: List<TeamReviewResultDTO>
fallbackModelInvocations: List<ModelInvokeResult>
```

### 8.7 路由

```text
route: TeamGraphRoute
retryTaskId: String
```

禁止放入 state：

```text
Spring Bean
Mapper
Entity
ThreadPool
MCP Client
TeamRunLimiter
TeamEventSink
TraceService
TokenUsageService
ObjectMapper
```

这些放入：

```text
TeamGraphRuntimeContext
```

## 9. Memory / RAG / Context 接入

结论：

```text
多 Agent 可以读记忆，但不能让每个 Agent 直接写长期记忆。
长期写入统一收口到 Reflect/Finalize 阶段。
```

第一版 LangGraph4j 改造不强制引入 ReflectNode。

但设计边界要保留：

```text
Planner 读摘要级 memory/rag，用于规划。
Executor 按 task query 读任务相关 memory/rag，用于执行。
Reviewer 读计划、结果、引用来源，用于审查。
Reflector 后续统一生成 MemoryWriteCandidate。
MemoryWriteService 做敏感扫描、准入、去重、冲突处理。
```

禁止：

```text
Planner 直接写 memory。
Executor 直接写 memory。
Reviewer 直接写 memory。
工具结果原文直接写 memory。
RAG chunk 原文塞入 graph state。
```

第一版建议：

```text
先复用当前 AgentContextBuilder 构建的 context。
不新增 Team 专用 ContextSchemaKind。
后续再扩展 TEAM_PLANNER / TEAM_EXECUTOR / TEAM_REVIEWER / TEAM_REFLECTION。
```

## 10. MCP / Skill 接入

结论：

```text
LangGraph4j 不直接接 MCP。
```

链路保持：

```text
ExecuteBatchNode
  -> TeamExecutor
  -> DefaultTeamExecutor
  -> AgentToolDispatcher
  -> Skill / MCP Tool Executor
  -> MCP Client
```

原因：

```text
MCP 涉及 server、tenant、tool id、Profile 授权、同名工具、refresh、Trace、敏感扫描。
这些是本项目平台能力，不能被 LangGraph4j 或 LangChain4j AgentExecutor 绕过。
```

## 11. SSE 事件契约

现有事件类型必须保持：

```text

team_plan
team_task_start
team_tool_call
team_tool_result
team_task_result
team_review
team_retry
team_final
```

正常路径：

```text
team_start
team_plan
team_task_start
team_task_result
team_review
team_final
```

TOOL_TASK：

```text
team_start
team_plan
team_task_start
team_tool_call
team_tool_result
team_task_result
team_review
team_final
```

retry：

```text
team_start
team_plan
team_task_start
team_task_result
team_review
team_retry
team_task_start
team_task_result
team_review
team_final
```

replan：

```text
team_start
team_plan
team_task_start
team_task_result
team_review
team_retry
team_plan
team_task_start
team_tool_call
team_tool_result
team_task_result
team_review
team_final
```

step 规则：

```text
step 从 1 开始。
每 emit 一个 TeamRuntimeEventDTO，step + 1。
不发 SSE 的 node 不增加 step。
```

## 12. Trace / Token 契约

保留当前 span 语义：

```text
team.run
context.build
team.plan
team.task.execute
team.review
team.fallback
```

第一版不依赖 LangGraph4j hook 承载业务 Trace。

建议：

```text
每个 node 或 TeamGraphSupport 显式 startSpan / finishSpan。
每个 ModelInvokeResult 显式调用 TokenUsageService.record。
Trace 失败不能影响 Team run。
Token usage 写入失败不能影响 Team run。
```

后续再考虑：

```text
TeamGraphTraceHook
TeamGraphEventRecorder
TeamGraphUsageRecorder
```

## 13. 限流与降级

继续使用：

```text
TeamRunLimiter
```

必须保留：

```text
maxTasks
maxRetries
maxToolCalls
maxModelCalls
timeoutMs
```

第一版不新增：

```text
maxParallelTasks
maxReplans
maxReActSteps
```

除非实现同步扩展测试。

降级规则保持：

```text
Planner 失败二次后降级单 MODEL_TASK。
Reviewer 要求 retry 时最多消耗一次 retry 预算。
Reviewer 要求 replan 时最多消耗一次 retry 预算。
replan 没有新任务时 fallbackModelAnswer。
finalAnswer 为空或像 prompt echo 时 fallbackModelAnswer。
```

```text

不做 ReActTaskGraph 子图。

不做 FinalizeMemoryWritesNode。
```

## 15. 第一版实施顺序

后续你明确下令实现时，按这个顺序：

```text
1. 引入 LangGraph4j 依赖并验证 API。
2. 新增 TeamGraphState / TeamGraphRuntimeContext / TeamGraphFactory / smoke test。
3. 迁移 build_context / plan / validate_plan / schedule。
4. 迁移 execute_batch。
5. 迁移 review / route_after_review。
6. 迁移 final_answer / fallback / usage 聚合。
7. DefaultTeamRuntimeService.run() 切换为 graph 主路径。
8. 跑 DefaultTeamRuntimeServiceTest。
9. 跑 Planner/Executor/Reviewer 测试。
10. 跑 Gateway/Web Team SSE 测试。
11. 最后再考虑全量 mvn test。
```

每一步都应保持可回归，不要一次性大重写。

## 18. 最终判断

最终采用：

```text
方案 A：保留 DefaultTeamRuntimeService 名字，内部换成 LangGraph4j。
```

最终拒绝：

```text
新增 LangGraphTeamRuntimeService 作为并行主入口。
引入 LangChain4j / Spring AI Alibaba Agent Framework 接管平台能力。
让 graph 直接调 MCP。
第一版同时做并发、checkpoint、human-in-the-loop。
```

一句话总结：

```text
保留入口，替换内核；保留角色服务，替换编排方式；保留平台能力，避免框架反客为主。
```
