# 记忆系统优化落地验收报告

> 当前分支：`feature/memory-rag-optimization`
> 最新提交：`b346108 feat(core): optimize memory and rag retrieval`
> 验收时间：2026-06-08

## 1. 验收结论

记忆系统优化主链路已经落地并推送到远端分支。当前实现覆盖 Memory v2、RAG MVP、Qdrant/Mock 向量检索、PostgreSQL tsvector 倒排召回、Memory/RAG 并行召回、Embedding 复用、Context Slot 注入、记忆管理接口和关键工程护栏。

当前代码可以作为后续 LangGraph4j 多 Agent 协作的底层记忆能力复用。多 Agent 中建议只让 Reflector/Orchestrator 走统一 MemoryWriteService 写入，Planner/Executor/Reviewer 只读经过裁剪和脱敏的记忆快照。

## 2. 已落地能力

### 2.1 Memory v2

- 支持长期记忆写入、召回、管理、禁用、更新和过期过滤。
- 支持 `summary`、`preference`、`fact`、`episodic` 等记忆分类。
- 支持 pinned 记忆保护，避免被自动衰减、归档或冲突覆盖。
- 支持显式记忆控制：
  - “记住 X”走准入后写入。
  - “忘记/删除 X”只在提取到明确关键词时禁用相关记忆。
  - 模糊“忘记”“删除记忆”不会批量误删最近记忆。
- 支持 READ_ONLY、READ_WRITE、SESSION_ONLY 等策略边界。
- 支持召回后的访问统计 best-effort 写入，统计更新失败不会导致召回结果丢失。

### 2.2 RAG MVP

- 支持知识文档入库、chunk 切分、搜索、删除和重新 ingest。
- PostgreSQL 作为事实源，Qdrant/MockVectorStore 作为派生向量索引。
- RAG citation 只来自知识库 chunk，不把 Memory 当外部引用来源。
- 重新 ingest 已存在文档时，会禁用旧 chunk 并删除旧向量，避免旧向量占满 topK。
- 向量检索失败时可降级到 PostgreSQL 关键词/tsvector 召回。

### 2.3 混合召回与性能优化

- Memory/RAG 共用同一次 query embedding，避免同一用户输入重复向量化。
- Memory/RAG 召回进入独立 retrieval executor，并受 Context 构建超时控制。
- embedding 预计算也被纳入 retrieval timeout，慢 embedding 服务不会绕过超时阻塞主链路。
- Memory 和 RAG 召回可以并行执行，任一路失败或超时都降级为空结果或关键词召回。
- Qdrant 搜索 scope 已和 PostgreSQL 召回规则对齐：当前应用/Profile 可以召回全局应用/Profile 记忆或资料。
- Qdrant 删除仍保持精确 scope，避免误删全局向量。

### 2.4 PostgreSQL tsvector

- 新增迁移：`V017__add_memory_rag_tsvector_search.sql`。
- `memories` 和 `knowledge_chunks` 增加 `search_vector` 和 GIN 索引。
- Memory 关键词召回优先走 tsvector，异常或空结果时保留原有 fallback。
- RAG 关键词召回走 tsvector，替代纯 LIKE 全表扫描方向。
- 问题台账已记录“PG tsvector 先行，Elasticsearch 后续可选”的决策。

### 2.5 Context 与 Trace

- Memory/RAG/Experience/Tools 等上下文通过 Slot 方式进入 AgentContext。
- Context 构建有 token 预算和召回降级边界。
- Memory recall/write、RAG ingest/search、Context slot 组合等关键过程已有测试覆盖。

## 3. 审查问题修复

本轮修复了 5 个高优先级问题：

| 问题 | 处理结果 |
|---|---|
| 模糊遗忘指令可能误删最近 20 条记忆 | 已修复。无明确关键词时不执行禁用查询 |
| Qdrant 向量 scope 和 PG scope 不一致 | 已修复。搜索支持全局 application/profile 可见性 |
| Context 超时不覆盖 embedding 预计算 | 已修复。embedding 进入 retrieval executor 并受 timeout 控制 |
| RAG 重新 ingest 留下旧 chunk 向量 | 已修复。禁用旧 chunk 后删除旧文档向量 |
| 召回读路径同步 touch 写失败导致召回丢失 | 已修复。访问统计写入改为 best-effort |

## 6. 后续接 LangGraph4j 的使用边界

记忆系统可以直接支撑 LangGraph4j 多 Agent，但需要遵守以下边界：

- Planner：只读任务相关摘要、偏好、事实，不直接写记忆。
- Executor：只读当前子任务相关记忆和 RAG，不直接写长期记忆。
- Reviewer：只读计划、执行结果和引用来源，不写记忆。
- Reflector：生成经验候选，由 MemoryWriteService 统一准入、脱敏、去重、冲突处理后写入。
- Orchestrator：控制最终答案和写入时机，不让多个 Agent 分散直接写库。

这样可以避免多 Agent 并发写记忆导致冲突、污染长期偏好或把敏感中间过程写入长期存储。
