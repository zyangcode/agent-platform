# 记忆与 RAG 检索评测集

> 用途：给 Memory/RAG 检索策略提供固定基线，避免 HyDE、MQE、Reranker、Neo4j 等增强能力上线时只凭主观感觉判断效果。

## 1. 评测口径

评测对象是“检索阶段返回的已排序 ID 列表”，不直接评估最终答案生成质量。

- Memory 用 `memory-{memoryId}` 作为评测 ID。
- RAG 用 `chunk-{chunkId}` 作为评测 ID。
- 一个 query 可以有多个正确 ID，命中任意一个即算 Hit@K。
- Recall@K 计算 TopK 内命中的正确 ID 数 / 该 query 的正确 ID 总数，再对可回答 case 求平均。
- MRR 使用 TopK 内第一个正确结果的倒数排名。
- 无答案问题不计入 Hit@K/Recall@K/MRR，由 no-answer precision 单独评估：`expectedIds` 为空且 TopK 返回为空算正确，否则算无答案误召回。

代码侧已新增纯计算评测核心：

- `RetrievalEvaluationCase`：评测用例，包含 `id/sourceType/query/expectedIds`。
- `RetrievalPrediction`：一次检索策略返回的有序结果 ID。
- `RetrievalEvaluationService`：计算 Hit@K、Recall@K、MRR@K、no-answer precision 和漏召回明细。

## 2. 业务评测集

当前 CI 默认使用业务评测集作为门禁。它不依赖线上数据库，而是把当前项目中已经明确的 Memory/RAG 事实抽成固定 corpus、case 和 baseline prediction，用来防止检索策略改动后发生明显退化。

- `retrieval-eval-corpus.business.jsonl`：业务 corpus，每行一个可检索记忆或 RAG chunk，固定评测 ID 和内容。
- `retrieval-eval-cases.business.jsonl`：业务评测用例，每行一个 `RetrievalEvaluationCase`。
- `retrieval-eval-predictions.business-baseline.jsonl`：当前基线检索排序，每行一个 `RetrievalPrediction`。
- `retrieval-eval-cases.seed.jsonl` / `retrieval-eval-predictions.baseline.jsonl`：保留为格式样例和 runner smoke test，不再作为 CI 默认门禁。

| caseId | sourceType | query | expectedIds | 覆盖场景 |
|---|---|---|---|---|
| memory-answer-style | memory | 用户喜欢什么回答风格 | memory-answer-style-current | 明确偏好召回 |
| memory-project-stack | memory | 当前项目后端框架和版本 | memory-project-stack | 长期事实召回 |
| memory-explicit-forget | memory | 用户已要求忘记的偏好 |  | 禁用记忆不应召回 |

## 3. 上线门槛

新增检索策略默认关闭，满足以下条件后再按 Profile 或配置开关启用：

| 策略 | 最低要求 |
|---|---|
| HyDE | Hit@5、Recall@5 或 MRR@5 明显提升，且平均延迟和成本可接受 |
| MQE | 复杂 query 的 Hit@5 / Recall@5 提升，简单 query 不明显退化 |
| Reranker | MRR@5 提升，P95 延迟不突破主链路预算 |
| Neo4j graph search | 实体关系类 query 命中提升，普通事实 query 不抢占 topK |
| Semantic Cache | 命中相似 query 时不污染 tenant/application/profile scope |

## 4. 当前状态

- 已完成评测指标计算核心和单元测试。
- 评测结果会同时返回 `totalCases`、`answerableCaseCount`、`noAnswerCaseCount`，避免无答案样本稀释可回答样本的 Hit@K/Recall@K/MRR。
- 已新增离线评测 runner：`RetrievalEvaluationFileRunner`，可读取 JSONL case/prediction 文件并输出稳定文本报告，也可通过阈值参数作为本地/CI 门禁。
- 已新增脚本入口：`scripts/run-retrieval-eval.ps1`，默认优先查找 business case/prediction 文件并执行门禁；找不到 business 文件时才回退 seed 样例。
- 已新增 GitHub Actions 工作流：`.github/workflows/retrieval-eval.yml`，当 Memory/RAG/Context/Evaluation 相关文件或评测集变更时自动执行评测门禁。
- 已新增业务评测集一致性测试 `RetrievalBusinessEvaluationDatasetTest`，校验 business case、prediction 和 corpus ID 对齐，并验证基线指标。
- Reranker、MQE 已具备默认关闭的 `openai-compatible` provider 接入口和 Mock 实现；Semantic Cache 已具备默认关闭的内存 provider；HyDE 仍为默认关闭的接口与 Mock。启用真实 Reranker/MQE/HyDE/Semantic Cache 前必须用同一 runner 对比 business baseline。

## 5. 本地运行方式

推荐直接用 Maven exec 运行，避免 Windows 中文路径下手工拼接 classpath 时出现编码问题：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run-retrieval-eval.ps1
```

也可以直接调用 Java runner：

```bash
mvn.cmd -s .mvn/settings.xml -pl agent-platform-core exec:java "-Dexec.mainClass=com.ls.agent.core.evaluation.application.RetrievalEvaluationFileRunner" "-Dexec.args=--cases 实际开发/优化阶段/记忆系统优化/retrieval-eval-cases.business.jsonl --predictions 实际开发/优化阶段/记忆系统优化/retrieval-eval-predictions.business-baseline.jsonl --topK 5 --minHitRate 0.90 --minMrr 0.85 --minRecall 0.90 --minNoAnswerPrecision 1.0"
```

Runner 退出码约定：

- `0`：命令执行成功；如果配置了阈值，表示所有门禁通过。
- `1`：参数、文件或 JSONL 格式错误。
- `2`：评测执行成功，但至少一个指标低于门禁阈值。

后续真实接入时，检索策略只需要为每个 case 输出一行 prediction：

```json
{"caseId":"rag-auth-api","returnedIds":["chunk-auth-api-key","chunk-other"]}
```

Runner 会统一计算 Hit@K、Recall@K、MRR@K、no-answer precision、缺失 prediction 和漏召回明细。
