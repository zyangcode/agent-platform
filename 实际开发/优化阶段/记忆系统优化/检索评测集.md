# 记忆与 RAG 检索评测集

> 用途：给 Memory/RAG 检索策略提供固定基线，避免 HyDE、MQE、Reranker、Neo4j 等增强能力上线时只凭主观感觉判断效果。

## 1. 评测口径

评测对象是“检索阶段返回的已排序 ID 列表”，不直接评估最终答案生成质量。

- Memory 用 `memory-{memoryId}` 作为评测 ID。
- RAG 用 `chunk-{chunkId}` 作为评测 ID。
- 一个 query 可以有多个正确 ID，命中任意一个即算 Hit@K。
- Recall@K 计算 TopK 内命中的正确 ID 数 / 该 query 的正确 ID 总数，再对所有 case 求平均。
- MRR 使用 TopK 内第一个正确结果的倒数排名。
- 无答案问题暂不计入 Hit@K/Recall@K/MRR，应单独用 no-answer precision 评估。

代码侧已新增纯计算评测核心：

- `RetrievalEvaluationCase`：评测用例，包含 `id/sourceType/query/expectedIds`。
- `RetrievalPrediction`：一次检索策略返回的有序结果 ID。
- `RetrievalEvaluationService`：计算 Hit@K、Recall@K、MRR@K 和漏召回明细。

## 3. 上线门槛

新增检索策略默认关闭，满足以下条件后再按 Profile 或配置开关启用：

| 策略 | 最低要求 |
|---|---|
| HyDE | Hit@5、Recall@5 或 MRR@5 明显提升，且平均延迟和成本可接受 |
| MQE | 复杂 query 的 Hit@5 / Recall@5 提升，简单 query 不明显退化 |
| Reranker | MRR@5 提升，P95 延迟不突破主链路预算 |
| Neo4j graph search | 实体关系类 query 命中提升，普通事实 query 不抢占 topK |
| Semantic Cache | 命中相似 query 时不污染 tenant/application/profile scope |
