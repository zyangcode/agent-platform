# 阶段 4 执行顺序

> 本文用于阶段 4 实际开发推进。设计边界见：`实际开发/阶段4/01-阶段4AgentTeam设计说明.md`。接口细节见：`实际开发/阶段4/02-阶段4接口设计.md`。

## 1. 总体目标

阶段 4 实现 Agent Team 最小稳定闭环：

```text
Profile executionMode=TEAM
  -> Chat 请求
  -> Planner 生成 TaskPlan
  -> Executor 调用授权 Skill/MCP 或执行 MODEL_TASK
  -> Orchestrator 生成 answerDraft
  -> Reviewer 审查 answerDraft
  -> Orchestrator 汇总最终答案
  -> SSE 阶段性实时展示过程
  -> Trace 展示 Team span
```

阶段 4 不做：

- 新 Team 数据表。
- 工作流设计器。
- 并发执行。
- 复杂 Pipeline 重构。
- 高危工具确认。
- token 级真实流式输出。

## 2. 开发原则

1. 先后端 Team Runtime，后前端展示。
2. 先 mock/规则兜底跑通活动策划 Demo，再接真实模型 JSON 输出。
3. 先复用现有 Skill/MCP 执行接口，再抽 AgentTool 统一视图。
4. Team 数据先落 Trace Span attributes，不新增表。
5. Team 事件必须阶段性实时推送，不做“执行完后一次性刷屏”的假进度。
6. 每个 Step 必须有测试和可验证结果。

## 6. Step 4：实现 Planner

目标：

- Planner 根据用户输入、Profile Prompt、AgentTool 列表生成 TaskPlan。
- 第一版必须有稳定兜底，保证任何 Demo 都不依赖真实模型 JSON 稳定性。

新增文件：

```text
agent-platform-core/src/main/java/com/ls/agent/core/team/api/TeamPlanner.java
agent-platform-core/src/main/java/com/ls/agent/core/team/command/PlanTeamCommand.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/DefaultTeamPlanner.java
```

策略：

```text
1. 优先调用模型生成 TaskPlan JSON。
2. JSON 解析或校验失败时，要求模型修正一次。
3. 仍失败时，生成通用单任务 fallback plan：MODEL_TASK + suggestedTool=null。
4. 活动策划三任务 plan 只作为测试 seed 或 Demo Profile 的推荐模板，不写死为 Planner 通用 fallback。
```

测试：

```text
DefaultTeamPlannerTest
```

重点测试：

- 模型返回合法 JSON。
- 模型第一次返回非法 JSON，第二次修正成功。
- 两次都失败时使用通用单任务 fallback plan。
- 模型返回空响应时使用 fallback plan。
- 模型返回超过 maxTasks 的 plan 时触发修正或 fallback。
- fallback plan 不依赖 weather/search/calculator 是否存在。

提交建议：


## 7. Step 5：实现 Executor

目标：

- 按依赖顺序执行任务。
- 对 `TOOL_TASK` 调用 AgentToolDispatcher。
- 对 `MODEL_TASK` 调用 ModelInvokeService，基于用户目标、当前任务、Profile Prompt 和已完成任务结果生成当前任务结果。

新增文件：

```text
agent-platform-core/src/main/java/com/ls/agent/core/team/api/TeamExecutor.java
agent-platform-core/src/main/java/com/ls/agent/core/team/command/ExecuteTeamCommand.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/DefaultTeamExecutor.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/TaskDependencySorter.java
```

规则：

```text
dependsOn 未成功时，当前任务 skipped。
TOOL_TASK 工具调用成功 -> success。
TOOL_TASK 工具调用失败 -> failed，但不中断整体。
MODEL_TASK 模型调用成功 -> success。
MODEL_TASK 模型调用失败 -> failed。
每个任务写 trace span。
每次工具调用写 team.tool.call span。
```

MODEL_TASK prompt 必须包含：

```text
1. 用户原始请求。
2. 当前任务 name/description。
3. 已完成任务的 ExecutionResult 摘要。
4. Profile 补充 Prompt。
5. 输出约束：只输出当前任务结果，不重新规划，不生成最终总结。
```

测试：

```text
DefaultTeamExecutorTest
TaskDependencySorterTest
```

重点测试：

- 依赖排序正确。
- 依赖失败时后续任务 skipped。
- TOOL_TASK 调用成功。
- TOOL_TASK 调用失败后返回 failed，Team 不直接崩。
- MODEL_TASK 调用模型成功。
- MODEL_TASK 模型异常后返回 failed。
- maxToolCalls 超限时后续工具任务 failed 或 skipped。

提交建议：


## 10. Step 8：实现 Team Orchestrator

目标：

- 串起 Planner -> Executor -> answerDraft -> Reviewer -> Retry -> Final。
- 统一保存会话消息和记忆。
- 统一记录 Team trace span。
- 返回 `AgentRunResult`，保证外部接口不变。

新增文件：

```text
agent-platform-core/src/main/java/com/ls/agent/core/team/api/TeamRuntimeService.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/DefaultTeamRuntimeService.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/TeamAnswerDraftBuilder.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/TeamFinalAnswerBuilder.java
agent-platform-core/src/main/java/com/ls/agent/core/team/application/TeamRunLimiter.java
```

流程：

```text
1. emit team_start。
2. Planner 生成 TaskPlan，emit team_plan。
3. Executor 逐任务执行，实时 emit task/tool 事件。
4. TeamAnswerDraftBuilder 生成 answerDraft。
5. Reviewer 审查 answerDraft，emit team_review。
6. 如需重试且未超过 maxRetries，emit team_retry 并重试指定任务。
7. TeamFinalAnswerBuilder 生成最终答案，emit team_final/message/done。
```

TeamRunLimiter 限制：

```text
maxTasks=8
maxRetries=1
maxToolCalls=8
maxModelCalls=6
timeoutMs=120000
```

集成点：

```text
DefaultAgentRuntimeService
AgentContextBuilder
ModelInvokeService
ConversationRepository
MemoryWriteService
TraceService
TokenUsageService
TeamEventSink
```

注意：

- 如果直接让 `DefaultAgentRuntimeService` 委托 Team，要避免循环依赖。
- 共享的保存会话、记录 token、trace span 小方法可以抽成 package-private helper。
- 不为了阶段 4 立刻做完整 Pipeline。

测试：

```text
DefaultTeamRuntimeServiceTest
DefaultAgentRuntimeServiceTest
TeamRunLimiterTest
```

重点测试：

- 正常 Planner -> Executor -> Reviewer -> Final。
- Reviewer 要求重试一次后通过。
- Reviewer 要求重试两次时第二次被限制。
- maxModelCalls 超限。
- timeoutMs 超限。
- 事件顺序符合接口文档。
- BASIC 模式仍走原 Agent Runtime。

提交建议：

```text
feat(team): orchestrate planner executor reviewer runtime
```

## 15. 阶段完成标准

满足以下条件即可认为阶段 4 完成：

- Team Profile 可以创建或更新。
- Team Chat 能跑通活动策划 Demo。
- SSE 能实时展示 Team 过程，而不是结束后一次性刷屏。
- Trace Detail 能看到 Team spans。
- Token Usage 不因 Team 模式断链。
- 基础 Agent BASIC 模式不回归。
- Team 上限和超时有测试覆盖。
- 后端全量测试通过。
- 前端 build 通过。
- 手工验收清单完成。

阶段 4 完成后：

```text
git tag phase-4-complete
git push origin main --tags
```
