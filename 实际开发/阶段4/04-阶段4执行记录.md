## 3. Step 4 复查结论

已复查 `DefaultTeamPlanner`、`PlanTeamCommand`、`TeamPlanner`、`TaskPlanValidator` 与相关测试：

- Planner 只负责拆任务，不调用 Skill/MCP，不生成最终答案。
- Planner 复用现有 `ModelInvokeService`，后续 Token / Trace 链路可以继续沿用模型调用层。
- Planner prompt 展示工具来源，如 `[SKILL] weather`、`[MCP] read_file`。
- 模型输出会先解析成 `TaskPlanDTO`，再交给 `TaskPlanValidator` 校验。
- 非法 JSON、非法计划、空响应、模型调用异常都会重试一次；二次失败后降级为通用单任务 `MODEL_TASK`。
- fallback 未硬编码 weather/search/calculator，普通问题仍能进入后续 Executor 模型任务。
- `TaskPlanValidator` 已注册为 Spring Bean，避免应用上下文启动时缺 Bean。
- Planner 结果通过 `TeamPlanResultDTO` 携带模型调用结果，后续 Orchestrator 可记录 Planner 阶段 token usage。
- retry prompt 会截断上一次错误原因，避免长异常或垃圾输出浪费上下文。

## 6. Step 5 复查结论

已复查 `DefaultTeamExecutor`、`ExecuteTeamTaskCommand`、`TeamTaskExecutionResultDTO` 与 `TaskDependencySorter`：

- Executor 执行单个任务，不负责完整 Team 编排。
- `TaskDependencySorter` 负责把 TaskPlan 按 dependsOn 排序，并拒绝缺失依赖和循环依赖。
- `TOOL_TASK` 调用 `AgentToolDispatcher` 前先校验授权 AgentTool 列表；未授权直接 failed，不调用 Dispatcher。
- `TOOL_TASK` 调用失败只返回 failed，不抛出中断整个 Team。
- `MODEL_TASK` 走 `ModelInvokeService`，prompt 包含用户原始请求、Profile Prompt、当前任务和已完成任务结果。
- Executor 不直接写 Trace / Token；模型调用和工具调用结果会返回给后续 Orchestrator 统一记录。

## 7. Step 6 复查结论

已复查 `DefaultTeamReviewer`、`ReviewTeamCommand`、`TeamReviewResultDTO` 与 `ReviewResultValidator`：

- Reviewer 只审查 TaskPlan、ExecutionResult 和 answerDraft，不调用 Skill/MCP，不重新规划任务。
- Reviewer 输出 `ReviewResultDTO`，并通过 `TeamReviewResultDTO.modelInvocations` 带回模型调用结果，供后续 Orchestrator 统一记录 Trace / Token。
- Reviewer JSON 非法、校验失败或模型异常时会重试一次；二次失败后 fallback 为 `passed=true` 且带 WARN issue。
- `ReviewResultValidator` 校验 `passed`、issue level、issue taskId、retryTasks 是否引用已有任务。
- answerDraft 为空时 fallback 会追加 `answerDraft is blank` warning。

## 10. Step 8 复查结论

已实现 `TeamRuntimeService` / `DefaultTeamRuntimeService`，串起 Planner -> Executor -> answerDraft -> Reviewer -> Retry -> Final。

- `DefaultTeamRuntimeService` 专注 Team 编排、事件、Trace / Token 记录，并返回 `AgentRunResult`；conversation message / memory 仍由 `DefaultAgentRuntimeService` 统一保存，避免 `core.team` 越界访问 `core.agent.entity`。
- Planner / Executor / Reviewer 的 `ModelInvokeResult.usage()` 统一汇总为 Team run usage，并逐条调用 `TokenUsageService.record(...)`。
- Team event 顺序覆盖 `team_start` -> `team_plan` -> `team_task_start` -> (`team_tool_call` -> `team_tool_result`) -> `team_task_result` -> `team_review` -> (`team_retry`) -> `team_final`。
- `TeamRunLimiter` 提供 maxTasks / maxRetries / maxToolCalls / maxModelCalls / timeoutMs 限制。
- `DefaultAgentRuntimeService` 已根据 `AgentContextDTO.profile.executionMode` 分发：`BASIC` 继续原单 Agent 链路，`TEAM` 委托 `TeamRuntimeService`。
- `AgentRuntimeService.run(command, TeamEventSink)` 和 `TeamRuntimeService.run(command, TeamEventSink)` 已留出请求级 event sink 入口，供 Step 9 Gateway SSE 适配使用。
