# 阶段 4 接口设计

## 1. 接口设计原则

阶段 4 尽量复用阶段 3 已有 Chat、Profile、Trace 接口，不新增复杂 API。

核心原则：

```text
Team 是 Profile 的 executionMode，不是新的入口类型。
浏览器仍调用 /api/chat/stream。
Web 仍转发 Gateway。
Gateway 仍调用 Core AgentRuntime。
Team 过程通过阶段性实时 SSE 和 Trace 展示。
```

## 2. Profile 接口调整

### 2.1 Profile DTO 增加 executionMode

当前数据库已有字段：

```text
agent_profiles.execution_mode varchar(32) default 'BASIC'
```

阶段 4 需要把该字段补到接口 DTO。

取值：

```text
BASIC
TEAM
```

Profile 响应：

```json
{
  "profileId": 50001,
  "applicationId": 1,
  "name": "活动策划助手",
  "profileType": "DOMAIN",
  "description": "适合活动策划、预算估算和方案推荐",
  "modelConfigId": 1,
  "promptExtra": "风格提示词：输出务实、结构清晰的活动方案。",
  "memoryStrategy": {"mode": "READ_WRITE"},
  "maxSteps": 5,
  "executionMode": "TEAM",
  "visibility": "PRIVATE",
  "status": "PUBLISHED",
  "skillBindings": [],
  "mcpToolBindings": []
}
```

### 2.2 创建 Profile

```text
POST /api/profiles
Authorization: Bearer <jwt>
Content-Type: application/json
```

请求：

```json
{
  "applicationId": 1,
  "name": "活动策划助手",
  "profileType": "DOMAIN",
  "description": "处理活动策划、天气、预算和方案搜索。",
  "modelConfigId": 1,
  "promptExtra": "输出时优先考虑天气、预算、交通便利性和活动强度。",
  "memoryStrategy": {"mode": "READ_WRITE"},
  "maxSteps": 5,
  "executionMode": "TEAM",
  "visibility": "PRIVATE"
}
```

响应：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {
    "profileId": 50001,
    "executionMode": "TEAM"
  }
}
```

校验：

- `executionMode` 为空时默认 `BASIC`。
- 只允许 `BASIC` / `TEAM`。
- User 只能创建自己的 private profile。
- Admin / Developer 权限按现有规则扩展。

### 2.3 更新 Profile

```text
PUT /api/profiles/{profileId}
Authorization: Bearer <jwt>
Content-Type: application/json
```

请求字段与创建一致，允许更新：

```json
{
  "name": "活动策划 Team",
  "description": "活动策划 Team Demo",
  "modelConfigId": 1,
  "promptExtra": "输出活动方案时说明预算、人均费用和天气风险。",
  "memoryStrategy": {"mode": "READ_WRITE"},
  "maxSteps": 5,
  "executionMode": "TEAM",
  "visibility": "PRIVATE"
}
```

## 3. Chat SSE 接口

### 3.1 请求

阶段 4 继续使用：

```text
POST /api/chat/stream
Authorization: Bearer <jwt>
Accept: text/event-stream
Content-Type: application/json
```

请求：

```json
{
  "applicationId": 1,
  "agentMode": "agent",
  "profileId": 50001,
  "conversationId": null,
  "message": "帮我策划一个本周六下午在重庆适合 20 人的团建活动，预算 3000 元以内，最好不要太累。",
  "enabledSkillIds": [1, 2, 3],
  "enabledMcpToolIds": [],
  "stream": true
}
```

说明：

- 是否走 Team 不由请求参数决定，而由 `profile.executionMode` 决定。
- `agentMode=none` 仍然是纯模型透传，不进入 Team。
- Team 模式只在 `agentMode=agent` 下生效。

### 3.2 SSE 通用格式

仍使用阶段 3 的 SSE data JSON：

```text
event: message
data: {"type":"message","traceId":"tr_xxx","content":"..."}
```

阶段 4 新增 type：

```text
team_plan
team_task_start
team_tool_call
team_tool_result
team_task_result
team_review
team_retry
team_final
```

阶段 4 的 Team SSE 必须阶段性实时输出，顺序约束如下：

```text
team_plan
team_task_start(task-1)
team_tool_call(task-1)      // TOOL_TASK 才有
team_tool_result(task-1)    // TOOL_TASK 才有
team_task_result(task-1)
...
team_review
team_retry                  // 需要重试时才有
team_final
message
done
```

不采用“TeamRuntime 完成后一次性吐出全部 team_* 事件”的方案。

### 3.3 team_plan

```json
{
  "type": "team_plan",
  "traceId": "tr_xxx",
  "step": 1,
  "role": "PLANNER",
  "content": "Planner 已生成 3 个任务",
  "plan": {
    "goal": "为 20 人策划重庆本周六下午团建活动，预算 3000 元以内",
    "tasks": [
      {
        "id": "task-1",
        "name": "查询天气",
        "description": "查询重庆本周六下午天气",
        "taskType": "TOOL_TASK",
        "suggestedTool": "weather",
        "dependsOn": []
      }
    ]
  }
}
```

### 3.4 team_task_start

```json
{
  "type": "team_task_start",
  "traceId": "tr_xxx",
  "step": 2,
  "role": "EXECUTOR",
  "taskId": "task-1",
  "content": "开始执行：查询天气"
}
```

### 3.5 team_tool_call

```json
{
  "type": "team_tool_call",
  "traceId": "tr_xxx",
  "step": 3,
  "role": "EXECUTOR",
  "taskId": "task-1",
  "toolType": "SKILL",
  "toolName": "weather",
  "content": "调用 weather"
}
```

不在 SSE 中发送敏感原始参数。必要时只发送脱敏摘要。

### 3.6 team_tool_result

```json
{
  "type": "team_tool_result",
  "traceId": "tr_xxx",
  "step": 4,
  "role": "EXECUTOR",
  "taskId": "task-1",
  "toolType": "SKILL",
  "toolName": "weather",
  "status": "success",
  "content": "weather 返回天气摘要"
}
```

### 3.7 team_task_result

`team_task_result` 表示一个任务已经结束。`TOOL_TASK` 和 `MODEL_TASK` 都必须发送这个事件。

```json
{
  "type": "team_task_result",
  "traceId": "tr_xxx",
  "step": 5,
  "role": "EXECUTOR",
  "taskId": "task-1",
  "taskType": "TOOL_TASK",
  "status": "success",
  "content": "任务完成：已获得重庆本周六下午天气摘要"
}
```

### 3.8 team_review

```json
{
  "type": "team_review",
  "traceId": "tr_xxx",
  "step": 8,
  "role": "REVIEWER",
  "content": "Reviewer 审查通过",
  "answerDraft": "根据天气、预算和活动强度，建议优先选择室内桌游馆或轰趴馆...",
  "review": {
    "passed": true,
    "issues": [],
    "retryTasks": [],
    "summary": "方案覆盖天气、预算和活动强度。"
  }
}
```

说明：

- `answerDraft` 由 Orchestrator 在 Reviewer 前生成。
- Reviewer 审查的是用户目标、TaskPlan、ExecutionResult 和 `answerDraft`。
- 最终 `message` 可以基于 `answerDraft` 和 ReviewResult 做轻微修正。

### 3.9 team_retry

```json
{
  "type": "team_retry",
  "traceId": "tr_xxx",
  "step": 9,
  "role": "ORCHESTRATOR",
  "taskId": "task-2",
  "content": "Reviewer 要求重试 task-2"
}
```

### 3.10 team_final

```json
{
  "type": "team_final",
  "traceId": "tr_xxx",
  "step": 10,
  "role": "ORCHESTRATOR",
  "content": "Team 已汇总最终答案"
}
```

之后继续发送现有：

```json
{
  "type": "message",
  "traceId": "tr_xxx",
  "content": "最终活动方案..."
}
```

最后：

```json
{
  "type": "done",
  "traceId": "tr_xxx",
  "conversationId": 27,
  "usage": {
    "promptTokens": 100,
    "completionTokens": 50,
    "totalTokens": 150,
    "estimated": true
  }
}
```

## 4. Trace 查询接口

阶段 4 不新增 Trace 接口，复用：

```text
GET /api/traces
GET /api/traces/{traceId}
```

新增 Team span name：

```text
team.run
team.plan
team.plan.validate
team.execute
team.task.execute
team.tool.call
team.review
team.retry
team.finalize
```

Trace Detail 页面根据 span attributes 识别：

```json
{
  "role": "PLANNER",
  "taskId": "task-1",
  "toolName": "weather",
  "retryIndex": 0
}
```

## 5. 前端类型

### 5.1 TeamTask

```ts
export type TeamTask = {
  id: string
  name: string
  description: string
  taskType: 'TOOL_TASK' | 'MODEL_TASK'
  suggestedTool?: string | null
  dependsOn: string[]
  status?: 'pending' | 'running' | 'success' | 'failed' | 'skipped'
}
```

### 5.2 TeamPlan

```ts
export type TeamPlan = {
  goal: string
  tasks: TeamTask[]
}
```

### 5.3 TeamReview

```ts
export type TeamReview = {
  passed: boolean
  issues: Array<{
    taskId?: string | null
    level: 'info' | 'warning' | 'error'
    message: string
  }>
  retryTasks: string[]
  summary: string
}
```

### 5.4 TeamStreamEvent

```ts
export type TeamStreamEvent = {
  type:
    | 'team_plan'
    | 'team_task_start'
    | 'team_tool_call'
    | 'team_tool_result'
    | 'team_task_result'
    | 'team_review'
    | 'team_retry'
    | 'team_final'
  traceId?: string
  step?: number
  role?: 'PLANNER' | 'EXECUTOR' | 'REVIEWER' | 'ORCHESTRATOR'
  taskId?: string
  toolType?: 'SKILL' | 'MCP'
  toolName?: string
  taskType?: 'TOOL_TASK' | 'MODEL_TASK'
  status?: string
  content?: string
  answerDraft?: string
  plan?: TeamPlan
  review?: TeamReview
}
```

## 6. 错误码建议

阶段 4 可先复用现有错误码：

```text
REQUEST_INVALID
MODEL_ERROR
AGENT_MAX_STEPS_EXCEEDED
INTERNAL_ERROR
```

如需新增，建议：

```text
TEAM_PLAN_INVALID
TEAM_REVIEW_FAILED
TEAM_TOOL_UNAVAILABLE
```

但第一版优先避免扩充过多错误码。

## 7. 手工验收接口顺序

1. 登录拿 JWT。
2. 创建或更新 Profile，设置 `executionMode=TEAM`。
3. 给 Profile 绑定 weather、search、calculator。
4. 调用 `/api/chat/stream`。
5. 检查 SSE 是否按执行过程实时出现 `team_plan`、`team_task_start`、`team_task_result`、`team_review`、`message`、`done`。
6. 打开 Trace Detail，检查 Team spans。
7. 打开 Token Usage，确认模型调用用量仍有记录。
