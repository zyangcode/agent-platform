# 阶段 4 问题台账

> 记录阶段 4 Agent Team 开发过程中的关键问题、决策和暂缓项。格式保持：问题、核心思路、核心方案。

## 1. Team 是独立 Agent 入口还是 Profile 的执行模式

**问题：** 如果为 Team 单独新增一套入口、页面和权限模型，用户会不清楚“普通 Agent”和“Team Agent”到底是两套产品还是同一个 Profile 的两种运行方式。

**核心思路：** Team 是 Agent Runtime 的增强执行模式，不是新产品线。外部 Chat 入口保持不变，降低 Web/Gateway 改动面。

**核心方案：** 在 `agent_profiles.execution_mode` 上区分 `BASIC` / `TEAM`。`BASIC` 继续走现有单 Agent Runtime；`TEAM` 后续由 Runtime 内部分发到 Planner -> Executor -> Reviewer。Step 1 已完成接口和前端表单闭环。

## 2. 普通用户能否创建和使用自己的 Team Profile

**问题：** 如果只有 Developer 能创建 Team Profile，普通用户就不能选择自己的 Team Agent；如果完全放开，又可能误以为普通用户可以修改公共 Profile 或高危工具策略。

**核心思路：** Profile 的归属和可见性要与执行模式解耦。用户可以创建自己的私有 Profile，但不能越权修改公共 Profile 或启用未授权工具。

**核心方案：** 普通用户可创建 `visibility=PRIVATE` 的 BASIC 或 TEAM Profile；Developer/Admin 可配置公共或领域 Profile。权限边界仍靠 application/profile owner、visibility、工具绑定和后端校验控制。Team 模式不等于 Developer 权限。

## 3. Team Demo 是否只能回答活动策划问题

**问题：** 活动策划是阶段 4 Demo 场景，但如果 Planner fallback 写死 weather/search/calculator 组合，其他问题会无法回答，Team 会变成一个只服务 Demo 的硬编码功能。

**核心思路：** Demo 可以有推荐场景，Runtime fallback 不能绑定场景。Team 失败降级要仍能处理普通用户问题。

**核心方案：** Planner 正常路径可以根据用户问题和可用工具生成任务；两次 JSON 解析/校验失败或模型调用异常后，统一降级为单个 `MODEL_TASK`，把原始问题交给后续 Executor 模型任务处理。活动策划三任务只能作为测试 seed 或演示 Profile 推荐模板，不写死在通用 fallback。

## 4. BASIC 模式是否仍然保留

**问题：** 引入 Team 后，如果所有 Agent 都默认走 Planner/Executor/Reviewer，会增加简单问答延迟，也会让已有单 Agent 测试和使用方式回归风险变大。

**核心思路：** 基础 Agent 是必做能力，Team 是高分增强项。Team 不能替换 BASIC。

**核心方案：** `executionMode=BASIC` 继续走现有单 Agent Runtime；`executionMode=TEAM` 才进入 Team Runtime。后续 Step 8 串 Orchestrator 时必须补 BASIC 不回归测试。

## 5. 单 Agent SSE 和 Team SSE 的优先级

**问题：** 用户希望回答时 token 级流式输出，但阶段 4 又要求 Team 过程实时展示。如果同时改 token 级 streaming 和 Team Orchestrator，调试范围会过大。

**核心思路：** 先完成 Team 事件级实时，再做 token 级流式。事件级实时解决“看得到 Planner/Executor/Reviewer 进度”；token 级流式解决“最终回答逐字输出”。

**核心方案：** 阶段 4 必须实现 Team 事件级 SSE：`team_plan`、`team_task_start`、`team_tool_call`、`team_tool_result`、`team_task_result`、`team_review`、`team_final`。单 Agent token 级流式和 Team token 级流式暂缓到 Team 闭环稳定之后单独做。

## 6. Codex Skill 与平台 Jar Skill 的边界

**问题：** 用户本机的 Codex Skill（例如前端设计 skill）和平台内的 Jar Skill 都叫 Skill，容易混淆为同一种能力，甚至误以为可以用平台 Jar 热加载来实现 Codex 开发 skill。

**核心思路：** Codex Skill 是开发助手的本地能力扩展；平台 Jar Skill 是 Agent Runtime 可调用的业务工具。两者运行环境、触发方式和安全边界完全不同。

**核心方案：** Codex Skill 存在于本机 Codex 配置目录，用来辅助写代码、设计和审查；平台 Jar Skill 由 agent-platform 后端通过上传、校验、ClassLoader 加载后，作为 Agent 可调用工具执行。Jar Skill 第一版只作为工具，不承担 Codex 开发技能或 Agent 角色模板。

## 7. 阶段 4 回退点如何保证

**问题：** 后续 Team Runtime 改动会逐步触碰 Agent 主链路，如果阶段 4 写坏了，需要能回到稳定版本。

**核心思路：** 用小步 commit 作为稳定回退点，不把多个大改动堆在一个提交里。

**核心方案：** Step 级别提交已形成回退点：`c786382 feat(profile): expose agent execution mode`、`8b5df5c feat(team): add task plan model and validation`。后续每个 Step 继续独立提交；阶段 4 完成后再打 `phase-4-complete` tag。

## 8. 当前非阻塞项

**问题：** 阶段 4 当前仍存在一些不影响 Step 1/2 的暂缓项，后续容易忘记。

**核心思路：** 非阻塞项不打断当前 Step，但要集中记录，避免误判为已完成。

**核心方案：** 当前暂缓项包括：Team Runtime 尚未接 Chat 分发；Team SSE 尚未实现；Trace Detail 尚未展示 Team span；Token 级流式输出暂缓；Admin Security/Users 仍是占位；Vite chunk size warning 仍存在；Jar Skill 上传和热加载 UI 未进入阶段 4 当前范围。

## 9. TaskPlanValidator 是否现在抽接口

**问题：** 审查建议把 `TaskPlanValidator` 拆成 `TeamPlanValidator` 接口和默认实现，理由是后续 Planner、Reviewer 或 Orchestrator 可能复用。

**核心思路：** 项目原则是不为想象中的变化点提前抽象。当前 Validator 是纯计算组件，不访问数据库、模型、外部服务，也只有一个明确调用方向，直接测试具体类更简单。

**核心方案：** 暂不抽接口。等 Planner/Reviewer/Orchestrator 至少两个真实调用方出现，或者需要替换 Mock 实现时，再抽 `TeamPlanValidator` 接口并把当前类改为默认实现。

## 10. Team Plan 校验是否现在新增专用错误码

**问题：** 审查建议增加 `TEAM_PLAN_INVALID` 错误码，用于区分用户请求错误和 Planner 输出 JSON 错误。

**核心思路：** 专用错误码有价值，但它属于 common 全局约定。当前 Step 只做 DTO 和 Validator，还没写 Planner JSON 解析、修正重试和 fallback，过早定错误码容易脱离真实消费方。

**核心方案：** 暂时继续使用 `REQUEST_INVALID`。到 Step 4 实现 Planner JSON 解析与修正重试时，再根据实际消费路径决定是否新增 `TEAM_PLAN_INVALID`，并同步更新 Web/Gateway 错误处理和测试。

## 11. AgentTool Resolver 是否重复查询 Skill/MCP

**问题：** Step 3 需要把 Skill/MCP 转成统一 AgentTool。如果 Resolver 自己再调用 SkillQueryService / McpToolQueryService，会和 AgentContextBuilder 重复查询，也可能导致 Planner 看到的工具列表与 Executor 实际上下文不一致。

**核心思路：** AgentContextDTO 已经是当前请求的授权上下文，里面的 availableSkills / availableMcpTools 已经经过 Profile、用户选择和权限校验。AgentTool 应基于这份上下文做视图转换。

**核心方案：** `AgentToolResolver.resolve(AgentContextDTO context)` 只读取 `context.availableSkills()` 和 `context.availableMcpTools()`，不重新查表。Planner、Validator、Executor 统一使用同一份 AgentTool 列表。

## 12. AgentTool 名称和风险等级第一版如何处理

**问题：** Skill 和 MCP 未来可能重名，且当前 SkillDTO / McpToolDTO 没有 riskLevel 来源。如果 Step 3 过早引入命名空间和风险策略，会扩大范围。

**核心思路：** 第一版先满足 Team Runtime 闭环，保留后续扩展点，不凭空增加配置来源。

**核心方案：** 第一版 Skill 工具名使用 `SkillDTO.code`，MCP 工具名使用 `McpToolDTO.name`，`displayName` 使用 DTO 的 `name`，`riskLevel` 统一 `LOW`。后续发现重名再升级为 `skill:xxx` / `mcp:xxx`；高危工具确认和安全策略接入后再计算真实 riskLevel。

## 13. Dispatcher 是否负责工具授权白名单

**问题：** `AgentToolDispatcher` 现在只根据 sourceType/toolName 分发到 Skill/MCP 执行器。如果 Planner 产出未授权工具，不能让 Dispatcher 直接执行。

**核心思路：** Dispatcher 是低层分发器，授权白名单应该由持有 TaskPlan 和 AgentTool 列表的 Executor 判断。

**核心方案：** Step 5 Executor 调用 Dispatcher 前必须校验 `TOOL_TASK.suggestedTool` 存在于 resolved AgentTool 列表；不在白名单时任务返回 failed，不调用 Dispatcher。

## 14. TeamEventSink 是否暴露 SSE 写出细节

**问题：** 文档中曾写“Gateway 写 SSE 时传入可写 OutputStream 的 sink”，容易被理解为 Core 层知道 OutputStream/SSE。

**核心思路：** Core 只 emit TeamRuntimeEventDTO，Gateway 负责 SSE 协议适配。

**核心方案：** Core 定义 `TeamEventSink.emit(TeamRuntimeEventDTO)`；Gateway 实现该接口并写 SSE。Core 不依赖 OutputStream、SseEmitter 或 HTTP API。

## 15. TaskPlan description 和 MODEL_TASK.suggestedTool 规则

**问题：** MODEL_TASK prompt 需要任务描述，但 Validator 当前只校验 name；MODEL_TASK 如果携带 suggestedTool，会产生“模型任务是否也该调工具”的歧义。

**核心思路：** Planner 输出越明确，Executor 越容易稳定执行。

**核心方案：** 后续补校验：task.description 必填；TOOL_TASK 必须有 suggestedTool 且存在于 AgentTool 列表；MODEL_TASK 不允许有 suggestedTool。

## 16. Team 纯逻辑类的 Spring Bean 注册边界

**问题：** `DefaultTeamPlanner` 作为 Spring Bean 依赖 `TaskPlanValidator`。如果 Validator 只在单元测试里手动 new，而不是 Spring Bean，单测可以通过，但应用上下文会启动失败。

**核心思路：** 被 Spring 构造注入的无状态协作者也要明确注册为 Bean；完全不参与注入的静态/纯函数工具才保持普通类。

**核心方案：** `TaskPlanValidator` 标注 `@Component`。后续 `ReviewResultValidator`、`TeamRunLimiter` 等如果被 Planner/Executor/Runtime 注入，也按同样规则注册，并用应用上下文测试兜底。
