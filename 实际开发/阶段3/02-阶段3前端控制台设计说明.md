# 阶段 3 前端控制台设计说明

## 1. 目标

阶段 3 的目标是把当前后端能力做成可演示、可操作的 Web 控制台。

MVP 需要覆盖：

- 登录并维持 JWT 会话。
- 查看控制台 Dashboard。
- 创建和查看 Application，并展示一次性 API Key。
- 创建、查看和配置 Agent Profile。
- 通过 Web 入口和 Agent 对话。
- 查看 Trace 列表和 Trace 明细。
- 查看 Token Usage 汇总和调用明细。
- 查看已有 Skill 和 MCP 工具信息。

阶段 3 不是完整 Admin 系统，暂不包含：

- Skill 上传。
- Jar Skill 热加载 UI。
- Team 编排。
- 飞书机器人配置。
- 完整模型供应商管理。
- 完整 RBAC / 用户管理。
- 完整安全策略管理。

## 2. 后端前提

前端只调用现有阶段 1、阶段 2 后端接口。

本地开发地址：

```text
Frontend: http://localhost:5173
Web:      http://localhost:8080
Gateway:  http://localhost:8081
```

Vite 代理：

```text
/api -> http://localhost:8080
```

浏览器端统一以 Web 模块作为入口。AI 对话请求从 Web 进入，再由 Web 转发到 Gateway；前端不直接调用 Core。

已知阶段 2 遗留问题：模型调用返回 `MODEL_ERROR` 时，`trace_roots` 可能没有写入。阶段 3 不因此阻塞，但 Trace 页面必须能展示空状态或缺失提示，不能崩溃。

## 4. 视觉方向

已确认的设计选择：

```text
整体风格：高级深色 AI 控制台
导航布局：固定左侧栏 + 顶部状态栏
Dashboard：工作台式信息密度
Agent Chat：中间对话区 + 右侧运行详情面板
Trace Detail：时间线 + 右侧详情面板
Token Usage：克制的数据看板图表
```

界面气质应像严肃的 AI 运维控制台，而不是普通白色后台或营销落地页。

设计规则：

- 使用深色分层界面，不做单调纯黑。
- 信息密度要适合管理和排查问题。
- 避免超大营销 Hero。
- 避免装饰性渐变光球。
- 避免一整套页面只靠紫色或蓝紫色撑视觉。
- 导航和紧凑操作优先使用图标。
- 卡片圆角保持克制，不做卡片套卡片。
- 空状态、加载中、错误、未授权、Token 过期状态必须明确。

## 5. 页面范围

### 5.1 登录页

用途：

- 用户输入用户名和密码登录。
- 保存后端返回的 JWT。
- 已登录用户进入控制台。

行为：

- 调用 `POST /api/auth/login`。
- 展示表单校验错误和后端认证失败信息。
- 如果第一版包含注册入口，则提供极简注册动作。

### 5.2 Dashboard

用途：

- 展示平台当前状态总览。

内容：

- 核心指标条：Applications、Profiles、recent Traces、Token Usage。
- 最近 Trace 列表。
- Token Usage 汇总图。
- 快捷操作：创建 Application、创建 Profile、开始 Chat。

如果后端没有专门的聚合接口，MVP 可以先基于列表接口在前端计算简单数量。

### 5.3 Applications / API Keys

用途：

- 创建和查看 Application。
- 创建后展示一次性 API Key。

行为：

- 调用 Web 模块下的 Application 接口。
- API Key 必须按敏感信息处理。
- 提供复制动作，并明确提示 Key 只展示一次。

### 5.4 Profiles

用途：

- 创建和查看 Agent Profile。
- 在现有接口支持范围内绑定 Skill 和 MCP 工具。

行为：

- 使用已有 Profile、Skill、MCP 查询和绑定接口。
- 第一版优先保证配置可读、可演示，不追求复杂编辑器。

### 5.5 Agent Chat

用途：

- 用户在浏览器里运行 Agent。
- 展示 SSE 事件和阶段 2 治理链信息。

布局：

- 中间：对话流和输入框。
- 右侧：当前 Profile、启用 Skill/MCP、Trace ID、运行状态、Token Usage、错误状态。

行为：

- 调用 Web AI Chat 接口。
- 处理 `thinking`、`action`、`observation`、`message`、`done`、`error` 等 SSE 事件。
- 流式输出过程中页面不能跳动或丢失已输出内容。
- SSE 失败时提供明确降级提示。

### 5.6 Trace List

用途：

- 展示当前用户最近 Trace Root。

行为：

- 如果后端支持，提供 status、application、profile、entrypoint 等基础筛选。
- 点击列表行进入 Trace Detail。
- 空数据是正常状态，不能当成错误。

### 5.7 Trace Detail

用途：

- 可视化阶段 2 的治理链路。

布局：

- 主区域：按顺序展示 Span 时间线。
- 右侧：选中 Span 的详细信息。

内容：

- Root 元信息：trace ID、状态、入口、耗时、模型模式、时间。
- Span 时间线：组件、名称、类型、状态、耗时、错误。
- 当前 Trace 的 Token Usage 信息。

页面必须兼容 `MODEL_ERROR` 导致 Trace 缺失的情况，展示明确提示。

### 5.8 Token Usage

用途：

- 展示 Token 消耗和配额结算结果。

布局：

- 汇总指标。
- 克制的 ECharts 图表区域。
- 调用明细表格。

如果后端只有列表或明细接口，第一版可以在前端计算简单总量。

### 5.9 Skills / MCP Tools

用途：

- 展示已有 Skill 和 MCP 工具。
- 帮助用户理解哪些工具可以绑定到 Profile。

不做：

- Skill 上传。
- Jar 热加载。
- Skill 审核和发布流程。

## 6. 前端目录建议

```text
agent-platform-frontend/
  src/
    app/
      router.tsx
      providers.tsx
    components/
      layout/
      ui/
      charts/
      feedback/
    features/
      auth/
      dashboard/
      applications/
      profiles/
      chat/
      traces/
      token-usage/
      tools/
    lib/
      api/
      auth/
      sse/
      format/
    styles/
```

边界约定：

- `lib/api` 负责 HTTP Client、响应解包、错误标准化、JWT Header 注入。
- `lib/auth` 负责 Token 持久化和当前用户状态。
- `features/*` 负责页面级组件和各自接口适配。
- 共享 UI 组件保持通用，不直接绑定后端 DTO。
- SSE 逻辑放在 `lib/sse` 或 Chat feature 内，不混进普通 HTTP Client。

## 7. 数据流

认证：

```text
Login form -> POST /api/auth/login -> 保存 JWT -> GET /api/auth/me -> 渲染控制台
```

普通接口：

```text
Page -> feature API wrapper -> shared API client -> Web backend
```

Chat：

```text
Chat page -> Web SSE endpoint -> event parser -> conversation state + runtime detail panel
```

Trace：

```text
Trace list -> select trace -> Trace detail -> timeline + token usage detail
```

## 8. 错误处理

全局处理：

- `401`：清理 Token，回到登录页。
- `403`：展示权限不足或安全策略拦截信息。
- `429`：展示配额不足状态。
- `5xx`：展示后端错误和重试动作。
- 网络失败：展示重连或重试状态。

页面处理：

- 空列表是正常状态。
- Trace 明细缺失是可预期状态。
- SSE 错误不能让 Chat 页面永久停留在 loading。

## 9. 验证方式

自动化检查：

- `npm run build`
- 如果配置了 lint，则跑 `npm run lint`
- 如果脚手架包含测试框架，再补关键 API 工具或组件测试

手工验收：

- 使用 `admin/admin123` 登录。
- 创建 Application 并复制 API Key。
- 查看模型、Profile、Skill、MCP 数据。
- 发起一次 Agent Chat，并观察 SSE 输出。
- 打开 Trace List 和 Trace Detail。
- 打开 Token Usage。
- 未登录访问控制台时应跳回登录页。
- Apifox 仍能导入 `http://localhost:8080/v3/api-docs`。

视觉检查：

- 先检查桌面端。
- 至少检查一个窄屏视口，确认文字不重叠。
- 检查深色主题对比度、导航清晰度、加载态和空状态。

## 11. 后续决策规则
