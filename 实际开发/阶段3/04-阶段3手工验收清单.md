# 阶段 3 手工验收清单

## 1. 验收目标

阶段 3 的验收目标是确认前端控制台 MVP 能在浏览器中跑通完整 Demo 流程：

- 能注册或登录。
- 能创建 Application，并看到一次性 API Key。
- 能创建 Profile，并绑定已有 Skill / MCP Tool。
- 能通过 Web 入口发起 Chat。
- 能查看 Trace 列表和 Trace 详情。
- 能查看 Token Usage 汇总和明细。
- 页面具备 loading / empty / error 状态，不因为空数据或后端错误崩溃。

阶段 3 不验收完整 Admin 后台、Skill 上传、Jar 热加载、Team 编排、nginx 部署和真实模型 token 级流式输出。

## 2. 启动前提

本地需要启动四类服务：

```text
PostgreSQL
Web:      http://localhost:8080
Gateway:  http://localhost:8081
Frontend: http://localhost:5173
```

前端启动命令：

```powershell
cd D:\study\蓝山最终考核项目\agent-platform\agent-platform-frontend
npm run dev
```

前端只请求 Web 的 `/api`，Vite 会代理到 `http://localhost:8080`。浏览器 Chat 请求也是先进 Web，再由 Web 转发 Gateway。

## 3. 自动化检查

在 `agent-platform-frontend` 目录执行：

```powershell
npm test -- pagination-utils token-usage-utils trace-utils tool-filters parse-sse
npm run build
npm run lint
```

通过标准：

- 测试全部通过。
- build 退出码为 0。
- lint 退出码为 0。
- Vite chunk 超 500KB warning 可以接受，不阻塞阶段 3。

## 4. 登录与注册

### 4.1 登录

访问：

```text
http://localhost:5173/login
```

使用默认账号：

```text
username: admin
password: admin123
```

验收标准：

- 登录成功后进入控制台。
- 刷新页面后仍保持登录态。
- 顶部状态栏能显示当前用户。
- 退出登录后回到登录页。

### 4.2 注册

在登录页点击注册入口，输入：

```text
displayName: 任意显示名
username: 任意未使用用户名
password: 至少 6 位
```

验收标准：

- 注册成功后自动登录。
- 进入控制台后能看到当前用户。
- 重复用户名时显示后端错误，不崩溃。

## 5. Application / API Key

进入：

```text
Applications
```

操作：

1. 创建一个 Application。
2. 查看创建成功弹窗。
3. 复制一次性 API Key。
4. 关闭弹窗后刷新页面。
5. 查看 API Key 列表。
6. 撤销一个 API Key。

验收标准：

- Application 创建成功。
- API Key 明文只在创建成功时展示一次。
- 刷新后只能看到 keyPrefix，不能看到明文。
- 撤销后状态刷新。
- 空列表时有明确空状态。

## 6. Profile

进入：

```text
Profiles
```

操作：

1. 选择 Application。
2. 创建 Profile。
3. 查看 Profile 详情。
4. 打开工具绑定区域。
5. 勾选 / 取消 Skill。
6. 勾选 / 取消 MCP Tool。
7. 保存绑定。

验收标准：

- 没有 Application 时提示先创建 Application。
- 能基于 Application 创建 Profile。
- Profile 创建后能出现在列表中。
- Skill / MCP Tool 绑定保存后，详情能反映最新绑定。
- 无可用工具时显示空状态。

## 7. Tools

进入：

```text
Tools
```

操作：

1. 查看 Skills 列表。
2. 查看 MCP Tools 列表。
3. 使用 scope / status 筛选。
4. 使用搜索框搜索。

验收标准：

- 页面能展示已有 Skill 和 MCP Tool。
- 筛选和搜索不会导致页面崩溃。
- 阶段 3 不出现 Jar Skill 上传入口。
- 空数据时有明确提示。

## 8. Chat

进入：

```text
Agent Chat
```

操作：

1. 选择 Application。
2. 选择 Agent 模式。
3. 选择 Profile。
4. 输入一条普通消息并发送。
5. 观察右侧 Runtime 面板。
6. 点击 Stop 测试中断。
7. 切换 Direct model 模式，选择 Model Config 后发送。

验收标准：

- 没有 Application 时不能发送，并提示原因。
- Agent 模式没有 Profile 时不能发送，并提示原因。
- Direct model 模式没有 Model Config 时不能发送，并提示原因。
- 发送后按钮切换为 Stop。
- 流式期间不能重复提交。
- Runtime 面板能展示事件、状态、Trace ID。
- 后端错误时页面退出 streaming 状态并显示错误。

说明：

当前后端是 SSE 事件流，不是模型 token 级流式。浏览器可能看到 `thinking -> message -> done`，但 `message` 内容通常一次性返回。token 级流式属于后续增强。

## 9. Trace

进入：

```text
Traces
```

操作：

1. 查看 Trace 列表。
2. 使用 Application / Status / Entrypoint 筛选。
3. 点击一条 Trace。
4. 查看右侧 Trace Detail。
5. 点击不同 Span。
6. 使用 Previous / Next 翻页。

验收标准：

- 能看到 Trace 列表或明确空状态。
- 筛选后列表刷新。
- 分页可查看超过 20 条的数据。
- Trace Detail 展示 traceId、status、latency、tokens、mode。
- Span timeline 按时间排序。
- Span detail 展示 attributes / error。
- `MODEL_ERROR` 或 Trace 缺失时页面不崩溃，显示缺失提示。

## 10. Token Usage

进入：

```text
Token Usage
```

操作：

1. 查看汇总指标。
2. 查看 Prompt / Completion 拆分。
3. 查看 Top models。
4. 查看 Usage records。
5. 使用 Application / Model Config 筛选。
6. 使用 Previous / Next 翻页。

验收标准：

- 能看到 token 汇总或明确空状态。
- estimated / real usage 区分清楚。
- 明细表展示 traceId、model、prompt、completion、total、source、createdAt。
- 分页可查看超过 20 条的数据。
- 没有数据时提示去发起 Chat。

## 11. 路由与权限

操作：

1. 未登录访问 `/`。
2. 未登录访问 `/chat`。
3. 登录后访问 Dashboard。
4. 点击左侧导航切换页面。
5. 退出登录。
6. 再次访问受保护页面。

验收标准：

- 未登录访问受保护页面会跳转到 `/login`。
- 登录后能回到目标页面。
- 退出登录后清理 token。
- 普通用户不应看到 Admin 导航项。

## 12. 异常场景

建议至少测以下异常：

| 场景 | 操作 | 期望 |
|---|---|---|
| 登录失败 | 输入错误密码 | 显示后端错误 |
| 无 Application | 新用户直接进 Chat | 提示先创建 Application |
| 无 Profile | Agent 模式发送 | 禁止发送并提示 |
| 敏感数据 | Chat 输入手机号 / 邮箱 / sk- 风格 Key | Gateway 返回阻断，前端显示安全策略错误 |
| 配额不足 | 使用低额度配置触发 429 | 前端显示 quota 不足 |
| Gateway 未启动 | 只启动 Web 后发送 Chat | Chat 显示流失败，不永久 loading |
| Trace 缺失 | 查询不存在 traceId 或 MODEL_ERROR 场景 | 页面显示缺失提示 |

## 13. 视觉检查

桌面端：

```text
1440px 宽度
```

```text
390px 或 430px 宽度
```

- 左侧导航和内容区不重叠。
- 表格可以横向滚动，不挤爆页面。
- 按钮文字不溢出。
- Dialog 在窄屏下可读。
- loading / empty / error 状态风格统一。
- 深色主题对比度足够。
