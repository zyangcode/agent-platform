## 2. 已修复问题

### 2.1 Dashboard Trace / Token 查询参数

问题：

Dashboard 调用 `/api/traces` 和 `/api/token-usages/summary` 时最初没有传 `applicationId`。后端参数虽然是 optional，不会导致 400，但首页语义会变成不明确的全局/默认查询，和其他页面“按应用查看”的逻辑不一致。

核心思路：

Dashboard 应先根据应用列表解析当前有效 Application，再把 `applicationId` 带入 Trace 和 Token 查询。没有应用时保持空查询和空状态。

核心方案：

已修复。新增：

```text
agent-platform-frontend/src/features/dashboard/dashboard-selection-utils.ts
agent-platform-frontend/src/features/dashboard/dashboard-selection-utils.test.ts
```

`getDashboardData()` 现在先加载 `/applications`，按上次选择或第一个有效应用确定 `applicationId`，再带入 `/traces` 和 `/token-usages/summary`。

问题：

Chat 和 Profiles 曾各自封装同一组后端接口，存在未来签名不一致风险。

核心思路：

跨 feature 复用的基础查询接口放到 `lib/api`。

核心方案：

### 2.8 后端分页插件缺失导致列表接口分页失效

问题：

真实联调时发现 `/api/applications`、`/api/profiles`、`/api/traces`、`/api/token-usages` 返回了 records，但 `total = 0`、`totalPages = 0`，并且 `pageSize` 没有生效。Trace 和 Token Usage 即使请求 `pageSize = 5`，仍会返回超过 5 条记录。

核心思路：

各 Service 已经使用 MyBatis-Plus `selectPage(Page.of(pageNo, pageSize), wrapper)`，问题不在业务分页参数，而是 MyBatis-Plus 分页拦截器没有注册。当前 `MybatisPlusConfiguration` 只做了 `@MapperScan`，没有注册 `MybatisPlusInterceptor`；同时 MyBatis-Plus 3.5.12 的分页 SQL 解析依赖需要显式引入 `mybatis-plus-jsqlparser`。

核心方案：

已补充：

```text
pom.xml
agent-platform-core/pom.xml
agent-platform-core/src/main/java/com/ls/agent/core/support/persistence/MybatisPlusConfiguration.java
agent-platform-core/src/test/java/com/ls/agent/core/persistence/MybatisPlusConfigurationTest.java
```

修复后注册 PostgreSQL `PaginationInnerInterceptor`，并增加配置测试防止回归。已通过后端全量 `mvn.cmd -s .mvn/settings.xml test`。

### 2.10 Chat 历史会话新建与删除闭环

核心思路：

核心方案：

## 3. 已知问题与暂缓项

### 3.1 Chat 不是 token 级流式输出

问题：

前端使用 `ReadableStream` 消费 SSE，但后端当前是等 `AgentRuntimeService.run()` 完成后一次性写出完整 `message` 事件。

核心思路：

当前实现是“事件级 SSE”，不是“模型 token 级 streaming”。前端能看到 `thinking -> message -> done`，但 `message` 内容通常一次性出现。

核心方案：

阶段 3 不改后端模型流式。后续有两种增强：

- 真流式：OpenAI-compatible provider 使用 `stream=true`，后端逐 chunk 转发 SSE。
- 假流式：前端收到完整 message 后做打字机效果，仅改善视觉体验。

建议放到阶段 4 或后续治理增强，不阻塞阶段 3 Demo。

### 3.2 ECharts 已安装但 Token Usage 未使用

问题：

前端依赖中有 `echarts`，但 Token Usage 当前使用 CSS 条形图。

核心思路：

阶段 3 的 Token 看板只要求基础汇总、明细和简单 TOP。强行引入 ECharts 会增加页面复杂度和 bundle 体积。

核心方案：

暂不使用 ECharts。当前 CSS 图表满足 MVP。复杂趋势图、多维图表和成本折算后续再做。

### 3.3 Vite build 存在 chunk 超 500KB warning

问题：

`npm run build` 通过，但 Vite 提醒部分 chunk 超过 500KB。

核心思路：

这是体积优化提醒，不是构建失败。当前阶段 3 不做 code splitting 优化，避免扩大范围。

核心方案：

记录为增强项。后续可按路由做 lazy loading，或移除未使用的大型依赖。

### 3.4 Admin 后台仍不是完整 CRUD

问题：

`/admin/models` 已可创建模型供应商和模型配置；`/admin/security`、`/admin/users` 已不再是占位页，但仍是只读收口页，不是完整后台管理系统。

核心思路：

阶段 3 是前端控制台 MVP，不是完整 Admin 系统。安全策略 CRUD、用户 CRUD、角色分配、禁用/重置、审计检索等仍需要后端接口和权限设计支撑。

核心方案：

阶段 3 保持当前边界：

- Admin Model Configs：可用，可创建 Provider / Model Config。
- Admin Security：只读展示已实现治理链和暂缓项。
- Admin Users：只读展示当前账号和角色模型。
- Admin 路由已加前端角色守卫，后端 Admin API 仍由 Spring Security 保护。

### 3.5 Skill 上传 / Jar 热加载 UI 未实现

问题：

阶段 3 Tools 页面只展示 Skill 和 MCP Tool，不支持上传 Skill 或 Jar 热加载。

核心思路：

Jar Skill 热加载是后端能力和后续高阶管理能力，不属于阶段 3 前端 MVP 的阻塞项。

核心方案：

阶段 3 明确暂缓。后续如进入 Skill 市场增强，再设计上传、校验、状态流转、审核和回滚 UI。
