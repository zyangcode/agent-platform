## 2. 已修复问题

### 2.1 Dashboard Trace / Token 查询参数

问题：

早期审查认为 Dashboard 调用 `/api/traces` 和 `/api/token-usages/summary` 缺少 `applicationId` 会导致空数据。

核心思路：

重新对照后端 Controller，当前两个接口的 `applicationId` 都是 `required = false`。

核心方案：

无需修复。Dashboard 查询当前用户可见范围内的最近 Trace 和 Token 汇总是可接受设计。

### 2.2 listModelConfigs / listProfiles 重复封装

问题：

Chat 和 Profiles 曾各自封装同一组后端接口，存在未来签名不一致风险。

核心思路：

跨 feature 复用的基础查询接口放到 `lib/api`。

核心方案：

已抽到：

```text
agent-platform-frontend/src/lib/api/model-configs.ts
agent-platform-frontend/src/lib/api/profiles.ts
```

Chat 和 Profiles 统一引用共享 API。

### 2.3 Chat 流式期间重复提交

问题：

流式输出期间，用户可能通过快捷键或重复触发再次提交，造成并发 Chat 请求。

核心思路：

UI 层禁用发送按钮之外，业务提交函数也要防御。

核心方案：

`ChatPage.sendMessage()` 增加 `runtimeStatus === 'streaming'` guard。

### 2.4 登录页缺少注册入口

问题：

阶段 3 文档和 MVP 设计要求用户通过登录页注册或登录，但前端最初只有登录。

核心思路：

不新增复杂注册页，直接在登录卡片内做登录 / 注册切换。

核心方案：

`LoginPage` 接入 `/api/auth/register`，注册成功后自动登录。

### 2.5 FilterSelect 三处重复

问题：

`ToolsPage`、`TraceListPage`、`TokenUsagePage` 曾各自定义相同的本地 `FilterSelect`。

核心思路：

这是通用表单控件，应该归入共享 UI 组件。

核心方案：

新增：

```text
agent-platform-frontend/src/components/ui/filter-select.tsx
```

三处页面统一使用共享组件。

### 2.6 Trace / Token Usage 无分页

问题：

Trace 和 Token Usage 列表曾固定 `pageNo = 1`，超过 20 条无法访问。

核心思路：

阶段 3 不需要复杂页码器，但至少要有 Previous / Next 和当前页信息。

核心方案：

新增：

```text
agent-platform-frontend/src/components/ui/pagination-controls.tsx
agent-platform-frontend/src/components/ui/pagination-utils.ts
agent-platform-frontend/src/components/ui/pagination-utils.test.ts
```

Trace 和 Token Usage 页面已接入分页。

### 2.7 Dashboard 初始化和重试逻辑重复

问题：

Dashboard 初始加载和 Retry 曾各有一套相似逻辑。

核心思路：

保留 `fetchDashboardData()` 作为数据获取函数，页面加载和 Retry 共用刷新路径。

核心方案：

已清理 `ConsoleHomePage` 中重复加载逻辑。

### 2.8 后端分页插件缺失导致列表接口分页失效

问题：

真实联调时发现 `/api/applications`、`/api/profiles`、`/api/traces`、`/api/token-usages` 返回了 records，但 `total = 0`、`totalPages = 0`，并且 `pageSize` 没有生效。Trace 和 Token Usage 即使请求 `pageSize = 5`，仍会返回超过 5 条记录。

核心思路：

各 Service 已经使用 MyBatis-Plus `selectPage(Page.of(pageNo, pageSize), wrapper)`，问题不在业务分页参数，而是 MyBatis-Plus 分页拦截器没有注册。当前 `MybatisPlusConfiguration` 只做了 `@MapperScan`，没有注册 `MybatisPlusInterceptor`；同时 MyBatis-Plus 3.5.12 的分页 SQL 解析依赖需要显式引入 `mybatis-plus-jsqlparser`。

核心方案：

已补充：

```text
pom.xml
agent-platform-core/pom.xml
agent-platform-core/src/main/java/com/ls/agent/core/support/persistence/MybatisPlusConfiguration.java
agent-platform-core/src/test/java/com/ls/agent/core/persistence/MybatisPlusConfigurationTest.java
```

修复后注册 PostgreSQL `PaginationInnerInterceptor`，并增加配置测试防止回归。已通过后端全量 `mvn.cmd -s .mvn/settings.xml test`。

## 3. 已知问题与暂缓项

### 3.1 Chat 不是 token 级流式输出

问题：

前端使用 `ReadableStream` 消费 SSE，但后端当前是等 `AgentRuntimeService.run()` 完成后一次性写出完整 `message` 事件。

核心思路：

当前实现是“事件级 SSE”，不是“模型 token 级 streaming”。前端能看到 `thinking -> message -> done`，但 `message` 内容通常一次性出现。

核心方案：

阶段 3 不改后端模型流式。后续有两种增强：

- 真流式：OpenAI-compatible provider 使用 `stream=true`，后端逐 chunk 转发 SSE。
- 假流式：前端收到完整 message 后做打字机效果，仅改善视觉体验。

建议放到阶段 4 或后续治理增强，不阻塞阶段 3 Demo。

### 3.2 ECharts 已安装但 Token Usage 未使用

问题：

前端依赖中有 `echarts`，但 Token Usage 当前使用 CSS 条形图。

核心思路：

阶段 3 的 Token 看板只要求基础汇总、明细和简单 TOP。强行引入 ECharts 会增加页面复杂度和 bundle 体积。

核心方案：

暂不使用 ECharts。当前 CSS 图表满足 MVP。复杂趋势图、多维图表和成本折算后续再做。

### 3.3 Vite build 存在 chunk 超 500KB warning

问题：

`npm run build` 通过，但 Vite 提醒部分 chunk 超过 500KB。

核心思路：

这是体积优化提醒，不是构建失败。当前阶段 3 不做 code splitting 优化，避免扩大范围。

核心方案：

记录为增强项。后续可按路由做 lazy loading，或移除未使用的大型依赖。

### 3.4 Admin 页面仍是占位

问题：

`/admin/models`、`/admin/security`、`/admin/users` 当前仍为 Placeholder。

核心思路：

阶段 3 是前端控制台 MVP，不是完整 Admin 系统。Admin 模型管理、安全策略管理、用户管理属于后续增强。

核心方案：

保持占位。左侧导航已按角色控制 Admin 菜单显示。

### 3.5 Skill 上传 / Jar 热加载 UI 未实现

问题：

阶段 3 Tools 页面只展示 Skill 和 MCP Tool，不支持上传 Skill 或 Jar 热加载。

核心思路：

Jar Skill 热加载是后端能力和后续高阶管理能力，不属于阶段 3 前端 MVP 的阻塞项。

核心方案：

阶段 3 明确暂缓。后续如进入 Skill 市场增强，再设计上传、校验、状态流转、审核和回滚 UI。

### 3.6 MODEL_ERROR 时 Trace root 可能缺失

问题：

阶段 2 已知遗留：某些模型错误路径可能没有写入 `trace_roots`。

核心思路：

Trace 是诊断数据，不能阻断 Chat 主链路。阶段 3 前端必须容忍 Trace 缺失。

核心方案：

Trace 页面已将缺失和详情加载失败作为正常 error / empty 状态处理。后端容错语义后续再统一增强。

### 3.7 真实浏览器联调尚未完成

问题：

代码级检查和自动化验证已完成，但尚未在用户本机完整启动 PostgreSQL、Web、Gateway、Frontend 后跑浏览器主流程。

核心思路：

阶段 3 完成标准应以 Demo 流程能否稳定跑通为准。

核心方案：

按 `04-阶段3手工验收清单.md` 执行真实联调。发现问题后补充到本文，并按优先级修复。
