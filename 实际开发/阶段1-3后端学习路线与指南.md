## 一、项目全景架构（先看懂地图）

### 1.1 四个 Maven 模块的职责

```
agent-platform/
├── agent-platform-common/    ← 纯工具层：错误码、统一响应、异常
├── agent-platform-core/      ← 核心业务层：13 个逻辑包，所有表和业务逻辑
├── agent-platform-web/       ← 浏览器入口：JWT 鉴权、REST API、SSE 透传
└── agent-platform-gateway/   ← AI 调用入口：治理链、OpenAI 兼容接口
```

**依赖方向（只能从上往下）：**

```
web ──HTTP调用──→ gateway ──Maven依赖──→ core ──→ common
 │                                           │
 └────────── Maven依赖（管理类请求）──────────┘
```

关键约束：
- `web` 和 `gateway` **不能拥有业务表**，表全在 `core` 里
- `web` 调 `gateway` 走 **HTTP 内部接口**，不是 Maven 依赖
- `gateway` 只调 `core.xxx.api` 接口，**不能直接访问 Mapper**

### 1.2 core 内部 13 个逻辑包

```
core.identity/     用户、角色、Application、API Key
core.model/        模型供应商、模型配置、调用入口
core.profile/      Agent Profile 管理
core.skill/        Skill 注册、执行
core.mcp/          MCP 工具
core.agent/        Agent ReAct 主循环（编排层）
core.memory/       长期记忆
core.context/      上下文组装（不拥有表）
core.trace/        Trace 链路记录
core.quota/        配额预扣 + Token 用量
core.security/     敏感数据扫描
core.alert/        告警事件
core.team/         多 Agent 协作（阶段 4）
```

每个包的标准内部结构：

```
core.xxx/
├── api/         对外暴露的接口
├── application/ 接口实现（DefaultXxxService）
├── entity/      数据库实体（有持久化职责的包才有）
├── mapper/      MyBatis-Plus Mapper（有持久化职责的包才有）
├── command/     入参对象（不可变 Record）
├── dto/         出参对象（不可变 Record）
└── internal/    包内私有，禁止跨包调用
```

**核心规则**：跨包调用只能通过 `api/` 接口 + `command/` + `dto/`，绝对禁止跨包直接访问 `entity/` `mapper/` `internal/`。像 `context` 这种不拥有表的包，不需要强行创建 `entity/mapper`。

## 二、阶段 1：基础 Agent 闭环（5 个 Flyway 迁移 + 15 个 Step）

阶段 1 目标：从零搭建到"能通过 Web → Gateway → Core 发起 Agent 对话并 SSE 返回"。

### Step 1：Maven 多模块工程

**学什么**：父 POM 的 `dependencyManagement` / `pluginManagement` 统一版本管理。

关键文件：根 `pom.xml`，四个子模块的 `pom.xml`。

要点：
- 父 POM 只管理版本，不放实际依赖（用 `dependencyManagement`）
- Spring Boot BOM + Spring AI BOM 统一在父 POM 导入
- 业务依赖放在各子模块的 `dependencies` 里

### Step 2：common 基础能力

**学什么**：统一错误码体系 + 统一响应结构。

关键文件：
```
agent-platform-common/src/main/java/com/ls/agent/common/
├── error/ErrorCode.java        ← 枚举式错误码（40001/50001/60001 分域）
├── error/BizException.java     ← 业务异常（携带 code + message）
├── response/ApiResponse.java   ← 统一响应 {success, code, message, data}
└── response/PageResult.java    ← 分页响应 {records, pageNo, pageSize, total, totalPages}
```

**设计要点**：
- `common` 只放纯工具类，**绝不放** Entity/DTO/Service
- `ErrorCode` 按域分段：4xxxx 客户端错误、5xxxx 服务端错误、6xxxx 第三方错误
- `BizException` 是整个项目的唯一业务异常类型，全局异常处理器统一捕获

### Step 3：Flyway 数据库迁移

**学什么**：用 Flyway 管理数据库版本，从 V001 到 V005 逐步建表。

关键文件：
```
agent-platform-core/src/main/resources/db/migration/
├── V001__init_identity.sql    ← tenants, users, roles, user_roles, applications, api_keys
├── V002__init_model_profile.sql ← model_providers, model_configs, agent_profiles, profile_skills, profile_mcp_tools
├── V003__init_skill_mcp.sql   ← skills, skill_versions, mcp_servers, mcp_tools
├── V004__init_agent_memory.sql ← conversations, conversation_messages, memories
└── V005__init_seed_data.sql   ← 默认 tenant、admin 用户、3 个内置 Skill、1 个 MCP 工具
```

**设计要点**：
- 迁移文件按依赖顺序编号（V001 → V002 → ...）
- Seed 数据放在最后，因为它依赖前面所有表
- `api_keys` 只存 `key_hash` + `key_prefix`，**不存明文**

### Step 4：Entity / Mapper 生成

**学什么**：MyBatis-Plus 的 Entity 写法 + Mapper 写法 + JSON 字段处理。

关键文件示例：
```
core/identity/entity/UserEntity.java    ← @TableName + @TableId + 字段映射
core/identity/mapper/UserMapper.java    ← 继承 BaseMapper<UserEntity>
```

**Entity 写法要点**：
```java
// 关键注解
@TableName(value = "users", autoResultMap = true)  // jsonb 字段必须开启
public class UserEntity {
    @TableId(type = IdType.AUTO)
    private Long id;
    // jsonb 字段必须声明 TypeHandler
    @TableField(typeHandler = JsonNodeTypeHandler.class)
    private JsonNode metadata;
}
```

**Mapper 写法**：
```java
@Mapper
public interface UserMapper extends BaseMapper<UserEntity> {
    // 简单 CRUD 用 MyBatis-Plus 自带方法
    // 复杂查询写自定义 SQL（用 @Select 或 XML）
}
```

### Step 5：ArchUnit 架构边界测试

**学什么**：用 ArchUnit 把包访问规则变成**测试失败**，而不是靠人工纪律。

关键文件：各模块的 `ArchitectureRulesTest`，包括：

```
agent-platform-common/src/test/java/com/ls/agent/common/ArchitectureRulesTest.java
agent-platform-core/src/test/java/com/ls/agent/core/ArchitectureRulesTest.java
agent-platform-web/src/test/java/com/ls/agent/web/ArchitectureRulesTest.java
agent-platform-gateway/src/test/java/com/ls/agent/gateway/ArchitectureRulesTest.java
```

核心规则：
```java
// 规则 1：web/gateway 禁止访问 core.*.mapper
noClasses()
    .that().resideInAnyPackage("com.ls.agent.web..", "com.ls.agent.gateway..")
    .should().accessClassesThat().resideInAnyPackage("com.ls.agent.core.*.mapper..")
    .check(classes);

// 规则 2：core 内部跨业务包禁止访问其他包的 entity/mapper/internal
// 规则 3：common 禁止包含业务 Entity/DTO/Service
```

**为什么重要**：没有 ArchUnit，包边界全靠 Code Review，人一多就失控。

### Step 6：identity（注册/登录/JWT/API Key）

**学什么**：Spring Security + JWT 的完整流程。

关键文件：
```
web/security/
├── SecurityConfig.java              ← HttpSecurity 配置
├── JwtAuthenticationFilter.java     ← OncePerRequestFilter 拦截器
├── JwtTokenService.java             ← 生成/校验 JWT
└── CurrentUser.java                 ← @AuthenticationPrincipal 获取当前用户

core/identity/application/
├── DefaultAuthService.java          ← 注册/登录逻辑
└── DefaultApplicationService.java   ← Application CRUD + API Key 生成
```

**核心流程**：
```
用户 POST /api/auth/login {username, password}
  → AuthService.login() 校验密码（BCrypt）
  → JwtTokenService.generate() 生成 JWT（含 userId, roles）
  → 返回 {accessToken, expiresIn}

后续请求带 Authorization: Bearer <jwt>
  → JwtAuthenticationFilter 解析 JWT
  → 注入 SecurityContext
  → Controller 通过 @AuthenticationPrincipal CurrentUser 获取
```

**API Key 设计**：
- 绑定 Application，不直接绑定 User
- 创建时生成 `sk-` 前缀随机串，通过 BCrypt 哈希存储
- **明文只在创建时返回一次**，后续无法取回

### Step 7：model（模型供应商 + 统一调用入口）

**学什么**：多供应商模型调用的抽象层。

关键文件：
```
core/model/api/ModelInvokeService.java       ← 对外门面接口
core/model/application/DefaultModelInvokeService.java ← 内部委托给 Provider
core/model/provider/
├── ModelProvider.java              ← 供应商抽象接口
├── ModelProviderRegistry.java      ← 按 providerType 查找 Provider
├── MockModelProvider.java          ← Mock 实现（无 API Key 也能跑）
└── OpenAiCompatibleProvider.java   ← 真实 OpenAI 兼容调用
```

**设计要点**：
- Spring AI 只封装在 `core.model` 内部，不向 agent/gateway 暴露
- `MockModelProvider` 确保**无外部 API Key 也能跑通全链路**
- 模型 API Key 通过 `SecretEncryptor` 加密存储

### Step 8：profile（Agent Profile CRUD）

**学什么**：标准的三层 CRUD 模式（Controller → Application Service → Mapper）。

关键文件：
```
web/controller/ProfileController.java       ← REST 接口
web/dto/CreateProfileRequest.java           ← Web 层入参
core/profile/api/ProfileService.java        ← 接口
core/profile/application/DefaultProfileService.java ← 实现
core/profile/command/CreateProfileCommand.java     ← Core 层入参
core/profile/dto/ProfileDTO.java            ← Core 层出参
```

**调用链示例**：
```
POST /api/profiles {name, modelConfigId, ...}
  → ProfileController.create()
  → 从 CurrentUser 取 tenantId, userId
  → 构造 CreateProfileCommand
  → DefaultProfileService.createProfile(command)
    → 校验 applicationId 归属（调 ApplicationService）
    → 构造 AgentProfileEntity → profileMapper.insert()
    → 返回 ProfileDTO
  → 包装成 ApiResponse<ProfileDTO>
```

**设计要点**：
- Controller 入参（`CreateProfileRequest`）和 Core 入参（`CreateProfileCommand`）**是两套**，Controller 负责转换
- 归属权校验在 Application Service 层做：`applicationService.ensureApplicationOwned()`
- 状态机：`DRAFT → PUBLISHED → DISABLED`

关键文件：

core/mcp/api/McpToolExecutor.java       ← 执行 MCP 工具

**MCP 工具执行接口**：

### Step 12：agent（ReAct 主循环）★ 核心

**学什么**：自研 ReAct while 循环——这是整个项目的核心编排逻辑。

关键文件：`core/agent/application/DefaultAgentRuntimeService.java`（约 500 行）

**完整执行流程**：

```
AgentRuntimeService.run(command)
  │
  ├─ 1. validate(command) — 参数校验
  │
  ├─ 2. ensureConversation(command) — 创建/复用会话
  │
  ├─ 3. saveMessage(user) — 保存用户消息
  │
  ├─ 4. contextBuilder.build() — 组装上下文（调 profile/skill/mcp/memory）
  │      └─ span: context.build
  │
  ├─ 5. runModelLoop(command, context) — ReAct 循环 ← 核心
  │      │
  │      ├─ 5a. detectDirectToolCall() — 关键词匹配直接调工具（本阶段实现）
  │      │      "天气北京" → 直接调 @skill:weather
  │      │      "128*36+59" → 直接调 @skill:calculator
  │      │      非工具请求 → 直接走模型
  │      │
  │      └─ 5b. for step < MAX_AGENT_STEPS (3):
  │            ├─ invokeModel() — 调 LLM
  │            │    └─ span: model.invoke + record token usage
  │            ├─ parseToolCall() — 解析模型回复中的 @skill:xxx {...}
  │            │    如果模型说 "@skill:calculator {...}" → 提取为 ToolCall
  │            ├─ executeTool() — 执行工具
  │            │    └─ span: tool.execute
  │            └─ 如果没有工具调用 → 返回最终回复
  │
  ├─ 6. saveMessage(assistant) — 保存 AI 回复
  │
  ├─ 7. memoryWriteService.record() — 写入长期记忆（SUMMARY 类型）
  │
  └─ 8. return AgentRunResult(conversationId, message, usage, toolEvents)
```

**工具调用格式**（模型回复中识别）：
```
@skill:calculator {"expression": "128 * 36 + 59"}
@mcp:filesystem {"path": "/tmp/readme.txt"}
```

**设计要点**：
- `agent` 是**编排者**，不是所有逻辑的拥有者：查 Skill 调 `SkillRegistry`，查模型调 `ModelInvokeService`
- 所有 Span 写入失败**不能影响主链路**（`safeStartSpan` / `safeFinishSpan` 内部 catch）
- Token 用量记录失败同样不影响主链路

### Step 13：gateway 空壳

**学什么**：Gateway 作为 AI 请求统一入口，校验 + 路由 + SSE 输出。

关键文件：`gateway/controller/InternalAiController.java`（约 300 行）

**三个入口**：

| 接口 | 鉴权方式 | 用途 |
|------|---------|------|
| `GET /internal/ai/stream-test` | X-Internal-Token | 测试固定 SSE 流 |
| `POST /internal/ai/chat/stream` | X-Internal-Token | Web 转发 AI 请求 |
| `POST /api/ai/chat/stream` | Bearer sk-xxx | 外部 API Key 请求 |

**SSE 输出方式**：Spring MVC `StreamingResponseBody`

```java
// 每写一个事件就 flush，保证实时性
output.write(("event: thinking\n").getBytes(StandardCharsets.UTF_8));
output.write(("data: " + json + "\n\n").getBytes(StandardCharsets.UTF_8));
output.flush();
```

**agent_mode 路由**：
- `agent`（默认）→ `AgentRuntimeService.run()` → 完整 ReAct 链路
- `none` → `ModelInvokeService.invoke()` → 纯模型透传

### Step 14：Web SSE 透传

**学什么**：Web 如何**边读边写**透传 Gateway SSE，不被缓冲。

关键文件：`web/client/HttpGatewayClient.java`

**SSE 三层透传架构**：
```
浏览器 ←SSE← Web:8080 ←HTTP边读边写← Gateway:8081 ←StreamingResponseBody← Core
```

**核心实现**：Web 用 JDK `HttpClient` 发送异步请求到 Gateway，读一行写一行 flush 一行。

**关键配置**：
- Web 用 `HttpClient` 连接 Gateway，设置 3 秒连接超时
- `SecurityConfig` 放行 `DispatcherType.ASYNC`，避免异步 Dispatch 被安全链二次拦截
- 连接异常时写 `event: error` 到 SSE 流

### Step 15：总体验收

手工验证：注册 → 登录 → 创建 Application → 创建 Profile → Chat SSE → 查看数据库落库。

## 三、阶段 2：AI Infra 网关治理（4 个 Flyway 迁移 + 4 个治理模块）

阶段 2 在 Gateway 空壳上补全治理链，使其变成真正的 **生产级 AI 网关**。

**治理链全景（当前实现）**

当前 Gateway 治理逻辑分布在 3 个独立 Filter + InternalAiController 中：

```
请求进入 Gateway
  │
  ├─ 1. 生成 traceId（tr_xxxxxxxx）
  │
  ├─ 2. SensitiveDataFilter — 敏感数据扫描 → 命中则 403 阻断
  │
  ├─ 3. QuotaFilter — Token 配额预扣 → 超限则 429
  │
  ├─ 4. AgentRuntimeService.run() — 执行 Agent
  │      └─ 内部记录 span + token usage（由 InternalAiController 编排）
  │
  ├─ 5. QuotaFilter.commit() — 按真实用量结算 / release() — 失败释放
  │
  └─ 6. AlertFilter — 异常时记录告警事件
```

设计目标（CLAUDE.md 定义）为 7 个独立 Filter 的完整链：
```
TenantFilter → TraceFilter → SensitiveDataFilter → QuotaFilter
→ RuntimeInvokeFilter → SensitiveResponseFilter → TokenRecordFilter → AlertFilter
```
当前 Trace/Token 的 span 创建和记录逻辑暂嵌在 `InternalAiController` 方法中，未抽成独立的 `TraceFilter` / `TokenRecordFilter`。

### Step 2.1：Trace + Token 用量记录

**V006 迁移**：新建 3 张表
```
trace_roots       ← 一次请求一个 root（traceId, entrypoint, status, latency_ms）
trace_spans       ← 请求内每个步骤一个 span（span_name, span_type, latency_ms, attributes jsonb）
token_usage_logs  ← 每次模型调用一条记录（prompt_tokens, completion_tokens, estimated）
```

**TraceService 接口**：
```java
public interface TraceService {
    TraceRootDTO startRoot(StartTraceRootCommand command);
    void finishRoot(FinishTraceRootCommand command);
    TraceSpanDTO startSpan(StartTraceSpanCommand command);
    void finishSpan(FinishTraceSpanCommand command);
}
```

**Span 类型定义**：
```
gateway.chat_stream     Gateway 收到请求
agent_runtime.run       Core Agent 执行
context.build           组装上下文
model.invoke            调用模型
tool.execute            执行 Skill/MCP
```

**设计要点**：
- traceId 由 Gateway 生成，传给 Core 使用
- Span 写入异常**不影响 Chat 主链路**（所有 safeXxx 方法内部 catch）
- trace_roots 先 `startRoot(RUNNING)`，最终 `finishRoot(SUCCESS/FAILED)`
- `estimated=true` 表示 Token 是本地估算（如 mock-chat），`false` 表示来自模型真实返回

### Step 2.2：Token 配额预扣

**V007 迁移**：新建 2 张表
```
quota_configs       ← 配额配置（日限额/月限额，按 tenant/app/user 维度）
quota_reservations  ← 预扣记录（traceId, estimated_tokens, actual_tokens, status, version）
```

**QuotaService 三方法**：
```java
void reserve(ReserveQuotaCommand)   // 请求前预估 1000 token 预扣，用 version CAS
void commit(CommitQuotaReservationCommand) // 完成后按真实 usage 修正
void release(ReleaseQuotaReservationCommand) // 失败时释放预扣
```

**并发安全**：用 PostgreSQL `version` 字段实现乐观锁（CAS），`UPDATE ... WHERE version = ?`。

**预扣/结算/释放流程**：
```
请求进来 → reserve (status=RESERVED, version=1)
  │
  ├─ 成功 → 调 AgentRuntime
  │          └─ commit (status=COMMITTED, actual_tokens=真实用量, version=2)
  │
  └─ 失败 → release (status=RELEASED, version=2)
```

### Step 2.3：敏感数据治理

**V008 迁移**：`security_events` 表。

**SensitiveDataScanner 接口**：
```java
public interface SensitiveDataScanner {
    List<SensitiveDataFindingDTO> scan(String content, String location);
}
```

**四条正则规则**：
```
PHONE    — 手机号（支持无分隔符中文场景）
EMAIL    — 邮箱
ID_CARD  — 身份证号
API_KEY  — API Key 格式（sk- 前缀）
```

**关键修复**：Java `\b`（word boundary）在中文场景失效，全部替换为 lookahead/lookbehind 实现。

**处理方式**：命中即 403 阻断（BLOCK 模式），同时记录 `security_events`。

### Step 2.4：飞书告警

**V009 迁移**：`alert_events` 表。

**三类告警**：
```
SECURITY_BLOCKED  — 敏感数据被阻断（WARN）
TOKEN_EXCEEDED    — Token 配额超限（WARN）
MODEL_ERROR       — 模型调用失败（ERROR）
```

**告警内容**：traceId + tenantId + applicationId + alertType + level + title + suggestion。

**设计要点**：
- 告警写入失败**不能影响用户 AI 响应**（内部 catch + log.warn）
- 完整飞书消息发送放阶段 5

## 五、贯穿三个阶段的核心设计模式

### 5.1 "api 接口 + application 实现" 两层分离

```
core/skill/api/SkillExecutor.java        ← 接口（对外契约）
core/skill/application/DefaultSkillExecutor.java ← 实现（可替换）
```

这样 agent 包只依赖 `SkillExecutor` 接口，不依赖具体实现。Mock 替换和测试都很方便。

### 5.2 "Repository 抽象" 隔离 Mapper 依赖

当 Agent 需要操作会话数据时，不直接依赖 `ConversationMapper`，而是通过一个 Repository 接口：

```java
// core/agent/api/ConversationRepository.java — 接口（对外契约）
public interface ConversationRepository {
    ConversationEntity findConversationById(Long conversationId);
    void insertConversation(ConversationEntity conversation);
    void insertMessage(ConversationMessageEntity message);
    // ...
}

// core/agent/application/DefaultConversationRepository.java — 实现（内部用 Mapper）
@Service
public class DefaultConversationRepository implements ConversationRepository {
    private final ConversationMapper conversationMapper;  // ← Mapper 被封在实现里
    // ...
}
```

**为什么需要这一层**：项目早期 `DefaultAgentRuntimeService` 直接注入了 `ConversationMapper`，在架构审查中被发现违规（Agent 包不应直接依赖别的包的 Mapper）。抽一个 Repository 接口放在 `api/` 包，实现放在 `application/` 包，就符合了"跨包只能调 api/"的规则。这个改动本身也是 ArchUnit 推动架构改善的真实案例。

### 5.3 "Command → DTO" 不可变对象

入参用 Command Record，出参用 DTO Record，都是不可变对象：

```java
// Command（入参）
public record CreateProfileCommand(Long tenantId, Long ownerUserId, ...) {}

// DTO（出参）
public record ProfileDTO(Long profileId, String name, ...) {}
```

### 5.4 "安全兜底" try-catch 模式

所有非核心路径（Trace、Token、Alert）的写入都用 safe 模式：

```java
private TraceSpanDTO safeStartSpan(...) {
    try {
        return traceService.startSpan(...);
    } catch (Exception ex) {
        return null;  // 不影响主链路
    }
}
```

### 5.5 Flyway 渐进式建表

从不一次性建完所有表，而是按阶段逐步迁移：
```
V001→V005: 阶段 1（18 张核心业务表，其中 V005 为种子数据）
V006:      阶段 2.1 Trace + Token 表
V007:      阶段 2.2 配额预扣表
V008:      阶段 2.3 安全事件表
V009:      阶段 2.4 告警事件表
```

### 5.6 SSE 事件格式

所有 SSE 事件统一格式：
```json
{
  "type": "thinking|action|observation|message|done|error",
  "traceId": "tr_xxx",
  "conversationId": 123,
  "step": 1,
  "content": "消息内容",
  "metadata": {}
}
```

### 5.7 多入口统一路由

无论是浏览器 JWT 还是外部 API Key，最终都进入同一个 Agent Runtime：

```
通道 A: 浏览器 → JWT → Web → HTTP → Gateway → Core AgentRuntime
通道 B: 外部代码 → API Key → Gateway → Core AgentRuntime
```

## 七、常见问题

**Q: 为什么 web 调 gateway 要走 HTTP 而不是 Maven 依赖？**
A: 因为 web 和 gateway 是不同的部署单元。web 面向浏览器（JWT），gateway 面向外部 API 调用（API Key），它们有不同的安全配置和部署策略。HTTP 调用保持了部署独立性。

**Q: 为什么要在 core 内部还用 api/application 两层，同一个 Maven 模块不能直接调吗？**
A: 技术上可以，但这是工程纪律。api/ 层定义了包的对外契约，application/ 是实现细节。一旦跨包直接调 mapper/internal，表结构一变就大面积炸。ArchUnit 会在测试阶段拦住这种调用。

**Q: ReAct 循环为什么不用 LangChain4j 或 Spring AI 的 Agent？**
A: 两个原因：(1) 本项目作为考核项目，需要展示对 Agent 循环的底层理解；(2) 框架的 Agent 循环不透明，出了问题很难排查。自研 ReAct 虽然多写几百行，但每行都可控。

**Q: 为什么 Trace 记录失败不能影响对话？Token 配额也能忽略吗？**
A: 不能混在一起看。Trace Span 主要是诊断数据，写入失败应该尽力记录日志，但不应直接中断用户对话。Token 配额不同，它是网关治理核心能力：请求前的 reserve、完成后的 commit、失败时的 release 都必须可靠处理，不能因为实现方便就静默丢失。
