# 阶段 2 开发计划

> 本文档用于阶段 2 写代码前定边界、定顺序、定验收方式。阶段 2 目标是把阶段 1 已跑通的 Chat 主链路变成可观测、可统计、可排查的链路。

---

## 1. 阶段目标

阶段 2 先完成最小可用可观测闭环：

```text
每次 Chat 请求生成 traceId。
记录一次请求的开始、结束、状态、耗时。
记录 Agent Runtime 和 Model Invoke 的关键 Span。
记录应用、用户、Profile、模型、Provider 维度。
记录 promptTokens、completionTokens、totalTokens。
能在 PostgreSQL 中按 traceId 查询完整链路。
为后续前端 Trace 页面和 OTLP 导出预留接口。
```

阶段 2 不急着做：

```text
完整 SkyWalking / OTLP 接入。
复杂成本计费。
复杂告警。
复杂链路图渲染。
token 级真实流式输出。
多 Provider 自动故障切换。
```

这些能力可以放到阶段 2 后半段或阶段 3。

## 4. 阶段 2 模块划分

建议新增 `core.trace` 包：

```text
agent-platform-core/src/main/java/com/ls/agent/core/trace/api
agent-platform-core/src/main/java/com/ls/agent/core/trace/application
agent-platform-core/src/main/java/com/ls/agent/core/trace/command
agent-platform-core/src/main/java/com/ls/agent/core/trace/dto
agent-platform-core/src/main/java/com/ls/agent/core/trace/entity
agent-platform-core/src/main/java/com/ls/agent/core/trace/mapper
```

职责：

```text
api             对外接口，例如 TraceService、TokenUsageService。
application     默认实现，负责写表和状态流转。
command         写入命令对象。
dto             查询返回对象。
entity          MyBatis-Plus 实体。
mapper          MyBatis-Plus Mapper。
```

不要让 Web/Gateway/AgentRuntime 直接操作 Mapper。业务代码只调用 TraceService。

## 8. 验收标准

自动化测试：

```powershell
mvn.cmd -s .mvn/settings.xml test
```

本地手工验收：

```text
1. 启动 PostgreSQL。
2. 启动 Gateway。
3. 启动 Web。
4. 登录获取 accessToken。
5. 创建 Application。
6. 创建 Profile。
7. 调用 mock-chat。
8. 在 SSE 中拿到 traceId。
9. DBeaver 查询 trace_roots，能看到 SUCCESS。
10. 查询 trace_spans，能看到 agent_runtime 和 model_invoke。
11. 查询 token_usage_logs，能看到 mock token 用量。
12. 调用 DeepSeek。
13. 再查 token_usage_logs，能看到真实模型 token 用量。
```

成功标准：

```text
Chat 响应不受 Trace 写入影响。
Trace 写入失败不能导致模型请求失败。
每个 traceId 至少有 1 条 trace_roots。
agent 模式至少有 agent_runtime、model_invoke 两类 span。
token_usage_logs 能按 applicationId、modelConfigId 聚合。
```

## 9. 风险控制

阶段 2 最容易踩的坑：

```text
在业务主链路里直接写太多 Mapper，导致代码混乱。
Trace 失败反过来影响 Chat 成功。
表字段过细，第一版难以落地。
把 token 级流式和 Trace 同时做，范围失控。
在 Gateway 和 Core 同时生成不同 traceId，链路断开。
```

处理原则：

```text
traceId 由 Gateway 生成。
业务代码只调用 TraceService。
Trace 写入异常先记录日志，不中断 Chat。
先做事件级 Trace，再做 token 级流式。
先保证 DBeaver 能查，再做前端可视化。
```
