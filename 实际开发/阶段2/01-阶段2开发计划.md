# 阶段 2 开发计划

> 本文档用于阶段 2 写代码前定边界、定顺序、定验收方式。阶段 2 目标是把阶段 1 已跑通的 Chat 主链路变成可观测、可统计、可排查的链路。

---

## 1. 阶段目标

阶段 2 先完成最小可用可观测闭环：

```text
每次 Chat 请求生成 traceId。
记录一次请求的开始、结束、状态、耗时。
记录 Agent Runtime 和 Model Invoke 的关键 Span。
记录应用、用户、Profile、模型、Provider 维度。
记录 promptTokens、completionTokens、totalTokens。
能在 PostgreSQL 中按 traceId 查询完整链路。
为后续前端 Trace 页面和 OTLP 导出预留接口。
```

阶段 2 不急着做：

```text
完整 SkyWalking / OTLP 接入。
复杂成本计费。
复杂告警。
复杂链路图渲染。
token 级真实流式输出。
多 Provider 自动故障切换。
```

这些能力可以放到阶段 2 后半段或阶段 3。

## 8. 验收标准

自动化测试：

```powershell
mvn.cmd -s .mvn/settings.xml test
```

本地手工验收：

```text
1. 启动 PostgreSQL。
2. 启动 Gateway。
3. 启动 Web。
4. 登录获取 accessToken。
5. 创建 Application。
6. 创建 Profile。
7. 调用 mock-chat。
8. 在 SSE 中拿到 traceId。
9. DBeaver 查询 trace_roots，能看到 SUCCESS。
10. 查询 trace_spans，能看到 agent_runtime 和 model_invoke。
11. 查询 token_usage_logs，能看到 mock token 用量。
12. 调用 DeepSeek。
13. 再查 token_usage_logs，能看到真实模型 token 用量。
```

成功标准：

```text
Chat 响应不受 Trace 写入影响。
Trace 写入失败不能导致模型请求失败。
每个 traceId 至少有 1 条 trace_roots。
agent 模式至少有 agent_runtime、model_invoke 两类 span。
token_usage_logs 能按 applicationId、modelConfigId 聚合。
```

## 9. 风险控制

阶段 2 最容易踩的坑：

```text
在业务主链路里直接写太多 Mapper，导致代码混乱。
Trace 失败反过来影响 Chat 成功。
表字段过细，第一版难以落地。
把 token 级流式和 Trace 同时做，范围失控。
在 Gateway 和 Core 同时生成不同 traceId，链路断开。
```

处理原则：

```text
traceId 由 Gateway 生成。
业务代码只调用 TraceService。
Trace 写入异常先记录日志，不中断 Chat。
先做事件级 Trace，再做 token 级流式。
先保证 DBeaver 能查，再做前端可视化。
```

## 11. 阶段 2.2 Token 配额预扣结论

阶段 2.2 使用 PostgreSQL 事务版配额预扣，不引入 Redis。

新增范围：

```text
V007__init_quota_reservation.sql
quota_configs
quota_reservations
core.quota.api.QuotaService
gateway.filter.QuotaFilter
```

实现规则：

```text
无 quota_configs 配置时默认允许通过，保持阶段 1/2.1 Demo 可跑。
有 ACTIVE 配置时，先检查 single_request_limit_tokens。
超出单次限额直接返回 HTTP 429，不调用 AgentRuntime。
AgentRuntime 成功后按 AgentRunResult.usage.totalTokens commit。
AgentRuntime 失败时 release。
commit/release 按 traceId + RESERVED + version 做幂等更新。
```

本轮不做：

```text
Redis/Lua 原子预扣。
Admin 配额配置页面。
复杂日/月累计统计扣减。
支付/充值/成本折算。
配额耗尽飞书告警。
```
