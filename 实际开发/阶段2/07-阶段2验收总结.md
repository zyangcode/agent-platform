# 阶段 2 验收总结

## 1. 当前结论

阶段 2 主功能已完成到 2.4，可以进入阶段 3 前的整理阶段。

本阶段已经形成最小治理闭环：

1. Trace 记录一次 Chat 请求的主链路。
2. Token Usage 记录模型调用消耗。
3. Quota 在 Gateway 入口做 Token 预扣和超限阻断。
4. Sensitive Data 在请求入口做敏感数据扫描和阻断。
5. Alert 在关键治理事件发生时写入告警事件。

本阶段不等于完整监控平台，也不等于完整计费系统。它的目标是先把可观测、可统计、可治理的最小闭环跑通。

## 2. 已完成内容

### 2.1 Trace + Token Usage

已完成：

- `trace_roots` 表
- `trace_spans` 表
- `token_usage_logs` 表
- `core.trace.api.TraceService`
- `core.quota.api.TokenUsageService`
- AgentRuntime 中的 agent/context/model span
- `agentMode=none` 直连模型路径中的 `model.invoke` span
- ModelInvokeResult 中的 provider/model/usage 信息
- `agentMode=none` 直连模型路径中的 Token Usage 记录
- Web Trace 查询接口
- Web Token Usage 查询接口

关键边界：

- Trace 写入失败不影响 Chat 主链路。
- Token Usage 写入失败不影响 Chat 主链路。
- `token_usage_logs` 归属 `core.quota`，不是 `core.trace`。

### 2.2 Token 配额预扣

已完成：

- `quota_configs` 表
- `quota_reservations` 表
- `core.quota.api.QuotaService`
- `reserve / commit / release` 三段式预扣流程
- Gateway 入口超限返回 429
- 配额预扣失败时不调用 AgentRuntime

当前能力边界：

- 当前只实现单请求预估上限检查。
- `daily_limit_tokens` 和 `monthly_limit_tokens` 已建模，但暂未做日/月累计扣减。
- Gateway 预估 Token 暂时使用固定值 `1000`。

### 2.3 敏感数据治理

已完成：

- `security_events` 表
- `core.security.api.SensitiveDataScanner`
- `core.security.api.SecurityEventService`
- Gateway 请求入口扫描
- 命中敏感数据后返回 403
- 审计事件只记录类型、位置、hash 和脱敏样例，不记录敏感原文

当前能力边界：

- 请求入口已扫描 Agent 模式的 `message`，以及 `agentMode=none` 直连模型模式的 `messages[].content`。
- 暂不扫描模型响应、工具结果、历史上下文。
- 扫描失败采用 fail-close，返回安全阻断。

### 2.4 告警最小闭环

已完成：

- `alert_events` 表
- `core.alert.api.AlertEventService`
- Gateway `AlertFilter`
- 配额超限记录 `TOKEN_EXCEEDED`
- 敏感数据阻断记录 `SECURITY_BLOCKED`
- AgentRuntime 或模型异常记录 `MODEL_ERROR`
- 告警写入失败不影响原始响应

当前能力边界：

- 当前只写入告警事件。
- 暂不做通知发送、重试 worker、告警规则配置、告警列表接口。

## 4. 架构边界检查

当前已补齐以下架构约束：

- Web 模块不能访问 `core.*.entity`。
- Web 模块不能访问 `core.*.mapper`。
- Gateway 模块不能访问 `core.*.entity`。
- Gateway 模块不能访问 `core.*.mapper`。
- Core 内部不同业务包之间不能跨包访问对方的 Entity/Mapper。
- Mapper 只能存在于 Core 各业务包的 `mapper` 目录下。

之前发现的 Gateway/Web 测试直接 mock Core Mapper 的问题已经修复。

当前 Gateway/Web 只依赖 Core 的 api、command、dto。

## 5. 仍需注意的问题

### 5.1 Trace latency 已修复

`DefaultTraceService.finishRoot` 和 `finishSpan` 已按 `startedAt` 与当前 `endedAt` 计算 `latencyMs`。

验证：

- `DefaultTraceServiceTest` 已覆盖 root latency 计算。
- `DefaultTraceServiceTest` 已覆盖 span latency 计算。
- `agent-platform-core` 模块测试通过。

### 5.2 Quota 还不是完整用量配额系统

当前 Quota 只完成单请求预扣和单次限额阻断。

影响：

- 可以证明 Gateway 治理链能阻断超限请求。
- 不能宣称已经完成日/月累计额度扣减。

建议：

- 阶段 3 Token 看板或阶段 4 治理增强时，再实现日/月累计统计和扣减策略。

### 5.3 Gateway 治理链仍在 Controller 内编排

当前 `InternalAiController` 中串联：

- SensitiveDataFilter
- QuotaFilter
- Trace root
- AgentRuntime
- AlertFilter

影响：

- 阶段 2 可接受。
- 后续继续加入治理能力时，Controller 会继续膨胀。

建议：

- 阶段 3 先不急着重构。
- 如果阶段 3 后 Controller 明显变重，再单独抽 Gateway 治理链或 AgentRuntime Pipeline。

### 5.4 阶段 2 文档存在时间线不一致

`04-阶段2执行顺序.md` 开头仍写着 2.2、2.3、2.4 不放进本轮。

原因：

- 该文档最初只服务 2.1。
- 后续 2.2、2.3、2.4 是单独推进的。

建议：

- 保留旧文档作为执行记录。
- 后续以本文档作为阶段 2 最新验收状态入口。

## 6. 阶段 3 前建议处理

进入阶段 3 前，Trace latency 已修复，两个直连模型路径的小闭环问题也已处理：

1. 敏感数据扫描已覆盖 `agentMode=none` 的 `messages`。
2. `agentMode=none` 已补齐 model span 和 token_usage_logs。

不建议在阶段 3 前做：

- AgentRuntime Pipeline 大重构
- Gateway 完整 Filter 链重构
- 日/月配额完整计费
- 告警通知 worker
- 安全响应扫描

这些都可以作为阶段 3 或阶段 4 的增强项，避免阶段 2 收尾时扩大范围。
