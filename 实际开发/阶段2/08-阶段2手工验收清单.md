# 阶段 2 手工验收清单

> 用途：阶段 2 push 前手工验收。  
> 目标：确认 Gateway 治理链已经形成最小闭环：Trace、Token Usage、Quota、Sensitive Data、Alert 都能在真实接口链路中工作。

## 0. 验收结论先看这里

阶段 2 通过标准：

- 自动化测试 `mvn.cmd -s .mvn/settings.xml test` 通过。
- Web 登录态 Chat 能正常返回 SSE。
- 每次成功 Chat 都能查到 `trace_roots`、`trace_spans`、`token_usage_logs`。
- Trace 查询接口和 Token Usage 查询接口能查到刚才的调用。
- Quota 超限返回 HTTP 429，并且不进入 AgentRuntime。
- 敏感数据命中返回 HTTP 403，并写入 `security_events` 和 `alert_events`。
- `agentMode=none` 直连模型路径也能记录 `model.invoke` span 和 `token_usage_logs`。
- 模型或运行时异常能把 trace 标记为 `FAILED`，并写入 `MODEL_ERROR` 告警。

当前阶段不要求：

- 完整日/月配额扣减。
- 真实告警通知发送 worker。
- 响应内容敏感扫描。
- Gateway Filter 链完整重构。
- AgentRuntime Pipeline 重构。

## 10. Quota 超限验收

阶段 2 的 Quota 只验证单请求预扣和 429 阻断。默认预估 token 是 1000，所以插入一个低于 1000 的限制：

```sql
insert into quota_configs (
  tenant_id, subject_type, subject_id,
  daily_limit_tokens, monthly_limit_tokens, single_request_limit_tokens,
  status
)
values (
  1, 'APPLICATION', 你的applicationId,
  0, 0, 10,
  'ACTIVE'
)
on conflict (tenant_id, subject_type, subject_id)
do update set single_request_limit_tokens = 10,
              status = 'ACTIVE',
              updated_at = current_timestamp;
```

再发起一次 Web Chat：

```powershell
curl.exe -i `
  -X POST `
  -H "Authorization: Bearer $token" `
  -H "Content-Type: application/json" `
  -d $chatBody `
  http://localhost:8080/api/chat/stream
```

通过标准：

```text
HTTP 429
响应包含 QUOTA_EXCEEDED
不会返回正常 message/done
```

检查告警：

```sql
select alert_type, level, notify_status, message
from alert_events
where application_id = 你的applicationId
order by created_at desc
limit 5;
```

通过标准：

```text
能看到 TOKEN_EXCEEDED
notify_status = PENDING
```

恢复配额，避免影响后续测试：

```sql
update quota_configs
set single_request_limit_tokens = 0,
    status = 'DISABLED',
    updated_at = current_timestamp
where tenant_id = 1
  and subject_type = 'APPLICATION'
  and subject_id = 你的applicationId;
```
