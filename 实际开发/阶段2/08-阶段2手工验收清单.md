# 阶段 2 手工验收清单

> 目标：用最少重复步骤验收阶段 2 的 Gateway 治理闭环。
> 原则：普通接口用 Apifox，SSE 用 PowerShell，落库用 SQL，回归用 Maven。

## 0. 工具分工

不要在 PowerShell 和 Apifox 里把所有接口重复测一遍。推荐这样分工：

| 工具 | 负责内容 | 为什么 |
|---|---|---|
| Maven 自动化测试 | 代码逻辑、Controller 行为、ArchUnit 架构边界 | 快速防回归 |
| Apifox | 登录、创建 Application、创建 Profile、Trace/Token 查询、403/429 错误码 | 请求体、Header、环境变量、JSON 展示更方便 |
| PowerShell `curl.exe -N` | 关键 SSE 链路：agent、agentMode=none、API Key | 最容易看出是不是逐条流式返回 |
| DBeaver / SQL | Trace、Token、Quota、Security、Alert 是否真实落库 | HTTP 工具不能替代查库 |

最终验收通过标准：

- `mvn.cmd -s .mvn/settings.xml test` 通过。
- Apifox 能完成登录、创建 Application、创建 Profile、Trace/Token 查询。
- PowerShell 能看到 SSE `thinking -> message -> done`。
- SQL 能查到 `trace_roots`、`trace_spans`、`token_usage_logs`、`quota_reservations`。
- Quota 超限返回 429，并写入 `TOKEN_EXCEEDED` 告警。
- 敏感数据返回 403，并写入 `security_events` 和 `SECURITY_BLOCKED` 告警。
- `agentMode=none` 直连模型路径也有 `model.invoke` span 和 `token_usage_logs`。

阶段 2 不要求：

- 完整日/月配额扣减。
- 告警通知 worker。
- 响应内容敏感扫描。
- Gateway Filter 链重构。
- AgentRuntime Pipeline 重构。

## 5. PowerShell 验证关键 SSE

这一节只用 PowerShell 测 SSE。不要把前面的登录、创建 Application、创建 Profile 在 PowerShell 里再做一遍。

先把 Apifox 里的变量复制到 PowerShell：

```powershell

$profileId = 替换为 Apifox 环境变量 profileId
$apiKey = "替换为 Apifox 环境变量 apiKey"

```

### 5.1 Web Agent SSE

```powershell

  applicationId = $applicationId
  agentMode = "agent"
  profileId = $profileId
  conversationId = $null
  message = "hello phase 2"
  enabledSkillIds = @()
  enabledMcpToolIds = @()

  modelConfigId = 1
```

## 6. SQL 验证正常链路落库

先确认阶段 2 表存在：

```sql
select table_name
from information_schema.tables
where table_schema = 'public'
  and table_name in (
    'trace_roots',
    'trace_spans',
    'token_usage_logs',
    'quota_configs',
    'quota_reservations',
    'security_events',
    'alert_events'
  )
order by table_name;
```

通过标准：7 张表都存在。

替换 `tr_xxx` 为 PowerShell SSE 返回的 traceId：

```sql
select trace_id, status, entrypoint, agent_mode, tenant_id, application_id,
       user_id, profile_id, conversation_id, latency_ms, error_code, error_message
from trace_roots
where trace_id = 'tr_xxx';
```

通过标准：

```text
有 1 条记录
status = SUCCESS
entrypoint = INTERNAL_WEB 或 API_KEY
agent_mode = agent 或 none
application_id/user_id 有值
latency_ms >= 0
```

查 span：

```sql
select span_name, span_type, component, status, latency_ms, attributes
from trace_spans
where trace_id = 'tr_xxx'
order by started_at asc;
```

通过标准：

```text
agent 模式包含 agent_runtime.run、context.build、model.invoke
none 模式至少包含 model.invoke
成功 span 的 status = SUCCESS
latency_ms >= 0
```

查 Token Usage：

```sql
select trace_id, model_config_id, model_name, provider_type,
       prompt_tokens, completion_tokens, total_tokens, estimated
from token_usage_logs
where trace_id = 'tr_xxx';
```

通过标准：

```text
至少 1 条记录
model_name = mock-chat
provider_type = MOCK
total_tokens > 0
estimated = true
```

查 Quota reservation：

```sql
select trace_id, estimated_tokens, actual_tokens, status
from quota_reservations
where trace_id = 'tr_xxx';
```

通过标准：

```text
有 1 条记录
estimated_tokens = 1000
status = COMMITTED
actual_tokens >= 0
```

## 9. Apifox + SQL 验证 Quota 429

先用 SQL 给当前 Application 加一个很小的单次限额。默认预扣值是 1000，所以设置为 10：

```sql
insert into quota_configs (
  tenant_id, subject_type, subject_id,
  daily_limit_tokens, monthly_limit_tokens, single_request_limit_tokens,
  status
)
values (
  1, 'APPLICATION', 你的applicationId,
  0, 0, 10,
  'ACTIVE'
)
on conflict (tenant_id, subject_type, subject_id)
do update set single_request_limit_tokens = 10,
              status = 'ACTIVE',
              updated_at = current_timestamp;
```

Apifox 请求：

```text
POST {{webBaseUrl}}/api/chat/stream
```

Body：

```json
{
  "applicationId": {{applicationId}},
  "agentMode": "agent",
  "profileId": {{profileId}},
  "conversationId": null,
  "message": "hello quota",
  "enabledSkillIds": [],
  "enabledMcpToolIds": [],
  "clientRequestId": "phase2-quota-exceeded",
  "modelConfigId": 1,
  "messages": null,
  "stream": true
}
```

通过标准：

```text
HTTP 429
响应包含 QUOTA_EXCEEDED
不会返回正常 message/done
```

查告警：

```sql
select alert_type, level, notify_status, content
from alert_events
where application_id = 你的applicationId
order by created_at desc
limit 5;
```

通过标准：

```text
能看到 TOKEN_EXCEEDED
notify_status = PENDING
```

恢复配额，避免影响后续测试：

```sql
update quota_configs
set single_request_limit_tokens = 0,
    status = 'DISABLED',
    updated_at = current_timestamp
where tenant_id = 1
  and subject_type = 'APPLICATION'
  and subject_id = 你的applicationId;
```

```text

error_code/error_message 有值
```

继续查：

```sql
select trace_id, status
from quota_reservations
where trace_id = '上一步traceId';

select alert_type, level, notify_status, content
from alert_events
where trace_id = '上一步traceId';
```

通过标准：

```text
quota_reservations.status = RELEASED
alert_events.alert_type = MODEL_ERROR
```
