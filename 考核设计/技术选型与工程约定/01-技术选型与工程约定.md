# 技术选型与工程约定

> 本文档用于约束后续代码实现方式。架构边界以 `考核设计/架构与模块设计/04-模块目录与包结构设计.md` 为准，数据库字段以 `考核设计/数据库模型设计/01-数据库模型设计.md` 为准；本文只规定“代码怎么落地、怎么检查、怎么验收”。

---

## 1. 文档定位

本文回答以下问题：

```text
用哪些技术。
工程如何建。
包、类、DTO、Entity、Mapper、SQL 放在哪里。
接口返回、错误码、日志、Trace、SSE 怎么统一。
哪些边界必须用测试或检查工具兜住。
阶段 1 开发时最低验证标准是什么。
```

本文不重复详细业务设计。遇到冲突时按下面优先级处理：

```text
1. 数据库字段、表归属：数据库模型设计文档
2. 模块、包、调用方向：模块目录与包结构设计文档
3. 阶段范围和实现顺序：MVP 分阶段实现路线图
4. 具体编码规范、工程落地：本文档
```

## 2. 技术选型

### 2.1 后端

| 类别 | 选型 | 说明 |
|---|---|---|
| JDK | Java 17+ | Spring Boot 3.x 基线 |
| 后端框架 | Spring Boot 3.4.x | 平台主工程、gateway、feishu-bot 均使用；与 Spring AI 1.0.x 兼容 |
| 构建工具 | Maven 多模块 | 后端主工程使用父 POM 管理版本 |
| Web 框架 | Spring MVC | REST、SSE、内部 HTTP 接口 |
| 安全 | Spring Security | JWT、API Key、内部服务 Token |
| ORM | MyBatis-Plus | Entity / Mapper 按表归属包放置 |
| 数据库 | PostgreSQL | 平台主库和 feishu-bot 独立库 |
| 数据库迁移 | Flyway | SQL 脚本随 core / feishu-bot 管理 |
| 参数校验 | Jakarta Validation | Controller Request、Command 入参校验 |
| JSON | Jackson | Spring Boot 默认 |
| 日志 | SLF4J + Logback | 日志必须带 trace_id |
| 架构测试 | ArchUnit | 检查模块和包访问边界 |
| 单元测试 | JUnit 5 | 阶段 1 至少覆盖关键边界测试 |

### 2.2 Agent / AI

| 类别 | 选型 | 说明 |
|---|---|---|
| LLM 调用 | 魔改 Spring AI ChatClient 或 OpenAI 兼容 Client | 统一封装在 `core.model` |
| Agent Runtime | 自研 ReAct while 循环 | 放在 `core.agent` |
| 上下文组装 | 自研 AgentContextBuilder | 放在 `core.context` |
| Skill | 内置 Skill + 后续配置型/Jar Skill | 阶段 1 先做内置 Skill |
| MCP | MCP Client 示例工具 | 阶段 1 至少实现 1 个 |
| SSE | SseEmitter / StreamingResponseBody 二选一 | 基于 Spring MVC，以“不缓冲、可逐事件透传”为验收标准 |

说明：

- 阶段 1 不强求完整 Jar 热加载沙箱。
- 阶段 1 不强求完整 RAG / 向量库。
- 阶段 1 不强求完整 Trace / Token / 安全策略表，但接口边界必须预留。
- Spring AI 只作为模型调用适配层，不接管 Agent Runtime、ReAct 循环、Skill/MCP 执行和多 Agent 调度。
- Spring AI 保留的能力主要是 OpenAI 兼容请求构建、模型响应解析、usage 读取、流式 token 适配和多供应商 ChatClient 封装；这些能力统一包在 `core.model`，不向 `web`、`gateway`、`core.agent` 泄漏 Spring AI 细节。
- 如使用 Spring AI tool calling，默认不要启用内部自动工具执行；工具调用结果应交给 `core.agent` / `core.team` 控制，以便发送 `action`、`observation` 等实时 SSE 事件。
- 阶段 4 多 Agent 由 `core.team.TeamOrchestrator` 控制状态机，Planner / Executor / Reviewer 仍通过 `core.model` 调模型，不把 Spring AI 当作多 Agent 编排器。

### 2.3 前端

| 类别 | 选型 | 说明 |
|---|---|---|
| 构建 | Vite | 前端独立工程 |
| 框架 | React | 阶段 3 实现控制台 |
| 样式 | Tailwind CSS | 控制台 UI |
| 组件 | shadcn/ui | 表单、弹窗、表格等 |
| API | fetch / axios 二选一 | 统一封装 API Client |

阶段 1 可先不做完整前端，用 curl / Postman 验证后端主链路。

## 3. 本地开发环境准备

### 3.1 必备软件

阶段 1 开发前建议准备：

| 软件 | 建议版本 | 用途 |
|---|---|---|
| JDK | 17+ | 编译和运行 Spring Boot 后端 |

| PostgreSQL | 15 或 16 | 平台主库、Flyway 建表、接口联调 |

| Postman / Apifox | 任一即可 | 手工接口调试 |

可选软件：

| 软件 | 用途 |

```text
Redis
Jaeger
Milvus / Qdrant

```

### 3.2 PostgreSQL

推荐使用 PostgreSQL 15 或 16。

用途：

```text
运行平台主库。
验证 Flyway SQL。
保存用户、应用、Profile、Skill、会话消息和长期记忆。
阶段 2 保存 Trace、Token、Security、Alert 表。
```

安装方式二选一：

```text
1. 本机安装 PostgreSQL 官方安装包。
2. 使用 Docker Desktop 启动 PostgreSQL 容器。
```

阶段 1 只要能提供一个本地数据库即可，后续 Docker Compose 再统一整理启动方式。

### 3.3 curl 是什么

`curl` 是命令行 HTTP 客户端，用来直接请求后端接口。

它适合做两件事：

```text
1. 快速验证 REST API，不需要打开浏览器或 Postman。
2. 验证 SSE 是否实时流式返回。
```

Windows 10/11 通常已经自带 `curl.exe`。检查命令：

```powershell
curl.exe --version
```

如果没有，可以通过以下方式安装：

```text
1. 安装 Git for Windows，通常会附带 curl。
2. 使用 Windows 包管理器 winget 安装 curl。
3. 从 curl 官方网站下载 Windows 版本。
```

阶段 1 常用示例：

```powershell
curl.exe -X GET http://localhost:8080/actuator/health
```

验证 SSE 时要禁用缓冲，推荐：

```powershell
curl.exe -N http://localhost:8080/api/chat/stream-test
```

`-N` 表示不缓冲输出。如果服务端每 500ms 推一个事件，终端也应该逐条显示，而不是最后一次性显示。

## 4. 工程结构约定

### 4.1 后端主工程

后端主工程只包含 4 个 Maven 子模块：

```text
agent-platform/
├── pom.xml
├── agent-platform-common/
├── agent-platform-core/
├── agent-platform-web/
└── agent-platform-gateway/
```

其中：

- `agent-platform-common`：公共基础能力。
- `agent-platform-core`：核心业务能力和业务表归属。
- `agent-platform-web`：浏览器端入口。
- `agent-platform-gateway`：AI 调用网关和治理链入口。

禁止把 `agent-platform-frontend` 放进 Maven modules。

### 4.2 独立工程

```text
agent-platform-frontend/    React 前端工程
feishu-bot/                 独立 Spring Boot 服务
```

`feishu-bot` 不依赖 `agent-platform-common`、`agent-platform-core`、`agent-platform-web`、`agent-platform-gateway` 的任何代码。

### 4.3 Maven 依赖方向

允许：

```text
web -> core
web -> common
gateway -> core
gateway -> common
core -> common
```

禁止：

```text
禁止 common 依赖 core/web/gateway
禁止 core 依赖 web/gateway
禁止 web 以 Maven 代码依赖方式依赖 gateway
禁止 feishu-bot 依赖 agent-platform-* 代码
```

说明：

- `web -> gateway` 只能通过 HTTP 内部接口调用。
- `gateway -> core` 使用 Maven 依赖和方法调用。
- Web AI 请求必须走 `web -> gateway -> core`，不能为了省事走 `web -> core.agent`。

## 5. Java 包约定

### 5.1 包名前缀

平台主工程：

```text
com.ls.agent
```

飞书机器人：

```text
com.ls.feishu
```

建代码后不要再随意改包名前缀。

### 5.2 common 包

允许：

```text
common.error
common.response
common.enums
common.util
common.constants
```

禁止：

```text
common.entity
common.mapper
common.service
common.dto.AgentProfileDto
common.dto.UserDto
```

`common` 只放无业务归属的基础能力，不放业务对象。

### 5.3 core 包

`core` 内部按业务域分包：

```text
core.identity
core.model
core.profile
core.skill
core.mcp
core.agent
core.memory
core.context
core.quota
core.trace
core.security
core.alert
core.team
core.support
```

`core.support` 只放 core 内部基础支撑能力，例如持久化基类、MyBatis TypeHandler、core 内部通用转换器；不能放具体业务 DTO、Entity、Mapper 或业务 Service。

业务包内部推荐结构：

```text
api/            对其他包暴露接口
application/    应用服务和用例编排
domain/         领域对象、策略、领域服务
entity/         数据库 Entity
mapper/         MyBatis-Plus Mapper
dto/            对外返回 DTO
command/        对外接收 Command
repository/     数据访问封装，可选
internal/       包内私有实现
```

跨 core 业务包只能访问：

```text
api
dto
command
```

禁止跨包访问：

```text
entity
mapper
internal
```

### 5.4 web 包

推荐结构：

```text
web.auth
web.controller
web.dto
web.sse
web.gatewayclient
web.security
web.config
```

禁止：

- `web` 写 Mapper。
- `web` 拥有业务表。
- `web` 直接调用 `core.agent` 处理 AI 执行请求。
- `web` 复用 `gateway.dto`。

### 5.5 gateway 包

推荐结构：

```text
gateway.controller
gateway.dto
gateway.auth
gateway.context
gateway.filter
gateway.runtime
gateway.stream
gateway.notifyclient
gateway.config
```

要求：

- `RuntimeInvokeFilter` 放在 `gateway.filter`。
- `gateway.runtime` 只放调用 core runtime/model 的适配逻辑。
- Gateway 不拥有业务表。
- Gateway 不访问任何 `core.*.mapper`。

## 6. DTO、Command、Entity 约定

### 6.1 基本规则

```text
Controller 入参：Request
Controller 出参：Response
core 用例入参：Command
core 用例出参：Result / DTO
数据库对象：Entity
领域/系统事件：Event
```

示例：

```text
web.dto.LoginRequest
web.dto.LoginResponse
gateway.dto.ChatCompletionRequest
core.agent.command.AgentRunCommand
core.agent.dto.AgentRunResult
core.identity.entity.UserEntity
core.quota.event.TokenQuotaExceededEvent
```

### 6.2 禁止规则

- Controller 不直接接收 Entity。
- Controller 不直接返回 Entity。
- Web DTO 和 Gateway DTO 不互相复用。
- common 不放业务 DTO。
- Entity 不跨业务包传递。

### 6.3 Entity 基类

平台主工程 Entity 基类放在 `agent-platform-core` 的 `core.support.persistence`，不要放进 `common`。

推荐位置：

```text
com.ls.agent.core.support.persistence.BaseEntity
com.ls.agent.core.support.persistence.VersionedEntity
com.ls.agent.core.support.persistence.SoftDeleteEntity
```

推荐字段：

```text
BaseEntity:
  id
  createdAt
  updatedAt

VersionedEntity extends BaseEntity:
  version

SoftDeleteEntity extends BaseEntity:
  deletedAt
```

使用规则：

- 只有实际拥有 `version` 字段的表继承 `VersionedEntity`。
- 只有实际拥有 `deleted_at` 字段的表继承 `SoftDeleteEntity`。
- 不要为了统一继承关系给数据库表额外加不存在的字段。
- `feishu-bot` 是独立服务，如需基类，在 `com.ls.feishu` 下单独定义。

## 7. 数据库与 Flyway 约定

### 7.1 SQL 脚本位置

平台主库 Flyway 脚本放在：

```text
agent-platform-core/src/main/resources/db/migration/
```

飞书机器人独立库 Flyway 脚本放在：

```text
feishu-bot/src/main/resources/db/migration/
```

### 7.2 命名规则

推荐：

```text
V001__init_identity.sql
V002__init_model_profile.sql
V003__init_skill_mcp.sql
V004__init_agent_memory.sql
V005__init_quota_trace_security_alert.sql
V006__init_seed_data.sql
```

规则：

- 文件名使用 `V数字__描述.sql`。
- `V数字` 和描述之间必须是双下划线 `__`，例如 `V001__init_identity.sql`，不能写成 `V001_init_identity.sql`。
- 描述使用小写英文和下划线。
- 一个脚本只负责一组相关表或初始化数据。
- 已经执行过的版本脚本不要修改，新增变更用新版本。

### 7.3 阶段 1 建表范围

阶段 1 只强制创建数据库模型文档中列出的阶段 1 必建表：

```text
tenants
users
roles
user_roles
applications
api_keys
model_providers
model_configs
agent_profiles
skills
skill_versions
profile_skills
mcp_servers
mcp_tools
profile_mcp_tools
conversations
conversation_messages
memories
```

`quota_configs`、`quota_reservations`、`token_usage_logs`、`trace_roots`、`trace_spans`、`security_policies`、`security_events`、`alert_rules`、`alert_events` 放阶段 2；如果阶段 1 提前建，字段必须和数据库模型文档一致。

### 7.4 Entity / Mapper

要求：

- Entity 字段以数据库模型文档为准。
- Mapper 放在表归属包的 `mapper` 下。
- Mapper 只服务本包 application/repository。
- 其他包不得直接注入 Mapper。

## 8. 接口响应与错误码约定

### 8.1 Web 管理类接口统一响应

Web 管理、配置、查询类接口统一使用：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "success": false,
  "code": "AUTH_UNAUTHORIZED",
  "message": "未登录或登录已过期",
  "data": null
}
```

### 8.2 OpenAI 兼容接口

`gateway` 的 `/v1/chat/completions` 优先保持 OpenAI 兼容格式。

要求：

- TraceID 放响应头 `X-Trace-Id`。
- 严格兼容模式下不强行往 body 塞平台字段。
- 增强模式可以额外返回 `trace_id`，但必须由请求参数显式开启。

### 8.3 错误码命名

格式：

```text
模块_错误含义
```

示例：

```text
AUTH_UNAUTHORIZED
AUTH_FORBIDDEN
API_KEY_INVALID
QUOTA_EXCEEDED
SECURITY_BLOCKED
MODEL_INVOKE_FAILED
AGENT_MAX_STEPS_EXCEEDED
SKILL_EXECUTE_FAILED
MCP_TOOL_FAILED
INTERNAL_ERROR
```

### 8.4 HTTP 状态码

| 场景 | HTTP 状态码 |
|---|---|
| 未登录 / API Key 无效 | 401 |
| 无权限 | 403 |
| 请求参数错误 | 400 |
| 配额超限 | 429 |
| 敏感数据阻断 | 400 或 403 |
| 模型调用失败 | 502 |
| 系统内部错误 | 500 |

## 9. 鉴权与上下文约定

### 9.1 浏览器 JWT

浏览器入口由 `web` 处理 JWT。

用途：

- 登录注册。
- 管理配置。
- 个人中心。
- 浏览器 AI 对话入口。

浏览器 AI 对话进入 `web` 后，必须由 `web.gatewayclient` 通过 HTTP 转发到 `gateway`。

### 9.2 API Key

外部调用直接进入 `gateway`：

```text
Authorization: Bearer sk-xxx
```

API Key 校验必须通过 `core.identity.api.ApiKeyService`，Gateway 不直接查表。

### 9.3 Web 调 Gateway 内部鉴权

Web 调 Gateway 内部接口必须带内部服务 Token 或签名 Header。

推荐 Header：

```text
X-Internal-Token: xxx
X-Internal-Timestamp: 1710000000000
X-Internal-Nonce: uuid
```

阶段 1 可以先使用固定内部 Token；阶段 2 再升级为签名和防重放。

### 9.4 GatewayContext 与 AgentContext

`GatewayContext` 只保存治理上下文：

```text
tenant_id
application_id
user_id
trace_id
channel
agent_mode
request_path
```

`AgentContext` 保存模型执行上下文：

```text
System Prompt
Profile Prompt
用户输入
历史消息
长期记忆
可用 Skill
可用 MCP Tool
安全策略摘要
```

强约束：

- Gateway 不组装 Prompt。
- Gateway 不做上下文压缩。
- core 不读取 Gateway ThreadLocal。
- Gateway 必须把 `GatewayContext` 显式转换成 `AgentRunCommand` 或 `ModelInvokeCommand`。

## 10. SSE 与流式响应约定

### 10.1 事件类型

平台内部 SSE 事件统一使用：

```text
thinking
action
observation
message
done
error
```

推荐事件结构：

```json
{
  "type": "message",
  "trace_id": "tr_xxx",
  "step": 3,
  "content": "部分回复",
  "metadata": {}
}
```

### 10.2 Web 转发 Gateway

浏览器 AI 对话链路：

```text
frontend -> web -> gateway -> core
```

要求：

- Web 必须逐事件透传 Gateway SSE。
- 不能等 Gateway 完整返回后一次性写给浏览器。
- Gateway 出错时，Web 要转成 SSE `error` 事件。
- 浏览器断开后，Web/Gateway/Core 要释放资源。

### 10.3 阶段 1 必测

Gateway 空壳完成后，必须先做固定流测试：

```text
Gateway 每 500ms 输出一个 SSE 事件。
Web 逐事件转发。
curl 或浏览器能实时看到事件。
```

这个测试早于完整 Agent Runtime。

## 11. 日志、Trace 与敏感数据约定

### 11.1 日志字段

后端日志至少包含：

```text
trace_id
tenant_id
application_id
user_id
module
operation
```

阶段 1 如果 Trace 表未落库，也要生成 `trace_id` 并写入日志。

### 11.2 敏感数据

禁止原文写入：

```text

security_events
conversation_messages
memories
alert_events
```

敏感扫描范围：

```text
用户请求
工具入参

模型响应
记忆写入
Trace/日志
```

阶段 1 可先做简单正则脱敏，阶段 2 再接完整策略表。

## 12. Token 配额约定

### 12.1 阶段 1

阶段 1 `core.quota` 至少提供默认允许策略：

```text
没有配置时返回 allow。
不做完整预扣和结算。
接口形态保留。
```

禁止在主链路中写大量：

```java
if (quotaService != null) {
    quotaService.check(...);
}
```

应该注入一个真实 Bean，哪怕它是 `DefaultAllowQuotaService`。

### 12.2 阶段 2

阶段 2 必须实现：

```text
请求前预扣
失败释放
成功按真实 usage 修正
用量记录
基础 Token 看板
```

并发安全要求：

- 使用 `version` CAS 或 `SELECT ... FOR UPDATE`。
- `quota_reservations.trace_id` 唯一。
- 释放和结算按 `trace_id + status + version` 幂等更新。
- 不能只靠异步任务扣 Token。

## 13. ArchUnit 约定

### 13.1 加入时机

生成 Entity / Mapper 后，必须立刻加入 ArchUnit 或等价架构测试。

不要等代码写多了再补。

### 13.2 最低规则

必须覆盖：

```text
common 不依赖 core/web/gateway
core 不依赖 web/gateway
web 不以 Maven 方式依赖 gateway
web/gateway 不访问 core.*.mapper
core 内部跨业务包不访问其他包的 entity/mapper/internal
Controller 不直接访问 Mapper
common 不包含业务 Entity/DTO/Mapper/Service
```

### 13.3 示例规则意图

```text
core.agent 可以访问 core.skill.api.SkillRegistry
core.agent 不可以访问 core.skill.mapper.SkillMapper
gateway 可以访问 core.agent.api.AgentRuntimeService
gateway 不可以访问 core.agent.mapper.ConversationMapper
```

### 14.1 Maven 命令

后端统一验证命令：

```bash

```

- 编译通过。
- ArchUnit 规则通过。
- 关键 Service 单测通过。
- 纯单元测试不依赖本地 PostgreSQL。
- 涉及 Flyway、Mapper、Repository 或 `@SpringBootTest` 的集成测试，需要本地 PostgreSQL 或 Testcontainers 环境。

```text

历史消息查询
长期记忆写入

至少 1 个 MCP 示例工具调用
```

阶段 1 完成标准：

```text
Postman 或 curl 能通过 Web -> Gateway -> Core 路径跑通 Agent 对话。
SSE 是实时流式返回。

Agent 能调用至少一个 MCP 示例工具。

Gateway 空壳路径保留，阶段 2 不需要重构入口。
```

## 17. 代码风格约定

### 17.1 命名

| 类型 | 后缀 |
|---|---|
| Controller 入参 | `Request` |
| Controller 出参 | `Response` |
| core 用例入参 | `Command` |
| core 用例出参 | `Result` / `DTO` |
| 数据库实体 | `Entity` |
| Mapper | `Mapper` |
| 应用服务 | `Service` |
| 策略类 | `Policy` / `Strategy` |
| 适配器 | `Adapter` / `Client` |
| 领域/系统事件 | `Event` |

### 17.2 查询方法命名

查询方法必须区分“必须存在”和“允许不存在”：

```text
getXxx：必须返回结果，不存在时抛业务异常。
findXxx：允许不存在，返回 Optional 或 null。
existsXxx：只判断是否存在，返回 boolean。
checkXxx：执行业务检查，不通过时抛业务异常。
listXxx：返回列表。
pageXxx：返回分页结果。
```

示例：

```text
getProfile(profileId)          查不到就是异常
findProfile(profileId)         查不到允许为空
existsApiKey(prefix)           只判断是否存在
checkApplicationActive(appId)  应用不可用则抛异常
pageConversations(query)       分页查询会话
```

不要用一个 `queryXxx` 同时表达多种语义。

### 17.3 分层

推荐调用顺序：

```text
Controller -> Application Service -> Domain/Repository/Mapper
```

禁止：

```text
Controller -> Mapper
Controller -> Entity 作为响应
Filter -> Mapper
```

### 17.4 三层校验

校验按层分工：

```text
Controller 层：
  只做格式校验、必填校验、长度校验。
  使用 @Validated / Jakarta Validation。

Application 层：
  做业务规则校验、权限校验、状态流转校验、数据存在性校验。

Domain 层：
  做领域不变条件、策略规则、状态对象内部一致性校验。
```

禁止：

```text
Controller 直接判断复杂业务状态。
Mapper 查询结果直接决定接口语义。
Domain 层依赖 Web/Gateway 请求对象。
```

### 17.5 jsonb 字段

数据库 `jsonb` 字段在 Entity 中优先使用明确的配置对象。

推荐：

```text
memory_strategy -> MemoryStrategyConfig
capabilities -> ModelCapabilities
permission_declaration -> PermissionDeclaration
runtime_config -> SkillRuntimeConfig
metadata / attributes -> JsonNode 或明确对象
```

约定：

- 稳定结构用明确 Java 对象。
- 不稳定扩展字段可以用 `JsonNode`。
- 不推荐到处使用 `Map<String, Object>`。
- MyBatis TypeHandler 统一注册，不在各个 Mapper 里分散处理。
- 写入 Trace、日志、记忆前必须先脱敏。

### 17.6 事务

事务只放在 application service 或明确的业务写服务上。

不要在 Controller、Filter、DTO 转换层开启事务。

Token 预扣、释放、结算必须单独设计事务边界，不能混在模型流式响应长事务里。

## 19. 最终原则

实现时只守住三条线：

```text
入口层不持有业务数据。
核心业务只通过 api/dto/command 跨包协作。
关键边界必须能被测试或验收命令发现。
```

如果某个实现让 `web/gateway/common` 开始承载业务规则，或者让 `core.agent` 直接查别的包的表，就说明工程边界已经偏了，需要立刻调整。
