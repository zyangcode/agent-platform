# 阶段 2 接口设计

> 本文档定义阶段 2 Trace / Token 查询接口。阶段 2 的写入链路由 Gateway/Core 内部调用完成；对外新增的主要是 Web 登录态下的查询接口，为阶段 3 前端 Trace 页面和 Token 看板预留稳定响应结构。

---

## 1. 设计边界

阶段 2 接口只覆盖：

```text
按 traceId 查询完整链路。
按 application/profile/model/status 分页查询 trace 列表。
在 trace 详情中查看 span 和 token usage。
按 application/model/provider 做基础 token 用量聚合。
```

阶段 2 暂不做：

```text
复杂趋势图。
成本折算和真实计费。
Trace 链路图渲染。
OTLP / SkyWalking 查询代理。
多租户 Admin 跨租户审计页面。
```

权限原则：

```text
Admin：可查看当前 tenant 下所有 Trace/Token。
Developer：可查看自己应用、自己创建 Profile 相关 Trace/Token。
User：只能查看自己的 Application、Conversation、Trace/Token。
所有查询必须带 tenantId 过滤。
Web/Gateway 不直接访问 Mapper，只调用 `core.trace.api` 和 `core.quota.api`。
```

---

## 2. 通用响应

沿用现有 `ApiResponse<T>`：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

分页沿用 `PageResult<T>`：

```json
{
  "records": [],
  "pageNo": 1,
  "pageSize": 20,
  "total": 0
}
```

错误响应继续由 `GlobalExceptionHandler` 输出。阶段 2 新增错误建议：

```text
TRACE_NOT_FOUND      traceId 不存在或无权限。
TRACE_QUERY_DENIED   无权查询该 trace。
```

如果不新增错误码，第一版可以复用：

```text
RESOURCE_NOT_FOUND
AUTH_FORBIDDEN
REQUEST_INVALID
```

第一版建议直接复用现有错误码，避免为了查询接口扩大 common 改动：

```text
traceId 不存在或无权限：REQUEST_INVALID 或 AUTH_FORBIDDEN。
查询参数非法：REQUEST_INVALID。
未登录：AUTH_UNAUTHORIZED。
```

```text

```

第一版必选或推荐参数：

```text

modelConfigId
```

## 8. Web Controller

建议新增：

```text
agent-platform-web/src/main/java/com/ls/agent/web/controller/TraceController.java
```

接口：

```java
@RestController
public class TraceController {
    private final TraceService traceService;

    public TraceController(TraceService traceService) {
        this.traceService = traceService;
    }

    @GetMapping("/api/traces/{traceId}")
    public ApiResponse<TraceDetailDTO> getTrace(CurrentUser currentUser, @PathVariable String traceId) {
        return ApiResponse.success(traceService.getTrace(currentUser.tenantId(), currentUser.userId(), traceId));
    }

    @GetMapping("/api/traces")
    public ApiResponse<PageResult<TraceSummaryDTO>> pageTraces(
            CurrentUser currentUser,
            @RequestParam(name = "applicationId", required = false) Long applicationId,
            @RequestParam(name = "profileId", required = false) Long profileId,
            @RequestParam(name = "modelConfigId", required = false) Long modelConfigId,
            @RequestParam(name = "status", required = false) String status,
            @RequestParam(name = "entrypoint", required = false) String entrypoint,
            @RequestParam(name = "pageNo", defaultValue = "1") int pageNo,
            @RequestParam(name = "pageSize", defaultValue = "20") int pageSize
    ) {
        return ApiResponse.success(traceService.pageTraces(new QueryTracePageCommand(
                currentUser.tenantId(),
                currentUser.userId(),
                applicationId,
                profileId,
                modelConfigId,
                status,
                entrypoint,
                pageNo,
                Math.min(pageSize, 100)
        )));
    }
}
```

说明：

```text
Controller 不访问 Mapper。
权限和资源归属校验放 TraceService。
第一版 User/Developer 都按 tenantId + userId + applicationId 过滤；Admin 放宽到 tenant 级可后续增强。
```

## 9. SSE 事件中的 Trace 字段

阶段 2 继续沿用阶段 1 SSE 格式：

```text
event: thinking|action|observation|message|done|error
data: {"type":"thinking","traceId":"tr_xxx","conversationId":1,"step":1,"content":"request accepted","metadata":{}}
```

要求：

```text
每个事件都尽量带 traceId。
error 事件如果异常发生在 traceId 生成前，traceId 可以为 null。
done 事件 metadata 可带 totalTokens、estimated、modelConfigId。
```
