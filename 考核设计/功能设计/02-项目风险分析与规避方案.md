# 全面风险分析报告

---

## 一、架构合理性

### 风险 1：BlockingQueue 导致 Agent Team 阻塞不可观测

当前设计 Team 之间用 `BlockingQueue<TeamMessage>` 通信。在单 JVM 内，如果 Executor 执行 Skill 超时（比如调外部 API 卡住 60 秒），`queue.take()` 会一直阻塞调用线程，前端收不到任何 SSE 事件——用户会看到页面"卡死"，不知道发生了什么。

**解决方案**：BlockingQueue 的 `poll(timeout, unit)` 替代 `take()`，超时后推一个 SSE timeout 事件给前端，让用户知道"子任务执行中，等待中"。同时在 Trace 里记录等待 Span。

### 风险 2：ThreadLocal GatewayContext 在异步场景泄漏

GatewayContext 靠 `ThreadLocal` 贯穿拦截器链。但对话是 SSE 长连接，内部可能用 `@Async` 做长期记忆摘要、异步告警。一旦从主线程切换到异步线程池，ThreadLocal 的值就丢了——子线程拿不到 traceId，日志断链。

**解决方案**：所有异步方法显式传递 traceId + appId，或使用 `InheritableThreadLocal` + `TaskDecorator` 在 Spring 线程池里自动传递 MDC 上下文。MVP 阶段至少保证异步方法签名里带 traceId 参数。

### 风险 3：Maven 多模块循环依赖

web 和 gateway 都需要调用 core；feishu-bot 是独立服务，不依赖 agent-platform 的 core/common 代码。如果 core 模块里不小心引用了 web 或 gateway 的类，Maven 编译就会循环依赖报错。随着开发深入，core 很容易膨胀。

**解决方案**：core 是核心业务能力层，可以放 Agent Runtime、Skill、Memory、Profile、Quota、Trace 等业务实现；web/gateway 只做入口适配和协议转换，不拥有业务表。严格禁止 core 依赖 web/gateway；feishu-bot 只能通过 HTTP 告警接口被调用，不做 Maven 代码依赖。用 `mvn dependency:analyze` 定期检查。

### 风险 4：基础 Agent 与 Team 模式切换点在代码里散落

设计说 Profile 有个开关选基础 Agent 或 Team。但如果这个 `if/else` 散落在 ChatService、Controller、SSE 推送三处，未来加第三种执行模式时改一处漏一处。

**解决方案**：抽取 `AgentRuntime` 接口，`BasicAgentRuntime` 和 `TeamAgentRuntime` 各自实现。Profile 加载后通过工厂返回对应的 Runtime 实例。

## 二、安全漏洞

### 风险 5：URLClassLoader 加载的 Jar 没有沙箱

CLAUDE.md 明确写了"ClassLoader 只做依赖隔离，不等价于安全沙箱"。但功能上 Developer/Admin 上传的 Jar Skill 会被反射执行 `execute()`。如果有人上传了一个 `execute()` 里写 `Runtime.getRuntime().exec("rm -rf /")` 的 Jar——ClassLoader 隔离根本拦不住。

**解决方案**：MVP 阶段至少做到：(1) 高危权限声明制，Skill 必须声明需要哪些权限；(2) Executor 调用前检查权限声明，高危 Skill 需 Admin 确认；(3) 长期：用容器隔离、独立进程、Wasm 或受限执行服务限制 Jar Skill 的系统调用能力。不要把 Java 17 的 `SecurityManager` 当成主要沙箱方案。

### 风险 6：API Key 明文存储和传输

设计里用户在个人中心为 Application 生成 API Key，Admin 后台可代创建或吊销作为兜底。如果 API Key 明文存在数据库、明文过网络，一旦数据库泄露或中间人攻击，所有用户的 Key 全部暴露。

**解决方案**：(1) API Key 只在生成时返回一次，存数据库的是 bcrypt 哈希；(2) 鉴权时比对哈希而非明文；(3) 所有 HTTP 通信（包括 Gateway 调 feishu-bot）走内网或 HTTPS。

### 风险 7：JWT 没有过期和刷新机制

MVP 只做极简用户名密码注册登录，不做找回密码、OAuth、复杂账号体系。但如果 JWT 一旦签发就永不过期，一旦泄露就永久有效。

**解决方案**：JWT 设置合理过期时间（Admin 端 24h，User 端 7d）。Admin 后台提供"吊销 Key/Token"按钮。API Key 支持在数据库标记 revoked。

### 风险 8：敏感数据扫描的正则只做前端验证

敏感数据正则可能写得不够严谨，攻击者可以构造变形数据绕过检测（如用全角数字替换手机号中间几位）。

**解决方案**：(1) 正则规则覆盖常见变形（全角/半角、分隔符）；(2) 不仅正则，加上简单的熵检测或模式匹配；(3) 脱敏后的内容再跑一次扫描确认无遗漏。

### 风险 9：用户上传的配置型 Skill（HTTP API Skill）可做 SSRF

配置型 Skill 允许 User 填写 URL、Method、Header。如果 URL 填 `http://169.254.169.254/latest/meta-data/`（云环境元数据地址），Executor 调这个 Skill 时就会把内部敏感信息返回。

**解决方案**：(1) HTTP Skill 执行层做 URL 白名单或黑名单过滤（禁内网 IP、禁 localhost、禁云元数据地址）；(2) 高危 URL 命中时记录安全日志并拒绝执行。

## 三、并发性能

### 风险 10：Redis Token 预扣与修正的竞态条件

设计是"请求前 Redis 原子预扣 → 完成后修正 → 失败释放"。但如果两个并发请求同时预估 5000 tokens、当前 Token 剩余额度只剩 8000，两个请求都通过预扣检查——总共消耗 10000 tokens 超出配额。`DECRBY` 原子减没问题，但"先检查额度"和"执行 DECRBY"之间不是原子的。

**解决方案**：用 Lua 脚本把"检查额度 + 预扣"封装为单次原子操作。Redis 执行 Lua 期间其他命令排队。

### 风险 11：SSE 连接数耗尽

SSE 是长连接，HTTP 线程被一直占用。如果 200 个用户同时打开对话页面，Tomcat 默认 200 线程池直接打满，新请求全部排队。

**解决方案**：(1) Spring Boot 配置 `server.tomcat.threads.max` 调大到合理值；(2) SSE 用 `SseEmitter` 的超时参数，前端断连时释放线程；(3) 极限场景考虑用 WebFlux 替代 Spring MVC（但这是重大架构变更，MVP 先调线程池参数）。

### 风险 12：SkillRegistry 的 ConcurrentHashMap 读多写少没问题，但上传 Jar 时锁粒度过大

如果上传一个大 Jar（几十 MB），在 `loadClass` 过程中所有读 SkillRegistry 的请求都在排队等同一把锁。

**解决方案**：SkillRegistry 用 `ReadWriteLock`——加载 Jar 时加写锁，对话调用时用读锁不互斥。或者加载新 Skill 时先写入临时 Map，加载成功后原子替换引用。

### 风险 13：BlockingQueue 单通道容量上限

Team 通信全靠一个 BlockingQueue。如果 Planner 拆了 20 个子任务一次性 `put` 进队列，而 Executor 消费速度慢，队列满后 Planner 线程阻塞。

**解决方案**：BlockingQueue 设置合理容量（如 100），超过时 Planner 分批放入。或者用 `LinkedBlockingQueue` 无界队列——风险是内存溢出，折中用 `ArrayBlockingQueue(200)` 加 `offer(timeout)`。

## 四、数据稳定性

### 风险 14：物模型切换时中途留下不完整数据

配置文件里改个模型 provider 的连接信息，正在执行的对话请求突然拿不到 ChatClient，抛异常后那轮对话的 message 只写了一半——user 消息入库了，assistant 回复没入库。

**解决方案**：整轮对话（messages 的完整周期）包在一个事务里。如果中途异常，回滚整轮对话的 message 记录。

### 风险 15：长期记忆的异步摘要可能丢失

对话结束后异步生成摘要、写入长期记忆。如果异步任务执行失败（线程池满、数据库连接断开），摘要就丢了。设计里没有失败重试队列。

**解决方案**：(1) 异步任务捕获异常后写入"待重试"表；(2) 定时任务扫表重试（最多 3 次）；(3) 最终失败记录日志，不影响主链路。

### 风险 16：Token 统计丢失导致配额不准确

请求完成后 TokenRecordFilter 负责统计入库。但如果这步抛异常（比如数据库连不上），真实 Token 用量没记录——预算配额就少扣了，用户实际可超用。

**解决方案**：Token 记录失败时，(1) 至少把 usage 写本地日志，后续补录；(2) Redis 预扣保留一段时间，数据库入库成功后才真正提交 Redis 扣减。

### 风险 17：数据库没有版本号导致并发覆盖

Agent Profile 编辑没有乐观锁。Admin A 和 Admin B 同时打开同一个 Profile 编辑——先提交的被后提交的静默覆盖，数据丢失。

**解决方案**：关键实体表（agent_config、skill、model_provider）加 `version` 字段。MyBatis-Plus 的 `@Version` 注解自动做乐观锁，更新时 WHERE version = oldVersion，更新失败返回给前端提示"数据已被他人修改"。

## 五、异常容错

### 风险 18：LLM 返回非 JSON 时 ReAct 循环无退出

PLanner 输出要求 TaskPlan JSON，但如果 LLM 返回了非 JSON 文本（比如"我理解你的需求，让我规划一下..."然后没输出 JSON），Schema 校验失败后会要求 LLM 修正——但如果 LLM 持续返回非 JSON，while 循环里套校验循环会无限循环。

**解决方案**：LLM 输出解析失败时：(1) 第一次 → 在 messages 里追加"请严格按照 JSON Schema 输出"后重试；(2) 第二次 → 尝试从文本中正则提取 JSON 片段；(3) 仍失败 → 记录错误 Trace，返回给用户"规划失败，请简化问题后重试"，断开 SSE。

### 风险 19：飞书告警接口调用失败导致 Gateway 响应延迟

如果 AlertFilter 在响应返回之前同步调 feishu-bot，feishu-bot 服务挂了或网络不通时，HTTP 连接超时会拖慢整个请求响应。这和当前“告警通知可以异步”的设计口径不一致。

**解决方案**：AlertFilter 只负责生成告警事件并写入队列/事件表；飞书发送由异步任务执行。HTTP 调用设置短超时（如 2 秒），失败只记录日志和告警状态，不阻塞主响应。

### 风险 20：Skill 热卸载时正在执行的调用会抛 NullPointerException

假设对话正在执行 WeatherSkill，此时 Admin 点下架 → SkillManager 卸载 ClassLoader → 缓存置 null。正在执行 `skill.execute()` 的线程可能访问到已卸载的类对象，JVM 抛 `NoClassDefFoundError`。

**解决方案**：卸载时只标记状态为 DISABLED，不立即 close ClassLoader。后台线程等 5 分钟后（确保所有进行中的调用已完成）再真正 close。或者用引用计数——每进入一次 Skill 调用 +1，调用结束 -1，引用计数归零才允许卸载。

### 风险 21：MCP Client 连接外部 Server 失败无降级

如果 Agent 的 MCP Tool 连了一个外部 Server，该 Server 宕机。每次对话构建工具列表时，MCP Client 尝试连接失败，是忽略该工具继续构建还是抛异常中断整个对话？当前设计没说。

**解决方案**：MCP Client 连接失败时，该工具从本次可用工具列表中移除（标记 unavailable），记录 Trace MCP Span 状态为 UNAVAILABLE，不中断对话。连续 N 次失败后 Admin 仪表盘告警。

## 六、开发维护

### 风险 22：功能点 216 个，MVP 路线图只有 5 阶段——中段可能范围蔓延

功能清单 216 个点，路线图 5 个阶段只覆盖了骨架。阶段 2 "网关治理"里有 30 个功能点，如果每个都追求完整实现（比如 Jaeger 深度集成、Token 仪表盘复杂图表），阶段 2 会拖成整个项目的最长阶段。

**解决方案**：每个阶段开始前，从功能清单里挑出该阶段"必须完成才能跑 Demo"的最小功能子集（不超过 15 个点），其余标记为"本阶段暂缓"。阶段 2 的 MVP 最小集就是路线图 4.1 列的那些——不要自行加量。

### 风险 23：没有自动化测试，回归要靠手动

CLAUDE.md 只是暂缓完整单元测试体系，但仍要求接口验证、手工验证和 Demo 验证。如果 5 个阶段叠加后完全没有自动化 Smoke Test，阶段 5 改飞书机器人时可能引入 bug 导致阶段 1 的对话功能挂了——没有自动化回归，只能手工按一遍所有 Demo 场景。

**解决方案**：(1) 至少给核心链路写 Smoke Test——一个 `@SpringBootTest` 启动整个上下文，发一个 HTTP 请求，断言响应 200；(2) 每次阶段完成后跑一次 Smoke Test。5 分钟写完，避免 2 小时排查。

### 风险 24：包结构里 Skill 接口定义在 core 模块，但内置 Skill 实现在哪？

内置 Skill（weather、calculator、search）如果放在 core 模块，它们依赖外部 API 的 HTTP 客户端（OkHttp/WebClient），core 就会引入网络层依赖。如果放在 web 模块，ReAct 引擎在 core 里怎么找到它们？

**解决方案**：内置 Skill 实现在 core 的 `skill/builtin/` 下，但用接口隔离——Skill 内部通过 `ToolClient` 接口做 HTTP 调用。MVP 可在 core 的基础设施包提供默认 `HttpToolClient` 实现，或由 gateway 启动服务注入实现；不要让 web 成为 AI 执行链路的 HTTP 客户端提供者。

### 风险 25：React 前端状态管理未定

功能清单列了 19 个前端页面，但没说用 Redux、Zustand、React Query、还是 Context。如果每个页面各自用 `useEffect` 直接 fetch，SSE 连接管理、Token 刷新、错误处理会在每个页面重复实现。

**解决方案**：MVP 阶段用 React Query（TanStack Query）管理服务端状态，EventSource 封装成自定义 hook `useSSE`。不做全局状态管理——19 个页面之间没有太多共享状态。

## 七、生产上线

### 风险 26：Docker Compose 启动顺序未保证

`docker-compose up` 同时启动 5 个容器。PostgreSQL 启动需要 5 秒初始化，但 Spring Boot 在第 2 秒就连数据库——连不上就 crash。

**解决方案**：(1) `docker-compose.yml` 里配 `depends_on` + `healthcheck`，后端容器等 PostgreSQL 健康检查通过才启动；(2) Spring Boot 配 `spring.datasource.hikari.initialization-fail-timeout: 30000`，30 秒内重试连接。

### 风险 27：api_key 和飞书 secret 通过 .env 管理，但 .env 可能被提交到 Git

.env 文件如果没加到 `.gitignore`，git push 后所有 Key 泄露。

**解决方案**：(1) `.gitignore` 加 `.env`；(2) 仓库里放 `.env.example` 模板（只有字段名没有真实值）；(3) README 说明如何创建 `.env`。

### 风险 28：Jaeger 和 OpenTelemetry 是生产级依赖——但 MVP 可能吃资源

Jaeger Collector + Query + UI 至少吃掉 300MB 内存。如果学生服务器只有 4GB 内存，加上 PostgreSQL + Redis + 3 个 Java 服务 + 前端 Node，内存直接跑满 OOM。

**解决方案**：MVP 阶段本地 Trace 存储就用 PostgreSQL 的 trace_span 表，Docker Compose 里 Jaeger 作为可选容器（profile: optional），默认不启动。演示需要时再开。

### 风险 29：没有做 API 版本控制

`/v1/chat/completions` 直接写死 v1。如果考核需要展示兼容不同 OpenAI 版本的请求格式，改 v1 接口就会破坏已有调用方。

**解决方案**：Controller 上加 `@RequestMapping("/v1")` 作为前缀，未来加 `/v2` 时只需新建 Controller。MVP 先只做 v1。

## 八、依赖隐患

### 风险 30：Spring AI 1.0.x API 不稳定

Spring AI 1.0.x 在考核时点可能不是 GA 版本。API 可能在你开发中途变更（比如 `ChatClient` → `ChatModel`，或 Function Calling 的参数结构变）。一旦依赖升级，你自研的 ReAct 引擎可能编译不过。

**解决方案**：(1) `pom.xml` 里锁定 Spring AI 具体版本号，不用 `latest`；(2) ReAct 引擎里不要深入到 Spring AI 内部类，只依赖公开 API；(3) 准备一个"裸"HTTP 调 OpenAI 兼容 API 的备选实现——如果 Spring AI 不可用，至少能直接发 HTTP 请求跑通对话。

### 风险 31：MyBatis-Plus 和 PostgreSQL 的兼容性——JSON 字段和 upsert 语法

设计里 `skill` 表有 `parameter_schema(JSON)`，PostgreSQL 的 JSON 类型在 MyBatis-Plus 里需要用 `@TableField(typeHandler = JacksonTypeHandler.class)` 处理。如果没配 typeHandler，读写 JSON 字段会抛类型转换异常。

**解决方案**：所有 JSON 字段配 JacksonTypeHandler，建表 SQL 里用 `jsonb`（PostgreSQL）而非 `json`（性能更好）。

### 风险 32：飞书 SDK 版本与 API 变更

飞书开放平台 API 会更新，SDK 可能滞后。比如飞书卡片消息的 JSON Schema 变了，SDK 还没支持新字段——你按旧格式发的消息在飞书群显示异常。

**解决方案**：(1) 飞书消息发送不依赖 SDK 的高级封装，用 OkHttp 直接拼 JSON 调飞书 API；(2) 写一个 `FeishuClient` 封装类，只做签名和 HTTP，消息体自己拼 JSON。SDK 升级不影响你的代码。

### 风险 33：shadcn/ui 组件版本与 React 19 兼容

选型是 React + shadcn/ui。shadcn/ui 依赖 Radix UI + Tailwind CSS，React 主版本升级时 Radix 可能会有 breaking change。如果你在项目中间升级了 React 版本，部分 shadcn 组件可能报 `forwardRef` 或 `createRoot` 相关错误。

**解决方案**：`package.json` 锁定 react 18.x + shadcn/ui 对应版本。MVP 阶段不追新版本。

### 风险 34：Jaeger Client / OpenTelemetry SDK 与 Spring Boot 3.4.x 的版本匹配

OpenTelemetry 的 Spring Boot starter 版本如果和 Spring Boot 3.4.x 不匹配，auto-configuration 会失败——Trace 功能注入不了，整个 Gateway 的 Trace 能力空白。

**解决方案**：(1) 查 [Spring Initializr](https://start.spring.io) 或 OpenTelemetry 官方兼容矩阵确认 Spring Boot 3.4.x 对应的 OpenTelemetry 推荐版本；(2) MVP 阶段优先用自建 trace_span 表存储，Jaeger 导出作为可选增强——避免被 SDK 版本冲突阻塞主链路。
