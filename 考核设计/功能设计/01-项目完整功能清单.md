## 功能域总览（11 个域）

整个项目拆成 11 个功能域，后续逐个域做设计、建表、写接口、做页面：

```
0. 角色与权限体系
1. 模型与统一调用入口
2. Agent Profile 体系
3. Skill / Tool 市场
4. MCP 工具体系
5. Agent Runtime 执行引擎（基础 Agent + Team 模式）
6. 记忆系统
7. AI Infra 治理能力
8. 飞书超级助手机器人
9. 前端控制台
10. 部署与交付
```

### 优先级分层

为避免功能范围失控，后续实现按三档推进：

| 优先级 | 含义 | 典型功能 |
|------|------|------|
| MVP 必做 | 满足考核硬性要求，必须优先完成 | 基础 Agent、模型切换、Skill 市场、MCP 示例、记忆、OpenAI 兼容入口、Trace、Token 配额、Token 基础看板、脱敏、飞书告警、Docker Compose |
| 高分增强 | 对应考核高分项或生产级亮点 | Planner/Executor/Reviewer Team、Mermaid 协作可视化、告警升级、代码审查、Token 趋势图与多维分析 |
| 可选扩展 | 时间充足时再做，避免拖慢主线 | MCP Server 暴露平台 Skill、普通用户上传 Jar Skill、模型自动降级、智能降噪、Jira/Monitor 深度接入 |

## 域 0：角色与权限体系

> 考核要求：项目 3 需要支持多租户/应用隔离、应用级/用户级 Token 配额。
> 建议方案：Spring Security + JWT + RBAC。管理端和用户端用 JWT，OpenAI 兼容接口用 API Key。
> 做极简登录注册页面（用户名+密码），保留用户、应用、租户模型。API Key 绑定到 Application，用户创建应用时自动生成，一个用户可创建多个应用，每个应用自带一个 Key。
> 
> **双通道架构**：用户注册后获得 JWT（浏览器端），创建 Application 后获得 API Key（代码端调用）。最终按请求类型分流：AI 执行请求统一进入 Gateway 治理链，再进入同一 Agent Runtime；管理、配置、统计、查询类请求走 Web -> Core。两类 AI 入口扣减同一 Token 配额。请求参数 `agent_mode` 控制走 Agent 链路（默认）还是透传模型。

| 编号 | 功能 | 说明 |
|------|------|------|
| R1 | Admin 角色 | 管理模型供应商、全局 Skill 校验/上架/下架、MCP Server、全局安全策略、Token 配额策略、飞书告警配置、查看所有 Trace/Token/告警 |
| R2 | Developer 角色 | MVP：创建垂直领域 Agent Profile、配置领域 Prompt、配置默认/可选 Skill、上传领域 Skill、查看自己应用的 Trace/Token。增强：配置 Team Profile |
| R3 | User 角色 | 选择通用/垂直 Agent、输入请求、选择本次启用 Skill、上传个人私有 Skill、保存个人偏好、查看自己的 Trace/Token/历史对话 |
| R4 | User 禁止操作 | 不能修改公共 Profile、不能修改核心 System Prompt、不能启用未授权高危 Skill、不能影响其他用户、不能绕过安全检查 |
| R5 | 权限入口区分 | 登录注册页面 → JWT（用户名+密码）；Admin 端 → JWT（高权限角色）；对话页面 → JWT；OpenAI 兼容接口 → API Key。JWT 用于浏览器端交互，API Key 用于第三方代码/工具调用 |
| R6 | 凭证获取方式 | 用户注册后自动获得 JWT 登录能力。API Key 绑定到 Application：用户在个人中心创建应用时自动生成 sk- 开头的 Key，一个用户可创建多个应用（如"VS Code 插件""自己脚本"），每个应用一个 Key，可随时吊销 |
| R7 | 用户管理 | 极简登录注册页面（用户名+密码）。Admin 后台可手动创建/禁用/删除用户和应用，分配角色和权限，作为兜底管理手段。找回密码、OAuth 登录等功能后续补 |
| R8 | 多租户/应用隔离 | 保留 tenant_id、app_id、user_id，Trace、Token、Profile、Skill 权限都按租户/应用/用户隔离 |

### 安全策略定义

> 安全策略是一组可配置的规则，作用于 Profile 级别和 Skill 调用级别，是"安全策略"概念的具体化。

| 编号 | 策略项 | 说明 |
|------|------|------|
| SEC1 | 敏感数据规则 | 定义哪些正则规则启用（手机号/身份证/邮箱/银行卡/IP/自定义），作用于请求和响应扫描 |
| SEC2 | 调用频率限制 | 单个用户每分钟/每小时最大调用次数，超限后短时间拒绝 |
| SEC3 | 高危 Skill 确认 | 标记某些 Skill 为"高危"（如部署、创建分支），调用前需用户二次确认 |
| SEC4 | Skill 黑白名单 | 可按 Profile 配置禁止调用的 Skill 列表，覆盖全局 Skill 合成结果 |
| SEC5 | 单次最大 Token | 限制单次请求允许消耗的最大 Token 数，超出则截断或拒绝 |
| SEC6 | 脱敏/阻断模式 | 敏感数据命中后的处理方式：脱敏继续 或 阻断拒绝 |
| SEC7 | 安全事件日志 | 记录所有安全触发事件（脱敏后存储），同步到 Trace |

安全策略配置位置：
- 全局安全策略：Admin 在平台级别配置，所有 Profile 默认继承
- Profile 安全策略：Developer 可在 Profile 中覆盖部分规则（如 Skill 黑白名单、确认策略）
- 用户级安全策略：User 可在个人偏好中从严配置（只能收紧，不能放宽）

## 域 2：Agent Profile 体系

> Profile 是智能体配置，不是业务代码。基础 Agent 是必做能力，Team 模式是高分项；通用模式和垂直模式都可以选择基础 Agent 或 Team 执行链路。

### 2.1 Profile 配置项

| 编号 | 功能 | 说明 |
|------|------|------|
| P1 | 新建 Profile | 配置：名称、描述、模型、补充 Prompt、默认 Skill、可选 Skill、MCP Tool、记忆策略、安全策略、Token 限额、输出风格 |
| P2 | 编辑 Profile | 修改上述配置项 |
| P3 | 删除 Profile | 同时清理关联数据 |
| P4 | 查询 Profile 列表 | 区分通用 Profile 和垂直领域 Profile |
| P5 | 启用/禁用 Profile | 禁用后不可使用 |
| P6 | 复制 Profile | 基于已有 Profile 快速创建 |
| P7 | 通用 Agent 模式 | 灵活开放，用户可选全局 Skill + 个人 Skill，平台内置通用 System Prompt |
| P8 | 垂直领域 Agent 模式 | Developer 预配置领域 Prompt + 领域 Skill，用户开箱即用 |
| P9 | Profile 发布 | Developer 将垂直 Profile 发布为可被 User 发现和使用的领域 Agent |
| P10 | 保存个人偏好 | User 可保存自己的 Profile 设置（不影响公共 Profile） |
| P11 | 执行模式选择 | Profile 预留执行模式字段。MVP 固定使用基础 Agent；Team 执行用于复杂任务和高分演示，阶段 4 再开放配置 |

### 2.2 Prompt 分层

| 编号 | 功能 | 说明 |
|------|------|------|
| P12 | 核心 System Prompt | 平台内置，锁定基础 Agent 或 Planner/Executor/Reviewer 的职责 + 输出格式 + 权限边界，不允许普通用户覆盖 |
| P13 | Profile 补充 Prompt | Developer 或 User 配置的领域偏好、业务规则、输出风格 |
| P14 | 任务上下文 | 运行时动态注入：用户请求、会话记忆、可用 Skill 列表、TraceID、Token/安全策略上下文 |

```
最终 Prompt = 核心 System Prompt（平台锁定）
            + Profile 补充 Prompt（可配置）
            + 当前任务上下文（运行时注入）
```

## 域 3：Skill / Tool 市场

### 3.1 三层 Skill 体系

| 编号 | 功能 | 说明 |
|------|------|------|
| S1 | 全局通用 Skill | Admin 维护，全平台可见，通用和垂直模式都可复用。内置：天气、计算器、日历、翻译、搜索、地图、文件总结 |
| S2 | 垂直领域 Skill | Developer 配置/上传，绑定到特定领域 Agent |
| S3 | 用户自定义 Skill | User 上传个人私有 Skill，默认只对上传者可见，不影响其他用户。MVP 优先支持配置型 Skill，Jar 型 Skill 作为增强能力 |

### 3.2 Skill 类型安全边界

| 编号 | 功能 | 说明 |
|------|------|------|
| S4 | 配置型 Skill | 普通 User 可上传，例如 HTTP API Skill：填写 URL、Method、Header、参数 Schema、返回 Schema，不上传任意代码 |
| S5 | 代码型 Jar Skill | Developer/Admin 上传，适合复杂领域能力。普通 User 上传 Jar 需要沙箱、管理员确认或作为后续增强 |
| S6 | 高危权限声明 | Skill 必须声明是否需要网络、文件、数据库、命令执行等权限，高危权限默认需要 Admin 确认 |

### 3.3 Skill 元数据 & 生命周期

| 编号 | 功能 | 说明 |
|------|------|------|
| S7 | Skill 元数据 | 名称、描述、版本、参数 Schema、返回 Schema、依赖、作用域(scope)、所属者(owner)、权限声明 |
| S8 | Jar 上传 | 接收文件上传，存到磁盘，解析元数据。MVP 主要面向 Developer/Admin |
| S9 | 自动校验 | 上传后进入 VALIDATING：校验元数据完整性、参数 Schema、Skill 接口实现、依赖允许列表、权限声明。注意：自动校验不能完全证明 Jar 没有恶意行为 |
| S10 | 状态流转 | UPLOADED → VALIDATING → INSTALLED → ENABLED → LOADED；可流转到 DISABLED / FAILED / UNINSTALLED |
| S11 | 热加载 | 独立 ClassLoader 加载 Jar → 校验实现 Skill 接口 → 反射实例化 → 注册到 SkillRegistry |
| S12 | 热卸载 | 关闭 ClassLoader → 从缓存移除 → GC 回收 |
| S13 | 类加载器隔离 | 每个 Skill 独立 ClassLoader 实例，打破双亲委派。该能力解决依赖隔离，不等价于安全沙箱 |
| S14 | 启用/禁用 | 控制 Skill 在市场中是否可用 |
| S15 | Admin 上下架 | Admin 可将已校验通过的 Skill 发布到全局市场，也可下架。人工审核作为增强能力，不阻塞 MVP |

### 3.4 Skill 合成规则

| 编号 | 功能 | 说明 |
|------|------|------|
| S16 | 通用模式合成 | 可用 Skill = 全局通用 Skill + 用户自定义 Skill |
| S17 | 垂直模式合成 | 可用 Skill = 全局通用 Skill + 当前领域 Skill + 用户自定义 Skill |
| S18 | 最终过滤 | 上述候选 Skill ∩ 用户权限 ∩ 应用权限 ∩ 安全策略 ∩ 用户本次勾选 |
| S19 | 默认/可选区分 | 领域 Skill 中，必选的自动启用，可选的由 User 本次勾选 |

## 域 5：Agent Runtime 执行引擎

> 基础 Agent 是必做能力，用于普通对话和简单工具调用；Team 模式是高分项，用于复杂任务协作。两者共用模型调用、Skill 合成、记忆、Trace、Token、脱敏和告警能力。

### 5.1 基础 Agent 执行链路

| 编号 | 功能 | 说明 |
|------|------|------|
| BA1 | 默认 Agent Runtime | 接收用户请求，加载 Profile、记忆和可用 Skill，调用模型生成回复 |
| BA2 | 工具调用 | 通过 Function Calling 或 ReAct 调用 Skill / MCP Tool |
| BA3 | 运行时动态切换模型 | 根据 Profile 或请求参数选择模型 |
| BA4 | 上下文构建 | 拼接核心 System Prompt + Profile 补充 Prompt + 历史消息 + 用户消息 + 工具列表 |
| BA5 | 基础 Agent 场景 | 适合普通问答、翻译、总结、简单工具调用，满足项目 1 基础 Agent 架构要求 |

### 5.2 ReAct 循环

| 编号 | 功能 | 说明 |
|------|------|------|
| RX1 | while 循环控制 | 最大步数限制(默认10)、终止条件判断(finish_reason=stop) |
| RX2 | Think 阶段 | 调用 LLM，解析返回内容，判断是回复还是需要调工具 |
| RX3 | Act 阶段 | 解析 tool_calls → 查 SkillRegistry → 执行 → 拿结果 |
| RX4 | Observe 阶段 | 把工具执行结果拼成 role=tool 消息追加到 messages |
| RX5 | SSE 推送 | 每一步实时推：thinking / action / observation / message / done / error |
| RX6 | 错误处理 | LLM 调用失败重试、工具执行异常捕获、超时处理、模型降级 |
| RX7 | 上下文构建 | 拼接核心 System Prompt + Profile 补充 Prompt + 历史消息 + 用户消息 + 工具列表 |
| RX8 | 工具列表生成 | 根据 Skill 合成结果生成 Function Calling 格式的工具定义 |

### 5.3 Team 协作流程（高分项）

| 编号 | 功能 | 说明 |
|------|------|------|
| T1 | 三角色定义 | Planner(规划) / Executor(执行) / Reviewer(审查) ——平台内置，用户不需要编写 Agent 代码 |
| T2 | 角色职责 | Planner：只拆任务、不调工具、不生成最终答案。Executor：只执行分配的子任务、不重新规划。Reviewer：只审查、不规划、不执行 |
| T3 | 核心 System Prompt | 每个角色有平台内置的锁定 Prompt，固定职责+输出 Schema+权限边界 |
| T4 | Skill 绑定方式 | Skill 不分别绑给三个角色，而是统一绑到任务/领域/用户。三个角色里只有 Executor 拿到实际调用权限 |
| T5 | 消息通道 | BlockingQueue<TeamMessage> 实现角色间消息传递（任务分发、结果汇报） |

### 5.4 Planner

| 编号 | 功能 | 说明 |
|------|------|------|
| T6 | 任务拆解 | 输入：用户请求 + Profile + 可用 Skill 列表(名称/描述/参数Schema) + 记忆 + Planner System Prompt → LLM 分析并拆解 → 输出 TaskPlan JSON |
| T7 | TaskPlan Schema | `{goal, tasks:[{id, name, description, suggestedSkill, arguments, dependsOn}]}` |
| T8 | Schema 校验 | 后端校验：JSON合法、taskId唯一、suggestedSkill在可用列表、dependsOn引用存在、无循环依赖、任务数不超限 |
| T9 | 校验失败处理 | 首次失败→返回错误要求 Planner 修正；二次失败→降级为单任务或返回规划失败 |

### 5.5 Executor

| 编号 | 功能 | 说明 |
|------|------|------|
| T10 | 按依赖执行 | 解析 dependsOn → 拓扑排序 → 按序执行子任务，调用对应 Skill |
| T11 | 调用前校验 | 任务是否可执行、依赖是否完成、Skill 是否授权、参数是否合 Schema、是否触发敏感策略、是否需二次确认、是否超 Token/次数限制 |
| T12 | ExecutionResult Schema | `{taskId, status(success|failed|skipped), result, usedTools, errorMessage}` |

### 5.6 Reviewer

| 编号 | 功能 | 说明 |
|------|------|------|
| T13 | 审查执行结果 | 检查是否完成用户目标、有无遗漏/冲突/错误 → 输出 ReviewResult |
| T14 | ReviewResult Schema | `{passed, issues:[{taskId, level(info|warning|error), message}], retryTasks, summary}` |
| T15 | 返工机制 | Reviewer 指出需重试的 taskId → Executor 重新执行 → 再次提交审查 |
| T16 | ReviewResult 校验 | 后端校验：符合 Schema、有明确通过/不通过、retryTasks 引用真实任务 |

### 5.7 结果汇总

| 编号 | 功能 | 说明 |
|------|------|------|
| T17 | 最终汇总 | Planner 收齐全部 ExecutionResult 和 ReviewResult 后调用 LLM 生成最终回答 |
| T18 | LLM 动态决策 | 不写死脚本流程，Planner 用 LLM 拆任务而非 if-else，每个角色根据上下文自主决策 |
| T19 | 仲裁机制 | Reviewer 不通过时先让 Executor 重试；重试仍失败则由 Orchestrator 仲裁，决定继续重试、降级基础 Agent、返回部分结果或返回失败原因 |

| 编号 | 功能 | 说明 |
|------|------|------|
| SD1 | 正则规则扫描 | 手机号、身份证、邮箱、银行卡、IP、自定义关键词 |
| SD2 | 请求入站扫描 | 用户消息进入时扫描 |
| SD3 | 响应出站扫描 | LLM 返回内容发出前扫描 |
| SD4 | 脱敏模式 | 命中后替换为占位符（138****1234），继续处理 |
| SD5 | 阻断模式 | 命中后直接拒绝请求，返回错误 |
| SD6 | 事件日志 | 记录命中事件（脱敏后存储，不存原始敏感数据） |
| SD7 | 扫描范围扩展 | 扫描用户请求、Planner 输出、Executor 工具入参、Skill/MCP 返回、Reviewer 输出、最终响应、记忆写入内容、Trace 和日志内容 |

### 7.4 告警联动

| 编号 | 功能 | 说明 |
|------|------|------|
| AL1 | 告警触发条件 | 模型调用失败 / Skill 调用失败 / MCP 调用失败 / 错误率超阈值 / Token 超限 / 敏感数据阻断 / 敏感数据逃逸 / 延迟过高 |
| AL2 | 告警消息 | TraceID、告警级别(INFO/WARN/CRITICAL)、条件描述、建议动作 |
| AL3 | 调项目 2 发消息 | HTTP 调 feishu-bot 内部接口，发飞书卡片消息 |
| AL4 | 静默期 | 同一 app 同一告警类型 N 分钟内不重复发送 |
| AL5 | 升级策略 | 持续告警超过阈值自动提升级别 |

### 7.5 网关拦截器链

```
请求 → TenantFilter(租户识别)
     → TraceFilter(创建TraceID+rootSpan)
     → SensitiveDataFilter(入站敏感扫描)
     → QuotaFilter(Token配额检查)
     → RuntimeInvokeFilter(调用 core AgentRuntime / 模型透传)
     → SensitiveResponseFilter(出站敏感扫描)
     → TokenRecordFilter(Token统计入库)
     → AlertFilter(告警条件判断)
     → 响应
```

GatewayContext 用 ThreadLocal 贯穿整条链，请求结束在 finally 中清理。

## 与早期草案的关键差异

| 差异点 | 早期草案 | 当前结论 |
|--------|-----------|--------|
| 登录机制 | Spring Security + JWT + RBAC（需登录页面） | 做极简用户名密码注册登录；找回密码、OAuth、复杂账号体系暂缓 |
| 用户模型 | 单用户或弱用户模型 | 保留 tenant_id / app_id / user_id，满足多租户、应用隔离、用户级配额 |
| 使用模式 | 单 Agent + Team 两种 | 基础 Agent 必做，Team 模式作为高分项；通用/垂直模式都可选择执行链路 |
| Profile 类型 | 通用 / 垂直 / Team / Personal 分开建模 | 不再把 Team、Personal 做成独立 Profile 类型；统一为通用 Profile / 垂直 Profile + 执行模式选择，Personal 作为 User 偏好设置 |
| 功能域划分 | 13 节 | 11 个域，每个域下编号可追踪，新增部署与交付 |
| Skill 上架 | 人工审核流程 | MVP 自动校验通过即可用，Admin 发布全局市场作为增强能力 |
| Agent Team 校验 | 有校验列表 | 增加了 JSON Schema、降级机制、重试策略的详细描述 |
| 拦截器链顺序 | Tenant→Trace→Sensitive→Quota→... | 修正为 Tenant→Trace→Sensitive→Quota→...（一致） |
