# 双通道入口与权限设计

> 本文档汇总了「登录方式、权限入口、双通道架构、Application/Key 关系、agent_mode 行为、Token 配额、429 拦截、后续支付扩展」的最终设计结论。对应 01-项目完整功能清单 的域 0（角色与权限）和域 1（模型与统一调用入口）中的双通道部分。

---

## 1. 用户注册与登录

```
用户打开平台首页
  → 极简登录页（用户名 + 密码）
  → 没有账号 → 点"注册" → 填写用户名、密码、确认密码
  → 注册成功，自动登录，拿到 JWT
```

- MVP 只做用户名+密码登录注册，不做验证码、不接 OAuth。
- 找回密码、美化登录页、第三方登录 → 后续增强。
- Admin 初始账号通过配置文件预设。

## 2. 双通道入口与最终路由

用户注册登录后，通过两个入口访问平台。最终路由按“请求类型”分流，而不是按“角色”分流：

- AI 执行请求：统一进入 Gateway 治理链，再进入 core Agent Runtime。
- 管理、配置、统计、查询请求：走 Web -> Core，不经过 Gateway。
- 两类 AI 入口共用同一 Agent Runtime、同一 Token 配额、同一 Trace/脱敏/告警治理能力。

```
用户注册（极简页：用户名+密码）
  → 登录拿 JWT
  → 创建 Application → 自动生成 API Key（sk- 开头）
                         一个用户可创建多个应用，每个应用一个 Key，可单独吊销
  │
  ├── 通道 A：JWT + 浏览器
  │    鉴权方式：登录后浏览器自动带 JWT
  │    用途1：人在网页上选 Agent、勾 Skill、对话
  │    体验：SSE 实时流（thinking/action/observation/message/done）
  │    计费：扣减用户 Token 配额
  │    路由：AI 对话/测试调用 → Web → Gateway → Core
  │    用途2：Admin/User/Developer 管理配置与统计查询
  │    路由：管理配置查询 → Web → Core
  │
  └── 通道 B：API Key + 代码/工具
       鉴权方式：Authorization: Bearer sk-xxx
       用途：开发者在自己的代码/脚本/工具里调 /v1/chat/completions
       体验：OpenAI 兼容格式，现有生态工具直接接入
       计费：扣减同一用户的 Token 配额
       路由：外部 AI 调用 → Gateway → Core
```

**结论：不是所有 Web 请求都过 Gateway，只有 AI 执行类请求过 Gateway。Gateway 是统一 AI 调用治理入口，Web 是浏览器和管理后台入口。**

### 2.1 Web 转发 Gateway 的机制

浏览器 AI 对话/测试请求由 Web 通过 HTTP 内部接口转发到 Gateway，携带内部服务 Token 或签名 Header。Gateway 校验通过后才接受 user_id/app_id/tenant_id，防止外部直接伪造内部身份。

阶段 1 先建立 Gateway 空壳：Web 转发 Gateway、Gateway 再转发到 Core，仅做请求透传，不启用完整治理链（Trace/Token/脱敏/告警）。阶段 2 在 Gateway 内部补上 8 步拦截器链。阶段 1 的 Web → Gateway → Core 路由和阶段 2 完全一致，只是 Gateway 内部实现从空壳换成完整治理链，对外部调用方完全透明。这样阶段 1 已经定了最终路由方向，降低后续重构成本。

## 3. 权限入口区分

| 入口 | 鉴权方式 | 适用角色 |
|------|---------|---------|
| 登录注册页面 | JWT（用户名+密码） | 所有用户 |
| Admin 管理端 | JWT（高权限角色） | Admin |
| User 对话页面 | JWT（登录后自动带） | User / Developer |
| OpenAI 兼容接口 | API Key（sk- 开头） | 第三方代码/工具 |

- JWT 用于浏览器端的人机交互。
- API Key 用于代码端的自动化调用。
- 浏览器端 AI 对话、Developer 测试 Profile、Admin 测试模型/Agent 都属于 AI 执行请求，需要由 Web 转发到 Gateway 治理链。
- Admin 管理模型、Skill、MCP、配额、安全策略，Developer 管理 Profile/Skill，User 查看历史/Trace/Token/API Key，属于管理配置查询请求，走 Web -> Core。
- 权限落地方式：Spring Security + JWT + RBAC + API Key。

## 4. Application 与 API Key

### 4.1 关系模型

```
用户 zhangsan
  │
  ├── Application "我的 VS Code 插件"
  │     → API Key: sk-abc123
  │
  ├── Application "公司内部脚本"
  │     → API Key: sk-def456
  │
  └── Application "测试用"
        → API Key: sk-ghi789
```

### 4.2 关键规则

- Key 绑定 Application，不直接绑 User。
- 用户创建 Application 时自动生成 Key，一个应用一个 Key。
- 一个用户可创建多个 Application，不同应用不同 Key。
- Token 消耗按 Application 分别统计。
- Key 泄露只吊销那一个，不影响其他 Application。
- 用户可随时在个人中心吊销任意 Key。

### 4.3 为什么用 Application 而不是个人 Key

- 泄漏隔离：一个 Key 泄漏，吊销它即可，不影响其他。
- 消费可见：每个应用的 Token 消耗单独统计，哪个应用花得多一目了然。
- 配额控制：可针对某个 Application 单独设限额。
- 典型用法：不同工具/脚本用不同 Key，出问题好定位。

## 5. Agent 模式 vs 透传模式

请求参数 `agent_mode` 控制执行路径：

| 值 | 行为 | 默认 |
|----|------|------|
| `agent` | 走完整 Agent Runtime（Profile + Skill 合成 + ReAct 循环 + 记忆） | 是 |
| `none` | 纯模型转发，不走 Agent，直接调模型返回 | 否 |

### 5.1 各通道的模式限制

| 通道 | agent_mode 限制 |
|------|----------------|
| JWT + 浏览器（通道 A） | 始终走 agent 模式（人在网页上使用必然走完整 Agent） |
| API Key + 代码（通道 B） | 请求参数可选 agent 或 none |

说明：通道 A 的 `agent_mode=agent` 限制只作用于浏览器 AI 执行请求；管理配置查询请求不进入 Agent Runtime，也不需要 `agent_mode`。

### 5.2 透传模式用途

开发者在某些场景只想当纯中转站用：
- 简单问答不需要 Skill
- 已有自己的 Agent 逻辑，只想调模型
- 对比测试：裸模型 vs Agent 的输出差异

透传模式与 Agent 模式共用同一 Token 配额和 Trace。

## 6. Token 配额策略

### 6.1 配额层级

| 层级 | 说明 |
|------|------|
| 用户级配额 | 每个 user 配置日/月 Token 上限 |
| 应用级配额 | 每个 application 配置日/月 Token 上限 |

### 6.2 扣减流程

```
请求进入 → QuotaFilter 检查配额
  → 按预估 Token 通过数据库事务或 Redis 原子预扣
  → 放行进入 Agent Runtime 或模型透传
  → 请求完成 → 按模型返回的真实 usage 修正扣减
  → 请求失败 → 释放预扣额度
```

### 6.3 Token 来源优先级

```
1. 模型返回 usage → 直接使用
2. 模型不返回 usage → 用 tokenizer 估算，标记 estimated=true
```

### 6.4 配额用尽

- 两类 AI 入口同一配额，任意入口耗尽后 AI 调用全停；管理配置查询仍可登录查看和调整配额。
- 超限返回 **HTTP 429 + 错误信息**：
  ```json
  {
    "error": "quota_exceeded",
    "limit": 100000,
    "used": 100000,
    "reset_at": "2026-05-19T00:00:00Z"
  }
  ```
- 配额用尽触发飞书告警。

## 7. Token 配额与后续支付扩展

- **MVP 不做真实支付和充值页面。**
- Token 配额由 Admin 后台按用户/应用手动配置日限额和月限额。
- 后续可扩展：
  - 对接支付宝/微信真实支付
  - 用户自助充值页面
  - 按 Token 计费（¥/1K tokens）
  - 发票、账单明细

## 8. 平台定位总结

```
你的平台 = 中转站的能力 + Agent 执行引擎 + AI Infra 治理

  ┌────────────┐     ┌────────────────────┐
  │  中转站能力  │     │  Agent 超集能力     │
  │            │     │                    │
  │ 用户注册    │     │ ReAct while 循环    │
  │ 生成 Key   │     │ Skill/Tool 调用     │
  │ 按量计费    │     │ Profile 管理        │
  │ 配额拦截    │     │ 长期记忆            │
  │ OpenAI 兼容 │     │ Team 协作           │
  │ Trace 追踪  │     │ 全链路 Trace        │
  │            │     │ 敏感数据脱敏         │
  └────────────┘     │ 飞书告警            │
                      └────────────────────┘

  不是"不需要中转站"，而是中转站变成了平台的一个功能子集。
```
