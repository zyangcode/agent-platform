# 模块目录与包结构设计方法

> 本文沉淀本项目在设计 `04-模块目录与包结构设计.md` 过程中形成的原则、经验和检查方法。后续新增模块、写代码或调整目录时，优先按本文判断。

---

## 1. 核心结论

模块目录设计不是为了把文件摆整齐，而是为了控制依赖方向、数据归属和后续变更成本。

本项目最终采用：

```text
少量 Maven 工程单元
+ core 内部清晰业务包
+ ArchUnit 包边界测试
```

也就是：

```text
Maven 模块不要过度拆
Java 包边界不能放松
表归属必须唯一
跨包调用必须走 api
入口层不能拥有业务表
```

---

## 2. 为什么不把每个能力都拆成 Maven 模块

完整能力边界可以拆出很多模块：

```text
auth
user
model
agent
skill
mcp
memory
quota
trace
security
alert
team
```

但 MVP 阶段一个人或小团队开发时，过早拆成大量 Maven 模块会带来额外成本：

- `pom.xml` 和依赖管理变复杂。
- DTO / Command 容易被提前抽到 common，污染公共模块。
- 包扫描、启动配置、测试配置变重。
- 小改动经常跨多个 Maven 模块。
- 主链路还没跑通，工程复杂度先上来了。

所以当前采用四个后端 Maven 模块：

```text
common
core
web
gateway
```

再把业务能力收敛到 `core` 内部包：

```text
core.identity
core.model
core.profile
core.skill
core.mcp
core.agent
core.memory
core.context
core.quota
core.trace
core.security
core.alert
core.team
```

经验：物理模块可以少，但逻辑边界必须清楚。

---

## 3. 模块拆分的判断原则

### 3.1 先问职责

每个包必须能用一句话说清楚职责。

示例：

```text
core.profile 负责 Agent Profile 配置。
core.agent 负责 Agent Runtime 编排。
core.context 负责模型执行上下文组装。
core.quota 负责 Token 配额、预扣和结算。
```

如果一句话说不清，说明包边界可能太散。

### 3.2 再问表归属

有表的业务能力，必须明确表归属。

```text
skills -> core.skill
model_configs -> core.model
conversations -> core.agent
memories -> core.memory
token_usage_logs -> core.quota
trace_spans -> core.trace
```

原则：

```text
谁拥有表，谁拥有 Entity / Mapper / Repository。
其他包只能通过 api 查询或操作。
```

### 3.3 最后问调用方向

调用方向必须稳定。

允许：

```text
web -> core
gateway -> core
core -> common
core.agent -> core.skill.api
core.agent -> core.model.api
```

禁止：

```text
core -> web
core -> gateway
common -> core
core.agent -> core.skill.mapper
gateway -> core.quota.mapper
web -> gateway Java 代码依赖
```

## 4. common 的边界

`common` 只放真正通用的基础能力：

```text
错误码
异常
统一响应
分页结果
时间工具
ID 工具
JSON 工具
```

禁止放：

```text
业务 Entity
业务 DTO
业务 Command
业务 Service
Mapper
```

经验：common 一旦放业务对象，很快会变成所有模块都依赖的杂物模块。

---

## 5. web、gateway、core 的分工

### 5.1 web

`web` 是浏览器入口。

负责：

```text
JWT 登录态
Admin/User/Developer REST API
浏览器 SSE 转发
文件上传
AI 请求转发到 Gateway
```

不负责：

```text
Agent Runtime
Token 结算
Trace 落库
直接查业务表
```

### 5.2 gateway

`gateway` 是 AI 执行请求的统一治理入口。

负责：

```text
OpenAI 兼容接口
API Key 鉴权
Web 内部 AI 请求接入
GatewayContext
Trace / Token / 脱敏 / 告警治理链
Agent 模式和纯模型透传分流
SSE / OpenAI Stream 协议适配
```

不负责：

```text
组装 Agent Prompt
读取 Profile 表
读取 Skill 表
保存 conversation_messages
直接写 token_usage_logs
```

### 5.3 core

`core` 是核心业务层。

负责：

```text
身份、模型、Profile、Skill、MCP
Agent Runtime
上下文组装
记忆
配额
Trace
安全
告警事件
```

core 内部包之间也要遵守边界，不能因为都在一个 Maven 模块里就互相乱调。

## 6. GatewayContext 和 AgentContext 的区别

这两个名字都叫 Context，但含义不同。

### 6.1 GatewayContext

属于 `gateway.context`。

保存请求治理信息：

```text
tenant_id
user_id
application_id
channel
agent_mode
trace_id
```

用途：

```text
鉴权
Trace
Quota
脱敏
告警
调用 core 前构造 Command
```

### 6.2 AgentContext

属于 `core.context`。

保存模型执行上下文：

```text
System Prompt
Profile Prompt
用户输入
历史消息
短期记忆
长期记忆召回
可用 Skill 列表
可用 MCP Tool 列表
安全策略摘要
```

用途：

```text
给模型调用提供完整上下文
控制上下文 token 预算
决定压缩、裁剪、保留哪些内容
```

原则：

```text
Gateway 不组装 AgentContext。
core 不读取 Gateway 的 ThreadLocal。
GatewayContext 必须显式转换成 AgentRunCommand / ModelInvokeCommand。
```

## 7. 上下文 Token 超额压缩放在哪里

上下文 Token 超额压缩属于 `core.context`。

推荐结构：

```text
core.context
├── api/
│   └── AgentContextBuilder
├── application/
│   └── DefaultAgentContextBuilder
└── domain/
    ├── ContextBudget
    ├── ContextWindowManager
    └── ContextCompressor
```

处理流程：

```text
加载 Profile Prompt
加载用户输入
加载历史消息
加载长期记忆
加载 Skill / MCP 工具描述
估算 token
超过模型上下文限制时压缩或裁剪
返回 AgentContext
```

优先保留：

```text
System Prompt
安全策略
当前用户输入
必要工具协议
最近几轮对话
```

可以压缩或裁剪：

```text
较早历史消息
长期记忆召回内容
非必选 Skill 描述
工具返回的长文本
```

不要把这类逻辑放进 Gateway。Gateway 只处理用户配额 Token，不处理模型上下文窗口。

## 8. SSE 转发经验

浏览器 AI 对话链路是：

```text
frontend -> web -> gateway -> core
```

如果 core/gateway 产生 SSE 流，web 还要再转发给浏览器，中间多一跳，风险是：

```text
事件被缓冲
连接超时
错误事件丢失
浏览器断开后后端没有取消执行
```

阶段 1 Gateway 空壳完成后，必须先做固定 SSE 流透传验证：

```text
gateway 每 500ms 输出一个事件
web 逐事件透传
curl 或浏览器能逐条收到
gateway 出错时 web 能返回 error 事件
```

不要只测“最后拿到完整回答”，那不能证明流式链路是通的。

## 9. Token 配额设计经验

Token 配额不是普通日志，必须保证：

```text
预扣原子
释放幂等
结算幂等
失败可重试
```

禁止：

```text
先查剩余额度
再普通 update 扣减
```

推荐两种方式：

```text
方式 1：version CAS
UPDATE quota_configs
SET used = used + ?, version = version + 1
WHERE id = ? AND version = ? AND remaining >= ?

方式 2：行锁
SELECT ... FROM quota_configs WHERE id = ? FOR UPDATE
```

`quota_reservations` 要支持幂等：

```text
trace_id 唯一
status 明确
version 乐观锁
```

状态流转：

```text
RESERVED -> COMMITTED
RESERVED -> RELEASED
RESERVED -> FAILED
```

释放和结算必须按条件更新：

```text
trace_id + status + version
```

避免重复释放或重复扣减。

## 10. ArchUnit 作为边界护栏

core 内部包边界不能只靠人工 code review。

必须用 ArchUnit 或等价架构测试约束：

```text
web 不能访问 core.*.mapper
gateway 不能访问 core.*.mapper
core.agent 不能访问 core.skill.mapper
core.agent 不能访问 core.model.mapper
core.agent 不能访问 core.quota.mapper
common 不能访问任何业务模块
```

允许访问：

```text
core.xxx.api
core.xxx.dto
core.xxx.command
```

禁止跨包访问：

```text
core.xxx.entity
core.xxx.mapper
core.xxx.internal
```

经验：生成 Entity / Mapper 后就加入 ArchUnit，不要等代码写多了再补。

## 11. 告警不要阻塞主链路

飞书告警属于边缘动作。

错误做法：

```text
用户请求等待 feishu-bot HTTP 返回
feishu-bot 失败导致 AI 对话失败
```

正确做法：

```text
AlertFilter 写 alert_events
通知适配器异步调用 feishu-bot
失败只更新 notify_status / retry_count
用户 AI 响应不受影响
```

原则：

```text
核心审计数据可靠写入。
外部通知可以异步重试。
```

## 12. 模块设计检查清单

新增一个类、包或表之前，先问：

```text
1. 它属于哪类业务能力？
2. 它操作哪张表？
3. 这张表归哪个 core 包？
4. 它是对外 api，还是内部实现？
5. 调用方是否应该知道它的 Entity / Mapper？
6. 是否可以通过 Command / Result 表达输入输出？
7. 是否会让 web/gateway/common 持有业务逻辑？
8. 是否需要 ArchUnit 增加边界规则？
```

如果一个类需要跨多个业务域、直接查多张不属于自己的表，或者让入口层开始持有业务逻辑，就说明设计已经开始失控。

---

## 13. 最终经验

本项目的模块设计经验可以概括为：

```text
MVP 不过度拆 Maven 模块。
业务边界必须在 Java 包里保留。
表归属决定 Entity / Mapper 位置。
跨包调用只走 api，不碰 mapper。
Gateway 管治理，不管 Agent 上下文。
core.context 管上下文，不管用户配额。
Token 配额要原子和幂等。
SSE 双跳要尽早做透传验证。
ArchUnit 把文档纪律变成测试失败。
```

这套方法的目标不是让目录看起来复杂，而是让后续功能增加时，代码不会因为依赖混乱而推翻重做。
