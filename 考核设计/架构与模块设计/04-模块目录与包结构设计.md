## 2. 总体结论

### 2.1 工程单元

MVP 阶段不把项目拆得过细，采用 6 个工程单元：

```text
agent-platform/
├── agent-platform-common/      公共基础模块
├── agent-platform-core/        平台核心业务模块
├── agent-platform-web/         浏览器端入口模块
├── agent-platform-gateway/     AI 统一调用网关模块
├── agent-platform-frontend/    React 前端工程
└── feishu-bot/                 飞书机器人独立服务
```

说明：

- `agent-platform-common`、`agent-platform-core`、`agent-platform-web`、`agent-platform-gateway` 是 Maven 模块。
- `agent-platform-frontend` 是前端工程，不是 Maven 模块。
- `feishu-bot` 是独立 Spring Boot 服务，不依赖 `agent-platform` 内部代码。

### 2.2 依赖方向

```text
frontend
  ↓ HTTP
web
  ├──→ core              Maven 代码依赖，用于管理/配置/查询类请求
  └──→ gateway           HTTP 内部调用，用于浏览器 AI 执行请求

gateway
  └──→ core              Maven 代码依赖，用于调用 core 对外 api 接口

core
  └──→ common

web / gateway
  └──→ common

feishu-bot
  └──→ 自己的 common / 自己的 DB

gateway / 告警通知适配器
  └──→ feishu-bot HTTP 告警接口
```

禁止方向：

```text
core 不能依赖 web
core 不能依赖 gateway
common 不能依赖任何业务模块
feishu-bot 不能直接依赖 agent-platform 代码
web / gateway 不能拥有业务表
web 不能以 Maven 依赖方式引用 gateway 内部代码或复用 gateway DTO
```

说明：

- `web -> gateway` 是运行时 HTTP 内部调用，不是 Maven 编译依赖。Web 侧使用自己的请求/响应 DTO，通过 HTTP Client 调 Gateway 内部接口。
- `gateway -> core` 是 Maven 代码依赖。Gateway 只依赖 `core.xxx.api` 暴露的接口和 Command/DTO，不直接访问 core 内部 Entity/Mapper。
- `core.alert` 负责平台侧告警规则和告警事件，不直接依赖 `feishu-bot` 代码；发送飞书消息由 Gateway、通知适配器或后续异步 worker 通过 HTTP 调用 `feishu-bot`。

## 3. 推荐目录结构

### 3.1 根目录结构

```text
agent-platform/
├── pom.xml
├── CLAUDE.md
├──考核设计/
├── agent-platform-common/
│   ├── pom.xml
│   └── src/main/java/com/ls/agent/common/
├── agent-platform-core/
│   ├── pom.xml
│   └── src/main/java/com/ls/agent/core/
├── agent-platform-web/
│   ├── pom.xml
│   └── src/main/java/com/ls/agent/web/
├── agent-platform-gateway/
│   ├── pom.xml
│   └── src/main/java/com/ls/agent/gateway/
├── agent-platform-frontend/
│   ├── package.json
│   └── src/
└── feishu-bot/
    ├── pom.xml
    └── src/main/java/com/ls/feishu/
```

包名前缀采用 `com.ls.agent`，其中 `ls` 表示蓝山项目。后续如果要改成 `com.blueshan` 或学校/团队域名，必须在正式建代码前统一替换，建代码后不要再分散修改。

`feishu-bot` 使用 `com.ls.feishu` 是因为它是独立服务，不复用 `agent-platform` 内部包；这样可以从包名上直接区分平台主工程和飞书机器人服务。

## 4. common 模块

### 4.1 职责

`common` 只放所有模块都能安全复用的基础能力。

```text
com.ls.agent.common
├── error/          错误码、业务异常、系统异常
├── response/       统一响应结构
├── enums/          通用枚举，不放强业务枚举
├── util/           纯工具类，如时间、ID、脱敏展示工具
└── constants/      通用常量
```

### 4.2 允许放入

- `BizException`
- `ErrorCode`
- `ApiResponse<T>`
- `PageResult<T>`
- `IdGenerator`
- `JsonUtils`
- `TimeUtils`

### 4.3 禁止放入

- 禁止放 Entity。
- 禁止放业务 DTO。
- 禁止放 Mapper。
- 禁止放 Service。
- 禁止放 `UserEntity`、`AgentProfileDto`、`SkillCommand` 这类属于具体业务域的对象。

原因：`common` 一旦放业务对象，很快会变成所有模块互相依赖的垃圾场。

## 5. core 模块

`core` 是平台核心业务模块。正式数据库模型中的平台核心业务表都归 `core` 内部逻辑包管理。

### 5.1 core 总体包结构

```text
com.ls.agent.core
├── identity/       租户、用户、角色、应用、API Key
├── model/          模型供应商、模型配置、ChatClient 创建
├── profile/        Agent Profile、Prompt 组装、Profile 工具绑定
├── skill/          Skill 注册、版本、执行、热加载
├── mcp/            MCP Server、MCP Tool、MCP Client
├── agent/          Agent Runtime、ReAct 主循环、会话消息
├── memory/         短期窗口、长期记忆
├── quota/          Token 配额、预扣、结算、用量统计
├── trace/          TraceID、Trace Root、Span、Trace 查询
├── security/       敏感数据检测、脱敏、阻断、安全事件
├── alert/          告警规则、告警事件、通知意图
├── team/           Planner / Executor / Reviewer（阶段 4）
├── context/        Agent 执行上下文组装
├── support/        core 内部基础支撑，如持久化基类、TypeHandler
└── config/         core 内部 Spring 配置
```

### 5.2 每个业务包的内部标准结构

业务包内部统一采用类似结构：

```text
core.xxx/
├── api/            对其他包暴露的接口
├── application/    应用服务，负责业务用例编排
├── domain/         领域对象、领域服务、策略
├── entity/         数据库 Entity
├── mapper/         MyBatis-Plus Mapper
├── dto/            本包对外返回 DTO
├── command/        本包对外接收 Command
├── repository/     数据访问封装，可选
└── internal/       包内私有实现，不允许跨包调用
```

阶段 1 可以简化，不要求每个包都立刻建满这些目录。推荐规则：

```text
有表的包：entity + mapper + application + api
有复杂策略的包：domain
只是适配外部调用的包：client / adapter
暂缓阶段的包：先不建，或只放接口占位
```

### 5.3 core 内部包边界强制规则

`core` 内部虽然在同一个 Maven 模块里，但不能只靠人工纪律维护边界。代码落地时必须加入 ArchUnit 或等价架构测试，把包访问规则变成测试失败。

核心规则：

```text
core.xxx.api / dto / command
  可以被其他 core 包访问

core.xxx.application / domain / repository
  原则上只允许本包内部使用，确需跨包调用时先抽 api

core.xxx.entity / mapper / internal
  禁止被其他业务包、web、gateway 直接访问
```

典型约束：

```text
core.agent 不能访问 core.skill.mapper
core.agent 不能访问 core.model.mapper
core.agent 不能访问 core.quota.mapper
core.gateway 不存在，gateway 模块不能访问任何 core.*.mapper
web 模块不能访问任何 core.*.mapper
common 不能访问 core/web/gateway/feishu-bot
```

阶段 1 生成 Entity / Mapper 后就要加入这些检查，避免代码量变大后再补规则导致大面积返工。

## 6. core 包职责、表归属与对外接口

### 6.1 core.identity

负责身份、租户、角色、应用、API Key。

归属表：

```text
tenants
users
roles
user_roles
applications
api_keys
```

推荐接口：

```text
IdentityQueryService
AuthService
ApplicationService
ApiKeyService
TenantContextService
```

说明：

```text
阶段 1 Step 6 可先只落地 AuthService / ApplicationService。
ApiKeyService 在 Gateway API Key 入口实现前补齐，用于认证外部 sk-xxx。
PermissionService 属于后续可拆能力，阶段 1 资源归属校验先放各业务 Application Service。
```

禁止：

- 禁止处理 Agent Runtime。
- 禁止直接扣 Token。
- 禁止直接写 Trace Span。

典型调用：

```text
gateway 通过 ApiKeyService 校验 API Key
web 通过 AuthService 登录注册
quota 通过 IdentityQueryService 校验用户/应用是否存在
```

### 6.2 core.model

负责模型供应商、模型配置、模型客户端创建。

归属表：

```text
model_providers
model_configs
```

推荐接口：

```text
ModelConfigService
ModelClientFactory
ModelInvokeService
```

禁止：

- 禁止直接读取 Profile 表。
- 禁止直接记录 Token 用量。
- 禁止直接处理 Gateway 鉴权。

说明：

`model` 只负责“如何调用模型”。至于“该用哪个模型”，由 `profile` 或运行时上下文决定。

### 6.3 core.profile

负责 Agent Profile、Prompt 补充配置、模型绑定、Skill/MCP 绑定关系。

归属表：

```text
agent_profiles
profile_skills
profile_mcp_tools
```

推荐接口：

```text
ProfileService
PromptAssemblyService
ProfileToolBindingService
```

禁止：

- 禁止执行 Skill。
- 禁止直接调用模型。
- 禁止写 conversation_messages。

说明：

`profile` 负责“配置是什么”，不负责“执行过程怎么跑”。

### 6.4 core.skill

负责 Skill 市场、Skill 元数据、版本、上传物、执行适配。

归属表：

```text
skills
skill_versions
skill_artifacts
```

推荐接口：

```text
SkillRegistry
SkillQueryService
SkillInstallService
SkillExecutor
SkillValidationService
```

禁止：

- 禁止反向依赖 `agent`。
- 禁止自己决定用户是否有权限使用某个 Profile。
- 禁止绕过 security 直接执行高危 Skill。

说明：

`agent` 只能通过 `SkillRegistry` 和 `SkillExecutor` 使用 Skill，不能直接查 `skills` 表。

### 6.5 core.mcp

负责 MCP Server 配置、工具发现、MCP Tool 调用。

归属表：

```text
mcp_servers
mcp_tools
```

推荐接口：

```text
McpServerService
McpToolRegistry
McpToolExecutor
McpDiscoveryService
```

阶段 1 要求：

```text
至少实现 1 个 MCP Client 示例工具，例如只读文件系统或只读 SQL 查询。
```

### 6.6 core.agent

负责 Agent Runtime、ReAct 主循环、会话和消息。

归属表：

```text
conversations
conversation_messages
```

推荐接口：

```text
AgentRuntimeService
ConversationService
MessageService
AgentStreamService
```

允许依赖：

```text
context.api
profile.api
model.api
skill.api
mcp.api
memory.api
trace.api
quota.api
security.api
```

禁止：

- 禁止直接查 `skills` 表。
- 禁止直接查 `model_configs` 表。
- 禁止直接写 `token_usage_logs`。
- 禁止直接发飞书告警。

说明：

`agent` 是主链路编排者，但不是所有业务的拥有者。它负责把 Profile、模型、Skill、记忆组织起来执行。上下文组装通过 `core.context.api.AgentContextBuilder` 完成，`agent` 只负责调用它；不要让上下文组装逻辑散落到 Controller 或 Gateway。

### 6.7 core.memory

负责短期记忆窗口、长期记忆持久化和简单召回。

归属表：

```text
memories
```

推荐接口：

```text
MemoryService
MemoryRecallService
MemoryWriteService
```

阶段 1：

```text
数据库持久化 + keywords 简单检索。
```

后续增强：

```text
pgvector / Milvus / Qdrant RAG 检索。
```

### 6.8 core.quota

负责 Token 配额、预扣、释放、结算和用量查询。

归属表：

```text
quota_configs
quota_reservations
token_usage_logs
```

推荐接口：

```text
QuotaService
TokenReservationService
TokenUsageService
TokenDashboardService
```

强约束：

- 请求前配额检查和预扣必须可靠。
- 请求失败必须释放预扣。
- 请求成功必须按真实 usage 修正。
- 不能只依赖异步任务扣 Token。

### 6.9 core.trace

负责 TraceID、Trace Root、Span 和 Trace 查询。

归属表：

```text
trace_roots
trace_spans
```

推荐接口：

```text
TraceService
SpanRecorder
TraceQueryService
```

强约束：

- TraceID 和 Trace Root 创建要同步。
- Trace Span 明细可以异步批量落库。
- Trace attributes 必须脱敏后存储。

### 6.10 core.security

负责敏感数据扫描、脱敏、阻断、安全事件记录。

归属表：

```text
security_policies
security_events
```

推荐接口：

```text
SecurityPolicyService
SensitiveDataScanner
SensitiveDataMasker
SecurityEventService
```

扫描范围：

```text
用户请求
工具入参
Skill/MCP 返回
最终响应
记忆写入
Trace/日志
```

### 6.11 core.alert

负责告警规则、告警事件和通知意图生成。

归属表：

```text
alert_rules
alert_events
```

推荐接口：

```text
AlertRuleService
AlertEventService
AlertNotifier
```

说明：

`alert` 只写平台侧告警事件和待通知状态，不直接持有 `feishu-bot` 的 HTTP Client。飞书消息发送由 Gateway、通知适配器或后续异步 worker 调用 `feishu-bot` 告警接口；发送结果由 `feishu-bot` 自己记录。

### 6.12 core.team

负责 Planner / Executor / Reviewer 多 Agent 协作。

阶段：

```text
阶段 4 实现，阶段 1 不进入主链路。
```

推荐接口：

```text
TeamOrchestrator
PlannerAgent
ExecutorAgent
ReviewerAgent
```

约束：

- Planner 只拆任务，不调工具。
- Executor 只执行任务，只有 Executor 能调用 Skill/MCP。
- Reviewer 只审查结果，不规划不执行。
- Skill 不按角色单独配置，统一来自 Profile / 任务 / 用户授权。

### 6.13 core.context

负责 Agent 执行上下文组装。

推荐职责：

```text
System Prompt
Profile Prompt
用户输入
短期记忆
长期记忆召回
可用 Skill 列表
可用 MCP Tool 列表
安全策略摘要
```

推荐接口：

```text
AgentContextBuilder
ToolContextBuilder
PromptContextBuilder
```

说明：

`context` 不拥有业务表，主要把各包查询到的信息组装成运行时上下文。

## 7. web 模块

### 7.1 职责

`web` 是浏览器端入口，负责 JWT 鉴权、Admin/User/Developer REST API、SSE 推送、文件上传和 AI 请求转发。

```text
com.ls.agent.web
├── auth/           登录、注册、JWT
├── controller/
│   ├── admin/      Admin 管理接口
│   ├── user/       User 个人中心、对话页面接口
│   └── developer/  Developer Profile / Skill 管理接口
├── dto/            Web 入参/出参 DTO
├── sse/            浏览器 SSE 推送
├── gatewayclient/  调 Gateway 内部 AI 接口的 HTTP Client
├── security/       Spring Security Web 配置
└── config/
```

### 7.2 路由规则

```text
浏览器 AI 对话 / Profile 测试 / 模型测试
  -> web
  -> gateway
  -> core

管理、配置、统计、查询类请求
  -> web
  -> core
```

### 7.3 禁止

- 禁止拥有业务表。
- 禁止写 Mapper。
- 禁止绕过 gateway 直接调用 Agent Runtime 处理 AI 执行请求。
- 禁止把 JWT 用户身份直接等同于 API Key 应用身份，必须转换成明确的 Application 上下文。

## 8. gateway 模块

### 8.1 职责

`gateway` 是统一 AI 调用入口，负责 OpenAI 兼容接口、API Key 鉴权、Web 内部 AI 请求接入和治理链。

```text
com.ls.agent.gateway
├── controller/       OpenAI 兼容接口、内部 AI 执行接口
├── dto/              ChatCompletionRequest / Response / Stream Event
├── auth/             API Key 鉴权、Web 内部 Token 校验
├── context/          GatewayContext
├── filter/           Gateway 治理链，包含 RuntimeInvokeFilter
├── runtime/          AgentRuntime / 纯模型透传调用适配
├── stream/           SSE / OpenAI Stream 事件适配
├── notifyclient/     可选外部通知 HTTP Client，例如 feishu-bot 告警接口
└── config/
```

### 8.2 治理链

```text
TenantFilter
  -> TraceFilter
  -> SensitiveDataFilter
  -> QuotaFilter
  -> RuntimeInvokeFilter
  -> SensitiveResponseFilter
  -> TokenRecordFilter
  -> AlertFilter
```

### 8.3 强约束

- `GatewayContext` 请求结束必须清理。
- API Key 只通过 core.identity 校验，不直接查表。
- Token 配额只通过 core.quota 处理，不直接写 `token_usage_logs`。
- Trace 只通过 core.trace 处理，不直接写 `trace_roots`。
- Runtime 调用只通过 `core.agent.AgentRuntimeService`。
- 纯模型透传调用只通过 `core.model.ModelInvokeService`，仍然经过 Gateway 治理链。
- Gateway 调 Core 是 Maven 代码依赖和方法调用，不是 HTTP；身份、租户、Trace、agent_mode 等上下文必须从 `GatewayContext` 显式转换到 `AgentRunCommand` 或 `ModelInvokeCommand`，不要让 core 读取 Gateway 的 ThreadLocal。
- Gateway 不拥有业务表。

### 8.4 阶段 1 做法

阶段 1 先做 Gateway 空壳：

```text
Web AI 请求 -> Gateway -> Core AgentRuntime
API Key agent 模式请求 -> Gateway -> Core AgentRuntime
API Key none 模式请求 -> Gateway -> Core ModelInvokeService
```

阶段 1 可以先不做完整治理链，但入口形态要保留，避免阶段 2 重构入口。

## 10. feishu-bot 服务

`feishu-bot` 是独立服务，不依赖 agent-platform 代码。

推荐目录：

```text
feishu-bot/
├── src/main/java/com/ls/feishu/
│   ├── config/          飞书应用配置
│   ├── webhook/         飞书事件回调
```

## 15. 常见错误与禁止清单

### 15.1 错误：Controller 直接调用 Mapper

错误：

```text
ChatController -> ConversationMapper
```

正确：

```text
ChatController -> GatewayInternalClient(HTTP) -> Gateway -> AgentRuntimeService -> MessageService
```

### 15.2 错误：agent 包直接查 skill 表

错误：

```text
AgentRuntimeService -> SkillMapper
```

正确：

```text
AgentRuntimeService -> SkillRegistry -> SkillMapper
```

### 15.3 错误：gateway 直接写 token_usage_logs

错误：

```text
TokenRecordFilter -> TokenUsageLogMapper
```

正确：

```text
TokenRecordFilter -> TokenUsageService -> TokenUsageLogMapper
```

### 15.4 错误：common 放业务对象

错误：

```text
common.dto.AgentProfileDto
common.entity.UserEntity
```

正确：

```text
core.profile.dto.AgentProfileDto
core.identity.entity.UserEntity
```

### 15.5 错误：feishu-bot 直接查平台 alert_events

错误：

```text
feishu-bot -> platform.alert_events
```

正确：

```text
告警通知适配器 -> HTTP -> feishu-bot.alert endpoint
feishu-bot -> feishu_alert_logs
```

### 15.6 错误：core 内部包绕过 api 直接访问 mapper

错误：

```text
core.agent.application.AgentRuntimeService -> core.skill.mapper.SkillMapper
core.agent.application.AgentRuntimeService -> core.model.mapper.ModelConfigMapper
```

正确：

```text
core.agent.application.AgentRuntimeService -> core.skill.api.SkillRegistry
core.agent.application.AgentRuntimeService -> core.model.api.ModelConfigService
```

要求：

```text
用 ArchUnit 在测试中禁止跨业务包访问 entity / mapper / internal。
```

### 15.7 错误：SSE 转发只测最终结果，不测流式透传

错误：

```text
浏览器 -> web -> gateway
只验证最后能拿到完整回答
```

正确：

```text
gateway 固定每 500ms 输出一个 SSE 事件
web 逐事件透传给浏览器或 curl
验证事件不是被缓冲后一次性返回
```

阶段 1 Gateway 空壳完成后必须先做这个验证，否则后面 Agent Runtime 做完才发现流式被缓冲，排查成本会很高。

### 15.8 错误：飞书告警失败影响用户主请求

错误：

```text
AlertFilter -> HTTP 调 feishu-bot 失败 -> 用户 AI 请求失败
```

正确：

```text
写入 alert_events
通知适配器异步调用 feishu-bot
调用失败只更新 notify_status / retry_count，不影响本次 AI 响应
```

## 17. 最终结论

本项目代码落地时坚持三条线：

```text
入口层：
  web / gateway 负责接请求、鉴权、协议转换和转发，不拥有业务表。

核心层：
  core 按 identity/model/profile/skill/mcp/agent/memory/context/quota/trace/security/alert/team 分包，有表的包拥有自己的表和接口；context 不拥有业务表，只负责运行时上下文组装。

独立服务：
  feishu-bot 独立部署、独立数据库，通过 HTTP 接收平台告警，不直接访问平台核心库。
```

判断代码是否放对位置，只问三个问题：

```text
1. 这个类负责哪类业务能力？
2. 它操作哪张表？这张表归哪个包？
3. 调用它的人是否需要知道它的内部实现？
```

如果一个类需要跨多个业务域、直接查多张不属于自己的表，或者让 `web/gateway/common` 开始持有业务逻辑，就说明模块边界已经开始失控，需要立刻调整。
