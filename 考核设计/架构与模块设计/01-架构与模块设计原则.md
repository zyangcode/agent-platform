## 1. 文档定位

本文用于沉淀当前项目在进入代码实现前已经讨论清楚的架构设计、模块设计原则和落地方法。

它不是功能清单，也不是数据库表设计，而是后续设计 Maven 模块结构、Service 边界、接口契约、数据库访问规则、异常处理规则时必须遵守的工程约束。

本项目功能范围很大，包括 Agent 平台、Skill 市场、MCP 工具、模型网关、Trace、Token 配额、敏感数据检测、飞书告警、多 Agent 协作等。如果没有提前定义模块边界，后续很容易出现循环依赖、跨模块查表、业务逻辑散落、接口不可替换、测试困难等问题。

所以本文的核心目标是：

- 让模块拆分有明确依据，而不是按文件夹随意分类。
- 让模块之间依赖方向稳定，避免循环依赖。
- 让主链路优先跑通，同时给后续扩展留下接口。
- 让每个模块可测试、可替换、可降级。
- 让后续写代码时知道什么能做、什么不能做。

## 2. 功能设计、架构设计、模块设计的关系

项目从需求到代码的落地关系如下：

```text
考核需求
  ↓
功能设计：系统要做什么
  ↓
架构设计：系统整体怎么拆、怎么连、怎么保证以后不乱
  ↓
模块设计：每个模块负责什么、暴露什么接口、依赖谁
  ↓
数据库设计：数据怎么存、表之间怎么关联、谁能访问哪些表
  ↓
代码实现：Controller / Service / Repository / Entity / DTO
```

因此，功能设计回答的是：

```text
要做哪些能力？
用户能怎么用？
管理员能管什么？
Agent、Skill、Trace、Token、告警分别有什么功能？
```

架构与模块设计回答的是：

```text
这些能力应该放在哪些模块？
模块之间谁调用谁？
哪些模块可以访问数据库？
哪些模块只能通过接口通信？
哪些逻辑必须同步完成？
哪些逻辑可以异步处理？
后期替换模型、替换 Skill 实现、接入飞书时怎么不改主链路？
```

## 3. 模块不是文件夹分类，而是责任边界

模块设计的第一原则是：模块不是简单的文件夹分类，而是责任边界。

可以把模块类比成公司的不同部门。每个部门负责一类事情，部门之间可以按流程协作，但不能互相乱指挥。前台可以把付款问题转给财务，把技术问题转给技术部；但财务不能反过来依赖前台的接待流程，否则前台流程一变，财务也会被迫修改。对应到代码里，就是每个模块要职责单一，模块之间要有固定调用方向。

这个类比背后对应两条核心原则：

```text
高内聚、低耦合：
  每个模块像一个职责明确的部门，只负责自己那类业务能力。

依赖单向：
  模块之间像公司流程一样有明确协作方向，不能你调用我、我又反过来依赖你。
```

一个模块必须能回答三个问题：

```text
1. 我负责什么业务能力？
2. 别人怎么使用我？
3. 我依赖谁，又不能依赖谁？
```

例如（以下为 core 内部逻辑包，不是独立 Maven 模块；Maven 模块拆分以 04 号文档为准）：

```text
core.identity
  负责：租户、用户、角色、应用、API Key、登录注册
  对外：提供 IdentityQueryService、AuthService、ApiKeyService、PermissionChecker
  依赖：无（基础层，不依赖其他业务包）
  禁止：直接处理 Agent Runtime、Skill 执行、Token 计费

core.agent
  负责：Agent 主运行链路、ReAct、Profile 装配、Skill 调用编排
  对外：提供 AgentRuntimeService
  依赖：context、profile、model、skill、mcp、memory、trace、quota、security 的 api 接口
  禁止：直接查 Skill 表、直接扣 Token、直接发飞书消息

core.skill
  负责：Skill 注册、发现、启用、禁用、执行适配
  对外：提供 SkillRegistry、SkillExecutor
  依赖：安全检查、配置存储
  禁止：反向依赖 agent 内部实现
```

判断一个模块是否合理，可以看它是否满足：

- 能用一句话说清楚责任。
- 主要类都围绕同一类业务能力。
- 对外接口少而稳定。
- 内部实现可以替换，不影响调用方。
- 不需要知道其他模块的数据库细节。

## 4. 推荐的模块拆分思路

后续 Maven 模块结构可以围绕以下能力边界展开：

```text
agent-platform-root
  ├─ common              通用异常、响应结构、错误码、通用枚举、基础工具
  ├─ persistence         数据访问基础能力、审计字段、分页模型、Repository 基础约束
  ├─ external-client     外部系统客户端基础封装，例如超时、重试、签名、错误转换
  ├─ auth                登录、JWT、API Key、权限校验
  ├─ user                用户、应用 Application、租户隔离
  ├─ model               模型供应商、模型配置、OpenAI 兼容调用
  ├─ agent               Agent Runtime、Profile 装配、ReAct 主链路
  ├─ skill               Skill 元数据、注册、启用、执行、热加载接口
  ├─ mcp                 MCP 服务发现、工具调用适配
  ├─ memory              短期记忆、长期记忆、持久化策略
  ├─ trace               TraceID、Span、链路追踪、查询接口
  ├─ quota               Token 统计、配额校验、用量记录
  ├─ security            敏感数据扫描、脱敏、阻断、安全事件
  ├─ alert               告警规则、静默期、飞书通知适配
  ├─ team                Planner / Executor / Reviewer 协作
  ├─ feishu-bot          飞书机器人后端
  ├─ admin-api           管理端接口
  ├─ user-api            用户端接口和 OpenAI 兼容接口
  └─ boot                Spring Boot 启动模块
```

说明：

- `common` 只放通用异常、响应结构、错误码、通用枚举和基础工具，不放具体业务 DTO，避免变成所有模块都依赖的杂物模块。
- 业务 DTO、Command、Result 应该放在各自业务模块内，例如 `agent` 放 `ChatCommand`，`skill` 放 `SkillExecuteCommand`。
- `persistence` 和 `external-client` 是基础设施能力边界，MVP 可以先做成公共包或轻量模块，但职责要清楚，不能让业务模块随意跨模块访问 Repository 或外部客户端实现。
- `feishu-bot` 是独立启动服务。Agent 平台、模型网关、告警模块只能通过 HTTP 接口或独立 Client 调用它，不能直接依赖飞书机器人内部 Service。

上面是按能力边界推演出的完整拆分形态，不是当前 MVP 的最终 Maven 模块方案。正式代码落地以 `04-模块目录与包结构设计.md` 为准：MVP 采用 `common/core/web/gateway` 四个后端 Maven 模块，把 `identity/model/profile/skill/mcp/agent/memory/context/quota/trace/security/alert/team` 作为 `core` 内部逻辑包管理；`persistence`、`external-client`、`boot` 等能力先不单独拆 Maven 模块，后续复杂度上来后再从现有模块中抽出。

MVP 阶段不一定一次性全部实现完整能力，但模块边界要先定清楚。即使某些模块内部先写简单实现，也不要把逻辑散落到其他模块里。

## 5. 核心设计原则与具体解决方案

### 5.1 高内聚、低耦合

含义：

```text
一个模块内部的代码应该围绕同一个业务目标组织。
模块之间只通过清晰接口协作，不互相了解内部细节。
```

风险示例：

```text
AgentRuntimeService 里同时写：
  - 查询用户
  - 查询 Skill 表
  - 拼 Prompt
  - 调模型
  - 扣 Token
  - 写 Trace
  - 发飞书告警
```

这样会导致 Agent 模块越来越大，任何功能变动都要改主链路。

解决方案：

```text
AgentRuntimeService 只负责编排主链路。

用户信息：调用 IdentityQueryService
Skill 列表：调用 SkillRegistry
模型调用：调用 ModelClient
Token 配额：调用 QuotaService
Trace 记录：调用 TraceRecorder
敏感检测：调用 SensitiveDataService
告警：发布 AlertEvent
```

### 5.2 依赖单向

含义：

```text
模块依赖必须有方向。
上层业务模块可以依赖下层能力模块。
下层基础模块不能反向依赖上层业务模块。
```

推荐方向（最终落地以 04 号文档为准）：

```text
web / gateway（入口层，Maven 模块）
  ├─ web → gateway：HTTP 内部调用（不做 Maven 依赖）
  └─ gateway → core：Maven 代码依赖

core（核心业务层，Maven 模块，内部 13 个逻辑包）
  ├── agent（编排层：组织主链路，不拥有表）
  ├── context（组装层：组装模型上下文，不拥有表）
  ├── skill / model / mcp / memory / quota / trace / security / alert（能力层：拥有表，对外 api）
  └── identity（基础层：被所有人查询，不依赖任何人）

common（公共基础，Maven 模块）
  被 web / gateway / core 依赖
```

错误示例：

```text
trace 模块依赖 agent 模块内部类
quota 模块调用 agent runtime
skill 模块调用 team orchestrator
```

解决方案：

```text
下层模块只定义通用接口或接收通用事件。
上层模块负责组织业务语义。
```

例如：

```text
agent 调用 trace.recordSpan(...)
trace 只认识 TraceSpan，不认识 AgentStep、PlannerTask 这些内部对象。
```

### 5.3 依赖倒置解决反向调用问题

当底层模块需要使用上层能力时，不能直接依赖上层实现，否则一定会产生循环依赖。

错误示例：

```text
security 模块发现敏感数据后，直接调用 agent 模块终止会话。

security -> agent
agent -> security

形成循环依赖。
```

正确做法：

```text
security 模块只返回扫描结果：

SensitiveCheckResult
  - action: PASS / MASK / BLOCK
  - sanitizedText
  - hitRules

agent 模块根据扫描结果决定继续、脱敏或阻断。
```

另一个例子：

```text
quota 模块不应该直接调用 alert 模块发告警。
quota 只发布 TokenQuotaExceededEvent。
alert 模块监听事件后发送飞书告警。
```

这样依赖关系保持为：

```text
agent -> quota
quota -> common-event
alert -> common-event
```

而不是：

```text
quota -> alert -> feishu -> agent
```

### 5.4 接口稳定，实现可替换

含义：

```text
调用方依赖接口，不依赖具体实现。
实现可以从本地模拟替换成真实服务，也可以从简单版本升级成生产版本。
```

本项目典型场景：

```text
模型调用：
  ModelClient 接口稳定
  OpenAiModelClient / QwenModelClient / LocalModelClient 可替换

记忆存储：
  MemoryStore 接口稳定
  FileMemoryStore / RedisMemoryStore / VectorMemoryStore 可替换

Skill 执行：
  SkillExecutor 接口稳定
  BuiltinSkillExecutor / HttpSkillExecutor / JarSkillExecutor 可替换

飞书告警：
  AlertNotifier 接口稳定
  MockAlertNotifier / FeishuAlertNotifier 可替换
```

MVP 阶段可以先写简单实现，但接口要先按未来形态设计。

### 5.5 不要跨模块直接查表

含义：

```text
每张表应该有明确归属模块。
其他模块不能绕过归属模块直接查表。
```

错误示例：

```text
agent 模块直接查 skills 表。
quota 模块直接查 users 表。
alert 模块直接查 traces 表。
```

问题：

- 表结构一变，多个模块都要改。
- 权限和租户隔离容易绕过。
- 数据含义散落，后期很难维护。

解决方案：

```text
skill 表归 core.skill。
agent 只能调用 SkillRegistry 查询可用 Skill。

users / applications 表归 core.identity。
quota 只能调用 IdentityQueryService 获取应用信息。

trace 表归 core.trace。
alert 只能通过 TraceQueryService 查询 Trace 摘要。
```

### 5.6 DTO 和 Entity 分离

含义：

```text
Entity 是数据库持久化对象。
DTO 是接口输入输出对象。
二者不要混用。
```

错误示例：

```text
Controller 直接返回 UserEntity。
前端提交 ProfileEntity。
```

问题：

- 数据库字段泄露给前端。
- 后期表结构变更会影响接口。
- 敏感字段容易误返回，例如 passwordHash、apiKeyHash。

解决方案：

```text
Entity：
  UserEntity
  AgentProfileEntity
  SkillEntity

Request DTO：
  CreateProfileRequest
  UpdateSkillConfigRequest
  ChatRequest

Response DTO：
  ProfileDetailResponse
  SkillListItemResponse
  ChatStreamEvent
```

### 5.7 主链路优先

含义：

```text
先保证用户请求能完整跑通，再逐步增强周边能力。
```

本项目主链路：

```text
用户请求
  → 鉴权
  → 应用与配额校验
  → 敏感数据扫描
  → 加载 Agent Profile
  → 加载 Skill / MCP 工具
  → 加载记忆
  → 调用模型
  → 执行 Skill
  → 生成回复
  → 记录 Trace / Token
  → 返回 SSE / JSON
```

MVP 优先级：

```text
必须先跑通：
  登录鉴权
  Agent 对话
  模型调用
  Skill 调用
  Trace 基础记录
  Token 基础统计

可以后补：
  可视化仪表盘
  Jaeger / OpenTelemetry 导出
  Jar Skill 热加载
  多 Agent 协作
  飞书高级自动化
```

### 5.8 边缘动作异步

含义：

```text
用户请求主链路中必须完成的事情同步执行。
不影响本次响应结果的事情尽量异步处理。
```

适合同步：

```text
鉴权
配额拦截
敏感数据阻断
模型调用
Skill 调用
最终响应
```

适合异步：

```text
Trace 明细落库
告警通知飞书
Token 历史趋势统计
安全事件归档
慢调用分析
```

边界：

```text
Trace 起止记录、Token 结算、配额扣减这类核心审计数据，不能简单丢给不可靠异步任务。
它们应同步写入，或者进入有确认机制的可靠队列。
详细 Span、趋势统计、告警通知、慢调用分析可以异步处理。
```

注意：

异步不是不可靠。异步任务要有失败日志、重试边界和降级策略。

### 5.9 输入输出清晰

每个模块的核心服务都必须定义清楚输入和输出。

示例：

```text
AgentRuntimeService.run(AgentRunCommand command): AgentRunResult / AgentStreamHandler

QuotaService.checkAndReserve(QuotaCheckCommand command): QuotaReservation

SensitiveDataService.scan(SensitiveScanCommand command): SensitiveScanResult

SkillExecutor.execute(SkillExecuteCommand command): SkillExecuteResult

ModelClient.chat(ModelChatCommand command): ModelChatResult
```

不要让方法参数无限膨胀，也不要到处传 Map。

可以用 Command / Result 对象承载清晰结构。

### 5.10 MVP 简单实现，但接口留扩展

含义：

```text
MVP 可以简单，但不能把未来扩展路线堵死。
```

例子：

```text
Skill 热加载：
  MVP 先支持内置 Skill + HTTP Skill 配置。
  接口上预留 JarSkillLoader。
  不要一开始就把 Skill 写死成 if-else。

记忆：
  MVP 先用数据库或文件存长期记忆。
  MemoryStore 接口保留 Redis / VectorStore 实现空间。

Trace：
  MVP 先落本地数据库。
  TraceExporter 接口保留 Jaeger / Zipkin / OTel Collector。
```

### 5.11 不过度抽象

含义：

```text
只为真实变化点抽象，不为想象中的变化点抽象。
```

本项目需要抽象的变化点：

- 模型供应商会变。
- Skill 类型会变。
- 记忆存储会变。
- Trace 导出目标会变。
- 告警通知渠道会变。
- 敏感规则会变。

不建议过早抽象的地方：

- 不要一开始做复杂插件生命周期框架。
- 不要一开始做多级租户组织架构。
- 不要一开始做完整工作流引擎。
- 不要一开始做真实支付系统。

### 5.12 模块要支持 Mock 和降级

含义：

```text
外部依赖不可用时，系统要能用 Mock 跑 Demo，或者清晰降级。
```

场景：

```text
模型 API 不可用：
  使用 MockModelClient 返回模拟结果，保证演示主链路。

飞书不可用：
  告警写入本地 alert_log，状态标记为 NOT_SENT。

天气 API 不可用：
  Skill 返回标准错误，不拖垮 Agent Runtime。

Trace 导出失败：
  本地 Trace 仍然落库，导出失败记录日志。
```

## 6. 本项目特别需要坚持的架构约束

### 6.1 TraceID 必须贯穿主链路

每次请求进入系统后必须生成或继承 TraceID。

TraceID 要贯穿：

```text
鉴权
配额
敏感扫描
Agent Runtime
模型调用
Skill 调用
MCP 工具调用
Token 统计
告警
日志
响应
```

否则后期排查问题会非常困难。

### 6.2 用户、应用、Profile、Skill 的权限边界要提前固定

本项目存在多种对象：

```text
User
Application
AgentProfile
Skill
ApiKey
Conversation
Trace
TokenUsage
```

后端必须统一校验：

```text
当前用户是否拥有该 Application？
该 Profile 是否属于该 Application 或可公开使用？
该 Skill 是否全局可用、领域可用，还是用户自定义？
该 API Key 是否绑定到正确 Application？
当前请求是否有权限查看 Trace / Token 记录？
```

不能只靠前端隐藏按钮。

### 6.3 历史记录要保存配置快照

Agent Profile、Skill 配置、模型配置都可能变化。

如果历史 Trace 只保存 profile_id，不保存当时的关键配置快照，后期就无法复现当时为什么这样回答。

建议在 Trace 或调用记录中保存：

```text
profile_id
profile_version
model_name
skill_ids
skill_versions
memory_strategy
safety_policy_version
agent_mode
```

### 6.4 外部调用必须有超时、重试和错误分类

外部调用包括：

```text
OpenAI / 通义千问 / 本地模型
天气 API
搜索 API
MCP Server
飞书 API
GitLab / GitHub / Jenkins / Prometheus
```

必须具备：

```text
连接超时
读取超时
有限重试
熔断或降级
标准错误码
Trace Span 记录
```

不能让一个 Skill 或一个外部 API 卡死整个 Agent 请求。

### 6.5 错误不能全部返回 500

需要明确错误类型：

```text
401 未登录或 API Key 无效
403 无权限
404 资源不存在
409 状态冲突，例如重复注册 Skill
422 参数合法但业务不可处理，例如 Skill 参数不满足 schema
429 Token 配额超限
500 系统内部错误
502 上游模型或外部工具错误
503 服务暂不可用
```

这样前端、API 调用方和告警系统才能正确处理。

### 6.6 列表接口必须分页

以下列表必须分页：

```text
Trace 列表
Token 用量记录
Skill 列表
Profile 列表
安全事件列表
告警列表
会话列表
API Key 列表
```

不要为了 MVP 直接返回全量列表，否则后期数据一多就会影响性能。

### 6.7 核心配置建议软删除

建议软删除的对象：

```text
AgentProfile
Skill
Application
ApiKey
ModelProviderConfig
SafetyPolicy
```

原因：

- 历史 Trace 需要关联旧配置。
- Demo 时误删可以恢复。
- 审计上需要知道曾经存在过什么配置。

API Key 需要单独注意：

```text
API Key 只保存 hash，不保存明文。
删除 API Key 的本质是吊销，而不是普通删除。
被吊销的 API Key 不能恢复明文，只能重新生成新的 Key。
```

### 6.8 高风险操作需要二次确认

高风险操作包括：

```text
删除 Skill
禁用 Profile
吊销 API Key
修改安全策略为放行
启用 Jar Skill
触发部署
创建 Git 分支
发送群广播
```

后端需要有明确的确认参数或审批状态，不能只靠前端弹窗。

## 7. 模块设计时的检查清单

每设计一个模块，都要回答以下问题：

```text
1. 这个模块的一句话职责是什么？
2. 这个模块拥有哪几张表？
3. 这个模块暴露哪些 Service 接口？
4. 其他模块是否只能通过接口访问它？
5. 它依赖哪些模块？
6. 有没有反向依赖或循环依赖风险？
7. 它的核心 DTO / Command / Result 是什么？
8. 它失败时返回什么错误码？
9. 它是否需要异步事件？
10. 它是否需要 Mock 实现？
11. 它是否需要降级策略？
12. 它是否需要记录 Trace Span？
13. 它是否涉及权限校验？
14. 它是否涉及租户或 Application 隔离？
15. 它是否需要保存历史配置快照？
```

如果这些问题答不出来，说明模块边界还没有设计清楚，不应该直接开始写代码。

## 8. 判断设计是否失控的信号

如果后续写代码时出现以下情况，说明架构边界可能已经开始失控：

```text
1. 一个 Service 超过 500 行，并且同时处理多个业务概念。
2. Controller 里出现复杂业务判断。
3. 某个模块直接引用了很多其他模块的 Entity。
4. Repository 被多个不属于它的模块直接调用。
5. 一个小需求要改 5 个以上不相关模块。
6. 新增一个 Skill 需要修改 Agent Runtime 核心代码。
7. 新增一个模型供应商需要改 Controller。
8. 告警失败会导致用户对话失败。
9. Trace 记录失败会导致模型调用失败。
10. 单元测试必须启动完整 Spring Boot 才能测一个普通业务类。
```

出现这些情况时，应优先回到模块边界，而不是继续堆代码。

## 9. 设计阶段回顾

以下为已完成的架构与模块设计步骤，对应产出文档：

```text
2. 数据库模型设计 ✅

4. 用户请求主链路设计 ✅

   明确 Web/JWT 和 API Key 双入口 → Gateway 治理链 → Core AgentRuntime 完整链路。

   明确阶段 1 必做范围、必做包和总体落地顺序；详细任务拆解与验收计划仍需单独补充。
```

## 10. 最终结论

当前项目最大的问题不是单个功能难，而是功能太多、链路太长、模块容易互相缠绕。

所以后续开发必须坚持：

```text
先定模块边界，再定表，再写代码。
先跑通主链路，再补管理端和高级能力。
上层编排，下层提供能力。
模块通过接口协作，不跨模块查表。
配置和历史记录要保留快照。
外部依赖必须可降级、可 Mock。
MVP 可以简单，但依赖方向不能乱。
```

只要这些原则守住，后续即使功能逐步增加，也不会轻易推翻重做。
