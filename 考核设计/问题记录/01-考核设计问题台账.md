# 考核设计问题台账

> 记录考核设计过程中遇到的关键问题。每条只保留：问题、核心思路、核心方案。

## 1. 模块是否拆得过细

**问题：** 一开始按能力拆出很多 Maven 模块，一个人开发维护成本高。

**核心思路：** 物理模块少一点，逻辑边界不能丢。

**核心方案：** MVP 用少量工程单元：common、core、web、gateway、frontend、feishu-bot；core 内部按 identity、agent、skill、quota、trace 等包保留边界。

## 2. web / gateway 是否拥有业务表

**问题：** 如果 users、applications、api_keys 归 web/gateway，core 会反向依赖入口层。

**核心思路：** 入口层只做协议适配，业务数据归核心业务层。

**核心方案：** web/gateway 不拥有核心业务表；users、applications、api_keys、quota、trace 等归 core 内部逻辑包。

## 3. common 是否放业务 DTO / Entity

**问题：** common 放业务对象后会变成杂物模块，所有模块都依赖它。

**核心思路：** common 只能放真正通用的东西。

**核心方案：** common 只放异常、错误码、通用响应、工具类；业务 DTO、Entity、Command、Result 放各自模块或包内。

## 4. Token 和 Trace 是否都异步

**问题：** Trace 全同步影响性能，Token 全异步又会导致配额不准。

**核心思路：** Token 是额度数据，不能丢；Trace 明细是排查数据，可以延迟。

**核心方案：** Token 预扣、结算、修正必须同步或可靠写入；Trace Span 明细、趋势统计、告警通知可以异步。

## 5. Token 预扣是否强绑定 Redis

**问题：** 文档写死 Redis 会让 MVP 被技术选型绑定。

**核心思路：** 原则文档不要过早绑定具体实现。

**核心方案：** 统一写成“数据库事务或 Redis 原子预扣”，后续按实现复杂度选择。

## 6. 飞书机器人是否依赖 Agent 平台代码

**问题：** 如果 feishu-bot 依赖 agent-platform common 或内部 Service，会破坏独立服务边界。

**核心思路：** 项目 2 是独立服务，项目 3 只通过接口调用它。

**核心方案：** feishu-bot 不依赖 agent-platform 代码；Gateway/Core 只能通过 HTTP 调飞书告警接口。

## 7. 双通道是否需要拆开

**问题：** Web/JWT 和 API Key/OpenAI 兼容接口如果混在一起，鉴权和拦截链会混乱。

**核心思路：** 两个入口调用方不同、协议不同；但 AI 执行治理能力必须统一。

**核心方案：** web 负责浏览器/JWT/管理入口；gateway 负责统一 AI 调用治理入口。浏览器 AI 对话/测试由 web 转发到 gateway，外部 API Key 直接进 gateway，最终都进入同一个 core Agent Runtime。

## 8. CLAUDE.md 是否要写完整设计

**问题：** 把所有设计文档塞进 CLAUDE.md 会导致上下文膨胀。

**核心思路：** CLAUDE.md 放短约束和文档索引，详细内容放设计文档。

**核心方案：** CLAUDE.md 只保留关键强约束、实现顺序和文档路径；具体解释按需读取对应 md 文件。

## 9. 下一步是模块设计还是数据库设计

**问题：** 工程单元和 core 包边界已经确定，继续重复模块设计会浪费时间。

**核心思路：** 当前最需要把数据归属和表关系定下来。

**核心方案：** 下一步进入数据库模型设计，再做 Skill 接口规范和用户请求主链路设计。

## 10. 用户注册登录模型

**问题：** 初版设计是"不做登录页、Admin 后台手动创建用户并分发 API Key"，这更像是内部管理系统而非面向用户的 SaaS 产品。

**核心思路：** 用户应能自助注册登录，API Key 应由用户自己管理而非 Admin 代发。

**核心方案：** MVP 做极简登录注册页（用户名+密码）；用户登录后在个人中心创建 Application，每个应用自动生成一个 API Key；Admin 后台手动创建用户仅作为兜底。

## 11. API Key 绑定模型

**问题：** 初版 Key 直接绑定 User，一个用户只有一个 Key。Key 泄漏意味着整个账户的调用权限泄漏。

**核心思路：** 参考中转站模式，Key 绑定 Application 而非 User。

**核心方案：** 一个用户可创建多个 Application，每个应用自动生成一个 API Key；不同工具/脚本用不同 Key；Key 泄漏只吊销那一个，不影响其他应用；Token 消耗按应用分别统计。

## 12. 计费模型

**问题：** 初版讨论中混用了"余额""充值""人民币扣费"等概念，容易把项目变成支付计费系统。

**核心思路：** 考核重点不是支付，而是 Token 治理能力。MVP 不应接真实支付。

**核心方案：** 统一为 Token 配额模型，不提余额/人民币；配额超限返回 HTTP 429；Admin 后台手动配置日/月额度；真实支付、充值页面放到后续增强。

## 13. Agent 执行模式

**问题：** 初版所有请求都走完整 Agent Runtime（Profile+Skill+ReAct+记忆），但有些用户只想当纯模型转发使用。

**核心思路：** 保留 Agent 能力作为默认，同时提供透传选项，让用户自行选择。

**核心方案：** 增加 `agent_mode` 参数：`agent`（默认）走完整链路，`none` 纯模型转发。浏览器端 AI 对话/测试始终走 agent 模式，API Key 通道可选两种模式。两模式共用 Token 配额和 Trace。

## 14. 文档双写与一致性问题

**问题：** CLAUDE.md 和设计文档多次修改后，两端出现措辞不一致（如"自行生成"vs"自动生成"、页面数 8 vs 9、功能点数统计过期）。

**核心思路：** 每次改完设计文档，必须回查 CLAUDE.md 和关联的功能清单是否同步。

**核心方案：** 建立了 CLAUDE.md ↔ 01-功能清单 ↔ 06-双通道设计 ↔ 架构原则文档 四份文件的交叉核对机制，关键字段统一为最终表述，功能点数随编辑同步更新。

## 15. Web 请求是否全部经过 Gateway

**问题：** 如果所有 Web 请求都走 Gateway，管理配置接口会被 AI 治理链拖复杂；如果 Web AI 对话不走 Gateway，又不符合项目 3 的统一治理定位。

**核心思路：** 按请求类型分流，不按角色分流。

**核心方案：** AI 执行请求统一走 Gateway；管理、配置、统计、查询请求走 Web -> Core。

## 16. 多 Agent 后置后 Developer 是否也后置

**问题：** Planner/Executor/Reviewer 放到后期做，会不会导致 Developer 模式也必须后置。

**核心思路：** Developer 基础能力和 Team 高分项要拆开。

**核心方案：** MVP 的 Developer 先负责配置单 Agent Profile、Prompt、模型、Skill、记忆和安全策略；Team 只是 Profile 的增强执行模式，后期再加。

## 17. Web 转发 Gateway 如何实现

**问题：** 浏览器 AI 请求走 Web -> Gateway -> Core 后，如果 Web 直接依赖 Gateway 代码，会造成入口层耦合。

**核心思路：** Web 和 Gateway 都是入口适配层，不能互相变成代码强依赖。

**核心方案：** Web 通过 HTTP 内部接口转发 AI 执行请求到 Gateway，并携带内部服务 Token 或签名 Header；Gateway 校验后才接受 user_id/app_id/tenant_id。

## 18. Admin 测试模型与测试 Agent 的路由区别

**问题：** Admin 测试模型连接和测试 Agent 效果如果都走同一路由，会让管理诊断和 AI 执行混在一起。

**核心思路：** 是否经过 Gateway 取决于是否触发完整 AI 执行治理链。

**核心方案：** 测试模型供应商连通性走 Web -> Core model；测试 Agent/Profile 效果走 Web -> Gateway -> Core。

## 19. 阶段 1 是否先建 Gateway

**问题：** 如果阶段 1 直接 Web -> Core，阶段 2 改成 Web -> Gateway -> Core 会改入口代码。

**核心思路：** 阶段 1 先保留最终调用路径，治理能力后补。

**核心方案：** 阶段 1 先建 Gateway 空壳，只做 AI 请求转发；阶段 2 在空壳上补 Trace、Token、脱敏、告警治理链。

## 20. Token 仪表盘是否后置

**问题：** 项目 3 要求 Token 仪表盘，如果全部放到高分增强会影响验收。

**核心思路：** 把“基础看板”和“复杂分析”拆开。

**核心方案：** MVP 做当前用量、调用明细、简单 TOP；历史趋势图、多维筛选、成本折算后续增强。

## 21. 飞书机器人完整功能与告警接口的阶段关系

**问题：** 项目 3 阶段 2 需要飞书告警，但完整飞书机器人放在阶段 5。

**核心思路：** 先做最小告警闭环，再做完整机器人能力。

**核心方案：** 阶段 2 先提供 feishu-bot 内部告警接口或 mock 告警接口；阶段 5 再补基础指令、Git/CR、DevOps。

## 22. MCP 示例是否容易漏掉

**问题：** 项目 1 明确要求 MCP 示例，但路线图容易只关注 Skill。

**核心思路：** MCP Client 示例是硬性验收点，MCP Server 是增强项。

**核心方案：** 阶段 1 至少实现 1 个 MCP Client 示例工具，如只读文件系统或只读 SQL；平台 Skill 暴露为 MCP Server 后续增强。

## 23. Gateway 图中是否能直接写 Agent Runtime

**问题：** 如果在 Gateway 模块里直接画 `Agent Runtime`，容易误解为 Runtime 属于 Gateway。

**核心思路：** Gateway 只负责统一入口和治理链，真正的 Agent Runtime 属于 core。

**核心方案：** 架构图中 Gateway 节点改为 `RuntimeInvokeFilter`，连线标注“调用 Core AgentRuntime”。

## 24. feishu-bot 是否属于平台引擎层

**问题：** 如果把 feishu-bot 放在引擎层，容易误解为它依赖 agent-platform 的 core/common/web/gateway。

**核心思路：** feishu-bot 是项目 2 的独立服务，只通过 HTTP 告警接口与平台通信。

**核心方案：** 架构图单独设置“独立服务层”，feishu-bot 使用独立配置库；平台核心表仍归 core 管理。

## 25. 架构图中告警链路是否同步调用飞书

**问题：** 如果图上写“HTTP 调告警接口”，容易让人以为 Gateway 响应前同步调用飞书。

**核心思路：** 告警通知属于边缘动作，不能阻塞主响应。

**核心方案：** 图中改为“写告警事件 / 异步 HTTP 通知”，AlertFilter 只生成事件，飞书发送异步执行。

## 26. 前端是否属于基础设施层

**问题：** 如果把 frontend 放在基础设施层，会混淆用户入口和运行基础设施。

**核心思路：** frontend 是浏览器用户入口的一部分，基础设施只放数据库、缓存、公共模块等支撑能力。

**核心方案：** 架构图调整为 `浏览器用户 -> frontend -> web`，frontend 不再放在基础设施层。

## 27. 模块目录与表归属如何落地到代码

**问题：** 只有架构原则和数据库表设计还不够，后续写代码时仍可能出现 Entity/Mapper 乱放、Controller 直接查表、gateway 直接写业务表等问题。

**核心思路：** 先固定工程模块、Java 包、表归属和调用方向，让“代码放哪里”有明确依据。

**核心方案：** 新增 `考核设计/架构与模块设计/04-模块目录与包结构设计.md`，规定 common/core/web/gateway/frontend/feishu-bot 工程边界，core 内部按 identity/model/profile/skill/mcp/agent/memory/context/quota/trace/security/alert/team 分包，并把每张表映射到唯一 Entity/Mapper 所属包。

## 28. core 内部包边界是否只靠自觉

**问题：** core 内部有多个业务包，但它们都在同一个 Maven 模块里，编译器不会阻止 `core.agent` 直接访问 `core.skill.mapper`。

**核心思路：** 文档纪律要尽量变成自动检查，否则代码量上来后很容易越界。

**核心方案：** 在 `04-模块目录与包结构设计.md` 中补充 ArchUnit 包边界规则，要求禁止跨业务包访问 `entity`、`mapper`、`internal`，只允许通过 `api`、`dto`、`command` 协作。

## 29. Web 转发 Gateway 的 SSE 是否会被缓冲

**问题：** 浏览器 AI 对话走 `frontend -> web -> gateway -> core`，SSE 中间多了一跳 Web 转发，可能出现缓冲、超时、错误事件丢失等问题。

**核心思路：** Gateway 空壳阶段就要验证流式透传，而不是等 Agent Runtime 完成后再发现链路问题。

**核心方案：** 在阶段 1 落地顺序中加入固定 SSE 流透传测试：Gateway 每隔固定时间输出 SSE 事件，Web 逐事件转发给浏览器或 curl，确认不缓冲、不超时、错误事件可传递。

## 30. 上下文 Token 超额压缩放在哪里

**问题：** 用户输入、历史消息、记忆、Skill/MCP 描述可能超过模型上下文窗口，压缩逻辑如果放在 Gateway 会让入口层理解 Agent 内部结构。

**核心思路：** 上下文压缩是 Agent 上下文组装策略，不是网关治理能力。

**核心方案：** 压缩和裁剪放在 `core.context`，由 `AgentContextBuilder`、`ContextWindowManager`、`ContextCompressor` 处理；Gateway 只负责 Token 配额预扣、治理链和协议转换。

## 31. Gateway 是否也组装上下文

**问题：** Gateway 有 `GatewayContext`，core 又有 `AgentContextBuilder`，容易误解为两边都在组装模型上下文。

**核心思路：** 两个 Context 的含义不同。GatewayContext 是请求治理上下文，AgentContext 是模型执行上下文。

**核心方案：** Gateway 只保存 tenant_id、user_id、application_id、trace_id、agent_mode 等治理信息，并显式转换成 `AgentRunCommand` 或 `ModelInvokeCommand`；真正的 Prompt、记忆、工具列表组装放在 `core.context`。

## 32. Token 预扣和结算如何避免并发错误

**问题：** 如果配额检查采用“先查余额再更新”，并发请求可能同时通过，导致超额使用；释放预扣失败也可能吃掉用户额度。

**核心思路：** Token 配额是额度数据，必须原子、幂等、可重试。

**核心方案：** 在数据库模型中给 `quota_configs` 和 `quota_reservations` 增加 `version`；配额预扣采用 CAS 或 `SELECT ... FOR UPDATE` 行锁；`quota_reservations.trace_id` 唯一，释放和结算按 `trace_id + status + version` 条件更新，避免重复释放或重复扣减。

## 33. 飞书告警失败是否影响聊天请求

**问题：** 如果 Gateway 在响应前同步调用 feishu-bot，飞书接口失败会拖垮用户 AI 对话。

**核心思路：** 告警通知是边缘动作，不能阻塞主链路。

**核心方案：** AlertFilter 只写 `alert_events` 或待通知状态，通知适配器异步调用 feishu-bot；飞书调用失败只更新 notify_status/retry_count，不影响本次 AI 响应。

## 34. Gateway Filter 与 Core Service 的分工边界

**问题：** Gateway 治理链的 Filter（QuotaFilter、TraceFilter、SensitiveDataFilter、TokenRecordFilter 等）和 Core 的 Service（QuotaService、TraceService、SensitiveDataScanner、TokenUsageService 等）成对出现、名称相近，容易让人产生"功能重复"的疑问，也可能导致 Filter 越过 Service 直接操作 Mapper。

**核心思路：** Filter 管"拦"（编排决策 + 请求阻断），Service 管"做"（业务执行 + 表写入）。两者通过 `core.xxx.api` 接口单向调用，Filter 永远不直接碰 Mapper。

**核心方案：** Gateway 每个 Filter 只调对应的 `core.xxx.api` 接口；每张表只有一个写入入口（归属包的 Service），不管谁触发都走同一个 Service；Filter 根据 Service 返回结果做阻断决策（如预扣失败返回 HTTP 429）。禁止 `TokenRecordFilter → TokenUsageLogMapper` 这类跨模块直接查表。

## 35. web→gateway 与 gateway→core 两种依赖方式的选择依据

**问题：** web→gateway 走 HTTP 内部调用，gateway→core 走 Maven 代码依赖。两种依赖方式的选择依据如果不说清楚，后续实现时可能走偏。

**核心思路：** 选 HTTP 还是 Maven 依赖，取决于调用方是否应受被调用方信任、以及被调用操作能否承受网络失败风险。

**核心方案：** web→gateway 走 HTTP——Gateway 必须独立校验内部服务 Token，不能信任 Web 传入的任何身份信息，HTTP 强制了安全边界。gateway→core 走 Maven 方法调用——Token 预扣/结算、Trace 创建等额度数据操作，不能承受 HTTP 额外的网络失败风险。已在 04 号文档 2.2 节和 8.3 节明确规定。

## 36. 技术选型与工程约定是否需要单独成文

**问题：** 架构、数据库和模块目录已经有文档，但后续真正写代码时仍可能忘记版本、Flyway 位置、错误码、SSE 透传、ArchUnit 检查、DTO/Entity 分层等落地细节。

**核心思路：** 架构文档回答“为什么这样拆”，工程约定回答“代码具体怎么写、怎么检查、怎么验收”。这类规则如果散落在聊天记录里，后面实现时最容易丢。

**核心方案：** 新增 `考核设计/技术选型与工程约定/01-技术选型与工程约定.md`，统一固化技术选型、模块依赖、包结构、DTO/Command/Entity、Flyway、统一响应、错误码、鉴权上下文、SSE、日志 Trace、Token、ArchUnit、测试和配置文件规则；并在 `CLAUDE.md` 中加入该文档索引。

## 37. 是否借鉴 AgentX 参考项目设计

**问题：** `AgentX-master/docs` 中有 Agent、Token 超限、计费、Trace、SQL、开发规范等文档，如果直接照搬会引入真实余额计费、订单、完整 RAG、Agent 市场等超出 MVP 的复杂度。

**核心思路：** 借鉴设计方法，不照搬业务范围。参考项目最有价值的是复杂问题单独成文、Trace 按阶段拆解、开发规范规则化，而不是具体表结构。

**核心方案：** 新增 `考核设计/参考项目分析/01-AgentX设计借鉴记录.md`，明确可借鉴和不可照搬内容；同时把查询命名、三层校验、jsonb 字段处理、事务边界等低风险工程规则补入工程约定。后续优先补阶段 1 接口设计和开发计划，再按需补上下文 Token 策略设计、Trace 设计。

## 38. Spring AI 是否接管 Agent Runtime

**问题：** Spring AI ChatClient 自带模型调用、streaming 和 tool calling 能力，如果直接让它自动执行工具调用，Agent 循环中的 action / observation / Trace / Token / SSE 事件会变得不可控。

**核心思路：** Spring AI 只适合做模型调用适配层，不适合接管本项目的 Agent Runtime。项目需要自己控制 ReAct while 循环、Skill/MCP 执行、多 Agent 编排和实时事件输出。

**核心方案：** Spring AI 或 OpenAI 兼容 Client 统一封装在 `core.model`，只保留请求构建、响应解析、usage 读取、流式 token 适配和多供应商 ChatClient 封装；`core.agent` 自己控制单 Agent ReAct 循环，`core.team` 后续自己控制 Planner / Executor / Reviewer 状态机。

## 39. SSE 是否引入 WebFlux / Flux

**问题：** 初版工程约定中把 `Flux` 和 Spring MVC SSE 写在一起，容易让实现者误以为阶段 1 要混用 WebFlux 和 Spring MVC。

**核心思路：** 当前后端栈是 Spring MVC + MyBatis-Plus，阶段 1 不引入响应式全家桶。SSE 的验收重点是不缓冲、可逐事件透传，而不是使用 Flux。

**核心方案：** 阶段 1 SSE 基于 Spring MVC，优先使用 `SseEmitter` 或 `StreamingResponseBody`；core 层用技术中立的流事件处理接口，不返回 `Flux<...>`，避免把 WebFlux 泄漏到核心业务层。

## 40. 阶段 1 接口是否提前做完整 Gateway 治理

**问题：** 阶段 1 需要保留 `Web -> Gateway -> Core` 路径，但如果接口设计直接铺开完整 OpenAI 兼容、Trace 查询、Token 看板、敏感策略、告警规则，会把阶段 2 的治理任务提前拉进来，拖慢基础 Agent 闭环。

**核心思路：** 阶段 1 接口只服务基础 Agent 闭环和后续可扩展边界：浏览器登录、Application/API Key、Profile、Skill/MCP 查询、对话 SSE、Web->Gateway 内部转发、Gateway 固定 SSE 测试、API Key 最小入口。

**核心方案：** 新增 `考核设计/接口设计/01-阶段1接口设计.md`。阶段 1 明确提供 Web API、Web->Gateway 内部接口、Gateway 空壳接口、SSE 事件格式和 `AgentRunCommand` / `ModelInvokeCommand` 字段；完整 `/v1/chat/completions`、Trace/Token/Security/Alert 查询和治理接口放阶段 2。

## 41. Web 接口与 Core 能力接口是否一一对应

**问题：** Web 面向 Admin、Developer、User、浏览器页面和外部调用场景，接口会比较多；如果 Core 也按角色和页面复制一套接口，会造成业务能力重复、权限逻辑分散、后续维护困难。

**核心思路：** Web 按访问场景和角色组织接口，Core 按业务能力组织接口。多个 Web 接口可以复用同一个 Core API 或 Application Service；Core 不因为不同角色使用同一能力就拆多套实现。

**核心方案：** Web Controller 可以按 `admin/user/developer` 和页面拆细接口；Core 维持稳定能力接口，如 `ModelConfigService`、`ProfileService`、`SkillRegistry`、`AgentRuntimeService`。权限由 Web 先拦截，Core application 层再基于 `tenantId/userId/applicationId/role` 做业务兜底，防止绕过入口层后越权。

## 42. 为什么不能只靠 URL 区分角色，必须用 Spring Security + RBAC

**问题：** 如果把接口地址按角色拆成 `/admin/api/traces`、`/user/api/traces`、`/developer/api/traces`，再手动解析 JWT 判断能否访问，看起来也能跑通。为什么还要用 Spring Security 和 RBAC？

**核心思路：** URL 能区分"谁能访问这个接口"，但不能区分"同一个接口，不同角色能查哪些数据"。按角色拆 URL 会导致同一张表的查询逻辑散布在多个 Controller 里，违背"数据写入口唯一"的原则。

**核心方案：** 统一 URL + Spring Security 做前端拦截（`@PreAuthorize`），RBAC 做权限到业务操作的映射（`TRACE:READ`），Core Service 只写一份查询逻辑，通过 `role` 参数控制 `where` 条件（Admin 查全部，User 只查自己的）。改查询逻辑只改一处。JWT 是身份证，Spring Security 是门禁系统，RBAC 是权限分级，三个东西各司其职。

## 43. 是否引入 Envoy AI Gateway / SkyWalking 做 GenAI 可观测

**问题：** Envoy AI Gateway + SkyWalking + BanyanDB 已经能提供 LLM 流量观测、Token 统计、TTFT/TPOT、Provider/Model 维度分析和 OTLP 接入。项目是否应该直接引入这套现成方案？

**核心思路：** 当前考核项目的目标是证明自研 Gateway、Agent Runtime、Trace、Token 配额的设计和实现能力，不应该在阶段 1/2 把核心链路交给外部基础设施。但这套方案的观测维度和 OTLP 接口边界值得借鉴。

**核心方案：** 阶段 2 仍以自研 `trace_roots`、`trace_spans`、`token_usage_logs` 落 PostgreSQL 为主；指标维度参考 OpenTelemetry GenAI 语义，至少覆盖 provider、model、application、user、profile、agent_mode、latency_ms、ttft_ms、tpot_ms、prompt_tokens、completion_tokens、total_tokens、status、error_code、tokens_estimated、stream。`core.trace` 预留 `TraceExporter` 接口，后续可选导出到 OTLP / SkyWalking / Jaeger / Zipkin；不把 Envoy AI Gateway、SkyWalking、BanyanDB 作为当前阶段必需依赖。

## 44. 阶段 2 是否先重构 Agent Runtime Pipeline

**问题：** emo 参考项目的 Pipeline 能把一次 Agent 执行拆成多个固定阶段，长期看有利于插入 Trace、Token、脱敏、SSE 和工具审计。但如果阶段 2 一开始就重构 `DefaultAgentRuntimeService`，会同时改变执行骨架和新增治理能力，问题定位成本上升。

**核心思路：** Pipeline 有长期价值，但阶段 2 的首要目标是 Trace / Token Usage 治理闭环。当前 `DefaultAgentRuntimeService` 还没有失控，先用小范围包装方法埋 span 和 token usage，更容易保持主链路稳定。

**核心方案：** 阶段 2 暂不做 Pipeline。先在现有 `DefaultAgentRuntimeService` 内用小方法封装 `startSpan`、`finishSpan`、`recordTokenUsage`；LLM Provider Registry 作为 `core.model` 内部增强优先落地，对外仍只暴露 `ModelInvokeService`。阶段 2 完成后根据 `run()` / `runModelLoop()` 膨胀程度，再单独评估 Pipeline 重构。

## 45. Application 管理为什么做禁用而不是物理删除

**问题：** 阶段 3 前端补 Application 管理闭环时，用户需要修改和删除能力。如果直接物理删除 Application，会影响 API Key、Profile、Conversation、Trace、Token Usage 等历史数据的归属和审计链路。

**核心思路：** Application 是应用级隔离和计量的核心资源，不适合作为普通表单数据直接删除。MVP 需要让用户能停止一个应用继续被使用，同时保留历史记录用于排查、看板和审计。

**核心方案：** Application 支持更新 `name/description`，删除语义落地为 `DISABLED` 状态。禁用后的应用仍在列表可见，但 `ensureApplicationOwned()` 继续只接受 `ACTIVE` 应用，因此 Chat、Profile、API Key 等后续操作会被后端拒绝；前端在选择应用时优先选择 `ACTIVE`，并对禁用应用展示只读/审计提示。

## 46. Profile 管理为什么做禁用而不是物理删除

**问题：** Profile 关联 Chat、工具绑定、Conversation、Trace 和 Token Usage。若直接物理删除，会让历史对话和运行审计失去配置来源，也会破坏问题排查时的上下文。

**核心思路：** Profile 是 Agent 运行配置快照的来源，不适合作为普通表单数据直接删除。用户需要的是让某个 Profile 不再被继续使用，同时保留历史可见性。

**核心方案：** Profile 删除语义落地为 `DISABLED` 状态。禁用后的 Profile 仍在列表可见，但前端只读展示；编辑、工具绑定保存和 Chat 运行选择都排除禁用 Profile；后端 `AgentContextBuilder` 兜底拒绝不可运行状态，避免绕过前端后继续执行。

## 47. 阶段 3 为什么先补模型管理页而不是 Jar Skill 热加载

**问题：** 用户已经需要在前端配置 DeepSeek、Qwen、OpenAI 等真实模型，并在 Profile / Direct model 中切换使用；同时 Jar Skill 热加载也是完整产品能力之一。若阶段 3 直接切入 Jar 热加载，会涉及上传、校验、ClassLoader、卸载、状态流转和安全边界，范围明显大于当前前端控制台收口目标。

**核心思路：** 阶段 3 的首要目标是让现有后端能力可视化、可操作、可演示。模型配置已有后端创建接口和运行时调用链路，只缺前端入口和供应商列表接口；Jar Skill 热加载还属于后续 Skill 市场增强，不应阻塞当前 Chat 体验。

**核心方案：** 当前先落地 `Model Management` 页面：Admin 可创建 OpenAI 兼容 Provider、创建 Model Config，Profile 和 Chat 继续复用已有模型配置列表。Jar Skill 热加载保持暂缓，后续单独设计上传、校验、加载、注册、禁用和审计闭环。

## 48. 是否借鉴 MVP Claude Code 项目

**问题：** `mvp-claude-code-main` 展示了类 Claude Code 的 Agent Harness，包括 token 级流式输出、ToolDispatcher、子 Agent、上下文压缩、本地 File/Bash 工具和文件型 Memory。如果直接照搬，会和当前平台的 Spring Boot 多模块、Web/Gateway/Core、PostgreSQL、多租户和权限体系冲突。

**核心思路：** 把它定位为 Agent Runtime / Harness 设计参考，而不是平台架构参考。当前项目继续保持既有工程架构，只吸收能增强 Agent 执行闭环的设计思想。

**核心方案：** 新增 `考核设计/参考项目分析/06-MVPClaudeCode设计借鉴记录.md`。后续可借鉴 token 级 streaming、ToolRegistry 统一工具视图、Planner/Executor/Reviewer 工具隔离、上下文压缩和高危工具确认；不引入 LangChain4j、Picocli、本地 File/Bash 裸工具能力、Markdown 文件型 Memory 或单体 Jar 结构。阶段 3 当前不因此扩范围，主要服务阶段 4 Team 和后续 Runtime 增强。

## 49. 阶段 4 Agent Team 是否新建完整 Team 表

**问题：** 阶段 4 要实现 Planner / Executor / Reviewer，如果一开始就新增 `team_runs`、`team_tasks`、`team_messages`、`team_reviews`，会带来 Entity、Mapper、查询接口、权限、前端列表和历史管理页面，容易把高分项扩成完整工作流产品。

**核心思路：** 阶段 4 的验收重点是 Team Demo 稳定跑通和 Trace 可观察，不是做 Team 历史管理后台。数据库模型已允许 Team MVP 先把运行数据放入 `trace_spans.attributes`。

**核心方案：** 阶段 4 第一版不建 Team 表。Team 运行过程通过 `team.run`、`team.plan`、`team.task.execute`、`team.tool.call`、`team.review` 等 Trace Span 记录，关键字段放 attributes。后续如果需要 Team 历史列表、任务重放、复杂审计，再单独新增 Team 表。

## 50. 阶段 4 是否引入 Graph / Workflow 引擎

**问题：** Spring AI Alibaba、JeecgBoot 等参考项目都有 Graph / Workflow / 流程编排思想，如果阶段 4 直接引入 Graph Core、Flowable 或低代码流程设计器，会明显改变当前 Agent Runtime 运行模型。

**核心思路：** 当前项目的考核主线是自研 Agent Runtime、Gateway 治理链和 PostgreSQL Trace。阶段 4 只需要证明平台能控制 Planner -> Executor -> Reviewer 协作，不需要通用工作流引擎。

**核心方案：** 阶段 4 只借鉴 Sequential / Routing / Loop 的编排思想，用 `core.team` 自研 Orchestrator 串起 Planner、Executor、Reviewer；不引入 Graph Core、Flowable、WebFlux/Reactor 或流程设计器。前端只展示 Mermaid 协作图，不做拖拽编排。

## 51. Team 工具权限如何设计

**问题：** Planner、Executor、Reviewer 如果都直接拿 Skill/MCP 执行能力，会出现角色越界：Planner 可能执行工具，Reviewer 可能绕过审查职责调用工具。

**核心思路：** 参考 MVP Claude Code 的子 Agent 工具隔离，工具权限要物理裁剪，而不是只靠 Prompt 约束。

**核心方案：** 阶段 4 在运行时引入轻量 `AgentTool` 统一视图，整合 Skill/MCP 的名称、描述、参数 schema、来源和风险等级。Planner 只能看工具描述，Executor 只拿授权后的可调用工具，Reviewer 默认不拿业务工具。底层仍复用现有 `SkillExecutor` / `McpToolExecutor`，不直接访问 mapper。

## 52. 阶段 4 是否做真正实时 Team SSE

**问题：** 理想情况下 Team 执行每一步都应实时推送 SSE，但当前阶段 3 的 Chat 是事件级 SSE，`AgentRunResult` 在执行完成后返回。如果阶段 4 同时改成运行中事件回调，会牵动 Web/Gateway/Core 的流式协议。

**核心思路：** 阶段 4 首要目标是 Team 协作闭环，流式体验可以逐步增强。先保证事件顺序、前端展示和 Trace 一致，再考虑实时 token / 实时 task 事件。

**核心方案：** 阶段 4 第一版必须做阶段性实时 Team SSE：Team 开始时输出 `team_start`，Planner 完成后立即输出 `team_plan`，每个任务开始/结束输出 `team_task_start` / `team_task_result`，工具调用输出 `team_tool_call` / `team_tool_result`，Reviewer 完成后输出 `team_review`。暂不做 token 级流式，但不接受 Team 全部结束后一次性刷屏。实现上 Core 只调用 `TeamEventSink.emit(TeamRuntimeEventDTO)`，Gateway 提供 SSE 适配实现，Core 不感知 OutputStream / SseEmitter。

## 53. Team 是否是普通用户也能使用的 Profile 模式

**问题：** Team 容易被误解成 Developer 专属能力，或者被误解成普通用户可以不受限制地配置团队、工具和核心提示词。

**核心思路：** Team 是 Profile 的执行模式，不是新的角色权限。能否使用 Team 与能否修改公共 Profile、核心 Prompt、高危工具授权是两件事。

**核心方案：** 普通用户可以创建和使用自己的私有 BASIC / TEAM Profile；Developer/Admin 可以配置公共或领域 Profile。普通用户仍不能修改公共 Profile、核心 System Prompt、未授权高危工具和全局安全策略。后端按 owner、visibility、role 和工具绑定规则继续做校验。

## 54. BASIC 和 TEAM 如何在运行时分流

**问题：** 如果为 Team 单独新增入口或让前端每次消息都传模式，会让 Web/Gateway/Core 分流逻辑变复杂，也容易绕过现有 Profile、Trace、Token 和权限链路。

**核心思路：** 执行模式由 Profile 决定，聊天入口保持不变。

**核心方案：** Chat 请求仍然传 `profileId`；Core 读取 Profile 的 `executionMode` 后分流：`BASIC` 走现有单 Agent Runtime，`TEAM` 走 Team Runtime。`agentMode=none` 仍是纯模型透传，不进入 Team。

## 55. Team Demo 是否会限制只能回答活动策划问题

**问题：** 阶段 4 Demo 以“重庆 20 人团建活动策划”为主，如果把 fallback 写死成 weather/search/calculator 活动策划模板，Team 就会变成只服务 Demo 的硬编码功能。

**核心思路：** Demo 可以有推荐场景，但 Runtime 必须保持通用。

**核心方案：** Planner 正常路径根据用户问题和可用工具生成计划；如果两次模型输出都不是合法 JSON、计划校验失败或模型调用异常，则统一降级为单个 `MODEL_TASK`，把原始用户问题交给后续 Executor 模型任务处理。活动策划只作为种子数据、演示 Profile 或验收脚本，不写死在通用 fallback 中。

## 56. 单 Agent token 级流式和 Team token 级流式什么时候做

**问题：** 用户希望回复像真实大模型一样逐 token 输出，但阶段 4 同时引入 Team Runtime、事件流和前端进度面板，如果一起做 token 级流式，排错范围会过大。

**核心思路：** 先做 Team 事件级实时，再做 token 级实时。

**核心方案：** 阶段 4 先实现 `team_plan`、`team_task_start`、`team_tool_call`、`team_tool_result`、`team_task_result`、`team_review`、`team_final` 等阶段性 SSE。单 Agent token 级流式和 Team 最终答案 token 级流式放到 Team 闭环稳定后的独立增强中处理。

## 57. Codex Skill 与平台 Jar Skill 是否是一回事

**问题：** Codex 本地 Skill 和 agent-platform 的 Jar Skill 都叫 Skill，容易混淆为同一种能力，甚至误以为可以用平台 Jar 热加载实现 Codex 开发 Skill。

**核心思路：** 两者运行环境、触发方式和安全边界完全不同。

**核心方案：** Codex Skill 是本机开发助手能力，用于辅助写代码、设计、审查；平台 Jar Skill 是后端通过上传、校验和 ClassLoader 加载后，供 Agent Runtime 调用的业务工具。Jar Skill 第一版只作为 Agent 工具，不承载 Codex 开发技能或 Agent 角色模板。

## 58. 阶段 4 如何保留稳定回退点

**问题：** Team Runtime 会逐步接触主聊天链路，如果阶段 4 后续实现破坏现有 BASIC 模式，需要能回退到稳定版本。

**核心思路：** 用小步 commit 作为回退点，阶段完成后再打 tag。

**核心方案：** 已形成的阶段 4 回退点包括 `c786382 feat(profile): expose agent execution mode` 和 `8b5df5c feat(team): add task plan model and validation`。后续每个 Step 继续独立提交；阶段 4 整体验收通过后再打 `phase-4-complete` 标签。

## 59. TaskPlanValidator 是否需要立即抽接口

**问题：** 阶段 4 审查中提出 `TaskPlanValidator` 是具体类，不符合常见的 api 接口 + application 实现分层，建议改成 `TeamPlanValidator` 接口和默认实现。

**核心思路：** 抽象应服务真实变化点。当前 Validator 是纯计算校验组件，不依赖外部资源，不跨模块暴露，也暂时只有一个调用方向。此时抽接口会增加样板代码，收益有限。

**核心方案：** 当前不抽接口，继续直接测试具体类。等 Planner、Reviewer、Orchestrator 中至少两个地方真实复用，或者需要为模型角色单测替换校验器时，再抽 `TeamPlanValidator` 接口并迁移为默认实现。

## 60. Team Plan 校验是否需要立即新增专用错误码

**问题：** 阶段 4 审查中提出 Planner 输出非法和用户请求非法都使用 `REQUEST_INVALID`，后续可能难以区分错误来源。

**核心思路：** 专用错误码有价值，但它属于 common 全局协议，应在真实消费路径明确后再加入。当前还没实现 Planner JSON 解析、修正 prompt 和 fallback，过早新增容易成为闲置约定。

**核心方案：** Step 2 继续使用 `REQUEST_INVALID`。到 Step 4 实现 Planner JSON 解析和修正重试时，再决定是否新增 `TEAM_PLAN_INVALID`，并同步测试 Planner fallback、Web/Gateway 错误映射和前端展示。

## 61. AgentTool Resolver 是否应该重新查询 Skill/MCP

**问题：** AgentTool 统一工具视图可以从 Skill/MCP 服务重新查询，也可以直接从 AgentContextDTO 转换。如果重新查表，可能和 AgentContextBuilder 的授权结果不一致。

**核心思路：** AgentContextDTO 已经代表当前请求的可用上下文，里面的 availableSkills / availableMcpTools 已经过 Profile 绑定、用户选择和可用性过滤。AgentTool Resolver 不应重复承担上下文构建职责。

**核心方案：** `AgentToolResolver.resolve(AgentContextDTO context)` 直接从 `context.availableSkills()` 和 `context.availableMcpTools()` 生成 AgentTool 列表，不调用 SkillQueryService / McpToolQueryService。Planner、Validator、Executor 共用这份列表。

## 62. AgentTool 第一版是否使用命名空间和真实 riskLevel

**问题：** Skill/MCP 未来可能出现同名工具，且 Team 执行需要风险等级；但当前 SkillDTO / McpToolDTO 没有 riskLevel 来源。

**核心思路：** 先做最小稳定契约，避免为了未来增强把 Step 3 变成安全策略重构。

**核心方案：** 第一版 Skill 工具名使用 `SkillDTO.code`，MCP 工具名使用 `McpToolDTO.name`，`displayName` 使用 DTO 的 `name`，`riskLevel` 统一 `LOW`。后续出现重名再升级为 `skill:xxx` / `mcp:xxx`，接入高危工具确认或安全策略后再计算真实风险等级。

## 63. AgentTool Dispatcher 是否负责授权校验

**问题：** `AgentToolDispatcher` 可以直接按 sourceType 调 Skill/MCP 执行器，但如果它不校验授权工具列表，Planner 输出未授权工具时可能被误执行。

**核心思路：** Dispatcher 是底层分发器，不应同时承担上下文授权职责。授权边界应该在 Executor 调用 Dispatcher 前完成，因为 Executor 同时持有 TaskPlan 和 resolved AgentTool 列表。

**核心方案：** Dispatcher 只做 sourceType 分发和基础参数校验。Step 5 Executor 必须用 resolved AgentTool 列表校验 `TOOL_TASK.suggestedTool`，未授权时任务返回 failed，不调用 Dispatcher。测试必须覆盖 Planner 给出未授权工具的场景。

## 64. TeamEventSink 是否暴露 OutputStream / SSE 细节给 Core

**问题：** 如果 TeamEventSink 被设计成直接写 OutputStream，Core 层会感知 Gateway 的 HTTP/SSE 实现细节，后续单测和复用都会变重。

**核心思路：** Core 只产生领域事件，Gateway 负责协议适配。

**核心方案：** Core 定义并调用 `TeamEventSink.emit(TeamRuntimeEventDTO)`；Gateway 实现一个 SSE 适配器，把事件写成 SSE。Core 不依赖 `OutputStream`、`SseEmitter` 或 HTTP API。SSE 写出失败由 Gateway 转换为 `error` 事件并结束流。

## 65. TaskPlan 的 description 和 MODEL_TASK.suggestedTool 如何校验

**问题：** Executor 的 MODEL_TASK prompt 依赖 task description，但当前 TaskPlanValidator 只校验 name；同时 MODEL_TASK 如果携带 suggestedTool，会让模型任务和工具任务语义混淆。

**核心思路：** TaskPlan JSON 应减少歧义，让 Planner 输出更容易修正，也让 Executor prompt 构建稳定。

**核心方案：** task.description 必填；`TOOL_TASK.suggestedTool` 必填且必须存在于 AgentTool 列表；`MODEL_TASK.suggestedTool` 必须为空。后续补 TaskPlanValidator 测试与实现。

## 66. Team 纯逻辑类是否需要注册为 Spring Bean

**问题：** `DefaultTeamPlanner` 作为 Spring Bean 依赖 `TaskPlanValidator`。如果 `TaskPlanValidator` 只在单元测试里手动 new，而不是 Spring Bean，单测能过但 Web/Gateway 应用上下文会启动失败。

**核心思路：** 会被 Spring 管理类构造注入的协作者，即使是无状态纯逻辑，也应该明确注册为 Bean；不参与注入的纯函数工具才保持普通类。

**核心方案：** `TaskPlanValidator` 标注 `@Component`。阶段 4 后续 `ReviewResultValidator`、`TeamRunLimiter` 等如果要被 Runtime/Planner/Executor 构造注入，也按同样规则注册，并用应用上下文测试兜底。
