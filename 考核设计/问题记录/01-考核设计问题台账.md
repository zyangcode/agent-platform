# 考核设计问题台账

> 记录考核设计过程中遇到的关键问题。每条只保留：问题、核心思路、核心方案。

## 1. 模块是否拆得过细

**问题：** 一开始按能力拆出很多 Maven 模块，一个人开发维护成本高。

**核心思路：** 物理模块少一点，逻辑边界不能丢。

**核心方案：** MVP 用少量工程单元：common、core、web、gateway、frontend、feishu-bot；core 内部按 identity、agent、skill、quota、trace 等包保留边界。

## 2. web / gateway 是否拥有业务表

**问题：** 如果 users、applications、api_keys 归 web/gateway，core 会反向依赖入口层。

**核心思路：** 入口层只做协议适配，业务数据归核心业务层。

**核心方案：** web/gateway 不拥有核心业务表；users、applications、api_keys、quota、trace 等归 core 内部逻辑包。

## 3. common 是否放业务 DTO / Entity

**问题：** common 放业务对象后会变成杂物模块，所有模块都依赖它。

**核心思路：** common 只能放真正通用的东西。

**核心方案：** common 只放异常、错误码、通用响应、工具类；业务 DTO、Entity、Command、Result 放各自模块或包内。

## 4. Token 和 Trace 是否都异步

**问题：** Trace 全同步影响性能，Token 全异步又会导致配额不准。

**核心思路：** Token 是额度数据，不能丢；Trace 明细是排查数据，可以延迟。

**核心方案：** Token 预扣、结算、修正必须同步或可靠写入；Trace Span 明细、趋势统计、告警通知可以异步。

## 5. Token 预扣是否强绑定 Redis

**问题：** 文档写死 Redis 会让 MVP 被技术选型绑定。

**核心思路：** 原则文档不要过早绑定具体实现。

**核心方案：** 统一写成“数据库事务或 Redis 原子预扣”，后续按实现复杂度选择。

## 6. 飞书机器人是否依赖 Agent 平台代码

**问题：** 如果 feishu-bot 依赖 agent-platform common 或内部 Service，会破坏独立服务边界。

**核心思路：** 项目 2 是独立服务，项目 3 只通过接口调用它。

**核心方案：** feishu-bot 不依赖 agent-platform 代码；Gateway/Core 只能通过 HTTP 调飞书告警接口。

## 7. 双通道是否需要拆开

**问题：** Web/JWT 和 API Key/OpenAI 兼容接口如果混在一起，鉴权和拦截链会混乱。

**核心思路：** 两个入口调用方不同、协议不同；但 AI 执行治理能力必须统一。

**核心方案：** web 负责浏览器/JWT/管理入口；gateway 负责统一 AI 调用治理入口。浏览器 AI 对话/测试由 web 转发到 gateway，外部 API Key 直接进 gateway，最终都进入同一个 core Agent Runtime。

## 8. CLAUDE.md 是否要写完整设计

**问题：** 把所有设计文档塞进 CLAUDE.md 会导致上下文膨胀。

**核心思路：** CLAUDE.md 放短约束和文档索引，详细内容放设计文档。

**核心方案：** CLAUDE.md 只保留关键强约束、实现顺序和文档路径；具体解释按需读取对应 md 文件。

## 9. 下一步是模块设计还是数据库设计

**问题：** 工程单元和 core 包边界已经确定，继续重复模块设计会浪费时间。

**核心思路：** 当前最需要把数据归属和表关系定下来。

**核心方案：** 下一步进入数据库模型设计，再做 Skill 接口规范和用户请求主链路设计。

## 10. 用户注册登录模型

**问题：** 初版设计是"不做登录页、Admin 后台手动创建用户并分发 API Key"，这更像是内部管理系统而非面向用户的 SaaS 产品。

**核心思路：** 用户应能自助注册登录，API Key 应由用户自己管理而非 Admin 代发。

**核心方案：** MVP 做极简登录注册页（用户名+密码）；用户登录后在个人中心创建 Application，每个应用自动生成一个 API Key；Admin 后台手动创建用户仅作为兜底。

## 11. API Key 绑定模型

**问题：** 初版 Key 直接绑定 User，一个用户只有一个 Key。Key 泄漏意味着整个账户的调用权限泄漏。

**核心思路：** 参考中转站模式，Key 绑定 Application 而非 User。

**核心方案：** 一个用户可创建多个 Application，每个应用自动生成一个 API Key；不同工具/脚本用不同 Key；Key 泄漏只吊销那一个，不影响其他应用；Token 消耗按应用分别统计。

## 12. 计费模型

**问题：** 初版讨论中混用了"余额""充值""人民币扣费"等概念，容易把项目变成支付计费系统。

**核心思路：** 考核重点不是支付，而是 Token 治理能力。MVP 不应接真实支付。

**核心方案：** 统一为 Token 配额模型，不提余额/人民币；配额超限返回 HTTP 429；Admin 后台手动配置日/月额度；真实支付、充值页面放到后续增强。

## 13. Agent 执行模式

**问题：** 初版所有请求都走完整 Agent Runtime（Profile+Skill+ReAct+记忆），但有些用户只想当纯模型转发使用。

**核心思路：** 保留 Agent 能力作为默认，同时提供透传选项，让用户自行选择。

**核心方案：** 增加 `agent_mode` 参数：`agent`（默认）走完整链路，`none` 纯模型转发。浏览器端 AI 对话/测试始终走 agent 模式，API Key 通道可选两种模式。两模式共用 Token 配额和 Trace。

## 14. 文档双写与一致性问题

**问题：** CLAUDE.md 和设计文档多次修改后，两端出现措辞不一致（如"自行生成"vs"自动生成"、页面数 8 vs 9、功能点数统计过期）。

**核心思路：** 每次改完设计文档，必须回查 CLAUDE.md 和关联的功能清单是否同步。

**核心方案：** 建立了 CLAUDE.md ↔ 01-功能清单 ↔ 06-双通道设计 ↔ 架构原则文档 四份文件的交叉核对机制，关键字段统一为最终表述，功能点数随编辑同步更新。

## 15. Web 请求是否全部经过 Gateway

**问题：** 如果所有 Web 请求都走 Gateway，管理配置接口会被 AI 治理链拖复杂；如果 Web AI 对话不走 Gateway，又不符合项目 3 的统一治理定位。

**核心思路：** 按请求类型分流，不按角色分流。

**核心方案：** AI 执行请求统一走 Gateway；管理、配置、统计、查询请求走 Web -> Core。

## 16. 多 Agent 后置后 Developer 是否也后置

**问题：** Planner/Executor/Reviewer 放到后期做，会不会导致 Developer 模式也必须后置。

**核心思路：** Developer 基础能力和 Team 高分项要拆开。

**核心方案：** MVP 的 Developer 先负责配置单 Agent Profile、Prompt、模型、Skill、记忆和安全策略；Team 只是 Profile 的增强执行模式，后期再加。

## 17. Web 转发 Gateway 如何实现

**问题：** 浏览器 AI 请求走 Web -> Gateway -> Core 后，如果 Web 直接依赖 Gateway 代码，会造成入口层耦合。

**核心思路：** Web 和 Gateway 都是入口适配层，不能互相变成代码强依赖。

**核心方案：** Web 通过 HTTP 内部接口转发 AI 执行请求到 Gateway，并携带内部服务 Token 或签名 Header；Gateway 校验后才接受 user_id/app_id/tenant_id。

## 18. Admin 测试模型与测试 Agent 的路由区别

**问题：** Admin 测试模型连接和测试 Agent 效果如果都走同一路由，会让管理诊断和 AI 执行混在一起。

**核心思路：** 是否经过 Gateway 取决于是否触发完整 AI 执行治理链。

**核心方案：** 测试模型供应商连通性走 Web -> Core model；测试 Agent/Profile 效果走 Web -> Gateway -> Core。

## 19. 阶段 1 是否先建 Gateway

**问题：** 如果阶段 1 直接 Web -> Core，阶段 2 改成 Web -> Gateway -> Core 会改入口代码。

**核心思路：** 阶段 1 先保留最终调用路径，治理能力后补。

**核心方案：** 阶段 1 先建 Gateway 空壳，只做 AI 请求转发；阶段 2 在空壳上补 Trace、Token、脱敏、告警治理链。

## 20. Token 仪表盘是否后置

**问题：** 项目 3 要求 Token 仪表盘，如果全部放到高分增强会影响验收。

**核心思路：** 把“基础看板”和“复杂分析”拆开。

**核心方案：** MVP 做当前用量、调用明细、简单 TOP；历史趋势图、多维筛选、成本折算后续增强。

## 21. 飞书机器人完整功能与告警接口的阶段关系

**问题：** 项目 3 阶段 2 需要飞书告警，但完整飞书机器人放在阶段 5。

**核心思路：** 先做最小告警闭环，再做完整机器人能力。

**核心方案：** 阶段 2 先提供 feishu-bot 内部告警接口或 mock 告警接口；阶段 5 再补基础指令、Git/CR、DevOps。

## 22. MCP 示例是否容易漏掉

**问题：** 项目 1 明确要求 MCP 示例，但路线图容易只关注 Skill。

**核心思路：** MCP Client 示例是硬性验收点，MCP Server 是增强项。

**核心方案：** 阶段 1 至少实现 1 个 MCP Client 示例工具，如只读文件系统或只读 SQL；平台 Skill 暴露为 MCP Server 后续增强。

## 23. Gateway 图中是否能直接写 Agent Runtime

**问题：** 如果在 Gateway 模块里直接画 `Agent Runtime`，容易误解为 Runtime 属于 Gateway。

**核心思路：** Gateway 只负责统一入口和治理链，真正的 Agent Runtime 属于 core。

**核心方案：** 架构图中 Gateway 节点改为 `RuntimeInvokeFilter`，连线标注“调用 Core AgentRuntime”。

## 24. feishu-bot 是否属于平台引擎层

**问题：** 如果把 feishu-bot 放在引擎层，容易误解为它依赖 agent-platform 的 core/common/web/gateway。

**核心思路：** feishu-bot 是项目 2 的独立服务，只通过 HTTP 告警接口与平台通信。

**核心方案：** 架构图单独设置“独立服务层”，feishu-bot 使用独立配置库；平台核心表仍归 core 管理。

## 25. 架构图中告警链路是否同步调用飞书

**问题：** 如果图上写“HTTP 调告警接口”，容易让人以为 Gateway 响应前同步调用飞书。

**核心思路：** 告警通知属于边缘动作，不能阻塞主响应。

**核心方案：** 图中改为“写告警事件 / 异步 HTTP 通知”，AlertFilter 只生成事件，飞书发送异步执行。

## 26. 前端是否属于基础设施层

**问题：** 如果把 frontend 放在基础设施层，会混淆用户入口和运行基础设施。

**核心思路：** frontend 是浏览器用户入口的一部分，基础设施只放数据库、缓存、公共模块等支撑能力。

**核心方案：** 架构图调整为 `浏览器用户 -> frontend -> web`，frontend 不再放在基础设施层。

## 27. 模块目录与表归属如何落地到代码

**问题：** 只有架构原则和数据库表设计还不够，后续写代码时仍可能出现 Entity/Mapper 乱放、Controller 直接查表、gateway 直接写业务表等问题。

**核心思路：** 先固定工程模块、Java 包、表归属和调用方向，让“代码放哪里”有明确依据。

**核心方案：** 新增 `考核设计/架构与模块设计/04-模块目录与包结构设计.md`，规定 common/core/web/gateway/frontend/feishu-bot 工程边界，core 内部按 identity/model/profile/skill/mcp/agent/memory/context/quota/trace/security/alert/team 分包，并把每张表映射到唯一 Entity/Mapper 所属包。

## 28. core 内部包边界是否只靠自觉

**问题：** core 内部有多个业务包，但它们都在同一个 Maven 模块里，编译器不会阻止 `core.agent` 直接访问 `core.skill.mapper`。

**核心思路：** 文档纪律要尽量变成自动检查，否则代码量上来后很容易越界。

**核心方案：** 在 `04-模块目录与包结构设计.md` 中补充 ArchUnit 包边界规则，要求禁止跨业务包访问 `entity`、`mapper`、`internal`，只允许通过 `api`、`dto`、`command` 协作。

## 29. Web 转发 Gateway 的 SSE 是否会被缓冲

**问题：** 浏览器 AI 对话走 `frontend -> web -> gateway -> core`，SSE 中间多了一跳 Web 转发，可能出现缓冲、超时、错误事件丢失等问题。

**核心思路：** Gateway 空壳阶段就要验证流式透传，而不是等 Agent Runtime 完成后再发现链路问题。

**核心方案：** 在阶段 1 落地顺序中加入固定 SSE 流透传测试：Gateway 每隔固定时间输出 SSE 事件，Web 逐事件转发给浏览器或 curl，确认不缓冲、不超时、错误事件可传递。

## 30. 上下文 Token 超额压缩放在哪里

**问题：** 用户输入、历史消息、记忆、Skill/MCP 描述可能超过模型上下文窗口，压缩逻辑如果放在 Gateway 会让入口层理解 Agent 内部结构。

**核心思路：** 上下文压缩是 Agent 上下文组装策略，不是网关治理能力。

**核心方案：** 压缩和裁剪放在 `core.context`，由 `AgentContextBuilder`、`ContextWindowManager`、`ContextCompressor` 处理；Gateway 只负责 Token 配额预扣、治理链和协议转换。

## 31. Gateway 是否也组装上下文

**问题：** Gateway 有 `GatewayContext`，core 又有 `AgentContextBuilder`，容易误解为两边都在组装模型上下文。

**核心思路：** 两个 Context 的含义不同。GatewayContext 是请求治理上下文，AgentContext 是模型执行上下文。

**核心方案：** Gateway 只保存 tenant_id、user_id、application_id、trace_id、agent_mode 等治理信息，并显式转换成 `AgentRunCommand` 或 `ModelInvokeCommand`；真正的 Prompt、记忆、工具列表组装放在 `core.context`。

## 32. Token 预扣和结算如何避免并发错误

**问题：** 如果配额检查采用“先查余额再更新”，并发请求可能同时通过，导致超额使用；释放预扣失败也可能吃掉用户额度。

**核心思路：** Token 配额是额度数据，必须原子、幂等、可重试。

**核心方案：** 在数据库模型中给 `quota_configs` 和 `quota_reservations` 增加 `version`；配额预扣采用 CAS 或 `SELECT ... FOR UPDATE` 行锁；`quota_reservations.trace_id` 唯一，释放和结算按 `trace_id + status + version` 条件更新，避免重复释放或重复扣减。

## 33. 飞书告警失败是否影响聊天请求

**问题：** 如果 Gateway 在响应前同步调用 feishu-bot，飞书接口失败会拖垮用户 AI 对话。

**核心思路：** 告警通知是边缘动作，不能阻塞主链路。

**核心方案：** AlertFilter 只写 `alert_events` 或待通知状态，通知适配器异步调用 feishu-bot；飞书调用失败只更新 notify_status/retry_count，不影响本次 AI 响应。

## 34. Gateway Filter 与 Core Service 的分工边界

**问题：** Gateway 治理链的 Filter（QuotaFilter、TraceFilter、SensitiveDataFilter、TokenRecordFilter 等）和 Core 的 Service（QuotaService、TraceService、SensitiveDataScanner、TokenUsageService 等）成对出现、名称相近，容易让人产生"功能重复"的疑问，也可能导致 Filter 越过 Service 直接操作 Mapper。

**核心思路：** Filter 管"拦"（编排决策 + 请求阻断），Service 管"做"（业务执行 + 表写入）。两者通过 `core.xxx.api` 接口单向调用，Filter 永远不直接碰 Mapper。

**核心方案：** Gateway 每个 Filter 只调对应的 `core.xxx.api` 接口；每张表只有一个写入入口（归属包的 Service），不管谁触发都走同一个 Service；Filter 根据 Service 返回结果做阻断决策（如预扣失败返回 HTTP 429）。禁止 `TokenRecordFilter → TokenUsageLogMapper` 这类跨模块直接查表。

## 35. web→gateway 与 gateway→core 两种依赖方式的选择依据

**问题：** web→gateway 走 HTTP 内部调用，gateway→core 走 Maven 代码依赖。两种依赖方式的选择依据如果不说清楚，后续实现时可能走偏。

**核心思路：** 选 HTTP 还是 Maven 依赖，取决于调用方是否应受被调用方信任、以及被调用操作能否承受网络失败风险。

**核心方案：** web→gateway 走 HTTP——Gateway 必须独立校验内部服务 Token，不能信任 Web 传入的任何身份信息，HTTP 强制了安全边界。gateway→core 走 Maven 方法调用——Token 预扣/结算、Trace 创建等额度数据操作，不能承受 HTTP 额外的网络失败风险。已在 04 号文档 2.2 节和 8.3 节明确规定。

## 36. 技术选型与工程约定是否需要单独成文

**问题：** 架构、数据库和模块目录已经有文档，但后续真正写代码时仍可能忘记版本、Flyway 位置、错误码、SSE 透传、ArchUnit 检查、DTO/Entity 分层等落地细节。

**核心思路：** 架构文档回答“为什么这样拆”，工程约定回答“代码具体怎么写、怎么检查、怎么验收”。这类规则如果散落在聊天记录里，后面实现时最容易丢。

**核心方案：** 新增 `考核设计/技术选型与工程约定/01-技术选型与工程约定.md`，统一固化技术选型、模块依赖、包结构、DTO/Command/Entity、Flyway、统一响应、错误码、鉴权上下文、SSE、日志 Trace、Token、ArchUnit、测试和配置文件规则；并在 `CLAUDE.md` 中加入该文档索引。

## 37. 是否借鉴 AgentX 参考项目设计

**问题：** `AgentX-master/docs` 中有 Agent、Token 超限、计费、Trace、SQL、开发规范等文档，如果直接照搬会引入真实余额计费、订单、完整 RAG、Agent 市场等超出 MVP 的复杂度。

**核心思路：** 借鉴设计方法，不照搬业务范围。参考项目最有价值的是复杂问题单独成文、Trace 按阶段拆解、开发规范规则化，而不是具体表结构。

**核心方案：** 新增 `考核设计/参考项目分析/01-AgentX设计借鉴记录.md`，明确可借鉴和不可照搬内容；同时把查询命名、三层校验、jsonb 字段处理、事务边界等低风险工程规则补入工程约定。后续优先补阶段 1 接口设计和开发计划，再按需补上下文 Token 策略设计、Trace 设计。

## 38. Spring AI 是否接管 Agent Runtime

**问题：** Spring AI ChatClient 自带模型调用、streaming 和 tool calling 能力，如果直接让它自动执行工具调用，Agent 循环中的 action / observation / Trace / Token / SSE 事件会变得不可控。

**核心思路：** Spring AI 只适合做模型调用适配层，不适合接管本项目的 Agent Runtime。项目需要自己控制 ReAct while 循环、Skill/MCP 执行、多 Agent 编排和实时事件输出。

**核心方案：** Spring AI 或 OpenAI 兼容 Client 统一封装在 `core.model`，只保留请求构建、响应解析、usage 读取、流式 token 适配和多供应商 ChatClient 封装；`core.agent` 自己控制单 Agent ReAct 循环，`core.team` 后续自己控制 Planner / Executor / Reviewer 状态机。

## 39. SSE 是否引入 WebFlux / Flux

**问题：** 初版工程约定中把 `Flux` 和 Spring MVC SSE 写在一起，容易让实现者误以为阶段 1 要混用 WebFlux 和 Spring MVC。

**核心思路：** 当前后端栈是 Spring MVC + MyBatis-Plus，阶段 1 不引入响应式全家桶。SSE 的验收重点是不缓冲、可逐事件透传，而不是使用 Flux。

**核心方案：** 阶段 1 SSE 基于 Spring MVC，优先使用 `SseEmitter` 或 `StreamingResponseBody`；core 层用技术中立的流事件处理接口，不返回 `Flux<...>`，避免把 WebFlux 泄漏到核心业务层。

## 40. 阶段 1 接口是否提前做完整 Gateway 治理

**问题：** 阶段 1 需要保留 `Web -> Gateway -> Core` 路径，但如果接口设计直接铺开完整 OpenAI 兼容、Trace 查询、Token 看板、敏感策略、告警规则，会把阶段 2 的治理任务提前拉进来，拖慢基础 Agent 闭环。

**核心思路：** 阶段 1 接口只服务基础 Agent 闭环和后续可扩展边界：浏览器登录、Application/API Key、Profile、Skill/MCP 查询、对话 SSE、Web->Gateway 内部转发、Gateway 固定 SSE 测试、API Key 最小入口。

**核心方案：** 新增 `考核设计/接口设计/01-阶段1接口设计.md`。阶段 1 明确提供 Web API、Web->Gateway 内部接口、Gateway 空壳接口、SSE 事件格式和 `AgentRunCommand` / `ModelInvokeCommand` 字段；完整 `/v1/chat/completions`、Trace/Token/Security/Alert 查询和治理接口放阶段 2。

## 41. Web 接口与 Core 能力接口是否一一对应

**问题：** Web 面向 Admin、Developer、User、浏览器页面和外部调用场景，接口会比较多；如果 Core 也按角色和页面复制一套接口，会造成业务能力重复、权限逻辑分散、后续维护困难。

**核心思路：** Web 按访问场景和角色组织接口，Core 按业务能力组织接口。多个 Web 接口可以复用同一个 Core API 或 Application Service；Core 不因为不同角色使用同一能力就拆多套实现。

**核心方案：** Web Controller 可以按 `admin/user/developer` 和页面拆细接口；Core 维持稳定能力接口，如 `ModelConfigService`、`ProfileService`、`SkillRegistry`、`AgentRuntimeService`。权限由 Web 先拦截，Core application 层再基于 `tenantId/userId/applicationId/role` 做业务兜底，防止绕过入口层后越权。

## 42. 为什么不能只靠 URL 区分角色，必须用 Spring Security + RBAC

**问题：** 如果把接口地址按角色拆成 `/admin/api/traces`、`/user/api/traces`、`/developer/api/traces`，再手动解析 JWT 判断能否访问，看起来也能跑通。为什么还要用 Spring Security 和 RBAC？

**核心思路：** URL 能区分"谁能访问这个接口"，但不能区分"同一个接口，不同角色能查哪些数据"。按角色拆 URL 会导致同一张表的查询逻辑散布在多个 Controller 里，违背"数据写入口唯一"的原则。

**核心方案：** 统一 URL + Spring Security 做前端拦截（`@PreAuthorize`），RBAC 做权限到业务操作的映射（`TRACE:READ`），Core Service 只写一份查询逻辑，通过 `role` 参数控制 `where` 条件（Admin 查全部，User 只查自己的）。改查询逻辑只改一处。JWT 是身份证，Spring Security 是门禁系统，RBAC 是权限分级，三个东西各司其职。

## 43. 是否引入 Envoy AI Gateway / SkyWalking 做 GenAI 可观测

**问题：** Envoy AI Gateway + SkyWalking + BanyanDB 已经能提供 LLM 流量观测、Token 统计、TTFT/TPOT、Provider/Model 维度分析和 OTLP 接入。项目是否应该直接引入这套现成方案？

**核心思路：** 当前考核项目的目标是证明自研 Gateway、Agent Runtime、Trace、Token 配额的设计和实现能力，不应该在阶段 1/2 把核心链路交给外部基础设施。但这套方案的观测维度和 OTLP 接口边界值得借鉴。

**核心方案：** 阶段 2 仍以自研 `trace_roots`、`trace_spans`、`token_usage_logs` 落 PostgreSQL 为主；指标维度参考 OpenTelemetry GenAI 语义，至少覆盖 provider、model、application、user、profile、agent_mode、latency_ms、ttft_ms、tpot_ms、prompt_tokens、completion_tokens、total_tokens、status、error_code、tokens_estimated、stream。`core.trace` 预留 `TraceExporter` 接口，后续可选导出到 OTLP / SkyWalking / Jaeger / Zipkin；不把 Envoy AI Gateway、SkyWalking、BanyanDB 作为当前阶段必需依赖。

## 44. 阶段 2 是否先重构 Agent Runtime Pipeline

**问题：** emo 参考项目的 Pipeline 能把一次 Agent 执行拆成多个固定阶段，长期看有利于插入 Trace、Token、脱敏、SSE 和工具审计。但如果阶段 2 一开始就重构 `DefaultAgentRuntimeService`，会同时改变执行骨架和新增治理能力，问题定位成本上升。

**核心思路：** Pipeline 有长期价值，但阶段 2 的首要目标是 Trace / Token Usage 治理闭环。当前 `DefaultAgentRuntimeService` 还没有失控，先用小范围包装方法埋 span 和 token usage，更容易保持主链路稳定。

**核心方案：** 阶段 2 暂不做 Pipeline。先在现有 `DefaultAgentRuntimeService` 内用小方法封装 `startSpan`、`finishSpan`、`recordTokenUsage`；LLM Provider Registry 作为 `core.model` 内部增强优先落地，对外仍只暴露 `ModelInvokeService`。阶段 2 完成后根据 `run()` / `runModelLoop()` 膨胀程度，再单独评估 Pipeline 重构。

## 45. Application 管理为什么做禁用而不是物理删除

**问题：** 阶段 3 前端补 Application 管理闭环时，用户需要修改和删除能力。如果直接物理删除 Application，会影响 API Key、Profile、Conversation、Trace、Token Usage 等历史数据的归属和审计链路。

**核心思路：** Application 是应用级隔离和计量的核心资源，不适合作为普通表单数据直接删除。MVP 需要让用户能停止一个应用继续被使用，同时保留历史记录用于排查、看板和审计。

**核心方案：** Application 支持更新 `name/description`，删除语义落地为 `DISABLED` 状态。禁用后的应用仍在列表可见，但 `ensureApplicationOwned()` 继续只接受 `ACTIVE` 应用，因此 Chat、Profile、API Key 等后续操作会被后端拒绝；前端在选择应用时优先选择 `ACTIVE`，并对禁用应用展示只读/审计提示。

## 46. Profile 管理为什么做禁用而不是物理删除

**问题：** Profile 关联 Chat、工具绑定、Conversation、Trace 和 Token Usage。若直接物理删除，会让历史对话和运行审计失去配置来源，也会破坏问题排查时的上下文。

**核心思路：** Profile 是 Agent 运行配置快照的来源，不适合作为普通表单数据直接删除。用户需要的是让某个 Profile 不再被继续使用，同时保留历史可见性。

**核心方案：** Profile 删除语义落地为 `DISABLED` 状态。禁用后的 Profile 仍在列表可见，但前端只读展示；编辑、工具绑定保存和 Chat 运行选择都排除禁用 Profile；后端 `AgentContextBuilder` 兜底拒绝不可运行状态，避免绕过前端后继续执行。

## 47. 阶段 3 为什么先补模型管理页而不是 Jar Skill 热加载

**问题：** 用户已经需要在前端配置 DeepSeek、Qwen、OpenAI 等真实模型，并在 Profile / Direct model 中切换使用；同时 Jar Skill 热加载也是完整产品能力之一。若阶段 3 直接切入 Jar 热加载，会涉及上传、校验、ClassLoader、卸载、状态流转和安全边界，范围明显大于当前前端控制台收口目标。

**核心思路：** 阶段 3 的首要目标是让现有后端能力可视化、可操作、可演示。模型配置已有后端创建接口和运行时调用链路，只缺前端入口和供应商列表接口；Jar Skill 热加载还属于后续 Skill 市场增强，不应阻塞当前 Chat 体验。

**核心方案：** 当前先落地 `Model Management` 页面：Admin 可创建 OpenAI 兼容 Provider、创建 Model Config，Profile 和 Chat 继续复用已有模型配置列表。Jar Skill 热加载保持暂缓，后续单独设计上传、校验、加载、注册、禁用和审计闭环。

## 48. 是否借鉴 MVP Claude Code 项目

**问题：** `mvp-claude-code-main` 展示了类 Claude Code 的 Agent Harness，包括 token 级流式输出、ToolDispatcher、子 Agent、上下文压缩、本地 File/Bash 工具和文件型 Memory。如果直接照搬，会和当前平台的 Spring Boot 多模块、Web/Gateway/Core、PostgreSQL、多租户和权限体系冲突。

**核心思路：** 把它定位为 Agent Runtime / Harness 设计参考，而不是平台架构参考。当前项目继续保持既有工程架构，只吸收能增强 Agent 执行闭环的设计思想。

**核心方案：** 新增 `考核设计/参考项目分析/06-MVPClaudeCode设计借鉴记录.md`。后续可借鉴 token 级 streaming、ToolRegistry 统一工具视图、Planner/Executor/Reviewer 工具隔离、上下文压缩和高危工具确认；不引入 LangChain4j、Picocli、本地 File/Bash 裸工具能力、Markdown 文件型 Memory 或单体 Jar 结构。阶段 3 当前不因此扩范围，主要服务阶段 4 Team 和后续 Runtime 增强。

## 49. 阶段 4 Agent Team 是否新建完整 Team 表

**问题：** 阶段 4 要实现 Planner / Executor / Reviewer，如果一开始就新增 `team_runs`、`team_tasks`、`team_messages`、`team_reviews`，会带来 Entity、Mapper、查询接口、权限、前端列表和历史管理页面，容易把高分项扩成完整工作流产品。

**核心思路：** 阶段 4 的验收重点是 Team Demo 稳定跑通和 Trace 可观察，不是做 Team 历史管理后台。数据库模型已允许 Team MVP 先把运行数据放入 `trace_spans.attributes`。

**核心方案：** 阶段 4 第一版不建 Team 表。Team 运行过程通过 `team.run`、`team.plan`、`team.task.execute`、`team.tool.call`、`team.review` 等 Trace Span 记录，关键字段放 attributes。后续如果需要 Team 历史列表、任务重放、复杂审计，再单独新增 Team 表。

## 50. 阶段 4 是否引入 Graph / Workflow 引擎

**问题：** Spring AI Alibaba、JeecgBoot 等参考项目都有 Graph / Workflow / 流程编排思想，如果阶段 4 直接引入 Graph Core、Flowable 或低代码流程设计器，会明显改变当前 Agent Runtime 运行模型。

**核心思路：** 当前项目的考核主线是自研 Agent Runtime、Gateway 治理链和 PostgreSQL Trace。阶段 4 只需要证明平台能控制 Planner -> Executor -> Reviewer 协作，不需要通用工作流引擎。

**核心方案：** 阶段 4 只借鉴 Sequential / Routing / Loop 的编排思想，用 `core.team` 自研 Orchestrator 串起 Planner、Executor、Reviewer；不引入 Graph Core、Flowable、WebFlux/Reactor 或流程设计器。前端只展示 Mermaid 协作图，不做拖拽编排。

## 51. Team 工具权限如何设计

**问题：** Planner、Executor、Reviewer 如果都直接拿 Skill/MCP 执行能力，会出现角色越界：Planner 可能执行工具，Reviewer 可能绕过审查职责调用工具。

**核心思路：** 参考 MVP Claude Code 的子 Agent 工具隔离，工具权限要物理裁剪，而不是只靠 Prompt 约束。

**核心方案：** 阶段 4 在运行时引入轻量 `AgentTool` 统一视图，整合 Skill/MCP 的名称、描述、参数 schema、来源和风险等级。Planner 只能看工具描述，Executor 只拿授权后的可调用工具，Reviewer 默认不拿业务工具。底层仍复用现有 `SkillExecutor` / `McpToolExecutor`，不直接访问 mapper。

## 52. 阶段 4 是否做真正实时 Team SSE

**问题：** 理想情况下 Team 执行每一步都应实时推送 SSE，但当前阶段 3 的 Chat 是事件级 SSE，`AgentRunResult` 在执行完成后返回。如果阶段 4 同时改成运行中事件回调，会牵动 Web/Gateway/Core 的流式协议。

**核心思路：** 阶段 4 首要目标是 Team 协作闭环，流式体验可以逐步增强。先保证事件顺序、前端展示和 Trace 一致，再考虑实时 token / 实时 task 事件。

**核心方案：** 阶段 4 第一版必须做阶段性实时 Team SSE：Team 开始时输出 `team_start`，Planner 完成后立即输出 `team_plan`，每个任务开始/结束输出 `team_task_start` / `team_task_result`，工具调用输出 `team_tool_call` / `team_tool_result`，Reviewer 完成后输出 `team_review`。暂不做 token 级流式，但不接受 Team 全部结束后一次性刷屏。实现上 Core 只调用 `TeamEventSink.emit(TeamRuntimeEventDTO)`，Gateway 提供 SSE 适配实现，Core 不感知 OutputStream / SseEmitter。

## 53. Team 是否是普通用户也能使用的 Profile 模式

**问题：** Team 容易被误解成 Developer 专属能力，或者被误解成普通用户可以不受限制地配置团队、工具和核心提示词。

**核心思路：** Team 是 Profile 的执行模式，不是新的角色权限。能否使用 Team 与能否修改公共 Profile、核心 Prompt、高危工具授权是两件事。

**核心方案：** 普通用户可以创建和使用自己的私有 BASIC / TEAM Profile；Developer/Admin 可以配置公共或领域 Profile。普通用户仍不能修改公共 Profile、核心 System Prompt、未授权高危工具和全局安全策略。后端按 owner、visibility、role 和工具绑定规则继续做校验。

## 54. BASIC 和 TEAM 如何在运行时分流

**问题：** 如果为 Team 单独新增入口或让前端每次消息都传模式，会让 Web/Gateway/Core 分流逻辑变复杂，也容易绕过现有 Profile、Trace、Token 和权限链路。

**核心思路：** 执行模式由 Profile 决定，聊天入口保持不变。

**核心方案：** Chat 请求仍然传 `profileId`；Core 读取 Profile 的 `executionMode` 后分流：`BASIC` 走现有单 Agent Runtime，`TEAM` 走 Team Runtime。`agentMode=none` 仍是纯模型透传，不进入 Team。

## 55. Team Demo 是否会限制只能回答活动策划问题

**问题：** 阶段 4 Demo 以“重庆 20 人团建活动策划”为主，如果把 fallback 写死成 weather/search/calculator 活动策划模板，Team 就会变成只服务 Demo 的硬编码功能。

**核心思路：** Demo 可以有推荐场景，但 Runtime 必须保持通用。

**核心方案：** Planner 正常路径根据用户问题和可用工具生成计划；如果两次模型输出都不是合法 JSON、计划校验失败或模型调用异常，则统一降级为单个 `MODEL_TASK`，把原始用户问题交给后续 Executor 模型任务处理。活动策划只作为种子数据、演示 Profile 或验收脚本，不写死在通用 fallback 中。

## 56. 单 Agent token 级流式和 Team token 级流式什么时候做

**问题：** 用户希望回复像真实大模型一样逐 token 输出，但阶段 4 同时引入 Team Runtime、事件流和前端进度面板，如果一起做 token 级流式，排错范围会过大。

**核心思路：** 先做 Team 事件级实时，再做 token 级实时。

**核心方案：** 阶段 4 先实现 `team_plan`、`team_task_start`、`team_tool_call`、`team_tool_result`、`team_task_result`、`team_review`、`team_final` 等阶段性 SSE。单 Agent token 级流式和 Team 最终答案 token 级流式放到 Team 闭环稳定后的独立增强中处理。

## 57. Codex Skill 与平台 Jar Skill 是否是一回事

**问题：** Codex 本地 Skill 和 agent-platform 的 Jar Skill 都叫 Skill，容易混淆为同一种能力，甚至误以为可以用平台 Jar 热加载实现 Codex 开发 Skill。

**核心思路：** 两者运行环境、触发方式和安全边界完全不同。

**核心方案：** Codex Skill 是本机开发助手能力，用于辅助写代码、设计、审查；平台 Jar Skill 是后端通过上传、校验和 ClassLoader 加载后，供 Agent Runtime 调用的业务工具。Jar Skill 第一版只作为 Agent 工具，不承载 Codex 开发技能或 Agent 角色模板。

## 58. 阶段 4 如何保留稳定回退点

**问题：** Team Runtime 会逐步接触主聊天链路，如果阶段 4 后续实现破坏现有 BASIC 模式，需要能回退到稳定版本。

**核心思路：** 用小步 commit 作为回退点，阶段完成后再打 tag。

**核心方案：** 已形成的阶段 4 回退点包括 `c786382 feat(profile): expose agent execution mode` 和 `8b5df5c feat(team): add task plan model and validation`。后续每个 Step 继续独立提交；阶段 4 整体验收通过后再打 `phase-4-complete` 标签。

## 59. TaskPlanValidator 是否需要立即抽接口

**问题：** 阶段 4 审查中提出 `TaskPlanValidator` 是具体类，不符合常见的 api 接口 + application 实现分层，建议改成 `TeamPlanValidator` 接口和默认实现。

**核心思路：** 抽象应服务真实变化点。当前 Validator 是纯计算校验组件，不依赖外部资源，不跨模块暴露，也暂时只有一个调用方向。此时抽接口会增加样板代码，收益有限。

**核心方案：** 当前不抽接口，继续直接测试具体类。等 Planner、Reviewer、Orchestrator 中至少两个地方真实复用，或者需要为模型角色单测替换校验器时，再抽 `TeamPlanValidator` 接口并迁移为默认实现。

## 60. Team Plan 校验是否需要立即新增专用错误码

**问题：** 阶段 4 审查中提出 Planner 输出非法和用户请求非法都使用 `REQUEST_INVALID`，后续可能难以区分错误来源。

**核心思路：** 专用错误码有价值，但它属于 common 全局协议，应在真实消费路径明确后再加入。Step 4 已实现 Planner JSON 解析、修正 prompt 和 fallback，当前非法输出由 Planner 内部消化，不需要暴露给外部错误分支。

**核心方案：** Step 4 继续使用 `REQUEST_INVALID`，不新增 `TEAM_PLAN_INVALID`。后续如果 Orchestrator、Gateway SSE 或前端展示需要区分 Planner 输出错误，再新增专用错误码并同步 Planner fallback、Web/Gateway 错误映射和前端测试。

## 61. AgentTool Resolver 是否应该重新查询 Skill/MCP

**问题：** AgentTool 统一工具视图可以从 Skill/MCP 服务重新查询，也可以直接从 AgentContextDTO 转换。如果重新查表，可能和 AgentContextBuilder 的授权结果不一致。

**核心思路：** AgentContextDTO 已经代表当前请求的可用上下文，里面的 availableSkills / availableMcpTools 已经过 Profile 绑定、用户选择和可用性过滤。AgentTool Resolver 不应重复承担上下文构建职责。

**核心方案：** `AgentToolResolver.resolve(AgentContextDTO context)` 直接从 `context.availableSkills()` 和 `context.availableMcpTools()` 生成 AgentTool 列表，不调用 SkillQueryService / McpToolQueryService。Planner、Validator、Executor 共用这份列表。

## 62. AgentTool 第一版是否使用命名空间和真实 riskLevel

**问题：** Skill/MCP 未来可能出现同名工具，且 Team 执行需要风险等级；但当前 SkillDTO / McpToolDTO 没有 riskLevel 来源。

**核心思路：** 先做最小稳定契约，避免为了未来增强把 Step 3 变成安全策略重构。

**核心方案：** 第一版 Skill 工具名使用 `SkillDTO.code`，MCP 工具名使用 `McpToolDTO.name`，`displayName` 使用 DTO 的 `name`，`riskLevel` 统一 `LOW`。后续出现重名再升级为 `skill:xxx` / `mcp:xxx`，接入高危工具确认或安全策略后再计算真实风险等级。

## 63. AgentTool Dispatcher 是否负责授权校验

**问题：** `AgentToolDispatcher` 可以直接按 sourceType 调 Skill/MCP 执行器，但如果它不校验授权工具列表，Planner 输出未授权工具时可能被误执行。

**核心思路：** Dispatcher 是底层分发器，不应同时承担上下文授权职责。授权边界应该在 Executor 调用 Dispatcher 前完成，因为 Executor 同时持有 TaskPlan 和 resolved AgentTool 列表。

**核心方案：** Dispatcher 只做 sourceType 分发和基础参数校验。Step 5 Executor 必须用 resolved AgentTool 列表校验 `TOOL_TASK.suggestedTool`，未授权时任务返回 failed，不调用 Dispatcher。测试必须覆盖 Planner 给出未授权工具的场景。

## 64. TeamEventSink 是否暴露 OutputStream / SSE 细节给 Core

**问题：** 如果 TeamEventSink 被设计成直接写 OutputStream，Core 层会感知 Gateway 的 HTTP/SSE 实现细节，后续单测和复用都会变重。

**核心思路：** Core 只产生领域事件，Gateway 负责协议适配。

**核心方案：** Core 定义并调用 `TeamEventSink.emit(TeamRuntimeEventDTO)`；Gateway 实现一个 SSE 适配器，把事件写成 SSE。Core 不依赖 `OutputStream`、`SseEmitter` 或 HTTP API。SSE 写出失败由 Gateway 转换为 `error` 事件并结束流。

## 65. TaskPlan 的 description 和 MODEL_TASK.suggestedTool 如何校验

**问题：** Executor 的 MODEL_TASK prompt 依赖 task description，但当前 TaskPlanValidator 只校验 name；同时 MODEL_TASK 如果携带 suggestedTool，会让模型任务和工具任务语义混淆。

**核心思路：** TaskPlan JSON 应减少歧义，让 Planner 输出更容易修正，也让 Executor prompt 构建稳定。

**核心方案：** task.description 必填；`TOOL_TASK.suggestedTool` 必填且必须存在于 AgentTool 列表；`MODEL_TASK.suggestedTool` 必须为空。后续补 TaskPlanValidator 测试与实现。

## 66. Team 纯逻辑类是否需要注册为 Spring Bean

**问题：** `DefaultTeamPlanner` 作为 Spring Bean 依赖 `TaskPlanValidator`。如果 `TaskPlanValidator` 只在单元测试里手动 new，而不是 Spring Bean，单测能过但 Web/Gateway 应用上下文会启动失败。

**核心思路：** 会被 Spring 管理类构造注入的协作者，即使是无状态纯逻辑，也应该明确注册为 Bean；不参与注入的纯函数工具才保持普通类。

**核心方案：** `TaskPlanValidator` 标注 `@Component`。阶段 4 后续 `ReviewResultValidator`、`TeamRunLimiter` 等如果要被 Runtime/Planner/Executor 构造注入，也按同样规则注册，并用应用上下文测试兜底。

## 67. Planner 模型调用的 token usage 由谁记录

**问题：** Planner 通过 `ModelInvokeService` 生成 TaskPlan 时会消耗 token。如果 `TeamPlanner.plan()` 只返回计划内容，后续 Team Runtime 无法记录 Planner 阶段的 usage，阶段 4 的 Trace/Token 配额链路会漏算。

**核心思路：** 模型角色服务不直接写配额表。Planner 不掌握完整运行时上下文和 Trace span 边界，直接记录会造成分层倒置；但它必须把模型调用结果交还给 Orchestrator。

**核心方案：** `TeamPlanner.plan()` 返回 `TeamPlanResultDTO`，其中 `modelInvocations` 保存 Planner 阶段所有已收到响应的 `ModelInvokeResult`。后续 Step 8 由 Team Orchestrator 统一创建 Trace span、记录 token usage 和 Team 事件，保证 Planner / Executor / Reviewer / Final Builder 的模型消耗都进入同一条治理链。

## 68. Executor 是否直接负责 Trace / Token 记录

**问题：** Team Executor 执行任务时会调用工具或模型，如果让 Executor 自己写 Trace span 和 token usage，它需要感知 traceId、parentSpanId、applicationId、profileId 等运行态字段，容易把角色执行逻辑和治理链记录耦合在一起。

**核心思路：** 阶段 4 的角色服务保持单一职责。Planner / Executor / Reviewer 产出结构化结果和调用元信息；Orchestrator 持有完整运行上下文，统一处理 Trace、Token、SSE 和重试。

**核心方案：** `DefaultTeamExecutor` 返回 `TeamTaskExecutionResultDTO`，包含 `ExecutionResultDTO`、`ModelInvokeResult` 和 `AgentToolDispatchResult`。后续 Step 8 Orchestrator 根据这些结果统一创建 task/tool/model span、记录 token usage，并通过 `TeamEventSink` 推送实时事件。

## 69. Reviewer 失败时 Team 是否直接失败

**问题：** 阶段 4 Reviewer 依赖模型输出结构化 JSON。如果 Reviewer 连续输出非法 JSON、校验失败或模型调用异常，直接让 Team 失败会浪费已经完成的 Planner/Executor 结果；静默通过又会隐藏质量风险。

**核心思路：** Reviewer 是质量审查层，不是安全阻断层。它失败时应显式降级，而不是阻断主链路或静默吞掉。

**核心方案：** Reviewer 正常路径输出 `ReviewResultDTO`；失败重试一次。二次失败后 fallback 为 `passed=true`，但带 `WARN` issue 和 fallback summary；answerDraft 为空时额外追加 warning。后续 Orchestrator 在最终答案中保留风险提示。

## 70. Reviewer 模型调用的 token usage 由谁记录

**问题：** Reviewer 调用 `ModelInvokeService` 会产生 token usage。如果 `TeamReviewer.review()` 只返回审查结论，Reviewer 阶段模型消耗会和 Planner 一样存在漏记风险。

**核心思路：** 保持角色服务一致性：Reviewer 不直接写 `token_usage_logs`，而是把模型调用结果交给 Orchestrator。

**核心方案：** `TeamReviewer.review()` 返回 `TeamReviewResultDTO`，其中包含 `ReviewResultDTO` 和 Reviewer 阶段所有已收到响应的 `ModelInvokeResult`。后续 Step 8 Orchestrator 统一写 Trace span、记录 token usage 和推送 Team 事件。

## 71. Agent ReAct 超限后无降级链路

**问题：** `DefaultAgentRuntimeService.runModelLoop()` 在 `MAX_AGENT_STEPS=3` 超限后直接 `throw BizException(AGENT_MAX_STEPS_EXCEEDED)`，异常一路冒泡到 Gateway → SSE error 事件 → 用户只看到错误提示，拿不到任何有用回复。同时模型输出非法工具调用格式时也会直接崩溃。

**核心思路：** ReAct 循环应把"工具尝试失败"和"完全无法回答"区分开。超限后不应放弃已收集的上下文，应做一次不带工具的纯模型降级调用。

**核心方案：**
- `runModelLoop` 循环内 `parseToolCall` 异常不崩溃，改为写入 tool error observation 并继续下一轮。
- 循环超限后调用新增的 `fallbackDirectAnswer()`：剥离 system prompt 中工具列表，追加"请基于已收集信息尽力回答"指令，做一次纯模型调用。模型调用也失败时返回中文兜底消息"抱歉，我暂时无法回答这个问题。请尝试换个方式提问，或提供更多信息。"
- 新增 `stripToolListings()` 用正则移除 system prompt 中 `Available skills:` / `Available MCP tools:` 段落。

## 72. 记忆召回完全忽略 query 参数

**问题：** `DefaultMemoryRecallService.recall()` 的 `query` 参数被完全忽略，只按 `tenantId + userId` 过滤，按 `updatedAt DESC` 排序取前 N 条。用户问"团建"和问"天气"召回的是一样的记忆，记忆与当前问题毫无关联。

**核心思路：** MVP 不引入向量库的前提下，至少做关键词匹配召回。

**核心方案：**
- `recall()` 从 query 提取关键词（按标点/空白分词，过滤长度 < 2 的短词），用 LIKE 条件过滤 `content` 字段。
- 先加大 limit 拉取候选（×10），再按关键词命中数 + 更新时间排序，截取前 N 条返回。
- 每条记忆内容截断到 300 字符，避免单条记忆撑爆上下文。

## 73. 记忆写入无去重无截断

**问题：** `DefaultMemoryWriteService.record()` 是纯 INSERT，每次 Agent 对话结束都写一条新记忆。同一主题反复对话会导致记忆表无限膨胀，且后续召回时可能拉回大量重复内容。

**核心思路：** 写入前检查是否有相似内容，有则更新而非新增；单条记忆限制长度。

**核心方案：**
- `record()` 写入前查最近 5 条同 tenant/user/app/profile/type 的记忆，用 Jaccard 词重叠相似度（阈值 0.6）判断是否重复。命中则追加到已有内容（总长不超过 4000 字符）并更新，不插入新记录。
- 新记录内容截断到 2000 字符。

## 74. System Prompt 缺少当前日期

**问题：** `DefaultAgentContextBuilder.PLATFORM_SYSTEM_PROMPT` 只写了"You are AgentX..."，没有注入当前日期。模型不知道今天是什么日子，用户问"建议哪一天团建"时模型会从训练数据里随机编一个过去日期（如 2025 年 4 月 19 日）。

**核心思路：** 每次构建上下文时注入真实当前日期。

**核心方案：** `PLATFORM_SYSTEM_PROMPT` 改为 `"Current date: %s."` 占位符，`buildSystemPrompt()` 中 `LocalDate.now()` 填充，如 `Current date: 2026-05-30`。测试 token 预算从 80→90。

## 75. 前端切换 Agent/Model 模式时清空对话记录

**问题：** 前端 `ChatPage.tsx` 模式切换时调用 `clearCurrentChatSession()`，清空 `conversationId`、消息列表和对话历史。用户从 Agent 模式切换到纯 Model 模式再切回来，之前的对话全丢。

**核心思路：** 模式切换只改运行模式，不应清空对话。清空对话应由用户主动点"新建对话"完成。

**核心方案：** 模式切换 handler 从 `clearCurrentChatSession() + 清空 messages` 改为只 `setAgentMode(value)`。消息列表和 conversationId 保留在状态中，`sessionStorage` 通过已有的 useEffect 自动同步持久化。

## 76. Team 模式 re-plan 后无输出（fallback plan task ID 冲突 + 硬编码团建答案）

**问题：** 两个关联问题：(A) `TeamFinalAnswerBuilder.fallbackAnswer()` 有一段硬编码代码，只要识别到"团建"关键词就返回固定文案，完全不调模型；(B) 当 Reviewer 要求重新规划时，若 Planner 模型调用失败触发 fallback plan，fallback 永远使用 task ID `"task-1"`，与原始计划冲突 → `executeNewTasks` 全部跳过 → 零个新任务执行 → `finalAnswer` 为空字符串 → 前端显示空消息。

**核心思路：** (A) FinalAnswerBuilder 不应硬编码任何领域答案，空结果就是空结果；(B) Orchestrator 在 re-plan 空 batch 时应做直接模型调用兜底。

**核心方案：**
- `TeamFinalAnswerBuilder.fallbackAnswer()` 删除硬编码团建分支，统一返回 `""`。
- `DefaultTeamRuntimeService` 新增 `ModelInvokeService` 依赖和 `fallbackModelAnswer()` 方法：汇总 plan + executionResults + 当前日期，做一次直接模型调用。
- re-plan 后 `newTaskBatch` 为空时跳过第二轮 review，直接走 `fallbackModelAnswer` 返回结果。
- 最终 `finalAnswer` 为空字符串时也走 `fallbackModelAnswer` 兜底。

## 77. Team fallback 模型调用未进入最终 usage 汇总

**问题：** `fallbackModelAnswer()` 调用模型后会写入 `token_usage_logs`，但 `AgentRunResult.usage()` 仍只汇总 Planner / Executor / Reviewer 的模型调用。Gateway 最终 `quotaFilter.commit(...)` 使用的是 `AgentRunResult.usage()`，会少结算 fallback 模型调用的 token。

**核心思路：** Team Runtime 里所有模型调用都必须同时进入明细记录和最终汇总 usage，不能只写明细不结算配额。

**核心方案：**
- `fallbackModelAnswer()` 返回 `FallbackModelAnswer(answer, modelInvocation)`，而不是只返回字符串。
- Team Orchestrator 收集 fallback `ModelInvokeResult`，传入 `totalUsage(...)` 一并汇总。
- fallback 调用前通过 `TeamRunLimiter.consumeModelCall()` 计入 `maxModelCalls` 上限，避免降级路径绕过 Team 整体模型调用限制。
- `DefaultTeamRuntimeServiceTest#includesFallbackModelUsageWhenFinalAnswerIsBlank` 覆盖 fallback answer、usage 汇总和 token usage 明细写入。

## 78. 内置 weather/search Skill 仍返回 mock 数据

**问题：** `DefaultSkillExecutor` 中 `weather` 返回 `"mock weather: sunny, 26C"`，`search` 返回 `https://example.com/search`，seed SQL 中 runtime_config 也标记为 `mock:weather` / `mock:search`。Agent/Team Demo 看起来能调用工具，但工具结果不是来自真实数据源。

**核心思路：** MVP 仍保持内置 Skill 简单可控，但把外部数据源换成无需 API Key 的真实公开接口；外部接口失败时返回失败结果，不伪造成功数据。

**核心方案：**
- `weather` 改为先调用 Open-Meteo Geocoding API 解析城市经纬度，再调用 Open-Meteo Forecast API 获取 current `temperature_2m/weather_code/wind_speed_10m`。
- `search` 改为调用 Wikipedia/MediaWiki OpenSearch API，解析标题、摘要和 URL；OpenSearch 返回空结果时，再调用 MediaWiki full-text search (`action=query&list=search`) 兜底。
- `DefaultSkillExecutor` 支持注入 `HttpJsonClient`，单元测试用 fake JSON 响应，不依赖外网。
- 新增 `V010__update_builtin_skill_real_handlers.sql`，把已有数据库里的 Skill 描述和 runtime_config 从 mock 更新为真实 handler。

## 79. 前端 Profile 与 Chat 页选中 Profile 状态不一致

**问题：** 用户在 Profile 页选中“生活小助手”后，切到 Agent 页或再切回 Profile 页，页面又回到原来的第一个 Profile。原因是 ProfilesPage 只在组件内存保存 selectedProfile，初始化时默认取列表第一项；ChatPage 使用自己的 sessionStorage，两个页面没有共享“当前应用下最后选中的 Profile”。

**核心思路：** Application 选择已经用 localStorage 做了跨页面偏好，Profile 也应按 applicationId 持久化最后选择；Chat 和 Profiles 都读写同一个偏好，避免页面间状态漂移。

**核心方案：**
- 新增 `profile-selection-storage`，按 applicationId 保存/读取最后选中的 profileId。
- ProfilesPage 初始化、手动选择、创建后都写入该偏好，并优先恢复对应 Profile。
- ChatPage 初始化、Profile 下拉切换、历史会话选择都同步该偏好；当外部页面已切换 Profile 时，不再强行恢复旧 session 的 Profile/会话。

## 80. Team 模式默认模型调用上限低于最大任务闭环需求

**问题：** Team 默认 `maxTasks=8`，但 `maxModelCalls=6`。一个正常多 Agent 流程至少需要 Planner 1 次 + 每个 MODEL_TASK 1 次 + Reviewer 1 次；当 Planner 把“明天 20 人团建计划”拆成 5 个以上模型任务时，会在中途触发 `Team model call count exceeds maxModelCalls`，前端表现为卡在某一步后回答不了。

**核心思路：** 上限仍然必须存在，但默认值要覆盖“允许的最大任务数”的正常闭环，再为重试、fallback 留出少量余量。

**核心方案：**
- 将 `TeamLimits.DEFAULT_MAX_MODEL_CALLS` 从 6 提高到 16。
- 新增 `TeamRunLimiterTest#defaultModelCallLimitSupportsPlannerAllModelTasksAndReviewer`，锁住 Planner + 8 个 MODEL_TASK + Reviewer 的基础闭环不应被默认上限误杀。

## 81. Team 多任务执行后缺少最终汇总导致纯规划问题回答不完整

**问题：** “我要组织团建，20人，给我计划”这类纯规划问题在 Team 模式下即使 Planner、Executor、Reviewer 都执行完成，最终答案也可能只是多个子任务结果的拼接，或者在 mock-chat 下变成提示词 Echo。根因是 `DefaultTeamRuntimeService` 的最终收口只调用 `TeamFinalAnswerBuilder` 做本地抽取，没有让 Orchestrator 基于所有执行结果再生成一份面向用户的完整答案。

**核心思路：** Planner/Executor/Reviewer 产出的是中间结构和素材，Orchestrator 必须负责最终收口；但不能在已有可用结果后再强制阻塞式调用模型，否则前端会停在最后一个 `team_task_result` 后等待。可用本地结果应立即返回；只有空答案或明显的 mock/prompt 回显才触发直接模型兜底。

**核心方案：**
- `DefaultTeamRuntimeService` 新增最终答案生成判断：仅空答案、或包含 `[mock-chat] Echo:` / `Original user request:` 等提示词回显时，调用 `fallbackModelAnswer()` 汇总生成最终回答；多任务已有可用结果时不再额外阻塞式汇总。
- 最终汇总模型调用继续纳入 `TeamRunLimiter`、token usage 明细和 `AgentRunResult.usage()` 汇总，避免绕过治理链。
- `MockModelProvider` 对团建最终汇总提示返回可用的 20 人团建计划，保证无外部模型时 Demo 也能稳定回答。
- 新增 `DefaultTeamRuntimeServiceTest#returnsUsableMultiTaskResultsWithoutBlockingFinalModelCall` 和 `DefaultModelInvokeServiceTest#mockChatReturnsUsableTeamBuildingPlanForDirectFinalAnswerPrompt` 锁住回归。

## 82. Team task3 汇总为 Markdown 列表时最终答案为空

**问题：** Team 模式下 task3 已经汇总出 Markdown 列表，但最终答案仍可能为空或继续等待 fallback 模型调用。根因是 `TeamFinalAnswerBuilder` 把任务结果里的子弹列表行也当成新的任务边界，导致 `- task-3 [SUCCESS]:` 后面的列表内容没有被纳入成功结果。

**核心思路：** 最终答案解析只应把真正的任务结果行当作边界，不能把用户可见答案里的 Markdown 列表、编号列表误判为内部结构。

**核心方案：** `TeamFinalAnswerBuilder` 使用任务结果行模式 `- <taskId> [STATUS]:` 判断边界；普通 Markdown 列表作为当前成功任务结果的续行保留。新增 `TeamFinalAnswerBuilderTest#keepsMarkdownBulletListFromSuccessfulTaskResult` 覆盖 task3 换行列表汇总场景，避免再次退回空答案/fallback 卡顿。

## 83. Jar Skill 热加载如何兼顾版本隔离和平台接口稳定

**问题：** Jar Skill 如果直接反射任意类，运行时缺少稳定调用契约；如果完全复用平台内部实现类，又会把外部 Jar 和 core 内部细节耦合在一起。Windows 下 URLClassLoader 还会持有 jar 文件句柄，影响版本替换和测试清理。

**核心思路：** 平台只暴露最小 SPI，Jar 侧实现 SPI；每个 Jar 版本用独立 URLClassLoader 加载，平台 DTO/Command/Jackson 等稳定类型仍走父加载器，业务实现类优先从 Jar 加载。加载失败不能切换 currentVersion。

**核心方案：** 新增 `JarSkillHandler` SPI 和 `JarSkillLoader`。注册成功时持久化 skill/version/artifact 后加载 handler 并注册到 `JarSkillRuntimeRegistry`；加载失败时将 skill/version 标记为 FAILED，不更新 currentVersion。Registry 支持替换时关闭旧 ClassLoader，并在 Spring 销毁时统一释放句柄。

## 84. 单 Agent 工具循环是否允许模型请求未授权工具后继续重试

**问题：** 用户只在前端选择了部分工具，或当前上下文没有任何可用工具时，模型仍可能输出 `@skill:` / `@mcp:` 工具调用。旧逻辑会把未授权工具写成 tool observation，再让模型继续 ReAct 重试，前端也会看到一次 action/observation，容易误解为平台尝试执行了该工具。

**核心思路：** 工具授权边界必须以本轮 `selected / authorized tools` 为准。模型请求不在授权列表中的工具时，不应进入 ToolExecutionPlanner / Dispatcher，也不应向前端暴露成工具执行事件；应停止工具循环，做一次不带工具的最终总结或降级回答。

**核心方案：**
- `DefaultAgentRuntimeService` 在执行 batch 前先统一调用 `ToolCallValidator` 预校验所有工具调用。
- 任一工具未授权或参数非法时，只把校验 observation 放入内部 fallback 上下文，不记录 `AgentToolEventDTO`，不调用 `ToolExecutionPlanner` / `AgentToolDispatcher`。
- 随后调用 `fallbackDirectAnswer()`，剥离工具列表并提示模型禁止继续输出 `@skill:` / `@mcp:`，基于已知信息返回用户可读答案。
- 主链路图明确两条分支：无工具调用直接退出 ReAct Loop；未授权工具停止工具循环并 fallback。SubAgent 仍标注为阶段 4 Team 扩展，不属于当前单 Agent 已完成链路。

## 85. 高危工具确认是否依赖模型重新生成工具调用

**问题：** 旧的高危工具确认流程只把 `confirmedToolKeys` 带回后端，并用一条“Confirm skill:deploy and continue...”消息重新触发 Agent。这样会依赖模型第二次再生成同一个工具调用，存在工具不再输出、参数变化或输出不稳定的问题。

**核心思路：** 高危确认应确认“第一次已经确定的工具调用”，而不是确认一个工具名后让模型重新规划。确认事件必须携带原始 pending tool call 的 sourceType、toolName 和 arguments 快照；用户确认后 Runtime 直接恢复执行这份快照。

**核心方案：**
- 新增 `PendingToolCallCommand`，扩展 `AgentRunCommand` / Gateway / Web / Frontend 请求字段，支持 `pendingToolCall` 透传。
- `tool_confirm_required` 事件的 metadata 中加入 `toolKey` 和 `pendingToolCall`，前端确认按钮读取该快照。
- 用户确认后前端回传 `confirmedToolKeys + pendingToolCall`，不再伪造一条普通用户确认消息让模型重新生成工具调用。
- `DefaultAgentRuntimeService` 在进入普通 ReAct loop 前检测 confirmed pending tool call：校验确认 key 后直接执行工具，把 observation 写回 messages，再调用模型生成最终回答。
- 主链路图中 Step 9 更新为“pending tool call 原地恢复已完成”。

## 86. 单 Agent 上下文预算阈值是否继续固定写死

**问题：** Step 4 初版把 `MEMORY_TOKEN_BUDGET=300`、`EXPERIENCE_TOKEN_BUDGET=500`、`AUTO_COMPACT_MAX_CHARS=360` 等阈值写死在 `DefaultAgentContextBuilder`。这会导致 4K、16K、128K 不同上下文窗口的模型使用同一套局部预算：小模型可能被记忆/经验撑爆，大模型又无法利用更大的上下文。

**核心思路：** Context Builder 已能从 `command.maxContextTokens` 或 `model_configs.max_context_tokens` 解析模型窗口，memory、experience 和 auto compact 摘要阈值应由该窗口派生，而不是固定常量。Runtime 里的 `DefaultMicroCompactService` 仍属于每轮 observation 防爆裁剪，先保留固定安全阈值，后续再独立扩展为可配置策略。

**核心方案：**
- 在 `DefaultAgentContextBuilder` 内新增 `ContextBudgetPolicy.from(maxContextTokens)`，按模型窗口计算局部预算：memory 约 8%、experience 约 10%，并设置上下限，避免极小窗口过度注入或超大窗口无限膨胀。
- Auto Compact 旧历史摘要的最大字符数和单条 sample 字符数同样由 `maxContextTokens` 派生，保留 `fallback trim` 兜底。
- 记忆块在预算装不下任何记忆时不再输出空的 `Long-term memories:` 标题，避免无效 prompt 噪声。
- 新增 `DefaultAgentContextBuilderTest` 覆盖小窗口收紧、大窗口放宽和大窗口 auto compact sample 扩展三类回归。

## 87. AGI-saber 记忆系统和 RAG 是否原样搬入 agent-platform

**问题：** AGI-saber 的记忆系统和 RAG 能力较完整，但 Java 版存在内存 List 记忆、Milvus/ES/Neo4j 组合依赖、`InfrastructureService` 大杂烩和缺少 tenant/application/user/profile 隔离等问题。如果原样搬入，会破坏当前 `core/web/gateway` 模块边界和多租户平台约束。

**核心思路：** 借鉴 AGI-saber 的能力切片，而不是复制实现。Memory 和 RAG 可以共用 embedding、召回过滤、预算和 Context Slot，但数据归属必须分开：Memory 归用户长期状态，RAG 归知识文档和 chunk。

**核心方案：**
- Step 13 先补 Trace Span 可观测性，为后续 memory/rag 调试打底。
- Memory v2 平台化落地 AGI-saber 的 `category/tags/importance/slotHint/RecallFilter/ConsolidationConfig/Preference` 思路，但使用 PostgreSQL 持久化和 tenant/application/user/profile 隔离。
- RAG v1 使用 `PostgreSQL 文档元数据 + VectorStore 接口 + Qdrant 默认实现`，保留 Milvus/ES/Neo4j/RRF 为后续增强。
- 不引入 AGI-saber 的内存 List 长期记忆、不照搬乱码 Preference 规则、不引入 `InfrastructureService` 式跨基础设施大服务。

## 88. Memory v2 第一层是否直接接向量库

**问题：** Step 12 既包含记忆升级，也包含 RAG / VectorStore。如果第一步就接 embedding 和 Qdrant，会同时改写记忆表、召回 API、上下文注入、外部服务和部署依赖，风险过大，也不利于确认长期记忆本身的多租户隔离和过滤策略是否正确。

**核心思路：** 先把记忆数据模型平台化，再把召回通道语义化。Memory v2 第一层应继续使用 PostgreSQL，先落地 category、tags、importance、slotHint 和 RecallFilter；RAG 和语义召回后续通过 VectorStore 接口独立接入。

**核心方案：**
- `memories` 新增 `memory_category`、`tags`、`importance`、`slot_hint`、`metadata` 字段，旧 `memory_type/content/keywords` 保留兼容。
- 新增 `MemoryRecallFilter`，支持 category、requireTags、minScore、topK、maxAge；召回仍先走关键词 LIKE 粗筛，再按 filter、关键词命中和 importance 排序。
- `DefaultAgentContextBuilder` 调用带 filter 的召回接口，只注入 `summary/preference/fact` 类长期记忆，仍受原有 token budget 控制。
- 下一步再补 Preference 提取、ConsolidationConfig、lastAccessedAt/衰减策略；RAG 的知识文档、chunk、VectorStore/Qdrant 不和用户长期记忆表混在一起。

## 89. Memory v2、Schema Context 和 RAG 是否应该合成一个大 Step

**问题：** 记忆系统和 RAG 都会影响上下文注入，如果在同一个 Step 里同时做偏好提取、记忆合并、Context Slot、知识文档、chunk、embedding、Qdrant 和 Trace，改动会横跨 `core.memory`、`core.context`、`core.rag`、`core.model`、前端和数据库，风险过大，也不利于定位“记忆召回不准”和“RAG 检索不准”的问题。

**核心思路：** 把能力拆成可独立验收的阶段：先把用户长期记忆稳定化，再把 Context 装配改成 Slot，再接 RAG MVP，最后接 Qdrant 和混合检索增强。Memory 解决用户长期状态，RAG 解决外部知识依据，两者只在 `core.context` 的 Slot 注入层汇合。

**核心方案：**
- 新增 `实际开发/优化阶段/单Agent优化/单Agent记忆与RAG设计方案.md`，明确 Memory/RAG 的数据边界、包边界、Trace Span 和五段落地顺序。
- Stage 1 做 Memory v2 稳定化：`last_accessed_at/access_count/confidence/expires_at`、`MemoryConsolidationConfig`、规则版 `PreferenceExtractor`、`memory.recall/write/consolidate` Span。
- Stage 2 做 Schema-driven Context：`ContextSlotKind`、`ContextSlot`、`ContextSlotFilter`、`ContextSchema`、`ContextSlotSource`，让 memory/experience/tools/rag 都受统一预算控制。
- Stage 3 做 RAG MVP：`knowledge_documents`、`knowledge_chunks`、`TextSplitter`、`EmbeddingService`、`VectorStore`、`MockVectorStore`、PostgreSQL keyword fallback、`RagEngine` 和 `RagSlotSource`。
- Stage 4 再接 Qdrant，Stage 5 再考虑 BM25/RRF/Neo4j/GraphMemory，不让外部复杂依赖阻塞单 Agent 主链路。
## 90. RAG MVP 第一版是否直接建表并接 Qdrant

**问题：** RAG Stage 3 需要体现知识库 ingest/search/delete、chunk 切分和 Context Slot 注入，但如果第一版就同时新增 `knowledge_documents/knowledge_chunks` 表、Flyway、Mapper、Qdrant collection、embedding provider 和部署配置，会把可演示主链路绑定到外部服务和数据库迁移上，排障范围过大。

**核心思路：** 先把 RAG 的业务边界和运行链路跑通，再替换存储和向量召回实现。`core.context` 只依赖 `RagSearchService`，不关心底层是 noop、内存、PostgreSQL 还是 Qdrant；`core.rag` 内部先用本地内存引擎证明 `TextSplitter -> ingest -> search -> RagSlotSource` 能闭环。

**核心方案：**
- Stage 3 第一段已接入 `RagSlotSource` 和 `rag.search` span，RAG 结果只注入 `apiMessages` 的 system prompt，不写入 `conversation_messages`。
- Stage 3 第二段先新增本地 `RagEngine`：`TextSplitter` 支持 Markdown 标题/段落优先切分和固定窗口 overlap 兜底，`DefaultLocalRagEngine` 支持内存版 `ingest/search/delete`、docHash 去重替换和 tenant/application/user/profile 边界过滤。
- 默认 Spring Bean 使用 `DefaultLocalRagEngine` 实现 `RagSearchService`，保证无 Qdrant、无 embedding API、无新表时主链路仍可演示；后续接 PostgreSQL/Qdrant 时替换 `RagEngine` 存储和召回实现，Context Builder 不需要改。
- 当前内存版 RAG 只作为 MVP 降级和测试骨架，不作为正式持久化知识库；正式知识文档表、chunk 表、VectorStore/Qdrant 和 embedding provider 放到下一小步继续落地。

## 91. RAG 正式持久化骨架是否直接绑定 Qdrant

**问题：** RAG 内存版已经能演示 `ingest/search/delete`，但正式知识库必须能落库。如果在落库骨架阶段就同时接 Qdrant、embedding 模型和 Docker 配置，排障范围会变成“数据库 + 向量库 + 模型接口 + 上下文注入”一起变化。

**核心思路：** PostgreSQL 先成为知识文档和 chunk 原文的唯一业务真相，Qdrant 只作为后续向量索引层。`core.rag` 内部优先使用 PG-backed `RagEngine`，没有 DataSource/Mapper 时仍降级到本地内存引擎；`core.context` 继续只依赖 `RagSearchService`。

**核心方案：**
- 新增 `knowledge_documents` 和 `knowledge_chunks`，保留 tenant/application/user/profile 隔离、docHash 去重、chunk 原文、metadata jsonb 和 `vector_id` 预留字段。
- 新增 `KnowledgeDocumentEntity` / `KnowledgeChunkEntity` / Mapper，并把 jsonb 字段纳入 `JsonNodeMappingTest`，把 `core.rag.mapper` 纳入 MyBatis MapperScan 和架构边界检查。
- 新增 `DefaultPostgresRagEngine`：ingest 时按 scope+docHash 替换旧 chunk，search 先走 PostgreSQL keyword fallback，delete 做软删除；没有 Mapper Bean 时仍使用 `DefaultLocalRagEngine` 降级。
- Qdrant、EmbeddingService、ANN 语义召回、HyDE 和 RRF 暂不在本小步实现，下一步在 PG 业务真相稳定后通过 VectorStore 接口接入。

## 92. RAG VectorStore 抽象是否直接接真实 Qdrant

**问题：** PG RAG 已经有正式文档/chunk 持久化，但如果下一步直接接 Qdrant 和真实 embedding provider，会把外部服务、模型配置、向量维度、collection 初始化和 RAG 回表链路同时引入，仍然不利于定位问题。

**核心思路：** 先把 VectorStore 的 Java 接口、默认 Mock 实现和 PG 回表链路打通。这样 Stage 3 可以本地演示“向量召回骨架”，Stage 4 再把 `MockVectorStore` 替换为 `QdrantVectorStore`，不改 `core.context` 和 `RagSlotSource`。

**核心方案：**
- 新增 `EmbeddingService` / `VectorStore` 接口和 `EmbeddingVectorDTO`、`VectorStoreDocumentDTO`、`VectorSearchQueryDTO`、`VectorSearchResultDTO`。
- 默认注册 `MockEmbeddingService` 和 `MockVectorStore`，不依赖外网、Docker 或真实 embedding API。
- `DefaultPostgresRagEngine` 支持可选向量路径：ingest 写入 VectorStore，search 优先向量召回并按 chunkId 回表，向量不可用或无结果时回退 PostgreSQL keyword fallback。
- Qdrant、真实 embedding provider、HyDE、RRF 和 citations 仍放到后续小步，不让外部依赖阻塞单 Agent 主链路。

## 93. Qdrant 接入第一段是否要求真实服务可用

**问题：** Stage 4 需要接 Qdrant 语义召回，但如果第一段就要求 Docker/Qdrant、collection 初始化和真实 embedding provider 同时可用，单元测试会依赖外部服务，主链路也可能因为向量库不可用而中断。

**核心思路：** 先把 Qdrant 适配器做成可配置、可测试、可降级的骨架。默认仍使用 `MockVectorStore`，只有显式配置 `agent.rag.vector-store=qdrant` 且 `agent.rag.qdrant.enabled=true` 时才启用真实 Qdrant HTTP 调用；Qdrant 调用失败时不向上抛出，交给 PG-backed RAG 的 keyword fallback 兜底。

**核心方案：**
- 新增 `QdrantVectorStoreProperties` 和 `QdrantVectorStore`，使用 JDK `HttpClient` 调 Qdrant REST API，不引入额外 SDK。
- `upsert` 写入 vector point 和 payload：Qdrant point id 使用由业务 `vectorId` 派生的稳定 UUID，业务层 `rag-chunk-*` / `memory-*` 保存在 `payload.vector_id`，并继续写入 `tenant_id/application_id/owner_user_id/profile_id/document_id/chunk_id/embedding_model`。不能直接用普通字符串业务 ID 作为 Qdrant point id，否则真实 Qdrant 会返回 400。
- `search` 使用 query vector + payload filter 检索，并把 Qdrant 返回的 `payload.vector_id` 映射为 `VectorSearchResultDTO.vectorId`，后续仍由 PostgreSQL 回表读取 chunk 原文。
- `deleteByDocument` 使用同 scope + document filter 删除向量索引；删除失败不影响 PostgreSQL 文档状态。
- 新增 `QdrantVectorStoreTest` 和配置测试，验证请求 JSON、过滤条件、默认 Mock、显式启用 Qdrant 和失败降级；collection 初始化、Docker Compose 和真实 embedding provider 放到下一小段。

## 94. Qdrant collection 初始化是否放在应用启动阶段

**问题：** Qdrant 接入后必须保证 collection 存在，但如果在 Spring Bean 创建或应用启动阶段强制初始化，开发环境没有启动 Qdrant 时后端会启动失败，违背“外部依赖可降级、不阻塞主链路”的约束。另一方面，如果每次请求都盲目 `PUT /collections/{name}`，已有 collection 时也可能因为重复创建或维度不一致导致误判失败。

**核心思路：** collection 初始化应由 `QdrantVectorStore` 在第一次真实操作前懒执行，并且先检查是否存在，不存在再创建。初始化失败只影响本次向量操作，不能影响 PostgreSQL 知识库业务真相和 keyword fallback。

**核心方案：**
- `QdrantVectorStore` 新增 collection ready 状态，第一次 `upsert/search/deleteByDocument` 前执行初始化，成功后同一实例不重复初始化。
- 初始化流程先调用 `/collections/{collectionName}/exists`，返回不存在时再 `PUT /collections/{collectionName}`，创建 body 使用 `vectors.size=dimension` 和 `vectors.distance=distance`。
- 初始化失败时本次 `upsert` no-op、`search` 返回空列表、`deleteByDocument` 返回 0，由 `DefaultPostgresRagEngine` 继续走 PostgreSQL keyword fallback。
- `docker-compose.dev.yml` 新增 Qdrant 开发容器和持久化卷，但默认 Spring 配置仍保持 Mock VectorStore；只有显式配置 Qdrant 时才走真实向量库。

## 95. RAG embedding provider 是否直接绑定 OpenAI 官方服务

**问题：** RAG 进入 Qdrant 后需要真实 embedding 向量，但如果直接绑定 OpenAI 官方服务、真实 API Key 和固定模型，会让本地测试依赖外网和账号，也会把后续接本地 bge / Qwen embedding 的路线封死。

**核心思路：** embedding provider 先做 OpenAI-compatible HTTP 骨架，而不是只做 OpenAI 专用实现。默认仍使用 `MockEmbeddingService`，只有显式配置 `agent.rag.embedding-provider=openai-compatible` 且 `agent.rag.embedding.enabled=true` 时才调用真实 provider；失败时返回空向量，让 RAG 继续走 PostgreSQL keyword fallback。

**核心方案：**
- 新增 `OpenAiCompatibleEmbeddingServiceProperties` 和 `OpenAiCompatibleEmbeddingService`，请求 OpenAI-compatible `/embeddings` endpoint。
- 请求 body 使用 `model`、`input`、`encoding_format=float`，可选 `dimensions`，响应解析 `data[0].embedding` 为 `EmbeddingVectorDTO`。
- 空输入、API Key 缺失、HTTP 失败或响应异常都返回空向量，不向上抛出阻塞主链路。
- `RagSearchServiceConfiguration` 增加显式配置切换；默认仍注册 `MockEmbeddingService`。
- 新增 `OpenAiCompatibleEmbeddingServiceTest` 和配置测试，覆盖请求映射、响应解析、失败降级和默认 Mock 兜底。

## 96. RAG 只记录 rag.search 是否足够支撑工作台排障

**问题：** `RagSlotSource` 已能记录 `rag.search` span，但真正影响召回质量和延迟的步骤在 `DefaultPostgresRagEngine` 内部：query embedding、VectorStore / Qdrant search、PG 回表和 keyword fallback。如果工作台只看到一个粗粒度 `rag.search`，很难判断是 embedding 失败、向量库无结果、Qdrant 不可用，还是 PG 回表过滤后为空。

**核心思路：** 保留 `core.context` 层的 `rag.search` 作为 RAG Slot 总 span，同时让 `core.rag` 在拿到 trace 上下文时记录内部子 span。这样 Context Builder 仍只依赖 `RagSearchService`，不会跨包访问 RAG 内部实现；Trace 写入失败仍然不能影响 RAG 召回和 PG keyword fallback。

**核心方案：**
- `RagSearchService` 增加带 `traceId/parentSpanId` 的默认重载，旧调用方继续使用原 search 签名。
- `RagSlotSource` 创建 `rag.search` span 后，把该 span id 作为 parent 传给 RAG 引擎，保证时间线层级清晰。
- `DefaultPostgresRagEngine` 可选注入 `TraceService`，在向量召回路径记录 `rag.embedding` 和 `rag.vector.search` 子 span。
- `rag.embedding` attributes 只记录 `queryChars/model/dimension`，不保存完整用户问题；`rag.vector.search` 只记录 `topK/dimension/profileScoped/vectorStore/resultCount`。
- Trace 失败被吞掉，embedding/vector 失败仍按现有策略回退 PostgreSQL keyword fallback，不阻塞单 Agent 主链路。

## 97. RAG 知识库手工联调接口是否应该走 Gateway

**问题：** RAG 需要提供文档录入、手工检索和删除接口，方便验证知识库、chunk、embedding、Qdrant 和 PostgreSQL fallback。但这些接口不是 AI 执行请求，如果也走 Gateway，会把管理型接口强行接入 AI 治理链，增加鉴权、SSE、Quota 和 Trace 转换复杂度。

**核心思路：** 按请求类型分流，而不是按功能名称分流。AI 对话、OpenAI-compatible 调用和 Agent Runtime 执行请求继续走 Gateway；知识库管理、配置和查询类请求走 Web -> Core。RAG 在 AI 对话中的注入仍由 Core Context Builder 调用 `RagSearchService` 完成。

**核心方案：**
- Web 新增 `RagKnowledgeController`，提供 `POST /api/rag/documents`、`GET /api/rag/search`、`DELETE /api/rag/documents/{documentId}` 三个最小联调接口。
- Controller 只注入 `RagEngine`，不访问 `core.rag.mapper/entity`，保持 Web -> Core API 边界。
- Web 从当前 JWT 用户注入 `tenantId/userId`，请求侧只传 `applicationId/profileId` 等 scope 参数，避免客户端伪造租户和用户归属。
- `GET /api/rag/search` 对 `topK` 做上限裁剪，避免手工测试接口一次拉取过多 chunk。
- Gateway 不新增 RAG 管理入口；后续前端知识库页面也调用 Web 管理接口，AI 对话里的 RAG 召回仍通过 Runtime 的 Context Slot 进入模型上下文。

## 98. RAG/Memory 外部反馈如何判定优先级

**问题：** 外部反馈指出 `memoryStrategy` 未生效、Docker Compose 缺 Qdrant、记忆语义检索未接 Qdrant、Planner 未输出 `needsRag/ragQuery`，并把 Gateway 测试失败归因到 RAG 条件表达式缺默认配置。如果不逐条验证，容易把已完成项重复实现，或把测试失败修错方向。

**核心思路：** 外部反馈只作为待验证线索。先对照代码、配置和测试报告确认事实，再按主链路风险排序：能影响单 Agent 运行语义的策略开关优先；已完成或仅属后续增强的项不抢占当前稳定化工作；测试失败必须修根因而不是修表象。

**核心方案：**
- `memoryStrategy` 属于真实缺口：`DISABLED/SESSION_ONLY` 不应读取或写入长期记忆，`READ_ONLY` 只读取不写入，`READ_WRITE` 和空配置保持默认读写。
- Qdrant Docker Compose 已完成：`docker-compose.dev.yml` 已包含 `qdrant/qdrant:v1.12.6`、`6333/6334` 端口和独立持久化卷，不再作为缺口重复实现。
- Gateway 测试失败根因不是 `RagSearchServiceConfiguration` 的 properties。RAG 条件表达式已有默认值并正常判 false；真实失败原因是 Gateway contextLoad 测试扫描到 core 的 `DefaultMemoryConsolidationService`，但无 `MemoryMapper` Bean，需要在 Gateway 测试中 mock 该服务保持入口层测试隔离。
- 记忆语义检索和 Planner `needsRag/ragQuery` 仍是后续增强：前者等 Memory v2 策略稳定后接 VectorStore/Qdrant；后者等 Planner/路由设计定稿后再做，避免把 Team/Planner 复杂度提前拉回单 Agent 主链路。

## 99. 长期记忆语义召回是否和 RAG 共用 VectorStore

**问题：** 长期记忆和 RAG 都需要 embedding 与向量召回。如果各自实现一套向量适配器，会重复维护 Qdrant、Mock、OpenAI-compatible embedding、Trace 和降级逻辑；但如果完全混在一起，又可能导致知识库 chunk 被当作用户记忆召回，或用户长期记忆被当作文档依据注入。

**核心思路：** 可以共用底层向量基础设施，但不能共用业务事实源，也不能混用召回域。Memory 的事实源仍是 `memories`，RAG 的事实源仍是 `knowledge_documents / knowledge_chunks`；VectorStore 只做可替换索引层，通过 `source_type` 区分 `memory` 与 `rag`。

**核心方案：**
- `VectorStoreDocumentDTO` 和 `VectorSearchQueryDTO` 增加 `sourceType`，旧构造默认 `rag`，保持 RAG 兼容。
- 长期记忆写入时可选生成 embedding，并以 `source_type=memory`、`vectorId=memory-{memoryId}` 写入 VectorStore；失败时吞掉异常，PostgreSQL 记忆写入仍成功。
- 长期记忆召回时优先 `EmbeddingService.embed(query)` + `VectorStore.search(source_type=memory)`，再按 id 回表读取 `memories`，继续执行 tenant/application/user/profile、category/tag/maxAge/minScore 过滤；无结果或失败时回退 PostgreSQL keyword/LIKE。
- Qdrant payload 和 search filter 都带 `source_type`，MockVectorStore 也按 `sourceType` 过滤，避免 RAG 与 Memory 交叉召回。
- Trace 中在 `memory.recall` 下记录 `memory.embedding` 和 `memory.vector.search` 子 span；Trace 写入失败、embedding 失败或 Qdrant 失败都不能中断单 Agent 主链路。
- `VectorStore.deleteByDocument` 已扩展为 source-aware 删除：旧签名默认删除 `source_type=rag`，长期记忆合并或过期删除时调用 `deleteByDocument("memory", ...)` 清理对应 memory 向量索引；删除失败被吞掉，PostgreSQL 记忆状态仍是事实源。
## 100. 长期记忆手动删除是物理删除还是软禁用

**问题：** 用户需要在工作台手动管理长期记忆，包括查看、改写、调整重要性和删除/禁用。若直接物理删除 `memories`，审计链路和误删恢复能力会变弱；若只改状态但不清理向量索引，后续语义召回仍可能命中已禁用记忆。

**核心思路：** PostgreSQL 的 `memories` 仍是长期记忆事实源，用户手动删除优先采用软禁用，保留审计痕迹；Qdrant/MockVectorStore 只是派生索引，禁用时同步清理 `source_type=memory` 的向量索引。向量清理失败不能阻塞主库状态变化。

**核心方案：** 新增 `MemoryManagementService`，提供当前用户作用域内的 `list/update/disable`；Web 新增 `GET /api/memories`、`PATCH /api/memories/{memoryId}`、`DELETE /api/memories/{memoryId}`，只从 JWT 注入 `tenantId/userId`，前端只能传 `applicationId/profileId` 等业务 scope。`disable` 将记忆状态改为 `DISABLED`，并调用 `deleteByDocument("memory", ...)` 清理向量索引；向量清理失败被吞掉，PostgreSQL 状态仍然生效。

## 101. SSE 流已开始后全局异常处理是否还能返回 JSON

**问题：** 浏览器对话和 API Key 流式调用使用 `text/event-stream`。当 Gateway/Core 运行中异常或 Spring MVC 异步超时发生在流响应已经开始之后，全局异常处理器如果继续返回 `ApiResponse` JSON，会在已设置的 `text/event-stream` Content-Type 下触发 `HttpMessageNotWritableException`，导致二次异常，前端看到空输出、半截回复或大量异常日志。

**核心思路：** 流开始前的鉴权、敏感词、配额等失败仍应返回标准 JSON 错误；流开始后的运行期错误应由 SSE 包装器写 `error` 事件或返回空 body 的 SSE 响应，不能再让全局异常处理器尝试写普通 JSON。Web 和 Gateway 两个入口层都要遵守同一规则，Core 不感知 HTTP/SSE 细节。

**核心方案：** Web `GlobalExceptionHandler` 和 Gateway `GatewayExceptionHandler` 根据 `HttpServletResponse` 判断响应是否已提交或 Content-Type 是否为 `text/event-stream`。普通请求继续返回 JSON `ApiResponse`；SSE/已提交响应只返回 `text/event-stream` 空 body，避免 JSON converter 二次失败。Gateway 同步补充 MVC async timeout 配置为 300 秒，与 Web 保持一致，降低模型慢响应时被容器默认超时截断的概率。新增 Handler 回归测试覆盖普通 JSON、SSE 已开始和 `AsyncRequestTimeoutException` 场景。

## 102. 空工具列表是否代表禁用所有工具还是使用默认工具

**问题：** 单 Agent 对话中前端会传 `enabledSkillIds` / `enabledMcpToolIds` 控制本轮允许使用的工具。若用户显式传空数组，Runtime 仍把空列表当作“不筛选”并回退到 Profile 默认工具，模型可能调用用户本轮没有授权的 weather / calculator / MCP 工具，破坏“循环中只能分析并使用用户输入工具列表”的规则。

**核心思路：** `null` 和空列表必须表达不同语义：`null` 表示请求方没有指定工具选择，可以使用 Profile 默认启用绑定；空列表表示请求方显式禁用本轮全部工具，Context Builder、SkillRegistry、McpToolRegistry 和 Web/Gateway DTO 传输链路都必须保留这个区别。

**核心方案：** `DefaultSkillQueryService.listAvailableSkills` 和 `DefaultMcpToolQueryService.listAvailableTools` 对非 `null` 的空 ID 列表直接返回空集合，不再查询所有可用工具；`null` 仍走默认可用工具查询。新增 Core 回归测试覆盖显式空列表不会注入工具 prompt，也不会生成 `availableSkills/availableMcpTools`；新增 Web/Gateway 控制器测试覆盖 JSON `[]` 能一路传递为 `AgentRunCommand.selectedSkillIds()/selectedMcpToolIds()` 空列表。真实 SSE 冒烟要求空工具请求只返回 `message/done`，不得出现 `event: action` 或未授权工具名。

## 103. 工具调用场景下 SSE message_delta 是否可以早于 action/observation

**问题：** 单 Agent 工具调用后，Core 的最终回答 token 会通过 `ModelStreamCallback` 回调到 Gateway；但工具事件仍在 `AgentRunResult.toolEvents()` 中随 `run()` 返回后统一输出。如果 Gateway 直接在 callback 中写 `message_delta`，前端时间线会先看到最终回答，再看到 `action/observation/tool_confirm_required`，表现为“还没查工具就已经回答”，破坏工作台可解释性。

**核心思路：** Core 不应感知 HTTP/SSE 输出细节，工具事件与最终回答 delta 的排序应由 Gateway 的 SSE 适配层负责。对于单 Agent 工具链路，用户更需要先看到工具过程，再看到基于工具结果生成的最终回答；否则 token 级流式虽然更快，但语义顺序是错的。

**核心方案：** `InternalAiController.runChatStream` 在 Agent Runtime 执行期间只缓存 `message_delta` token，不立即写出；`run()` 返回后按顺序输出 `toolEvents`，再刷出缓存的 `message_delta`，最后输出完整 `message` 和 `done`。新增 Gateway 回归测试模拟 Runtime 先触发 token callback、返回时带工具事件，断言 `action/observation` 必须早于 `message_delta`，且 `message_delta` 仍早于最终 `message`。

## 104. 工作台 Trace Span 是否只记录名称和耗时就足够

**问题：** 单 Agent 主链路已有 `context.build`、`model.invoke`、`tool.execute`、`memory.write`、`rag.search` 等 span，但如果 span attributes 只包含少量计数，工作台排障时仍难判断模型用了哪个 provider/model、token 消耗来自哪里、工具入参是什么、工具返回是否成功。只看 span 名称和 latency 不足以支撑考核要求中的全链路 Trace 追踪。

**核心思路：** Trace attributes 应记录排障必需的摘要信息，但不能把完整用户输入、完整工具结果或敏感内容无脑写入 Trace。模型 span 记录 provider/model/token/stream/toolCallCount；工具 span 记录 toolType/toolName/risk/readOnly/resourceKeys、参数摘要和结果摘要；Context/Memory/RAG 继续记录预算、召回数量和降级路径。

**核心方案：** `DefaultAgentRuntimeService` 在 `model.invoke` 完成时把 `providerId/providerType/modelName/promptTokens/completionTokens/totalTokens/estimatedTokens/toolCallCount/assistantChars` 写回 span attributes；在 `tool.execute` 开始时写 `argumentsSummary`，结束时写 `resultSummary/resultChars/success`。新增单 Agent Runtime 回归测试，断言 model/tool span 的可观测属性完整写入，同时保持 Trace 写入失败不影响主链路。

## 105. memoryStrategy 四种模式如何同时约束长期记忆读写

**问题：** 单 Agent 长期记忆既会在 Context Builder 阶段被召回注入，也会在 Runtime 结束阶段写入摘要和偏好。如果只在其中一处判断 `memoryStrategy`，会出现 `READ_ONLY` 仍写入、`SESSION_ONLY` 仍读取历史长期记忆，或 `DISABLED` 只禁写不禁读的问题。

**核心思路：** 长期记忆读写需要分层控制：Context Builder 只负责“是否读取并注入长期记忆”，Runtime 只负责“是否把本轮结果持久化为长期记忆”。短期上下文窗口始终保留，不受长期记忆策略影响。

**核心方案：** `DISABLED` 和 `SESSION_ONLY` 在 Context Builder 中跳过 `MemoryRecallService`，不注入长期记忆；`READ_ONLY` 和 `READ_WRITE` 允许召回。Runtime 只有 `READ_WRITE` 写入长期记忆，`READ_ONLY`、`DISABLED`、`SESSION_ONLY` 都跳过 `MemoryWriteService`。新增 Context/Runtime 回归测试覆盖四种模式，避免后续 RAG、前端配置或记忆管理接口改动破坏读写边界。

## 106. RAG Trace 是否只覆盖检索不覆盖入库

**问题：** RAG 检索阶段已经有 `rag.search`、`rag.embedding`、`rag.vector.search` span，但知识库文档入库阶段如果没有 Trace，工作台无法看到文档是否完成切割、入库了多少 chunk、是否写入向量索引，也无法判断后续检索为空是“没入库成功”还是“召回没命中”。

**核心思路：** RAG 入库属于 Core 内部知识库能力，应提供可选 trace 上下文重载，由调用方决定是否挂到某个 Trace root 下。Web 管理接口暂时仍保持简单调用，不强行经过 Gateway；后续如果工作台知识库操作也要可追踪，可由 Web 创建管理型 Trace root 后调用该重载。

**核心方案：** `RagEngine` 增加 `ingest(command, traceId, parentSpanId)` 默认重载；`DefaultPostgresRagEngine` 在带 trace 上下文时记录 `rag.ingest` span，attributes 只保存摘要字段：`titleChars/sourceType/sourceUriPresent/contentChars/chunkTokenBudget/overlapTokens/profileScoped/embeddingService/vectorStore/documentId/chunkCount/vectorIndexedCount/status`。Trace 写入失败不影响文档入库，向量索引失败仍按现有策略降级为 PostgreSQL keyword fallback。

## 107. 工作台 Trace 如何可读展示 Memory/RAG 细粒度 Span

**问题：** 后端已经补充了 `memory.embedding`、`memory.vector.search`、`rag.ingest`、`rag.embedding`、`rag.vector.search` 等细粒度 span，但前端如果只直接展示原始 spanName 和散乱 attributes，工作台仍难快速判断是入库、向量生成、向量检索还是召回结果问题。

**核心思路：** 后端 Trace attributes 负责保存稳定、脱敏的排障摘要；前端工作台负责把这些机器字段集中映射成人能读懂的标题和事实摘要。映射逻辑应集中在 Trace 标签层，避免时间线、详情面板各自硬编码。

**核心方案：** `trace-labels.ts` 增加 Memory/RAG span 标题映射，并把 `sourceType/titleChars/contentChars/documentId/chunkCount/vectorIndexedCount/queryChars/model/dimension/topK/resultCount/profileScoped` 等字段整理成稳定顺序的摘要；`contextBudgetSnapshot` 继续在同一入口扁平化展示。新增 `trace-labels.test.ts` 覆盖 Memory/RAG 标题、RAG 入库摘要和上下文预算摘要，保证后续后端 span 字段扩展时展示层不退化为原始 JSON。

## 108. RAG 管理入库和 Agent 检索是否必须使用同一个事实源

**问题：** HTTP 冒烟发现 `/api/rag/documents` 管理接口可以成功写入 PostgreSQL 知识库，但随后通过 `/api/chat/stream` 进入单 Agent 主链路时，Trace 中 `rag.search` 的 `searchService=DefaultLocalRagEngine` 且 `returnedCount=0`。这说明 Web 管理入库和 Gateway Agent 检索落到了不同 RAG 事实源：管理侧是 PostgreSQL，聊天侧是 Local 内存 fallback，导致“接口能用但 Agent 查不到知识库”。

**核心思路：** RAG 事实源必须跟运行环境的数据源绑定。只要应用启用了 `spring.datasource.url`，Web 和 Gateway 都应优先使用 PostgreSQL-backed RAG；只有没有数据库配置的测试/离线场景才允许退回 Local 内存 RAG。不能用“是否已经有 Mapper Bean”作为唯一条件，因为 Bean 条件评估顺序可能早于 MyBatis Mapper 注册。

**核心方案：** `RagSearchServiceConfiguration#postgresRagEngine` 改为 `@ConditionalOnProperty(prefix = "spring.datasource", name = "url")`，由数据库配置显式决定 PostgreSQL RAG。`localRagEngine` 继续 `@ConditionalOnMissingBean(RagEngine.class)` 作为无数据库 fallback。新增配置回归测试：有 datasource + mapper mock 时使用 `DefaultPostgresRagEngine`；无 datasource 即使存在 mapper mock 也使用 `DefaultLocalRagEngine`，避免主链路再次和管理接口分裂事实源。

## 109. Memory 主链路冒烟为什么在 SESSION_ONLY Profile 下看不到长期记忆 Span

**问题：** HTTP 冒烟最初使用 `phase4-team-validation` Profile，`memoryStrategy.mode=SESSION_ONLY`。该模式按设计只保留短期会话上下文，不读取 PostgreSQL 长期记忆，也不写入 `memories`，因此管理接口查不到本轮记忆，Trace 中也不会出现 `memory.recall`、`memory.embedding`、`memory.vector.search` 或 `memory.write`。如果不区分 Profile 策略，容易把正确的跳过逻辑误判为 Memory 主链路失效。

**核心思路：** Memory 验证必须先确认 Profile 的长期记忆策略。`DISABLED/SESSION_ONLY` 适合验证“长期记忆不介入”，`READ_ONLY` 适合验证召回不写入，`READ_WRITE` 才适合验证“写入长期记忆 -> 下一轮召回 -> Context Slot 注入 -> Trace 可观察”的完整链路。

**核心方案：** 后续 Memory 冒烟默认使用 `READ_WRITE` Profile，并在测试脚本输出 `profileId/memoryStrategy`。本轮用 `profileId=5` 的 `READ_WRITE` Profile 验证通过：首轮写入后 `/api/memories` 可查到 marker，第二轮 Trace 出现 `memory.recall`、`memory.embedding`、`memory.vector.search`、`memory.write`，且回答能引用已记住的篮球规划偏好。同时补强 `PreferenceExtractor`，让英文 `I prefer/I like/I want` 也写入 `PREFERENCE/preference` 记忆，避免英文用户偏好只落入 summary。

## 110. Jar Skill、经验型 Skill 和 MCP Tool 的 Demo 边界如何划分

**问题：** Jar Skill 热加载、经验型 Skill、工具型 Skill 和 MCP Tool 都会出现在 Skill/Tool 市场中。如果天气查询同时作为内置 Skill、Jar Skill 和 MCP Tool 展示，会让考核评审觉得能力重复，也会模糊“插件热加载”和“外部工具协议接入”的边界。

**核心思路：** 先按能力形态划分职责，再按全局/领域/个人做作用域管理。经验型 Skill 解决“经验如何注入上下文”，Jar Skill 解决“平台如何热加载插件能力”，MCP Tool 解决“外部工具和第三方服务如何接入”。Runtime 可以统一看到工具视图，但设计文档和 Demo 必须说明来源和职责不同。

**核心方案：** 经验型 Skill 只存储领域经验、回复风格、处理步骤和业务规则片段，只进入 `apiMessages`，不进入工具池，不走 ToolDispatcher，也不使用 Jar 热加载。Jar Skill 定位为插件型 Skill，推荐 Demo 用 `TextAuditSkill`、`CodeReviewSkill`、`JsonFormatterSkill` 或 `BusinessRuleSkill` 展示上传、校验、URLClassLoader 热加载、注册、启用、禁用/卸载闭环。天气、搜索、文件系统和第三方 API 优先作为 MCP Tool 展示，由 MCP Server 暴露工具、平台 MCP Client 发现和调用，避免把外部工具 Demo 和 Jar 热加载 Demo 混在一起。

## 111. MCP 消息格式和传输方式如何落到平台设计

**问题：** MCP 容易被误解成某一种网络接口或某个天气工具实现。实际上 MCP 的消息格式和传输方式是解耦的：本地工具常用 stdio，远程工具按 2025-03-26 规范使用 Streamable HTTP。如果文档不写清楚，后续实现可能把旧 HTTP+SSE 双端点、本地子进程通信和普通 REST 工具调用混在一起。

**核心思路：** MCP 在平台中作为“外部工具协议层”存在，不等于 Skill，也不等于具体业务工具。统一工具视图由 Agent Runtime 负责，MCP Client 负责协议适配：所有 MCP 消息遵循 JSON-RPC 2.0；传输层按本地/远程选择 stdio 或 Streamable HTTP。

**核心方案：** 本地 MCP Server 优先使用 stdio：平台启动子进程，通过 stdin/stdout 读写 JSON-RPC 2.0 消息，不走网络。远程 MCP Server 使用 Streamable HTTP 兼容子集：默认单端点 `POST /mcp`，普通请求返回 JSON，长任务或流式结果返回 `text/event-stream` SSE。当前阶段不再实现旧 HTTP+SSE 双端点方案。天气查询作为 MCP Demo，优先展示工具发现、参数 schema、调用、Trace 和降级；Jar Skill Demo 继续使用 `TextAuditSkill` 等插件型能力，避免职责重叠。

## 112. 高危工具确认为什么需要跨请求重发而不是在循环内暂停等待

**问题：** 高危工具确认的理想流程是：模型调工具 → Runtime 检测 HIGH → 暂停循环 → 前端弹窗 → 用户确认 → 继续循环执行工具。但当前实现必须结束 SSE 请求，等用户确认后重新 POST `/api/chat/stream`，重新构建上下文、重新进入 ReAct 循环。

**核心思路：** SSE 是单向协议（Server → Client），不支持客户端往服务端发消息。暂停-恢复模型需要双向通道。WebSocket 可以在同一连接内实现 "Server 推送 tool_confirm_required → Client 发回确认消息 → Server 继续循环"，不需要重新请求。

**核心方案：** MVP 保持 SSE + `confirmedToolKeys` 白名单方案——用户确认后重发请求，`confirmedToolKeys` 作为白名单带入新请求，模型进 ReAct 循环后如需调该 HIGH 工具，`doExecuteTool` 检查白名单自动通过。WebSocket 双向确认作为阶段 5 可选增强，不在当前 MVP 阶段引入 WebSocket 协议栈和 Gateway 代理改造。

## 113. pendingToolCall 已确认后是否应该短路执行工具

**问题：** 旧设计中 `resumePendingToolCallIfPresent` 在确认后直接执行工具 → 调一次模型（不带工具列表）→ return，完全跳过 ReAct 循环。这导致模型无法基于工具结果继续推理（如 deploy 成功后想调 verify 验证）。

**核心思路：** `confirmedToolKeys` 是白名单，不是执行指令。用户确认的是"允许用这个工具"，不是"必须立刻执行"。应该让模型进入 ReAct 循环后自己判断要不要调、什么时候调。不管工具是模型主动调的 还是用户确认后允许的，执行完都应该 continue 让模型自己决定下一步。

**核心方案：** `pendingToolCall` 已确认后不执行任何工具，只把 `confirmedToolKeys` 传入 `AgentRunCommand`，直接进入 ReAct 循环。模型正常推理 → 如调了该 HIGH 工具 → `doExecuteTool` 检查 `confirmedToolKeys` → 已确认 → 执行 → continue。如模型决定不调（比如先处理别的事）→ 正常继续。这样已确认路径和正常工具执行路径完全统一。

## 114. OpenAI function calling `tools` 参数导致模型返回空内容

**问题：** 单 Agent 优化阶段把 `invokeModel` 改为始终 `stream=false` 后，当有可用工具时，`OpenAiCompatibleProvider.openAiRequestBody()` 会向模型发送 OpenAI function calling 格式的 `tools` 和 `tool_choice: auto` 参数。DeepSeek 等模型看到这些参数后，可能尝试用 function calling 格式响应，但模型的 function calling 实现不兼容我们的编码格式（如 `SKILL__weather`），导致返回 `finish_reason=stop` 且 `content` 和 `tool_calls` 均为空，触发 `Model response message is missing` 异常。

**核心思路：** 平台的工具调用使用自研文本格式 `@skill:name {"arg":"value"}`，不依赖 OpenAI 原生 function calling。工具的可用性通过 System Prompt 文本告知模型，模型只需在回复中输出文本格式的工具调用即可。使用 OpenAI function calling 不仅引入兼容性风险，还把工具调用的解析控制权交给了模型 API 层，破坏了自研 ReAct 循环对工具执行的可控性。

**核心方案：**
- 从 `OpenAiCompatibleProvider.openAiRequestBody()` 中移除 `tools` 和 `tool_choice: auto` 参数，只保留 `messages`、`temperature`、`stream`。
- `DefaultModelInvokeService.invoke()` 增加两层兜底：Provider 调用异常时返回可读 fallback 消息；返回结果中 `assistantMessage` 为空时也返回 fallback 消息。
- System Prompt 增强：添加 `If a tool is unavailable, say so instead of guessing. Never fabricate facts.` 约束。
- `ToolsSlotSource` 当无工具时注入硬性指令 `No tools available. Answer from training knowledge only. For real-time or external data, do NOT guess; tell the user which tool they should enable.`，防止模型没有工具时伪造实时数据。

## 115. 模型无工具时仍然伪造实时数据

**问题：** 修复 #114 后，即使 Profile 已解绑天气工具，模型（DeepSeek）仍能回答"重庆天气"类问题，且不告知用户缺少工具。原因是模型训练数据中包含天气知识，它会自信地从记忆生成答案，忽略 System Prompt 中"不要编造"的约束。

**核心思路：** System Prompt 的约束对训练数据中高度自信的事实类知识效果有限。需要在上下文装配层做硬性干预——当工具列表为空时，注入一条**无条件执行指令**，要求模型在遇到实时/外部数据问题时明确告知用户缺少什么工具。

**核心方案：** `ToolsSlotSource.fetch()` 当 `skills` 和 `mcpTools` 都为空时，不再返回 `ContextSlotContent.empty()`，而是注入硬性指令：`"No tools available. Answer from training knowledge only. For real-time or external data, do NOT guess; tell the user which tool they should enable."`。模型看到这条指令后，对天气、新闻、股价等实时问题会告知用户需要启用对应工具。

## 116. MCP 工具状态不匹配导致 Profile 绑定页不显示

**问题：** 前端 Profile 绑定页调用 `GET /mcp-tools?status=ENABLED`，但数据库 MCP 工具状态为 `AVAILABLE`（来自 Flyway 迁移），前后端状态枚举不一致导致工具列表为空。

**核心思路：** MCP 工具状态应按实际数据库值对齐，前端使用 `AVAILABLE` 作为筛选条件。

**核心方案：** 前端 `listMcpTools()` 查询参数从 `status=ENABLED` 改为 `status=AVAILABLE`。

## 117. DemoWeatherMcpServer 返回硬编码 mock 数据

**问题：** MCP 天气工具 `DemoWeatherMcpServer` 返回固定 mock 数据（重庆固定 31°C），不是真实天气，和内置 Skill `weather`（调 Open-Meteo API）行为不一致。

**核心思路：** MCP 工具和内置 Skill 应使用相同的数据源（Open-Meteo 免费天气 API），保证一致性。

**核心方案：** `DemoWeatherMcpServer` 改为调 Open-Meteo Geocoding API 解析城市经纬度 + Open-Meteo Forecast API 获取实时天气。内置常用城市坐标缓存（重庆/北京/上海等 10 个），命中缓存时跳过 Geocoding 调用，加快速度。

## 118. StdioMcpClient 子进程启动失败（java.class.path 问题）

**问题：** `StdioMcpClient` 用 `System.getProperty("java.class.path")` 获取类路径启动子进程。Spring Boot 胖 JAR 环境下该属性返回启动器路径而非完整类路径，子进程找不到 `DemoWeatherMcpServer` 类。IDEA 中也受 devtools 影响可能不完整。

**核心思路：** 内置 MCP Server 应支持进程内直接调用（不走子进程），避免依赖类路径传递。`McpToolExecutor` 对 MCP 调用异常应返回可读错误而非抛异常。

**核心方案：**
- `StdioMcpClient` 超时从 8s 增加到 20s，子进程 stderr 合并到 stdout。
- `DefaultMcpToolExecutor` 不再重抛 `BizException`，统一将异常转为 error result，让模型看到工具失败消息并生成有用回复。
- Demo 工具优先通过子进程启动（保留标准 MCP 协议能力），失败时由 Agent Runtime 兜底。

## 119. MCP 工具错误处理导致模型回复为空

**问题：** `DefaultMcpToolExecutor.execute()` 对 `BizException` 做 `throw ex`，异常直接传播到 `runModelLoop` 导致工具调用崩溃。后续 `invokeModel` 返回空内容，触发 fallback 消息 "Sorry, the model did not return a valid response"。

**核心思路：** 工具调用层应该把异常转为结构化错误（`isError: true`），放入 messages 让模型看到并自然生成解释。MCP 异常（找不到 server、连接失败、命令缺失）都应走同一降级路径。

**核心方案：** `DefaultMcpToolExecutor` 删除 `catch (BizException ex) { throw ex; }` 分支，所有异常统一返回 `McpToolExecuteResult(success=false, errorMessage=...)`，通过 `AgentToolDispatcher` → `doExecuteTool` → `[tool failed]` observation 进入模型上下文。

## 120. Open-Meteo Geocoding API 找不到中文城市名

**问题：** `DemoWeatherMcpServer` 调用 Open-Meteo Geocoding API 时，中文城市名（如"重庆"）返回空结果，且 JDK HttpClient 默认不跟从重定向，导致 API 302 后无响应。

**核心思路：** 优先使用内置城市坐标缓存（与 `DefaultSkillExecutor` 一致），缓存未命中时再调 API 并开启重定向跟随。

**核心方案：** `DemoWeatherMcpServer` 内置 10 个常用中国城市经纬度缓存。API 调用时使用 `HttpClient.newBuilder().followRedirects(REDIRECT.NORMAL)`。Geocoding 请求加 `format=json` 参数。

## 121. StdioMcpClient.readJsonRpc() 在 Windows 上对 Python 子进程不可靠

**问题：** `readJsonRpc()` 使用 `BufferedReader.ready()` 轮询检查是否有数据可读。Windows 上 Python 子进程 stdout pipe 可能不会及时返回 ready 状态，导致超时。

**核心思路：** 改用阻塞式 `readLine()` 直接读取，不依赖 `ready()` 轮询。

**核心方案：** `readJsonRpc()` 删除 `reader.ready()` 检查和 `sleepBriefly()` 轮询循环，直接调用 `reader.readLine()` 阻塞等待数据到达。超时由外层 `CALL_TIMEOUT` 控制。

## 122. 用户输入的命令字符串未拆分导致进程启动失败

**问题：** 用户在 MCP Server 配置中填入 `"command": "python C:\Users\WYZ\my-tools.py"`，`StdioMcpClient` 把整串当成单个可执行文件名传给 `ProcessBuilder`，报 `CreateProcess error=2, 系统找不到指定的文件`。

**核心思路：** 命令字符串应支持空格分隔的完整命令行，与 Shell 行为一致。

**核心方案：** `StdioMcpClient.stdioCommand()` 对非内置命令按空白字符拆分 `configuredCommand.split("\s+")`，再传给 `ProcessBuilder`。也支持 `args` 数组方式显式指定参数。

## 123. HTTP MCP Client 未实现

**问题：** MCP 协议规范要求同时支持 stdio 和 HTTP(S) 两种传输方式。平台已实现 STDIO，但 HTTP 方式缺失，无法连接远程 MCP Server。

**核心思路：** HTTP MCP 使用 `POST /mcp` 单端点 + JSON-RPC 2.0 消息，支持普通 JSON 响应和 `text/event-stream` SSE 响应。

**核心方案：** 新增 `HttpMcpClient`，实现 `initialize` + `tools/call` JSON-RPC 调用。响应 Content-Type 为 `text/event-stream` 时走 SSE 解析，否则直接读 JSON。`DefaultMcpToolExecutor` 增加 `resolveClient()` 按 serverType 分流到 `StdioMcpClient` 或 `HttpMcpClient`。

## 124. MCP 工具自动发现未实现

**问题：** 创建 MCP Server 后，工具需要通过 Flyway 或 SQL 手动注册到 `mcp_tools` 表，用户无法在前端创建 Server 后立即看到可用工具。

**核心思路：** 创建 Server 时自动发送 JSON-RPC `tools/list` 请求，从 Server 响应中解析工具列表并注册到数据库。发现失败不影响 Server 创建。

**核心方案：**
- `StdioMcpClient` 和 `HttpMcpClient` 新增 `listTools()` 方法，发送 `initialize` + `tools/list` JSON-RPC。
- `DefaultMcpServerService.create()` 后调用 `discoverTools()`，自动将工具注册到 `mcp_tools` 表（去重）。
- `DemoWeatherMcpServer`、`DemoHttpMcpEndpoint` 增加 `tools/list` 处理返回工具列表。
- 工具发现失败只 `log.warn`，不阻塞 Server 创建。
## 125. Memory/RAG 倒排检索先用 PostgreSQL tsvector 还是直接接 Elasticsearch

**问题：** 记忆系统优化设计里要求 Memory/RAG 支持倒排检索和后续 RRF 融合。如果现在直接引入 Elasticsearch，会新增部署、索引同步、故障降级和评测成本；如果继续用 LIKE，则随着 memories 和 knowledge_chunks 增长会退化为全表扫描，也不利于后续和向量召回做融合。

**核心思路：** PostgreSQL 仍是 Memory/RAG 的事实源，优先用 PG 原生 `tsvector` + GIN 索引补齐倒排召回能力。这样零新增外部组件，和现有 Flyway/MyBatis 栈一致；Qdrant、Elasticsearch、Neo4j、RRF 可以继续作为后续增强，不阻塞当前主链路。

**核心方案：** 新增 `V017__add_memory_rag_tsvector_search.sql`，给 `memories` 和 `knowledge_chunks` 增加 `search_vector`、GIN 索引和触发器；`MemoryMapper.searchActiveMemories` 与 `KnowledgeChunkMapper.searchActiveChunks` 使用 `websearch_to_tsquery('simple', ...)` + `ts_rank_cd` 排名。Memory 召回保留异常/空结果降级到原有 keyword fallback，保证迁移或倒排查询异常时不影响 Agent 主响应。

## 126. Reranker、MQE、HyDE 是否现在直接接真实模型

**问题：** Reranker、MQE 和 HyDE 都能提升检索质量，但真实模型会增加延迟、成本、评测难度和外部依赖。如果在业务 corpus 和门禁不足时直接启用，可能只是在少量样例上看起来更准，实际却引入噪声、误召回或上下文污染。

**核心思路：** 先把检索增强做成可替换、默认关闭、可 Mock 的架构入口，并把评测 runner 接入脚本和 CI。真实模型接入必须先有固定评测集和基线对比，不能绕过门禁。

**核心方案：** 新增 `RetrievalReranker`、`QueryExpansionService`、`HypotheticalDocumentService` 三个接口，默认注册 Noop；只有显式配置时才启用 Mock 实现。`DefaultPostgresRagEngine` 在原始 query 的向量/关键词召回后，追加 MQE/HyDE 额外 query 的向量/关键词召回，统一进入 RRF 融合，再交给 Reranker 二次排序。新增 `scripts/run-retrieval-eval.ps1` 和 `.github/workflows/retrieval-eval.yml`，后续检索策略变更必须先跑 Hit@K、MRR、Recall 和 no-answer precision 门禁。

## 127. 本地开发重建数据库是否会误删模型和 Profile 配置

**问题：** 为了验证 Flyway clean migration 或重建开发环境执行 `docker compose down -v` 时，会删除 PostgreSQL 数据卷。模型供应商、模型配置、Application、Profile、记忆和 RAG 测试数据都会被清空，前端表现为“之前配置的模型没了”，容易误判为前端或模型管理功能异常。

**核心思路：** 本地开发数据卷属于可变运行数据，不应和迁移验证混用。需要把“清库重建验证”和“保留当前联调数据”分成两种脚本/流程，并在高风险命令前明确提示。

**核心方案：** 日常联调只重启容器，不使用 `down -v`；需要验证全新迁移时单独使用 clean migration 流程，并提前导出或记录必要的 Provider、Model Config、Application、Profile 初始化参数。后续可补充 dev seed 脚本，把常用 DeepSeek/OpenAI 兼容 Provider、测试应用和 READ_WRITE Profile 一键恢复，避免人工重复配置。

## 128. 记忆管理前端是否允许手填任意 Profile ID

**问题：** 长期记忆页原来让用户手填 `profileId`，应用选择又是 ToolsPage 自己的局部状态，没有沿用 Chat/Profile 页的最后选择。用户可能选中 `applicationId=3` 却手填 `profileId=3`，而真实记忆属于 `applicationId=4/profileId=3`，最终空搜显示 0 条。后端权限和 scope 是正确的，但前端组合出了不属于同一应用的筛选条件。

**核心思路：** Profile 是 Application 下的配置，前端不应让用户在记忆管理页自由组合任意应用和 Profile ID。应用选择和 Profile 选择需要共享同一套跨页面偏好，并且 Profile 下拉必须按当前应用加载。

**核心方案：** ToolsPage 读取并写入 `application-selection-storage`；记忆筛选区按当前 `applicationId` 调 `listProfiles()`，用 Profile 下拉替代手填数字；切换应用时清空旧记忆结果和旧 Profile 选择，并用 `profile-selection-storage` 恢复当前应用下最后选中的 Profile。新增 `memory-selection-utils` 测试，保证旧应用的 Profile 不会被带到新应用查询中。

## 129. Memory 关键词检索是否应该支持 `springboot` 匹配 `Spring Boot`

**问题：** 空搜能返回记忆，`Spring Boot` 能返回记忆，但用户输入 `springboot` 返回 0 条。原因是 Memory 管理查询只依赖 PostgreSQL `tsvector` 和普通文本匹配，内容中带空格的英文词组不会被无空格查询命中；RAG 侧已有文本 fallback，Memory 侧能力不一致。

**核心思路：** 记忆管理检索是面向用户排查和编辑的工具，应该容忍大小写、空格和简单输入差异。倒排检索仍作为主路径，但需要低成本 fallback，避免用户必须精确输入原文空格。

**核心方案：** `MemoryMapper.searchActiveMemories` 在 `tsvector` 命中之外增加 `m.content ilike` 文本 fallback，并增加紧凑匹配：`regexp_replace(lower(m.content), '[[:space:]]+', '', 'g') ilike '%term%'`。这样 `springboot` 可以命中 `Spring Boot`。同时更新 `MemoryMapperTest`，防止 Memory 检索再次退回只支持 tsvector 的行为。

## 130. 多模块本地启动是否可能加载旧 core 依赖

**问题：** 修改 `agent-platform-core` 后，只用 `mvn -pl agent-platform-web spring-boot:run` 重启 Web，运行中的接口仍然表现为旧逻辑。原因是 Web 单模块启动可能使用本地 `.m2` 中已安装的旧 `agent-platform-core` SNAPSHOT，而不是当前工作区刚编译的 core。若 Gateway 正在运行，还可能占用本地仓库里的 core jar，导致 `mvn install` 覆盖失败。

**核心思路：** 本地多模块开发时，入口模块和 core/common 必须作为同一个变更单元启动或先安装。否则会出现“测试通过、代码已改、接口不生效”的假象，排查成本很高。

**核心方案：** 修改 core 后验证运行效果时，优先使用 reactor 构建或先停止依赖 core jar 的 Web/Gateway 进程，再执行 `mvn -pl agent-platform-web -am -DskipTests install`，随后分别启动 Web/Gateway。后续可补充统一 dev 启动脚本：停止旧进程、安装 common/core、启动 web/gateway、轮询健康检查，避免手工命令遗漏。

## 131. `tools:enabled` Redis 缓存 Key 缺少租户隔离

**问题：** Redis 缓存设计方案中可用工具列表的 key 设计为 `tools:enabled`，没有 `tenantId` 维度。Skill 和 MCP 工具都带有 `tenant_id` 字段，不同租户可以启用不同的工具集。如果所有租户共用同一个 key，租户 A 启用的工具会被租户 B 读取到，造成数据隔离泄漏。

**核心思路：** 所有共享型缓存 key 必须包含完整的隔离维度（tenantId），不能假设所有租户的工具列表相同。

**核心方案：** 将 key 从 `tools:enabled` 改为 `tools:enabled:{tenantId}`。相应地，Skill 启用/禁用或 MCP 工具刷新时的缓存失效也改为 `DEL tools:enabled:{tenantId}`。

## 132. `rate:{userId}:minute` 限流计数器 INCR + EXPIRE 非原子

**问题：** Redis 设计方案中调用频率限制使用两步操作：先 `INCR rate:{userId}:minute`，如果是第一次调用再 `EXPIRE 60`。如果进程在 INCR 和 EXPIRE 之间崩溃，key 永远不会过期，用户被永久限流。此外，首次调用时如果两个并发请求同时 INCR，两个都认为是"第一次"，第二个 EXPIRE 会将 TTL 重新设为 60 秒，实际窗口被拉长。

**核心思路：** 限流计数器需要保证"首次创建 + 设置 TTL + 递增"的原子性，避免 key 泄漏或窗口变形。

**核心方案：** 使用 Lua 脚本或 `EVAL` 将 INCR + EXPIRE 合并为一次原子操作。Lua 脚本逻辑：`local count = redis.call('INCR', KEYS[1]); if count == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end; return count`。服务端判断返回值是否超限返回 429。

## 133. `agent:profile` 缓存失效未覆盖 bindSkills / bindMcpTools 独立接口

**问题：** Redis 设计方案只在 `updateProfile` 时 `DEL agent:profile:...`。但 `bindSkills()` 和 `bindMcpTools()` 是独立的 `@Transactional` 方法，各有自己的 REST 端点（`PUT /api/profiles/{id}/skills`、`PUT /api/profiles/{id}/mcp-tools`）。用户只调 bindSkills 不调 updateProfile 时，缓存中的 `skillBindings` 仍是旧值，Agent 继续使用旧的工具绑定列表。

**核心思路：** 任何修改 Profile 关联数据（技能绑定、MCP 工具绑定）的操作都必须失效对应的 Profile 缓存。

**核心方案：** 在 `DefaultProfileService.bindSkills()` 和 `bindMcpTools()` 方法末尾增加缓存失效逻辑，`DEL agent:profile:{tenantId}:{userId}:{profileId}`。或者后续将缓存失效抽到统一的 Profile 缓存管理层，由 `@Transactional` 的 post-commit 事件触发，避免遗漏。

## 134. 用户画像快照刷新缺少冷却期

**问题：** Redis 设计方案中 `refreshProfile` 由 LLM 每次对话结束时的隐藏决策触发。LLM 可能在连续对话中每次都输出 `refreshProfile=true`，导致每次对话结束都调轻量模型重新提炼画像。轻量模型调用有成本（token 消耗 + 延迟），且画像在短时间内不会有实质性变化，重复生成浪费资源。

**核心思路：** 用户画像刷新应加最小间隔（冷却期），避免短期高频对话重复提炼。

**核心方案：** 增加冷却检查：`EXISTS user:profile:cooldown:{userId}`，存在则跳过本次刷新。刷新成功后 `SET user:profile:cooldown:{userId} "1" EXPIRE {cooldownSeconds}`。默认冷却期参考 EchoMind 的 `summary-refresh-interval: 6`（每 6 轮对话最多刷新一次），可配置。冷却期 key 独立于画像 key，不受画像 TTL 影响。

## 135. `quota:{userId}:daily:used` 异步刷新导致预检值滞后

**问题：** Redis 设计方案中配额预检值在"PG 结算成功后（异步）SET"。从 PG 事务提交到 Redis 异步写入之间有延迟，且异步写入可能失败（Redis 临时不可用、网络超时）。在此期间，Redis 预检值低于 PG 真实值，已经接近满额的用户的后续请求仍会通过预检，穿透到 PG 事务被拒绝，多了一次无效的 PG 写入。

**核心思路：** 异步刷新只影响性能不破坏正确性（PG 事务仍然是最终防线）。但可以通过改为同步写入降低无效 PG 事务的概率。

**核心方案：** 当前方案可接受（只多一次 PG 事务，不被拒绝的请求不受影响）。如果后续发现无效 PG 事务比例过高，将 `SET quota:...` 改为 PG 事务提交后的同步写入（在 `@Transactional` 的 after-commit 回调中执行），写入失败只 log.warn 不抛异常。配额预检本质上是一个性能优化而非正确性保障，异步是合理的默认选择。

## 136. 短期对话窗口 List 并发写入可能乱序

**问题：** Redis 设计方案中对话结束后用 `LPUSH` + `LTRIM` + `EXPIRE` 三个独立命令更新短期窗口。如果同一个 session 有两次并发对话（如前端快速连续发两条消息），两个请求的 LPUSH 和 LTRIM 可能交叉执行，导致窗口内消息顺序与 PG 真实顺序不一致。虽然同一 session 的高并发在当前单用户场景下概率极低，但缺乏防御。

**核心思路：** 短期窗口是 LLM 上下文饲料，乱序会破坏对话连贯性。需要保证同一 session 的更新串行化或原子化。

**核心方案：** 使用 Lua 脚本将 LPUSH + LTRIM + EXPIRE 合并为一次原子操作。脚本参数：key、新消息 JSON、maxLen、ttl。逻辑：`redis.call('LPUSH', KEYS[1], ARGV[1]); redis.call('LTRIM', KEYS[1], 0, tonumber(ARGV[2]) - 1); redis.call('EXPIRE', KEYS[1], tonumber(ARGV[3])); return redis.call('LLEN', KEYS[1])`。同时确保异步写入线程池对同一 session 使用单线程或按 sessionId hash 分配。

## 137. Redis Key 以 userId 作为隔离维度但未显式声明 userId 全局唯一性假设

**问题：** Redis 设计方案中 `memory:recent:{userId}:{sessionId}`、`user:profile:{userId}`、`rate:{userId}:minute`、`quota:{userId}:daily:used` 等 key 都只使用 `userId` 作为用户隔离维度，没有 `tenantId`。当前系统设计中 userId 是否跨租户全局唯一？如果两个租户各自有 userId=1 的用户，它们的短期窗口、画像快照、限流计数和配额预检会互相覆盖。

**核心思路：** 确认 userId 的全局唯一性假设是否成立，如果成立则显式声明；如果不成立则 key 需要加 tenantId。

**核心方案：** 检查 `users` 表的 id 生成策略。如果是数据库 sequence 全局自增，userId 天然跨租户唯一，当前 key 设计安全，但需要在设计文档和代码注释中显式声明"userId 全局唯一，因此 Redis key 不需要 tenantId 维度"。如果 userId 只在租户内唯一（复合主键），则所有 Redis key 必须改为 `{tenantId}:{userId}:...` 格式。当前项目 CLAUDE.md 里用户表设计使用自增 ID，先按全局唯一处理，但在 Redis 接入代码中加注释说明这一假设。
