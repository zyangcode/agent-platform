# 阶段 1 问题台账

> 记录阶段 1 开发准备过程中遇到的关键问题。每条只保留：问题、核心思路、核心方案。

## 1. Maven 父 POM 和 BOM 容易混淆

**问题：** 一开始容易把父 POM、Spring Boot BOM、`dependencyManagement`、`dependencies` 混在一起，不清楚谁负责继承、谁负责版本、谁真正引入依赖。

**核心思路：** Maven 依赖管理要先分清三件事：父 POM 继承工程规则，BOM 提供版本表，`dependencies` 才真正引包。

**核心方案：** 在 `实际开发/阶段1/Maven学习笔记/` 下整理 Maven 完整笔记和 draw.io 图；阶段 1 父 POM 用 `dependencyManagement` 导入 Spring Boot BOM / Spring AI BOM，子模块在自己的 `dependencies` 中声明实际依赖。

## 2. `<modules>` 和 `<parent>` 不是一回事

**问题：** 容易误以为父 POM 写了 `<modules>` 后，子模块就自动继承父 POM 的依赖版本和插件配置。

**核心思路：** `<modules>` 只负责聚合构建；真正继承父 POM 要靠子模块里的 `<parent>`。

**核心方案：** 每个 Maven 子模块都必须显式写 `<parent>` 指向根 POM；根 POM 用 `<modules>` 声明 common/core/web/gateway 四个子模块。

## 3. 父 POM `dependencies` 会污染所有子模块

**问题：** 如果把 `spring-boot-starter-web`、MyBatis、Security 等依赖放进父 POM 的 `dependencies`，所有子模块都会继承，common/core 可能拿到不该有的 Web 或安全依赖。

**核心思路：** 父 POM 的 `dependencyManagement` 适合统一管版本；父 POM 的 `dependencies` 要谨慎，因为它会被子模块继承。

**核心方案：** 阶段 1 父 POM 优先只放 `dependencyManagement`、`pluginManagement` 和少量公共属性；具体依赖放到真正需要的子模块里。

## 4. Spring Security 到底用在哪里

**问题：** 一开始容易以为 Spring Security 参与注册、查库、生成 JWT 等全部逻辑，不清楚它在链路中的真实位置。

**核心思路：** Spring Security 主要在 Controller 之前工作，负责拦截请求、认证状态恢复、权限判断、401/403。

**核心方案：** 在 `web.security` 中配置 `SecurityFilterChain`、`JwtAuthenticationFilter`、异常处理器；注册登录业务仍由 `web.auth` 调 `core.identity` 完成。

## 5. JWT 和 Spring Security 的关系

**问题：** JWT 只是令牌格式，不会自己拦截请求；Spring Security 也不会天然知道项目 JWT 里放了什么。

**核心思路：** 需要一个桥把 JWT 翻译成 Spring Security 认识的 `Authentication`。

**核心方案：** 编写 `JwtAuthenticationFilter`：读取 `Authorization: Bearer token`，校验并解析 JWT，构造 `UsernamePasswordAuthenticationToken`，写入 `SecurityContextHolder`。

## 6. 是否使用 Spring Security 默认登录页和 Session

**问题：** 参考文章中包含默认登录页、`JSESSIONID`、Session 认证等内容，但本项目是前后端分离接口项目。

**核心思路：** 默认登录页和 Session 更适合传统后端页面项目；本项目阶段 1 应使用无状态 JWT。

**核心方案：** `SecurityConfig` 中关闭 CSRF，设置 `SessionCreationPolicy.STATELESS`，不启用默认表单登录；登录成功后返回 JWT，前端后续通过 Bearer Token 访问接口。

## 7. 密码存储不能明文

**问题：** 阶段 1 要做注册登录，如果密码明文入库，会直接破坏基础安全设计。

**核心思路：** 注册时加密，登录时用密码编码器匹配，数据库永远不存明文密码。

**核心方案：** 使用 `BCryptPasswordEncoder`；注册时 `encode` 后保存，登录时 `PasswordEncoder.matches(rawPassword, encodedPassword)` 校验。

## 8. Spring Security 与 core.identity 边界

**问题：** 如果 `JwtAuthenticationFilter` 直接查 `UserMapper`，或者 core 依赖 web 的 `CurrentUser`，模块边界会变乱。

**核心思路：** web 负责 HTTP 安全，core.identity 负责用户认证业务，两者通过 API/Command/Result 交互。

**核心方案：** `core.identity.api.AuthService` 提供注册、登录、当前用户查询能力；Web 层 Controller 调用 `AuthService`，并由 Web 安全组件负责 JWT 生成、解析和 Spring Security 上下文恢复。

## 9. 401 和 403 必须返回 JSON

**问题：** 前后端分离项目中，如果未登录或无权限时跳转默认登录页，前端无法按接口协议处理错误。

**核心思路：** API 项目要返回稳定 JSON 错误，不返回 HTML 登录页。

**核心方案：** 增加 `RestAuthenticationEntryPoint` 处理 401，`RestAccessDeniedHandler` 处理 403，并在 `SecurityConfig` 中配置。

## 10. 阶段 1 是否完整实现 RBAC

**问题：** 参考文章中有角色表、权限表、资源表、动态权限等完整 RBAC 内容，直接照搬会让阶段 1 过重。

**核心思路：** 阶段 1 先做最小角色控制，复杂权限后置；业务资源权限由 core 应用层兜底。

**核心方案：** 阶段 1 只区分 `USER` / `ADMIN`；Spring Security 做 `/api/admin/**` 粗粒度限制，Profile/Application 等资源归属校验放在 core 对应应用服务中。

## 11. Spring AI 不能接管 Agent Runtime

**问题：** Spring AI 提供 ChatClient、Tool Calling、Advisor、Memory、RAG、MCP 等能力，容易误以为可以直接让它接管 Agent 循环。

**核心思路：** 本项目是 Agent 平台，Agent 循环涉及上下文、Skill/MCP 权限、Token 配额、Trace、SSE、多 Agent 扩展，不能交给 Spring AI 默认流程黑盒处理。

**核心方案：** Spring AI 只作为 `core.model` 的模型调用适配层；保留请求构建、流式响应、usage 读取、供应商适配，不使用它接管 ReAct 循环、工具编排、记忆策略和平台治理。

## 12. 不使用 Spring AI 循环不是因为不能实时返回

**问题：** 容易误解为不用 Spring AI 内置循环，是因为它不能实时返回。

**核心思路：** Spring AI 支持流式模型输出；真正不使用默认循环的原因是平台需要实时返回更多类型事件，而不只是模型 token。

**核心方案：** 模型 token 流可以使用 Spring AI 流式能力，但 Agent 过程事件由自研 Runtime 控制：`context_building`、`tool_call`、`tool_result`、`token`、`done`、`error` 等统一转成 SSE。

## 13. Spring AI 的 Flux 不应扩散到平台接口

**问题：** Spring AI 流式接口可能使用 Reactor `Flux`，但阶段 1 技术栈选的是 Spring MVC。

**核心思路：** 内部适配可以使用库自身类型，但不能让它污染 web/gateway/core.agent 的稳定接口。

**核心方案：** `core.model` 内部封装 Spring AI 流式调用，向上暴露平台自定义事件或回调；Controller 层使用 `SseEmitter` 或 `StreamingResponseBody`，不把 `Flux` 作为正式接口类型。

## 14. SSE 转发不是一次性返回

**问题：** web 转发 gateway 的 SSE 时，如果被缓冲或超时，用户会感觉不是实时返回。

**核心思路：** SSE 需要逐事件透传，不能等完整回答生成后一次性返回。

**核心方案：** 阶段 1 先实现 `/internal/ai/stream-test` 和 `/api/chat/stream-test`，用固定 SSE 事件验证 web -> gateway -> frontend 的流式链路不会被缓冲。

## 15. Gateway 不负责组装 Agent 上下文

**问题：** 聊天链路中 gateway 和 core 都出现上下文相关概念，容易误以为 gateway 也组装 Prompt。

**核心思路：** Gateway 是 AI 入口治理层，Core 才是 Agent 业务执行层。

**核心方案：** Gateway 负责鉴权、限流、敏感词、配额、Trace、SSE 协议转换和调用适配；`core.context.AgentContextBuilder` 负责 System Prompt、Profile Prompt、Memory、Skill/MCP 列表和 token 窗口控制。

## 16. 上下文超 token 的压缩位置

**问题：** 如果上下文超过模型 token 限额，不清楚压缩逻辑应该写在 gateway、model 还是 agent。

**核心思路：** 上下文压缩属于上下文组装策略，不属于入口治理，也不属于底层模型调用。

**核心方案：** 压缩逻辑放在 `core.context`；`AgentContextBuilder` 负责估算上下文 token、裁剪历史、摘要压缩、保留关键系统提示和工具定义。

## 17. Web 接口多，Core 能力接口应更少

**问题：** web 会承接很多前端页面接口，容易误以为 core 也要一比一提供同样多的接口。

**核心思路：** web 是面向页面的入口，接口可以细；core 是能力层，接口应稳定、复用、少而清晰。

**核心方案：** web 可以有多个 Controller 和 Request/Response；core 通过 `api` 包暴露少量能力接口，如 `AuthService`、`ApplicationService`、`ProfileService`、`AgentRuntimeService`。

## 18. 技术学习笔记要和正式设计分开

**问题：** 学 Maven、Spring Security、Spring AI 的过程中会产生很多解释性内容，如果全部塞进正式设计文档，会让设计文档变臃肿。

**核心思路：** 正式设计文档保留决策和约束；学习解释放在实际开发阶段笔记中。

**核心方案：** 在 `实际开发/阶段1/` 下分别创建 Maven、SpringSecurity 等学习笔记目录；正式设计文档只同步会影响实现边界的结论。

## 19. 参考文章只能借鉴，不能照搬

**问题：** CSDN 文章内容较多，包含默认登录页、Session、完整 RBAC、MySQL 示例、页面登录等，不全适合本项目。

**核心思路：** 教程解决的是通用学习问题，项目设计要服务当前架构、阶段目标和模块边界。

**核心方案：** 只借鉴过滤器链、`UserDetailsService`、`PasswordEncoder`、`SecurityContextHolder`、401/403、角色控制等核心思想；不照搬完整 RBAC、默认登录页、Session 认证和页面跳转。

## 20. 阶段 1 最小闭环优先

**问题：** 阶段 1 同时涉及 Maven、多模块、数据库、登录、JWT、SSE、Agent Runtime、Spring AI，容易被单个技术点拖散。

**核心思路：** 先完成能验证主链路的最小闭环，再逐步补全治理能力。

**核心方案：** 按执行顺序推进：Maven 模块 -> 数据库/Flyway -> identity 登录闭环 -> model 调用 -> context -> agent -> gateway SSE 空壳 -> web SSE 转发 -> 完整聊天链路。

## 21. MyBatis-Plus 到底帮我写了多少代码

**问题：** 初看 MyBatis-Plus 文档，不清楚一张表哪些 SQL 要自己写、哪些它自动生成。经常问"insert 要写吗？selectById 要写吗？"

**核心思路：** 只要 Mapper 接口继承 `BaseMapper<XxxEntity>`，CRUD 全部自动生成。自己只需要用 LambdaQueryWrapper 写复杂条件查询。

**核心方案：** 每张表的 Mapper 只需 `extends BaseMapper<XxxEntity>` 一行代码；insert/selectById/selectList/selectPage/updateById/deleteById/selectCount 全部自带；复杂查询用 Lambda 条件 `.eq(Entity::getField, value)` 构造；不写 SQL 字符串，不用写 XML。已在 `实际开发/阶段1/` 下整理完整调用链路说明和 draw.io 流程图。

## 22. SSE 缓冲是什么意思、为什么是阶段 1 最大风险点

**问题：** 文档反复说"不能缓冲"，但缓冲具体在哪发生、怎么检测、后果是什么，一开始没有直观感受。

**核心思路：** 缓冲就是中间节点把多个事件攒起来一次性发出，导致浏览器先白屏等十几秒再突然全部蹦出来，SSE 流式效果完全消失。缓冲可能发生在 Nginx、Spring 的 response buffer、或者代码里没调 flush()。

**核心方案：** 阶段 1 先不做真实 Agent，先在 Gateway 写一个固定 SSE 测试接口（每 500ms 发一个事件），用 `curl -N` 验证逐条收到；然后再在 Web 做透传测试，确保三跳（Core→Gateway→Web→浏览器）都不缓冲。这个测试不通，Agent 对话的 SSE 体验就完全不可用。

## 23. Flyway 和 DBeaver 不是同一个东西

**问题：** 两个工具都碰数据库，容易把它们当同类工具。Flyway 是项目启动时自动执行 SQL 的版本管理工具；DBeaver 是你的手动可视化查询工具。

**核心思路：** Flyway 管"数据库结构怎么变"（建表、加字段、插种子数据），每次启动自动运行；DBeaver 管"你肉眼看看数据长什么样"（写 SELECT 验证、检查写入）。

**核心方案：** SQL 脚本放在 `core/src/main/resources/db/migration/V数字__描述.sql`，Flyway 自动按版本号顺序执行并记录；开发验证时用 DBeaver 连库查数据。

## 24. PostgreSQL 和 MySQL 的 SQL 几乎一样，差异点只在这几个关键字

**问题：** MySQL 用得多，担心换 PostgreSQL 要重学一套 SQL。

**核心思路：** SELECT/INSERT/UPDATE/DELETE/JOIN 完全一样。真正的差异集中在：自增主键 `BIGSERIAL` vs `BIGINT AUTO_INCREMENT`、JSON 字段 `jsonb` vs `JSON`、数组字段 `TEXT[]`（MySQL 没有）、DDL 事务支持。

**核心方案：** 项目 Flyway 建表 SQL 注意三个关键字：`BIGSERIAL`、`jsonb`、`TEXT[]`；MyBatis-Plus 写代码时基本不感知差异；jsonb 配合 GIN 索引可以高效查询 JSON 字段内容（如 memory_strategy、parameter_schema）。

## 25. Docker Desktop 在 Windows 上的架构

**问题：** 一开始以为 Docker 直接跑在 Windows 上，结果发现 `docker run` 要拉镜像拉不下来，不确定是不是 Docker 没装对。

**核心思路：** Docker Desktop 在 Windows 上通过 WSL2 跑一个 Linux 虚拟机，Docker Engine 在 Linux 里，docker CLI 在 Windows 上。你敲的命令在 Windows 上执行，容器在 Linux 虚拟机里启动。拉镜像失败是 Docker Hub 被墙的网络问题，不是 Docker 没装对。

**核心方案：** 在 Docker Desktop 设置中配置 `registry-mirrors` 使用国内镜像站（docker.1ms.run / docker.xuanyuan.me）解决拉镜像问题；`docker run` 后容器正常运行在 WSL2 里的 Linux 上，Windows 通过端口映射（-p 5432:5432）访问容器服务。

## 26. JsonNode 字段不能直接映射 PostgreSQL jsonb

**问题：** Entity 中使用 `JsonNode` 承接 `jsonb` 字段时，如果没有 MyBatis TypeHandler，后续插入或查询 model/profile/skill/mcp/agent 的 JSON 字段可能序列化失败或读不回来。

**核心思路：** `jsonb` 是 PostgreSQL 的 JSON 类型，Java 的 `JsonNode` 不是 JDBC 原生类型，必须显式定义转换规则，并让 MyBatis-Plus 对该 Entity 启用 ResultMap。

**核心方案：** 在 `core.support.persistence` 增加 `JsonNodeTypeHandler`，所有 `JsonNode` 字段用 `@TableField(typeHandler = JsonNodeTypeHandler.class)` 标注，对应 Entity 的 `@TableName` 启用 `autoResultMap = true`；新增 `JsonNodeMappingTest`，防止后续新增 jsonb 字段时漏配。

## 27. Model 阶段先 Mock First，不能被真实模型阻塞

**问题：** 阶段 1 的 Agent、Context、Gateway、SSE 都依赖模型调用能力。如果一开始就绑定真实模型 API Key、网络连通性和供应商细节，后续主链路会被外部依赖阻塞。

**核心思路：** `core.model` 先提供稳定的平台接口和 mock 模型实现，真实 Spring AI / OpenAI 兼容调用只作为内部适配替换，不影响上层 Agent Runtime。

**核心方案：** 使用 seed 数据中的 `mock-chat` 跑通 `ModelInvokeService`；`ModelConfigService` 管理 provider/config；模型 API Key 通过 `core.support.security.SecretEncryptor` 统一处理，阶段 1 用开发占位实现，后续可替换 AES/KMS。

## 28. Profile 绑定 Skill/MCP 不能直接跨包查 Mapper

**问题：** Profile 需要绑定 Skill 和 MCP Tool，最容易为了校验 ID 是否存在而在 `core.profile` 里直接注入 `SkillMapper` / `McpToolMapper`，这会破坏“跨业务包只能走 api/dto/command”的边界。

**核心思路：** 绑定关系表属于 `core.profile`，但 Skill/MCP 的可用性判断属于各自业务包。Profile 只发起能力调用，不碰别的包的持久化细节。

**核心方案：** 新增最小查询接口 `core.skill.api.SkillQueryService` 和 `core.mcp.api.McpToolQueryService`；`DefaultProfileService` 绑定前只调用这两个 API 校验可绑定性，ArchUnit 继续禁止跨包访问 entity/mapper。

## 29. 文档里的 Service 名称必须跟代码同步

**问题：** 早期设计文档中出现过 `IdentityAuthService`、`ProfileQueryService`、`ApiKeyService`、`PermissionService` 等候选接口名，但阶段 1 当前代码实际落地的是 `AuthService`、`ApplicationService`、`ProfileService`。如果文档继续保留旧名字，后续写 Gateway、Context、Agent 时容易按不存在的接口继续扩散。

**核心思路：** 正式开发阶段以已落地代码接口为准，设计文档只保留“后续要拆”的接口，并明确拆分时机。

**核心方案：** 阶段 1 当前统一使用 `AuthService`、`ApplicationService`、`ProfileService`；`ApiKeyService` 标注为 Step 13 Gateway API Key 入口实现前新增；`PermissionService` 暂不作为阶段 1 必做接口，资源归属校验先放各业务 Application Service。

## 30. Step 9 Skill 不能只做列表查询

**问题：** Profile 阶段已经为了绑定校验新增了 `SkillQueryService`，但 Step 9 如果只补 `GET /api/skills`，后续 Agent 仍然没有统一的 Skill 执行入口，容易在 `core.agent` 里直接查 `SkillMapper` 或手写工具分支。

**核心思路：** Skill 是独立能力包，不只是元数据表。查询、注册、执行要分清：Web 查询可用 Skill 走 `SkillQueryService`，Agent 获取可调用 Skill 走 `SkillRegistry`，真正执行工具走 `SkillExecutor`。

**核心方案：** Step 9 实现内置 `calculator/weather/search`。`calculator` 必须真实执行，`weather` 和 `search` 阶段 1 可先 mock。新增或扩展 `SkillQueryService` 支持列表查询；新增 `SkillRegistry` / `SkillExecutor` 作为后续 Agent 唯一调用入口；Web 增加 `GET /api/skills?scope=GLOBAL&status=ENABLED`；测试覆盖 Skill 列表、calculator 执行和 Web Controller。ArchUnit 继续保证 `core.agent` 不能访问 `core.skill.mapper`。

## 31. 为什么不能只靠 URL 区分角色，必须用 Spring Security + RBAC

**问题：** 如果把接口按角色拆成 `/admin/api/traces`、`/user/api/traces`、`/developer/api/traces`，看起来 URL 自己就区分了角色。为什么还要 Spring Security 和 RBAC？

**核心思路：** URL 能区分"谁能访问接口"，但不能区分"同一个接口，不同角色能查哪些数据"。按角色拆 URL 会导致同一张表的查询逻辑散布在多个 Controller 里，违背"数据写入/查询入口唯一"的原则。

**核心方案：** 统一 URL（如 `GET /api/traces`）+ Spring Security 做前端拦截（`@PreAuthorize / hasRole`），Core Service 只写一份查询逻辑，通过 `CurrentUser.role` 控制 where 条件（Admin 查全部，User 只查自己的）。当前代码 `ApplicationController` 和 `ProfileController` 就是例子——一个统一 URL，Controller 只做参数转换和入参校验，查询逻辑集中在 Service，权限由 SecurityConfig + `CurrentUser` 协作完成。改一个查询逻辑只需改一个地方。

## 32. 租户、平台、应用隔离单元的区分

**问题：** tenants、applications、users 三张表的关系不清晰，不清楚 platform/tenant/application/user 分别对应什么。

**核心思路：** 平台是 SaaS 系统本身；租户是数据隔离的最高层级（一个公司/团队一个租户）；应用是用户手里的"项目/用途单元"（一个用户可创建多个应用，每个一把 Key）；用户属于租户，Application 也属于租户。

**核心方案：** 当前 MVP 只有一个默认租户（`tenants.id=1`），但 `tenant_id` 字段全程保留。隔离链路：`tenant_id` 贯穿 users / applications / api_keys / conversations / trace_roots / token_usage_logs 等所有表，保证跨租户数据绝对隔离；`application_id` 贯穿 api_keys / conversations / token_usage_logs / trace_roots，保证同一租户下不同应用间 Token 消耗、Trace、会话各自独立。

## 33. Profile 阶段 1 权限粒度

**问题：** 当前 `POST /api/profiles`、`PUT /api/profiles/{profileId}/skills`、`PUT /api/profiles/{profileId}/mcp-tools` 没有走 `/api/admin/**` 或 `/api/developer/**` 前缀，`SecurityConfig` 目前只区分 `ADMIN` 和“所有登录用户”，因此普通 `USER` 也能创建 Profile 和修改绑定。但设计文档规定 Profile 主要由 Developer 创建和管理。

**核心思路：** 阶段 1 前半段优先跑通基础闭环，可以先让登录用户完成 Profile 创建和绑定，避免提前引入 Developer 授权流程阻塞联调。但阶段 1 收尾时必须收紧写权限，避免把临时放宽变成正式行为。业务资源归属校验（`getOwnedDraftProfile` 只允许创建者编辑）已在 core 应用层兜底。

**核心方案：** 暂不在 Step 8/Step 9 立即改代码。当前 SecurityConfig 规则：`/api/admin/**` → hasRole("ADMIN")，其他接口 → authenticated。Profile 创建/绑定受到归属权兜底（`ensureApplicationOwned` + `getOwnedDraftProfile`），即使恶意用户绕过前端创建了 Profile，也只能在自己所属 Application 下操作。Step 14 或阶段 1 收尾时补 Profile 写权限规则：`GET /api/profiles` 和 `GET /api/profiles/{profileId}` 保持登录用户可访问，并由 core 校验归属/可见性；`POST /api/profiles`、`PUT /api/profiles/{profileId}/skills`、`PUT /api/profiles/{profileId}/mcp-tools` 限制为 `DEVELOPER` 或 `ADMIN`。同时补默认 Developer 测试账号、SecurityConfig 规则和 Web Controller 权限测试。

## 34. common 放业务 DTO 的真正代价

**问题：** 经常想着"反正常用的 DTO 放 common 方便引用"。为什么不行？

**核心思路：** 不是编译问题——是改一行代码的爆炸半径。common 放了业务 DTO，所有依赖 common 的模块在 `mvn compile` 时都会重编译（Maven 比较依赖链，common 的 jar 变了 → 下游全重编）。项目后面 core 里几百个 Entity/Mapper/Service，每次改一个 DTO 就要等所有模块编译完。DTO 放各自模块里，改 web 的 Request 只有 web 重编译，gateway 不受影响。

**核心方案：** 当前代码严格遵守：`web.dto` 只放 Web 入参出参（LoginRequest、CreateProfileRequest），`gateway.dto` 只放 Gateway 入参出参（ChatCompletionRequest），`core.xxx.dto` 只放 core 内部用例出参。三个模块的 DTO 格式完全不同（LoginRequest 有 @NotBlank，gateway.dto 走 OpenAI 兼容格式），放 common 反而会让三个模块被迫共享同一个结构。

## 35. JWT + CurrentUser + ArgumentResolver 的完整请求链路

**问题：** 三个类（JwtAuthenticationFilter、CurrentUser、CurrentUserArgumentResolver）零散分布在 web.security 包下，不知道谁先执行、数据怎么传递、最终怎么进 Controller。

**核心思路：** 三个阶段接力——Filter 负责从 Header 取 token、验签名、把 CurrentUser 塞进 SecurityContextHolder；ArgumentResolver 负责在 Controller 方法被调用前，检查参数类型是否是 CurrentUser，从 SecurityContextHolder 取出 Filter 放进去的对象；Controller 直接用，不需要手动解析。

**核心方案：** 当前代码链路：`JwtAuthenticationFilter.doFilterInternal()` → `jwtTokenService.parse(token)` → 构造 `UsernamePasswordAuthenticationToken(CurrentUser, ...)` → `SecurityContextHolder.getContext().setAuthentication()` → `CurrentUserArgumentResolver.resolveArgument()` 检查参数类型 → 从 `SecurityContextHolder.getContext().getAuthentication().getPrincipal()` 取出 CurrentUser → Controller 直接使用 `currentUser.userId()/tenantId()/roles()`。三者靠 `SecurityContextHolder` 这个 ThreadLocal 全局暂存处串起来，Filter 写、Resolver 读、Controller 用。

## 36. Step 10 MCP 不能只做工具列表

**问题：** MCP 如果只实现 `GET /api/mcp-tools`，后续 Agent 仍然缺少统一的 MCP 工具发现和执行入口，容易在 `core.agent` 中直接查 `mcp.mapper` 或手写 MCP 工具分支，破坏包边界。

**核心思路：** MCP 与 Skill 一样是独立能力包。Web 查询工具列表走查询服务，Agent 获取可调用工具走 Registry，真正执行工具走 Executor，后续替换真实 MCP Client 时不影响 Agent Runtime。

**核心方案：** Step 10 增加 `McpToolRegistry` / `McpToolExecutor`，`DefaultMcpToolQueryService` 同时承担查询与注册发现能力；阶段 1 先提供 `read_file` 只读 mock 工具，不读取任意本地路径，只返回固定演示内容。Agent 后续只能通过 `core.mcp.api.McpToolExecutor` 调 MCP，ArchUnit 继续禁止跨包访问 `mcp.mapper`。

## 37. Step 11 Context 不能直接跨包查 Profile/Skill/MCP 持久化

**问题：** 上下文组装需要 Profile Prompt、短期历史、长期记忆、Skill 列表和 MCP Tool 列表。如果 `core.context` 为了方便直接注入 `profile.mapper`、`skill.mapper`、`mcp.mapper`，会破坏“跨业务包只能走 api/dto/command”的边界，后续 Agent Runtime 很容易继续沿用这种越界方式。

**核心思路：** Context 是组装层，不是所有数据的持久化拥有者。Profile、Skill、MCP 通过各自 API 获取；历史消息和记忆分别由 `core.agent` 与 `core.memory` 暴露最小 API，Context 只消费这些能力并负责 token 预算、历史裁剪和 Prompt 拼装。

**核心方案：** Step 11 增加 `core.context.api.AgentContextBuilder`、`BuildAgentContextCommand`、`AgentContextDTO`；同时补 `core.agent.api.MessageHistoryService` 和 `core.memory.api.MemoryRecallService` 作为历史消息与记忆召回入口。`DefaultAgentContextBuilder` 组装 System Prompt、Profile Prompt、用户输入、历史消息、长期记忆、Skill/MCP 工具列表，并在 `core.context` 内做阶段 1 简单 token 估算和历史裁剪。Gateway 不参与上下文组装，Agent 后续只调用 `AgentContextBuilder`。

## 38. Context 需要区分默认工具与用户本次选择工具

**问题：** 如果 `AgentContextBuilder` 只加载 Profile 绑定中 `enabledByDefault=true` 的 Skill/MCP，后续 Web 聊天页即使允许用户本次勾选工具，也没有地方把选择传入 Runtime，Agent 会看不到“已绑定但默认不启用”的工具。

**核心思路：** Profile 绑定表示“这个 Profile 允许使用哪些工具”，`enabledByDefault` 只表示默认勾选状态；用户本次选择表示“这一次实际启用哪些工具”。上下文组装必须以 Profile 绑定为授权上限，再叠加用户本次选择。

**核心方案：** `BuildAgentContextCommand` 增加 `selectedSkillIds` 和 `selectedMcpToolIds`。如果调用方传入选择列表，Context 使用“Profile 已绑定工具 ∩ 用户选择工具”；如果没传，则回退到 `enabledByDefault=true` 的默认工具。这样 Step 12 的 AgentRuntimeService 可以直接承接 Web/Gateway 传来的本次工具选择。

## 39. Context token 预算不能只靠调用方手填

**问题：** `model_configs` 已有 `max_context_tokens`，如果 `AgentContextBuilder` 只依赖 command 里的 `maxContextTokens`，Step 12 的 AgentRuntimeService 就必须自己查模型配置再转交给 Context，容易重复和漏传。

**核心思路：** 模型上下文窗口属于模型配置能力，Context 负责 token 裁剪，应能在调用方不显式传入时使用 Profile 对应模型的默认窗口。

**核心方案：** `ModelConfigService` 增加 `getActiveModelConfig(modelConfigId)`；`DefaultAgentContextBuilder` 在 `maxContextTokens` 未传或无效时，读取 `profile.modelConfigId` 对应的 `ModelConfigDTO.maxContextTokens`，再做历史裁剪。调用方仍可显式传入预算用于测试或特殊调用场景。

## 40. MCP Tool 绑定校验必须带租户隔离

**问题：** `DefaultMcpToolQueryService.areMcpToolsBindable` 如果只按 `mcp_tool.id` 和 `status=AVAILABLE` 判断，会导致租户 A 可能绑定租户 B 的 MCP Tool。因为 `mcp_tools` 自身没有 `tenant_id`，租户归属在 `mcp_servers.tenant_id` 上。

**核心思路：** MCP Tool 的租户边界必须通过所属 MCP Server 判断。任何绑定校验都不能只看 Tool 表本身，否则后续 Agent Runtime 会在错误授权范围内执行工具。

**核心方案：** `areMcpToolsBindable(tenantId, toolIds)` 先查询当前租户 `ACTIVE` 的 MCP Server ID，再限定 `mcp_tool.mcp_server_id in serverIds`、`status=AVAILABLE`、`id in toolIds`。如果当前租户没有 active server，直接返回 false。已补单测覆盖“当前租户无 active server 但 tool 查询命中时必须拒绝”的场景。

## 41. Agent Runtime 写 memory 不能直接访问 memory.mapper

**问题：** Step 12 Agent Runtime 需要在对话结束后写入 memories。如果 `core.agent` 直接注入 `MemoryMapper` 或使用 `MemoryEntity`，会违反“跨业务包只能走 api/dto/command”的 ArchUnit 规则。

**核心思路：** Agent 是流程编排者，不拥有 memory 的持久化细节。记忆召回由 `MemoryRecallService` 暴露给 Context；记忆写入也应该由 memory 包暴露最小写入 API。

**核心方案：** 新增 `core.memory.api.MemoryWriteService` 与 `RecordMemoryCommand`，`DefaultMemoryWriteService` 在 memory 包内部负责落库。`DefaultAgentRuntimeService` 只调用 `MemoryWriteService.record(...)`，不访问 `memory.entity` 或 `memory.mapper`。ArchUnit 已验证该边界。

## 42. 阶段 1 ReAct 循环采用文本协议占位，真实 function calling 后续替换

**问题：** 阶段 1 既要证明 Agent 能调用 Skill/MCP，又不能把 Spring AI 的默认 tool calling 或完整 OpenAI function calling 流程提前接进来，否则会阻塞 Gateway SSE 和 Web 透传主链路。

**核心思路：** 先把平台自研 Runtime 的控制权跑通：Context -> Model -> Tool -> Model -> Message/Memory 落库。模型如何表达工具调用可以先用轻量协议占位，后续真实模型接入时只替换解析层，不改变 Agent Runtime 对 Skill/MCP 的调用边界。

**核心方案：** `DefaultAgentRuntimeService` 阶段 1 支持模型返回 `@skill:calculator {"expression":"1+2"}` 或 `@mcp:read_file {"path":"/tmp/demo.txt"}`。Runtime 解析后分别调用 `SkillExecutor` / `McpToolExecutor`，把工具结果作为 `tool` message 加回上下文，再二次调用模型生成最终回答。循环最多 3 步，超过则抛 `AGENT_MAX_STEPS_EXCEEDED`。

## 43. Agent Runtime 复用 conversationId 必须校验归属

**问题：** 如果 `conversationId` 非空就直接复用，不校验它是否属于当前 `tenantId/applicationId/userId/profileId`，用户可能把消息写入别人的会话。即使历史查询会校验归属，写入动作已经越权发生。

**核心思路：** Conversation 是会话写入边界，Runtime 在保存 user message 前必须先确认会话归属。不能只依赖后续 `MessageHistoryService` 查询历史时兜底。

**核心方案：** `DefaultAgentRuntimeService.ensureConversation` 在复用已有会话时调用 `conversationMapper.selectById`，校验 `tenantId/applicationId/userId/profileId/status=ACTIVE`，不匹配直接抛 `REQUEST_INVALID`，并保证不写 message、不调 model。

## 44. Agent Runtime 执行工具必须校验 Context 授权工具列表

**问题：** Context 已经按“Profile 已绑定工具 ∩ 用户本次选择工具”算出本轮可用 Skill/MCP，但 Runtime 如果只按模型输出的 `@skill:` / `@mcp:` 名称直接执行，会绕过本轮工具授权边界。

**核心思路：** Runtime 可以解析模型工具调用，但不能相信模型输出。模型只能请求调用工具，真正是否允许执行必须由 Context 里已经授权的工具列表决定。

**核心方案：** `DefaultAgentRuntimeService.executeTool` 执行前校验：Skill 名必须存在于 `context.availableSkills().code()`；MCP 工具名必须存在于 `context.availableMcpTools().name()`。未授权则抛 `REQUEST_INVALID`，不调用对应 Executor。
