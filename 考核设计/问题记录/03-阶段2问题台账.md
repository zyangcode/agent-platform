## 2. 已解决问题

### 2.1 SensitiveDataScanner 正则 `\b` 中文失效

**问题：**
`DefaultSensitiveDataScanner` 中的 4 条敏感数据正则全部使用 `\b`（单词边界）。Java 的 `\w` 只认 `[a-zA-Z0-9_]`，中文字符被视为 `\W`，导致中文紧贴敏感数据时 `\b` 行为不符合预期，手机号等不会被匹配。

例如：`我的手机号是13812345678` 不会被拦截，但 `call 13812345678 now` 会被拦截。

**处理：**
将 `\b` 替换为更精确的边界断言：

- PHONE / ID_CARD → `(?<!\d)` / `(?!\d)`（数字边界）
- EMAIL → `(?<![A-Za-z0-9._%+-])` / `(?![A-Za-z0-9.-])`（邮箱字符边界）
- API_KEY → `(?<![A-Za-z0-9])` / `(?![A-Za-z0-9])`（字母数字边界）

**当前状态：**
已解决。

**验证方式：**
- `DefaultSensitiveDataScannerTest.scanFindsSensitiveDataAdjacentToChineseCharacters` 新增中文紧贴场景。
- Gateway 直连测试 `我的手机号是13812345678` 返回 403 SECURITY_BLOCKED。
- 全量 146 测试通过。

### 2.4 `agentMode=none` 的敏感数据扫描没有覆盖 `messages`

**问题：**
Gateway 当前的 `SensitiveDataFilter` 只扫描：

```java
request.message()
```

但 `agentMode=none` 直连模型路径使用的是：

```java
request.messages()
```

这意味着如果用户通过 `messages` 传入敏感内容，当前扫描可能不会命中。

**影响：**  
阶段 2.3 宣称“请求入口敏感数据阻断”，但实际只覆盖 Agent 模式的单条 `message`，没有完整覆盖纯模型透传模式。

**处理：**
已修复为同时扫描：

- `request.message()`，记录位置 `REQUEST_MESSAGE`
- `request.messages().content`，记录位置 `REQUEST_MESSAGES`

命中任一位置都会在 Quota 预扣和模型调用前阻断请求。

**当前状态：**
已解决。

**验证方式：**

- `InternalAiControllerTest.directModeScansMessagesAndBlocksBeforeModelInvoke` 覆盖 `agentMode=none` 的 `messages` 命中敏感数据时返回 403。
- 同一测试断言不会调用 `modelInvokeService.invoke`，也不会预扣 Quota。

### 2.5 `agentMode=none` 没有记录 model span 和 token_usage_logs

**问题：**  
Token Usage 记录逻辑目前在 `DefaultAgentRuntimeService.safeRecordTokenUsage` 内。  
`agentMode=none` 路径在 Gateway Controller 中直接调用 `ModelInvokeService`，不经过 AgentRuntime。

因此当前 `agentMode=none` 只有：

- trace root
- quota reserve/commit/release

但缺少：

- `model.invoke` span
- `token_usage_logs`

**影响：**  
阶段 2.1 的目标是 Trace + Token Usage 闭环；直连模型路径目前没有完整闭环。

**处理：**
已新增 Gateway 小封装 `DirectModelRunService`，避免把记录逻辑继续塞进 Controller。该封装负责：

- start `model.invoke` span
- 调用 `ModelInvokeService`
- record token usage
- finish span

Gateway Controller 的 none 路径只调用该封装。

**当前状态：**
已解决。

**验证方式：**

- `InternalAiControllerTest.directModeRecordsModelSpanAndTokenUsage` 覆盖 `agentMode=none` 成功调用时创建 `model.invoke` span。
- 同一测试断言写入 `RecordTokenUsageCommand`，并保留 `tenantId/applicationId/userId/modelConfigId/totalTokens` 等字段。

## 4. 可接受遗留

### 4.1 Quota 不是完整日/月配额扣减

**现状：**  
`quota_configs` 有：

- `daily_limit_tokens`
- `monthly_limit_tokens`
- `single_request_limit_tokens`

但当前 `DefaultQuotaService` 只检查 `single_request_limit_tokens`。

**影响：**  
当前只能说明“单请求预扣和超限阻断”已完成，不能说完整日/月 Token 配额系统已完成。

**严重度：**  
低到中。

**为什么阶段 2 可接受：**  
阶段 2.2 的目标是 Gateway 治理链能做 429 阻断，不是完整计费系统。

**后续处理：**  
阶段 3 Token 看板或阶段 4 治理增强时再做日/月累计统计和扣减。

### 4.2 Quota 预估 Token 固定为 1000

**现状：**  
`QuotaFilter` 中写死：

```java
DEFAULT_ESTIMATED_TOKENS = 1000L
```

**影响：**  
预扣值不够真实。

**为什么阶段 2 可接受：**  
阶段 2.2 先证明预扣链路，不做复杂 Token 预估。

**后续处理：**  
阶段 3 或阶段 4 从 Profile、ModelConfig 或请求上下文中取预估值。

### 4.3 Gateway 治理链仍在 Controller 内编排

**现状：**  
`InternalAiController` 内串联：

- SensitiveDataFilter
- QuotaFilter
- Trace root
- AgentRuntime / ModelInvokeService
- AlertFilter

**影响：**  
目前可读，但后续继续加治理能力会膨胀。

**为什么阶段 2 可接受：**  
之前已明确阶段 2 不提前做完整 Pipeline 或 Gateway Filter 链重构，优先跑通治理闭环。

**后续处理：**  
如果阶段 3 后 Controller 明显变重，再单独抽 Gateway 治理链。

### 4.4 MODEL_ERROR 极端路径 trace_roots 未写入

**现状：**
当 profileId 不存在时，`startTraceRoot()` 内部失败被 try-catch 吞掉（设计如此：trace 不能阻断主链路），后续 `finishTraceRoot()` 执行 UPDATE 但 root 从未创建。结果：quota_reservations RELEASED ✅、alert_events MODEL_ERROR ✅，但 trace_roots 无记录。

**影响：**
极端边缘路径下无法通过 traceId 查到失败记录。

**严重度：**
低。正常路径 trace_roots 写入完全正常。生产环境 profileId 来自下拉选择不会出现不存在的 ID。

**为什么阶段 2 可接受：**
正常路径全绿。修复可能涉及 start/finish 改为 insert-or-update 语义，改动面较大。

**后续处理：**
阶段 3 或阶段 4 治理增强时统一处理 trace 容错语义。
