# 阶段 2 问题台账

> 本台账用于记录阶段 2 收尾检查中发现的问题、矛盾和后续处理顺序。它不是开发计划，也不是验收总结；它只回答一个问题：当前还有哪些事情需要记住，哪些已经解决，哪些不能误称为已完成。

## 1. 当前结论

阶段 2 主功能已经完成到 2.4，并且可以进入阶段 3。

但阶段 2 仍有几个需要台账化的问题：

1. `agentMode=none` 直连模型路径的治理闭环弱于 AgentRuntime 路径。
2. 敏感数据扫描只覆盖 `message`，没有覆盖 `messages`。
3. Token Usage 只在 AgentRuntime 内记录，直连模型路径没有写 `token_usage_logs`。
4. Quota 当前是单请求预扣和单次限额，不是完整日/月配额扣减。
5. Gateway 治理链仍在 Controller 内编排，后续能力继续增加时会膨胀。
6. 部分阶段 2 文档是执行过程记录，存在时间线不一致，需要以验收总结和本台账为最新入口。

**处理：**

**当前状态：**  
已解决。

**验证方式：**

- Web 架构测试禁止访问 `core.*.entity` 和 `core.*.mapper`。
- Gateway 架构测试禁止访问 `core.*.entity` 和 `core.*.mapper`。
- Core 架构测试禁止跨业务包访问其他包 Entity/Mapper。

## 3. 需要修复的问题

### 3.1 `agentMode=none` 的敏感数据扫描没有覆盖 `messages`

**问题：**  
Gateway 当前的 `SensitiveDataFilter` 只扫描：

```java
request.message()
```

但 `agentMode=none` 直连模型路径使用的是：

```java
request.messages()
```

这意味着如果用户通过 `messages` 传入敏感内容，当前扫描可能不会命中。

**影响：**  
阶段 2.3 宣称“请求入口敏感数据阻断”，但实际只覆盖 Agent 模式的单条 `message`，没有完整覆盖纯模型透传模式。

**严重度：**  
高。

**建议处理：**  
阶段 3 开始前先修。

**建议方案：**

1. `SensitiveDataFilter.scanRequest` 同时扫描 `request.message()` 和 `request.messages()`。
2. 对 `messages` 只扫描 `content`，记录 location 时区分：
   - `REQUEST_MESSAGE`
   - `REQUEST_MESSAGES`
3. 命中任一位置都阻断请求。
4. 补 Gateway 测试：`agentMode=none` 的 `messages` 命中敏感数据时返回 403，且不调用 `modelInvokeService.invoke`。

### 3.2 `agentMode=none` 没有记录 model span 和 token_usage_logs

**问题：**  
Token Usage 记录逻辑目前在 `DefaultAgentRuntimeService.safeRecordTokenUsage` 内。  
`agentMode=none` 路径在 Gateway Controller 中直接调用 `ModelInvokeService`，不经过 AgentRuntime。

因此当前 `agentMode=none` 只有：

- trace root
- quota reserve/commit/release

但缺少：

- `model.invoke` span
- `token_usage_logs`

**影响：**  
阶段 2.1 的目标是 Trace + Token Usage 闭环；直连模型路径目前没有完整闭环。

**严重度：**  
中高。

**建议处理：**  
和 3.1 一起在阶段 3 前修复。

**建议方案：**

1. 不把记录逻辑塞进 Controller 大段代码。
2. 在 Gateway 或 Core 增加一个小的直连模型执行封装，例如 `DirectModelRunService` 或 `ModelGatewayRunner`。
3. 该封装负责：
   - start `model.invoke` span
   - 调用 `ModelInvokeService`
   - record token usage
   - finish span
4. Gateway Controller 只调用封装后的服务。

**注意：**  
不要为了这一个问题提前重构完整 Pipeline。

## 4. 可接受遗留

### 4.1 Quota 不是完整日/月配额扣减

**现状：**  
`quota_configs` 有：

- `daily_limit_tokens`
- `monthly_limit_tokens`
- `single_request_limit_tokens`

但当前 `DefaultQuotaService` 只检查 `single_request_limit_tokens`。

**影响：**  
当前只能说明“单请求预扣和超限阻断”已完成，不能说完整日/月 Token 配额系统已完成。

**严重度：**  
低到中。

**为什么阶段 2 可接受：**  
阶段 2.2 的目标是 Gateway 治理链能做 429 阻断，不是完整计费系统。

**后续处理：**  
阶段 3 Token 看板或阶段 4 治理增强时再做日/月累计统计和扣减。

### 4.2 Quota 预估 Token 固定为 1000

**现状：**  
`QuotaFilter` 中写死：

```java
DEFAULT_ESTIMATED_TOKENS = 1000L
```

**影响：**  
预扣值不够真实。

**为什么阶段 2 可接受：**  
阶段 2.2 先证明预扣链路，不做复杂 Token 预估。

**后续处理：**  
阶段 3 或阶段 4 从 Profile、ModelConfig 或请求上下文中取预估值。

### 4.3 Gateway 治理链仍在 Controller 内编排

**现状：**  
`InternalAiController` 内串联：

- SensitiveDataFilter
- QuotaFilter
- Trace root
- AgentRuntime / ModelInvokeService
- AlertFilter

**影响：**  
目前可读，但后续继续加治理能力会膨胀。

**为什么阶段 2 可接受：**  
之前已明确阶段 2 不提前做完整 Pipeline 或 Gateway Filter 链重构，优先跑通治理闭环。

**后续处理：**  
如果阶段 3 后 Controller 明显变重，再单独抽 Gateway 治理链。

建议阶段 3 开始前先做两个小修复：

## 7. 总体判断

阶段 2 的主线能力是成立的，但不能简单说“所有 Chat 路径都完全闭环”。

更准确的表述是：

> AgentRuntime 路径已经具备 Trace、Token Usage、Quota、Sensitive Data、Alert 的最小治理闭环；`agentMode=none` 直连模型路径具备 Trace root 和 Quota，但仍需补齐敏感数据 `messages` 扫描、model span 和 Token Usage 记录。
