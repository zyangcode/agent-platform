# 阶段 1 开发检查清单

> 阶段 1 目标是跑通“基础 Agent 闭环”。本清单用于防止实现时忘记关键边界和验证点。每完成一个小阶段，都应按对应章节回查。

---

## 1. 建工程前

- [ ] 后端只建立 `agent-platform-common`、`agent-platform-core`、`agent-platform-web`、`agent-platform-gateway` 四个 Maven 模块。
- [ ] `agent-platform-frontend` 是前端工程，不作为 Maven 模块。
- [ ] `feishu-bot` 是独立 Spring Boot 服务，不依赖 agent-platform 内部代码。
- [ ] 包名前缀统一：平台主工程使用 `com.ls.agent`，飞书机器人使用 `com.ls.feishu`。
- [ ] Web AI 请求路径按 `Web -> Gateway -> Core` 设计，不能为了阶段 1 省事改成 `Web -> Core`。

---

## 2. 生成 Entity / Mapper 后

- [ ] Entity / Mapper 按表归属放入对应 core 包。
- [ ] `common` 中没有任何业务 Entity / DTO / Mapper / Service。
- [ ] `web` 和 `gateway` 中没有业务 Mapper。
- [ ] 加入 ArchUnit 或等价架构测试。
- [ ] ArchUnit 禁止 `web/gateway` 访问任何 `core.*.mapper`。
- [ ] ArchUnit 禁止 core 内部跨业务包访问其他包的 `entity/mapper/internal`。
- [ ] core 内部跨包协作只访问 `api/dto/command`。

---

## 3. Gateway 空壳完成后

- [ ] 浏览器 AI 对话请求走 `frontend -> web -> gateway -> core.agent`。
- [ ] API Key `agent_mode=agent` 请求走 `gateway -> core.agent.AgentRuntimeService`。
- [ ] API Key `agent_mode=none` 请求走 `gateway -> core.model.ModelInvokeService`。
- [ ] Web 调 Gateway 使用 HTTP 内部接口，不共享 Gateway Java DTO 或 Controller。
- [ ] Gateway 校验 Web 内部服务 Token 或签名 Header。
- [ ] `GatewayContext` 请求结束后会清理。
- [ ] Gateway 将 `GatewayContext` 显式转换成 `AgentRunCommand` 或 `ModelInvokeCommand`。
- [ ] Core 不读取 Gateway ThreadLocal。

---

## 4. SSE 透传验证

- [ ] Gateway 提供固定 SSE 测试接口，每 500ms 输出一个事件。
- [ ] Web 提供固定 SSE 透传测试接口，例如 `/api/chat/stream-test`，内部调用 Gateway 测试流。
- [ ] Web 能逐事件透传 Gateway SSE 到浏览器或 curl。
- [ ] 事件不是被缓冲后一次性返回。
- [ ] Gateway 中途出错时，Web 能转成浏览器可理解的 SSE `error` 事件。
- [ ] 浏览器断开后，Web/Gateway/Core 能释放请求资源。

---

## 5. Agent 主链路完成后

- [ ] `AgentRuntimeService` 不直接查 `skills`、`model_configs`、`quota`、`trace` 等表。
- [ ] Skill 查询和执行只通过 `core.skill.api.SkillRegistry` / `SkillExecutor`。
- [ ] MCP 查询和执行只通过 `core.mcp.api.McpToolRegistry` / `McpToolExecutor`。
- [ ] 模型调用只通过 `core.model.api.ModelInvokeService`。
- [ ] 上下文组装放在 `core.context.AgentContextBuilder`。
- [ ] 上下文 token 预算、压缩、裁剪逻辑不放在 Gateway。
- [ ] 会话消息由 `core.agent.api.MessageHistoryService` 使用，写入由 `core.agent.api.ConversationRepository` 负责。
- [ ] 长期记忆由 `core.memory` 保存。

---

## 6. 阶段 1 quota / trace / security / alert

- [ ] `core.quota` 不做空实现，至少在无配置时返回“允许通过”的默认策略。
- [ ] 阶段 1 不强制建 `quota_configs`、`quota_reservations`、`token_usage_logs` 表。
- [ ] 如果提前建 `quota_configs` 表，创建 Application 时可生成默认无限额配置。
- [ ] `core.trace` 至少能生成 TraceID 或简单日志，不要求阶段 1 完整落库。
- [ ] `core.security` 可先做简单正则脱敏，不要求完整策略表。
- [ ] `core.alert` 可先做接口占位，不同步调用 feishu-bot 阻塞主请求。

---

## 7. 阶段 1 完成前 Demo 验收

- [ ] Postman 或 curl 能通过 `Web -> Gateway -> Core` 路径跑通 Agent 对话。
- [ ] 至少 1 个内置 Skill 可调用。
- [ ] 至少 1 个 MCP Client 示例工具可调用。
- [ ] SSE 能流式返回，而不是一次性返回。
- [ ] 历史会话消息可查询。
- [ ] 长期记忆有持久化记录。
- [ ] Gateway 空壳路径保留，阶段 2 可直接补治理链，不需要重构入口。

---

## 8. 禁止事项

- [ ] 禁止 `Controller -> Mapper`。
- [ ] 禁止 `web -> core.agent` 直接处理 AI 执行请求。
- [ ] 禁止 `gateway -> core.*.mapper`。
- [ ] 禁止 `core.agent -> core.skill.mapper`。
- [ ] 禁止把业务 DTO / Entity 放入 `common`。
- [ ] 禁止让飞书告警失败影响用户 AI 响应。
- [ ] 禁止为赶进度绕过 Gateway，后续再补。

---

## 9. 进入阶段 2 前确认

- [ ] 阶段 1 主链路没有绕过 Gateway。
- [ ] ArchUnit 包边界测试已经纳入测试命令。
- [ ] SSE 双跳透传已验证。
- [ ] AgentContext 和 GatewayContext 边界清楚。
- [ ] 阶段 2 需要补的治理点已经留好接口：Trace、Token、脱敏、告警。
- [ ] 没有大量临时 `if (xxxService != null)` 防御式代码。
